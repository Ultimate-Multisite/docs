# บทที่ 10: วันเปิดตัว

ทุกอย่างสร้างเสร็จแล้ว ก่อนที่คุณจะเปิดประตูต้อนรับลูกค้า บทเรียนนี้จะพาคุณไปดูรายการตรวจสอบ (checklist) ก่อนเปิดตัว เพื่อให้แน่ใจว่าไม่มีอะไรพัง ไม่มีอะไรขาดหาย และไม่มีอะไรน่าอาย

## จุดที่เราทำมาถึง {#where-we-left-off}

FitSite มีการตั้งค่า Template, แผนบริการ (plans), ระบบ Checkout, Branding, Onboarding, และ Pricing เรียบร้อยแล้ว ตอนนี้เราจะมาตรวจสอบว่าทุกอย่างทำงานได้จริง และพร้อมที่จะเปิดตัวแล้ว

## รายการตรวจสอบก่อนเปิดตัว (Pre-Launch Checklist) {#pre-launch-checklist}

ให้ทำตามทุกรายการ ห้ามข้ามรายการใด ๆ เด็ดขาด

### โครงสร้างพื้นฐานของ Platform (Platform Infrastructure) {#platform-infrastructure}

- [ ] Hosting มีความเสถียรและทำงานได้ดีภายใต้ภาระงาน (load)
- [ ] SSL แบบ Wildcard ต้องใช้งานได้ และ subdomain ทั้งหมดต้องใช้ HTTPS
- [ ] การแมปโดเมน (Domain mapping) ต้องใช้งานได้ — ทดสอบการสร้างเว็บไซต์และการแมปโดเมนแบบกำหนดเอง
- [ ] ระบบสำรองข้อมูล (Backups) ต้องตั้งค่าและทดสอบแล้ว (ควรทดสอบการกู้คืนอย่างน้อยหนึ่งครั้ง)
- [ ] ระบบ Monitoring ต้องพร้อมใช้งาน — คุณต้องรู้ว่าเมื่อไหร่ที่ Platform ล่ม

### Templates {#templates}

- [ ] Template ทั้งสามแบบต้องโหลดได้อย่างถูกต้องบนเว็บไซต์ใหม่
- [ ] เนื้อหาตัวอย่าง (Placeholder content) ต้องมีประโยชน์และไม่มีคำผิด
- [ ] รูปภาพทั้งหมดต้องมีลิขสิทธิ์ที่ถูกต้อง (ห้ามมีลายน้ำของภาพสต็อก)
- [ ] การแสดงผลบนมือถือ (Mobile responsiveness) ต้องใช้งานได้ในทุกหน้า Template
- [ ] ความเร็วในการโหลดหน้า (Page load speed) ต้องอยู่ในเกณฑ์ที่ยอมรับได้ (ทดสอบด้วยเครื่องมือเช่น GTmetrix หรือ PageSpeed Insights)
- [ ] ต้องไม่มีลิงก์เสียหรือ Asset ที่ขาดหายไปใน Template ใด ๆ

### แผนบริการและผลิตภัณฑ์ (Plans and Products) {#plans-and-products}

- [ ] แผนบริการทั้งสามแบบต้องเปิดใช้งานและมองเห็นได้
- [ ] คำอธิบายแผนบริการต้องถูกต้องและเจาะจงกลุ่มเป้าหมาย
- [ ] ราคาต้องถูกต้อง (ทั้งรายเดือนและรายปี)
- [ ] ค่าติดตั้ง (Setup fees) ต้องตั้งค่าไว้ในแผนบริการที่ถูกต้อง
- [ ] ช่วงทดลองใช้ (Trial period) ต้องทำงานได้ในแผน Starter
- [ ] Order bumps ต้องแสดงผลอย่างถูกต้องระหว่างการ Checkout
- [ ] ข้อจำกัดของ Plugin และ Theme ต้องถูกบังคับใช้ตามแผนบริการอย่างถูกต้อง

### กระบวนการ Checkout Flow {#checkout-flow}

- [ ] ทำการทดสอบการสมัครสมาชิกแบบเต็มรูปแบบในทุกแผน (ใช้โหมดชำระเงินทดสอบ)
- [ ] การเลือก Template ต้องแสดง Template ที่ถูกต้องตามแผนบริการ
- [ ] การชำระเงินต้องดำเนินการสำเร็จ
- [ ] ลูกค้าต้องได้รับอีเมลต้อนรับหลังการสมัคร
- [ ] เว็บไซต์ใหม่ต้องถูกสร้างขึ้นด้วย Template ที่ถูกต้อง
- [ ] ลูกค้าต้องสามารถเข้าสู่ระบบเว็บไซต์ใหม่ได้ทันที
- [ ] รหัสส่วนลด (Discount codes) ต้องทำงานได้อย่างถูกต้อง

### Branding {#branding}

- [ ] หน้า Login ต้องแสดง Branding ของ FitSite
- [ ] Dashboard ของ Admin ต้องแสดง Branding ของ FitSite
- [ ] อีเมลระบบทั้งหมดต้องใช้ Branding ของ FitSite และภาษาที่เกี่ยวข้องกับฟิตเนส
- [ ] ใบแจ้งหนี้ (Invoices) ต้องแสดงรายละเอียดธุรกิจของคุณอย่างถูกต้อง
- [ ] เว็บไซต์ Marketing ต้องเปิดใช้งานและลิงก์ไปยังแบบฟอร์ม Checkout

### Onboarding {#onboarding}

- [ ] Widget Quick Start ต้องปรากฏบน Dashboard ของลูกค้าใหม่
- [ ] ลิงก์ Quick Start ทั้งหมดต้องชี้ไปยังหน้าที่ถูกต้อง
- [ ] ลำดับอีเมลต้อนรับ (Welcome email sequence) ต้องตั้งค่าและทดสอบแล้ว
- [ ] บทความใน Knowledge base ต้องเผยแพร่และเข้าถึงได้
- [ ] หน้าบัญชี (Account page) ต้องแสดงข้อมูลแผนบริการที่ถูกต้องและตัวเลือกการอัปเกรด

### กฎหมายและธุรกิจ (Legal and Business) {#legal-and-business}

- [ ] ข้อกำหนดในการให้บริการ (Terms of service) ต้องเผยแพร่และลิงก์จากหน้า Checkout
- [ ] นโยบายความเป็นส่วนตัว (Privacy policy) ต้องเผยแพร่และเข้าถึงได้
- [ ] นโยบายการคืนเงิน (Refund policy) ต้องกำหนดและจัดทำเอกสารไว้
- [ ] ต้องตั้งค่าหน่วยงานธุรกิจเพื่อรับการชำระเงิน
- [ ] Payment gateway ต้องอยู่ในโหมดใช้งานจริง (live mode) ไม่ใช่โหมดทดสอบ (test mode)
- [ ] การตั้งค่าภาษี (Tax configuration) ต้องถูกต้องสำหรับเขตอำนาจศาลของคุณ

## Soft Launch vs. Hard Launch {#soft-launch-vs-hard-launch}

ควรพิจารณาการเปิดตัวแบบสองระยะ:

### ระยะที่ 1: Soft Launch {#phase-1-soft-launch}

เชิญเจ้าของสตูดิโอฟิตเนส 5-10 รายให้มาสมัครก่อนการเปิดตัวสู่สาธารณะ นี่คือลูกค้า Beta ของคุณ เสนอส่วนลดที่สำคัญ (เช่น ลด 50% ตลอดชีพ หรือฟรี 3 เดือน) แลกกับการที่พวกเขา:

- ให้ข้อเสนอแนะที่ตรงไปตรงมาเกี่ยวกับประสบการณ์การสมัครและการ Onboarding
- อนุญาตให้ใช้เว็บไซต์ของพวกเขาเป็นตัวอย่างโชว์เคส
- รายงานข้อบกพร่องหรือปัญหาที่พวกเขาพบ

สิ่งนี้จะทำให้คุณได้รับข้อเสนอแนะจากลูกค้าจริงและเว็บไซต์ที่ใช้งานได้จริงเพื่อนำมาโชว์ก่อนเปิดตัวสู่สาธารณะ

### ระยะที่ 2: Public Launch {#phase-2-public-launch}

เมื่อมีการนำข้อเสนอแนะจาก Soft Launch มาปรับใช้แล้ว:

- เปลี่ยน Payment gateway เป็นโหมดใช้งานจริง (live mode)
- เผยแพร่เว็บไซต์ Marketing ของคุณ
- เริ่มการหาลูกค้า (Customer acquisition) (บทเรียนที่ 11)
- ประกาศในช่องทางอุตสาหกรรมฟิตเนสที่เกี่ยวข้อง

## การดำเนินการในวันเปิดตัว (Launch Day Actions) {#launch-day-actions}

ในวันที่คุณเปิดตัวสู่สาธารณะ:

1. **สลับไปใช้การชำระเงินจริง** — ปิดโหมดทดสอบบน Payment gateway ของคุณ
2. **ตรวจสอบอีกครั้ง** — ทำการทดสอบการสมัครสมาชิกแบบเต็มรูปแบบด้วยการชำระเงินจริง (คืนเงินให้ตัวเองภายหลัง)
3. **เฝ้าติดตามอย่างใกล้ชิด** — คอยดูข้อผิดพลาด การสมัครที่ไม่สำเร็จ หรือปัญหาการชำระเงิน
4. **พร้อมให้ความช่วยเหลือ** — ลูกค้าจริงกลุ่มแรกของคุณอาจต้องการความช่วยเหลือ และการตอบสนองที่รวดเร็วจะสร้างความไว้วางใจ
5. **ฉลองสั้น ๆ** — จากนั้นกลับไปทำงานต่อ

## อะไรที่อาจผิดพลาดได้ (What Can Go Wrong) {#what-can-go-wrong}

เตรียมพร้อมสำหรับ:

- **ปัญหา Payment gateway**: เตรียมช่องทางการติดต่อฝ่ายสนับสนุนของผู้ให้บริการ gateway ของคุณ
- **ปัญหา SSL certificate**: รู้จักวิธีการตรวจสอบและต่ออายุ Certificate
- **ความล้มเหลวในการส่งอีเมล**: ทดสอบว่าอีเมลมาถึงจริงหรือไม่ (ตรวจสอบโฟลเดอร์สแปม)
- **ประสิทธิภาพภายใต้ภาระงาน**: หากมีการเข้าชมจำนวนมากอย่างกะทันหัน คุณต้องรู้ว่าจะขยาย Hosting ได้อย่างไร
- **ความสับสนของลูกค้า**: เตรียม Knowledge base และช่องทางสนับสนุนของคุณให้พร้อม

## สิ่งที่เราสร้างในบทเรียนนี้ {#what-we-built-this-lesson}

- **รายการตรวจสอบก่อนเปิดตัวที่ครอบคลุม** ซึ่งครอบคลุมโครงสร้างพื้นฐาน เนื้อหา การชำระเงิน และข้อกฎหมาย
- **กลยุทธ์ Soft Launch** เพื่อรับข้อเสนอแนะจริงก่อนเปิดตัวสู่สาธารณะ
- **ขั้นตอนการเปิดตัว** เพื่อให้คุณเปิดตัวได้อย่างมั่นใจ
- **แผนฉุกเฉิน** สำหรับปัญหาทั่วไปที่อาจเกิดขึ้นในวันเปิดตัว
