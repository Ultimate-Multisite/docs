# Ultimate Multisite 101 {#ultimate-multisite-101}

Ultimate Multisite هڪ ووردبريس (WordPress) هم پلاتين (plugin) آهي جيڪو توهان کي ڪلائين کي WaaS يا વેબસيتس ايز ا سيرفس (Websites as a Service) ڏيڻ جي اجازت ڏئي ٿو. ان ۾ گهريان ۽ سکون ڪرڻ کان اڳ، اسان کي ڪجهه بنيادي علم حاصل ڪرڻو پوندو آهي ته Ultimate Multisite توهان جي ڪاروبار ۽ ڪلائين کي ڪيئن مدد ڪري سگهي ٿو.

## ووردبريس هم پلاتين (The WordPress Multisite) {#the-wordpress-multisite}

هڪ سڀني ماڻهن کي عام طور تي اسان جو ووردبريس انٽالاسن سان واقف آهي. توهان يا ته پنهنجي هاوسنگ پروائيڊر جي ڪنٽرول پینل ذريعي اهو ٺاهين ٿا، يا جيڪڏهن توهان جرات مند آهيو، هڪ نئين ويب سرور ۽ ڈیٹا بيسٽ محفوظ ڪريو، کور فائلون ڈاؤنلوڊ ڪريو ۽ انٽالاسن جو عمل شروع ڪري سگهو ٿا.

هي دنيا ۾ ملڪ جي ترليڪن تي لکين جي ووردبريس سائٽن لاءِ ڪم ڪندو آهي پر هڪ ايجنسي يا هاوسنگ پروائيڊر جي نقطي ازل سان، اسان هڪ منٽ ان جي حجم (volumes) بابت بحث ڪريون.

جڏهن ته هڪ ووردبريس سائيٽ يا هيٺ هڪ سو سائٽن کي خودکار ڪنٽرول پینل ذريعي ٺاهڻ جو يقيني آهي، پر جڏهن اهو ان سائٽن جي انتظام (management) ۾ اچي ٿو ته جلدي ئي مسئلا ظاهر ٿيندا آهن. جيڪڏهن ان کي منظم نه ڪيو وڃي ته توهان مالوير (malware) لاءِ هڪ بنيادي نشان بڻجي ويندو. انتظام ڪرڻ کا مطلب ڪوشش ۽ وسيلن جو استعمال آهي، ۽ جيتوڻي ووردبريس سائٽن جي انتظام ۽ عمل کي آسان بڻائڻ ۾ مددگار ٻاهر ئي ٽولز ۽ پليجنز موجود آهن، پر ان واقعيت جو مطلب آهي ته جيڪڏهن ڪلائين کي اڪائيداري رسائي (administrative access) رکڻ گهرجي، ته اهڙا ڪوشون آساني سان شکست ملي سگهن ٿيون.

پنهنجي بنيادي حصي ۾، ووردبريس هڪ خاص هيٺ جو ڪم ڪري سگهي ٿو: هڪ اڪاديمي ادارو ووردبريس جو هڪ واحد انٽالاسن برقرار رکي ٿو پر هر فیکلٽي پنهنجي خودي ووردبريس سائٽ برقرار رکي ٿي.

اس جملے کو سمجھنے کے لیے آئیے کچھ بنیادی اصطلاحات دیکھتے ہیں جو نہ صرف Ultimate Multisite کی دستاویزات میں بلکہ WordPress کمیونٹی بھر میں موجود ہیں۔

### نیٹ ورک (The Network) {#the-network}

WordPress کے لحاظ سے، ایک ملٹی سائٹ نیٹ ورک وہ جگہ ہے جہاں آپ ایک ہی ڈیش بورڈ سے کئی سب سائٹس کو مینج کر سکتے ہیں۔ اگرچہ ہوسٹنگ فراہم کرنے والوں کے درمیان ملٹی سائٹ نیٹ ورک بنانے کا طریقہ مختلف ہوتا ہے، لیکن نتیجہ عام طور پر wp-config.php فائل میں کچھ اضافی ہدایات ہوتی ہیں تاکہ WordPress کو پتہ چل جائے کہ وہ اس خاص موڈ میں کام کر رہا ہے۔

ملٹی سائٹ نیٹ ورک اور ایک اسٹینڈ الون (stand-alone) WordPress انسٹالیشن کے درمیان کئی واضح فرق ہیں جن پر ہم مختصر بات کریں گے۔

#### سب ڈومین بمقابلہ سب ڈائریکٹری (Subdomain vs. Subdirectory) {#subdomain-vs-subdirectory}

سب سے اہم فیصلہ جو آپ کو کرنا ہوگا یہ ہے کہ کیا ملٹی سائٹ انسٹالیشن _سب ڈائریکٹریز_ (subdirectories) کے ساتھ چلے گی یا _سب ڈومینز_ (subdomains) کے ساتھ۔ Ultimate Multisite دونوں انتخاب کے ساتھ برابر کام کرتی ہے لیکن ان دونوں کنفیگریشنز کے درمیان کچھ آرکیٹیکچرل فرق ہیں۔

_سب ڈائریکٹری_ کنفیگریشن میں، نیٹ ورک سائٹس مین ڈومین نیم کی بنیاد پر ایک پاتھ (path) وراثت میں لیتی ہیں۔ مثال کے طور پر 'site1' نامی نیٹ ورک سائٹ کا مکمل URL https://domain.com/site1 ہوگا۔ _سب ڈومین_ کنفیگریشن میں، نیٹ ورک سائٹ کا اپنا _سب ڈومین_ مین ڈومین نیم سے اخذ کیا جائے گا۔ اس طرح 'site1' نامی سائٹ کا مکمل URL https://site1.domain.com ہوگا۔

اگرچہ دونوں آپشنز بالکل درست انتخاب ہیں، لیکن _سب ڈومینز_ کا استعمال کئی فوائد بھی دیتا ہے لیکن اس کی آرکیٹیکچر میں زیادہ سوچ و بچار اور منصوبہ بندی کی ضرورت ہوتی ہے۔

DNS جي لحاظ کان، _subdirectories_ جو استعمال ڪيو ٿو هڪ نسبتا سڌي چيلينج آهي. جهڙوڪ نٽوي سائيٽس پنهنجي ماءُ (parent path) جا ٻار آهن، ان لاءِ صرف مين ڈومین نام لاءِ هڪ ئي DNS entry موجود هجڻ گهرجي. _subdomains_ جي معاملن ۾ چيلينج ٿورو وڌيڪ جمان ٿا، جنهن لاءِ يا ته هر نٽوي سائيٽ لاءِ الگ CNAME entry گهرجي يا DNS records ۾ wildcard (*) entry گهرجي.

SSL ۽ SSL certificates جي ڇڏڻ ۽ استعمال جو هڪ ٻيو اهم حصو آهي. _subdirectory_ configuration ۾، هڪ ئي domain certificate جو استعمال ڪيو جا سگهجي ٿو جڏهن ته نٽوي سائيٽس مين domain name جا صرف پاتھ آهن. تنهن ڪري domain.com لاءِ جيڪا certificate هجي، اهو https://domain.com/site1، https://domain.com/site2 ۽ ان طرح جيئن آهي ان لاءِ SSL فراهم ڪندي کافی هوندي.

_subdomain_ configuration ۾، wildcard SSL certificate جو استعمال سڀ کان عام خيار آهي. هي SSL certificate هڪ domain ۽ سندس _subdomains_ کي encryption فراهم ڪري ٿو. تنهن ڪري، هڪ wildcard SSL certificate https://site1.domain.com، https://site2.domain.com ۽ HTTPS://domain.com لاءِ به encryption فراهم ڪندي.

ٻين خيار موجود آهن، پر اه عام طور تي محدود حد ۾ هوندا آهن ۽ انهن کي مناسبت جي لحاظ کان اضافي configuration ۽ غور و عمل گهرجي ٿو.

#### Plugins and Themes {#plugins-and-themes}

WordPress جو فائدو آهي ته اهو نقصان پڻ پيش ڪري ٿو، اقل ها ته ڪارئيندڙ (customer) جي نقطي نظر کان. جيڪڏهن هڪ stand-alone WordPress installation ۾ سائيٽ کاٻي (site administrator) خراب plugin نصب ڪري ڇڏي وڃي يا پنهنجي installation کي اپ ٹو ڈیٹ نه رکي، تڏهن ان عمل جو واحد متاثر ۽ نقصان خود ذات آهي. پر جڏهن ڪا site administrator multisite installation ۾ خراب plugin نصب ڪري ٿو، ته اهو نٽوي (network) جي هر سائيٽ جو متاثر بڻجي وڃي.

اهميت اس وجہ توں جب اسے ملٹی سائٹ ورڈپریس کے طور پر کنفیگر کیا جاتا ہے تو یہ سائٹ ایڈمنز کو پلگ انز اور تھیمز انسٹال کرنے کی صلاحیت ختم کر دیتا ہے اور اس صلاحیت کو ایک نئے بنائے گئے نیٹ ورک ایڈمنسٹریٹر یا 'سپر ایڈمن' رول میں منتقل کر دیتا ہے۔ پھر یہ خصوصی رول فیصلہ کر سکتا ہے کہ نیٹ ورک سائٹس کے ایڈمنز کو اپنے ڈیش بورڈ میں پلگ انز مینو دیکھنے یا تک رسائی دینے کی اجازت دی جائے یا نہیں، اور اگر ایسا ہو تو کیا ایسی اجازتیں پلگ انز کو _فعال کرنے_ یا _غیر فعال کرنے_ تک جاتی ہیں۔

اس حد تک نیٹ ورک ایڈمنسٹریٹر نیٹ ورک میں پلگ انز اور تھیمز انسٹال کرنے کا ذمہ دار ہوتا ہے اور ان پلگ انز اور تھیمز کو نیٹ ورک سائٹس کے لیے استعمال کرنے کی اجازت دینے کے لیے اجازتیں تفویض کرتا ہے۔ سائٹ ایڈمنز اپنی سائٹ سے منسلک نہ ہونے والے پلگ انز اور تھیمز انسٹال نہیں کر سکتے یا ان تک رسائی حاصل نہیں کر سکتے۔

#### صارفین اور ایڈمنز {#users-and-administrators}

ورڈپریس ملٹی سائٹ میں، تمام نیٹ ورک سائٹس ایک ہی ڈیٹا بیس شیئر کرتی ہیں اور اس لیے ایک ہی صارفین، رولز اور کیپابیلٹ شیئر کرتی ہیں۔ اسے سمجھنے کا سب سے اچھا طریقہ یہ ہے کہ تمام صارفین نیٹ ورک کے ممبر ہوتے ہیں نہ کہ کسی خاص سائٹ کے۔

اس سمجھ کو دیکھتے ہوئے، صارفین کو بنانے کی اجازت دینا اور اسی وجہ سے ورڈپریس ملٹی سائٹ اس صلاحیت کو سائٹ ایڈمنز سے ہٹا دیتا ہے اور اس صلاحیت کو نیٹ ورک ایڈمنسٹریٹر کے پاس منتقل کر دیتا ہے۔ اس طرح نیٹ ورک ایڈمنسٹریٹر ایک سائٹ ایڈمنسٹریٹر کو اپنی اپنی سائٹ کے لیے صارف اکاؤنٹس بنانے کی اجازت دینے کے لیے ضروری مراعات تفویض کر سکتا ہے۔

اوپر دی گئی بات کو دہراتے ہوئے، اگرچہ صارف اکاؤنٹس ان سائٹ سے متعلق نظر آتے ہیں حقیقت میں وہ نیٹ ورک کو مختص کیے جاتے ہیں اور اس لیے پورے نیٹ ورک میں منفرد ہونے چاہئیں۔ ایسے واقعات ہو سکتے ہیں جہاں اس وجہ سے یوزر نیم رجسٹر کرنے کے لیے دستیاب نہ ہوں۔

گر پر این شي کہ یہ انٹرپرائز سسٹم میں کوئی نئی بات نہیں ہے، لیکن صارف رجسٹریشن اور تصدیق کا یہ ایک ہی ذریعہ اکثر ان لوگوں کے لیے سمجھنا مشکل ہوتا ہے جو ایسے اسٹینڈ الون ورڈپریس انسٹالیشنز سے واقف ہیں جہاں یوزر ایڈمن بہت آسان ہوتی ہے۔

#### میڈیا (Media) {#media}

جہاں نیٹ ورک سائٹس وورڈپریس ملٹی سائٹ میں ایک ہی ڈیٹا بیس شیئر کرتی ہیں، وہ فائل سسٹم پر میڈیا فائلوں کے لیے الگ راستے رکھتی ہیں۔

سٹینڈرڈ وورڈپریس کی جگہ (wp-content/uploads) برقرار رہتی ہے؛ تاہم، اس کا راستہ نیٹ ورک سائٹ کی منفرد آئی ڈی کو ظاہر کرنے کے لیے تبدیل کر دیا جاتا ہے۔ نتیجے میں، نیٹ ورک سائٹ کے میڈیا فائلز wp-contents/uploads/site/[id] کے طور پر نظر آتی ہیں۔

#### پرمالینکس (Permalinks) {#permalinks}

ہم نے پہلے ذکر کیا تھا کہ _سبڈومین_ کنفیگریشن (_subdomain_) کی نسبت _سبسائڈ ڈائریکٹری_ (_subdirectory_) کی کچھ خاص فوائد ہیں اور یہ رہے: راستے (paths)۔

ایک _سبسائڈ ڈائریکٹری_ کنفیگریشن میں، مین سائٹ (وہ پہلی سائٹ جو نیٹ ورک قائم ہونے پر بنائی جاتی ہے) اور نیٹ ورک سب سائٹس کو ڈومین نیم سے آنے والے ایک ہی راستے کا شیئر کرنا پڑتا ہے۔ اس میں بہت سی ٹکراؤ (conflicts) پیدا ہونے کی صلاحیت ہوتی ہے۔

پوسٹس کے لیے، نیٹ ورک سائٹس کے ساتھ تصادم سے بچنے کے لیے مین سائٹ پر ایک لازمی /blog/ راستہ شامل کیا جاتا ہے۔ اس کا مطلب ہے کہ 'پوسٹ کا نام' جیسے خوبصورت پرمالینکس کو domain.name/blog/post-name/ کے طور پر پیش کیا جائے گا۔

ایک _سبڈومین_ کنفیگریشن میں یہ عمل ضروری نہیں ہے کیونکہ ہر نیٹ ورک سائٹ مکمل ڈومین علیحدگی سے فائدہ اٹھاتی ہے اور اس لیے انہیں ایک ہی راستے پر انحصار کرنے کی ضرورت نہیں ہوتی۔ وہ اپنی _سبڈومین_ کی بنیاد پر اپنے الگ راستے برقرار رکھتے ہیں۔

#### سٹیٹک پیجز (Static Pages) {#static-pages}

_subdirectory_ configuration ۾، سٽيڪ (static) صفحات جي نالا چونڊڻ جي مسئلي واري صورتحال ٿيندي آهي ڇاڪڏهن مٿي جو سيت (main site) ۽ ڪم ڪندڙ سيتون هڪ ئي پاتھ (path) استعمال ڪن.

ان کي روڪڻ لاءِ، WordPress وٽ هڪ طريقو آهي جنهن سان توهان ڪجهه سيت جي نالا بلڪي ليست (blacklist) ڪري سگهو ٿا ته جيئن اهي پهرين سيت جي نالا کان متصادم نه hon. عام طور تي، ڪم ڪندڙ اديثار (network administrator) مٿي جو سيت جي صفحات جا روٽ پاتھ (root paths) داخل ڪري ڇڏيندا آهن.

<!-- Screenshot unavailable: WordPress network settings showing banned site names to prevent conflicts -->

_subdomain_ configuration ۾، نالا چونڊڻ جي مسئلي کي _subdomain_ سان ختم ڪيو ويندو آهي ڇاڪله اهو سيت لاءِ مخصوص (unique) آهي ۽ مٿي جو سيت کان ڪنه به طرح تعلق نه رکي ٿو.

### Registration {#registration}

WordPress Multisite جا ڪم ڪندڙ ترتي ۾ ڪجهه نون صارف جي رجيشن (user registration) جا خيار موجود آهن، جيڪي نئين ۽ پراني ڪارن کي سيتون کڻڻ جو موقعو ڏين ٿا.

<!-- Screenshot unavailable: WordPress Multisite network settings showing registration options -->

انڪاري تي، اڪيلو WordPress installation جي برعهم عموما صارف جي رجيشن يا ان رجيشن کي رول (roles) سان جوڙڻ جا عام خيار برقرار رکي ٿو.

<!-- Screenshot unavailable: WordPress standalone site user registration settings showing limited options -->

جڏهن ڪارن جا حساب کڻيا ويندا آهن، تڏهن اهي حساب ترتي سطح تي جنريٽ (generate) ٿين ٿا. ان ڪري، هي ڪنهن خاص سيت جو ناهي، پر ڪم ڪندڙ سيت جي بنياد ۾ تعلق رکندا آهن. ان ۾ ڪجهه خاص مزو ۽ نقصان پيدا ٿين ٿا.

مثال کے طور پر، فرض کریں کہ آپ کا ورڈپریس ملٹی سائٹ نیوز اور معلومات کے کاروبار میں ہے۔ آپ ملٹی سائٹ قائم کریں گے اور پھر فنانس، ٹیکنالوجی، تفریح اور دلچسپی کے دیگر شعبوں کے لیے نیٹ ورک سائٹس بنائیں گے جبکہ پلگ انز اور تھیمز پر مجموعی کنٹرول برقرار رکھیں گے۔ ہر نیٹ ورک سائٹ کا اپنے نیٹ ورک سائٹ کی شکل و صورت اور صارف کے تجربے پر زیادہ کنٹرول ہوگا جو کہ کسٹم پوسٹ ٹائپس یا عام پوسٹ کی کیٹیگریز سے کہیں زیادہ ہوگا۔

اس حد تک، جب کوئی صارف لاگ ان کرتا ہے تو وہ نیٹ ورک میں لاگ ان ہوتا ہے اور بالآخر ایک ہموار تجربہ فراہم کرنے کے لیے ہر نیٹ ورک سائٹ پر بھی لاگ ان ہوتا ہے۔ اگر آپ کی نئی سائٹ سبسکرپشن پر مبنی تھی تو یہ بہترین حل اور نتیجہ ہوگا۔

اگر تاہم، ملٹی سائٹ کا مقصد مختلف نیٹ ورک سائٹس پیش کرنا ہو جن میں ایک دوسرے سے کوئی تعلق نہ ہو، تو تقریباً ہمیشہ ایسا ہی ہوتا ہے کہ صارف کے کرداروں (user roles) کو تبدیل کرنے کے لیے بیرونی یا اضافی پلگ ان کی ضرورت پڑتی ہے۔

### ڈومین اور SSL {#domain-and-ssl}

آئیے ورڈپریس ملٹی سائٹ انسٹالیشن کے بارے میں بات کرتے ہیں جو ہماری توجہ سے تقریباً چھپ جاتی ہے - WordPress.com۔ یہ ورڈپریس ملٹی سائٹ کا سب سے وسیع مثال ہے اور اس کی صلاحیتوں کو کسی مقصد کو پورا کرنے کے لیے کسٹمائز اور ڈھالنے کی صلاحیت کو ظاہر کرتا ہے۔

آج کل جدید انٹرنیٹ پر SSL کا استعمال تقریباً لازمی ہو گیا ہے اور ورڈپریس ملٹی سائٹ کے نیٹ ورک ایڈمنز کو جلد ہی ان چیلنجز کا سامنا کرنا پڑتا ہے۔

_subdomain_ کنفیگریشن میں سائٹس رُوٹ ڈومین نیم کی بنیاد پر بنائی جاتی ہیں۔ اس لیے 'site1' نامی سائٹ 'site1.domain.com' کے طور پر بنائی جائے گی۔ وائلڈ کارڈ SSL سرٹیفکیٹ کا استعمال کرتے ہوئے، ایک نیٹ ورک ایڈمن اس چیلنج کا کامیابی سے سامنا کر سکتا ہے اور نیٹ ورک کے لیے SSL انکرپشن کی صلاحیت فراہم کر سکتا ہے۔

WordPress Multisite ۾ هڪ دومين ميپنگ فنڪشن (domain mapping function) آهي جيڪو نٽ ورڪ سائٽس کي ڪسٽم دومين نامن يا نٽ ورڪ جي ريٽ دومين کان مختلف دومين نامن سان جوڙڻ جي اجازت ڏئي ٿو.

نٽ ورڪ اڊمنٽرن لاءِ، هي دومين نامن جي ترتیب (domain name configuration) ۽ SSL سرٽيفڪيٽس جي ڇڏڻ ۽ برقرار رکڻ ۾ ٻيو اضافي چولنج (complexity) پيش ڪري ٿو.

هي حد تائين جڏهن ته WordPress Multisite [www.anotherdomain.com](http://www.anotherdomain.com) کي 'site1' سان جوڙڻ جي وسيل فراهم ڪري ٿو، ان وقت نٽ ورڪ اڊمنٽر کي DNS entries کي bahri طور تي منظم ڪرڻ ۽ SSL سرٽيفڪيٽس جو عملدرآمد ڪرڻ جي چيلينج رڳو بچائي ڇڏي ٿو.

## Ultimate Multisite {#ultimate-multisite}

هڪ آزاد WordPress installation ۽ Multisite installation بجيڪن وچ ۾ فرق سمجهڻ کان پوءِ، اچون ٿا ته ڏسو ته Ultimate Multisite ڪيئن Websites as a Service (WaaS) فراهم ڪرڻ لاءِ حقيقي آرماني آهي.

### تعارف {#introduction}

Ultimate Multisite توهان جو سويسيئر آرماني آهي جڏهن توهان Website as a Service (WaaS) ٺاهڻ جي باري ۾ سوچيو ٿا. Wix.com، Squarespace، WordPress.com ۽ پوءِ پنهنجي خود سروس کي ملڻ جو سوچيو ته.

ان اندر Ultimate Multisite WordPress Multisite جو استعمال ڪري ٿو پر اهو ان طرح ڪري ٿو ته جنهن طريقن سان نٽ ورڪ اڊمنٽرن کي multisite installation سان لاڳاپيل ڪيترن ئي چيلينجن کان بچائڻ گهرجي، ۽ صلاحيتون کي بهتر بڻائي ٿو جيڪا وڏي قسم جا استعمال (use cases) سپورٽ ڪرڻ لاءِ اجازت ڏئي ٿي.

انٺين حصن ۾ اسين ڪجهه عام استعمال جي صورتحالن ۽ ان کي سپورٽ ڪرڻ لاءِ گهرجن تمام ڪندڙن باريڳيان ڏسنداسون.

### استعمال جي صورتحال (Use Cases) {#use-cases}

#### صورتحال 1: هڪ ايجنسي (An Agency) {#case-1-an-agency}

عام طور تي ايجنسي جا بنيادي صلاحيتون وڏين سائٽس جو تصميم ڪرڻ ۾ آهن جن ۾ ان جي هاستيزي (hosting) يا مارڪيٽنگ کي اضافي خدمتن جي صورت ۾ لکڻ جاسته شامل آهي.

एजेन्सीहरूका लागि, अल्टीमेट मल्टिसाइट (Ultimate Multisite) एउटा अविश्वसनीय मूल्य प्रस्ताव प्रस्तुत गर्छ जसले एकै प्लेटफर्ममा धेरै वेबसाइट होस्ट गर्न र व्यवस्थापन गर्न सक्ने क्षमता राख्छ। विशेष गरी ती एजेन्सीहरूको लागि जो आफ्नो डिजाइनलाई कुनै खास थीम जस्तै GeneratePress, Astra, OceanWP वा अरूमा मानकीकृत (standardize) गरेका छन्, तिनीहरूले अल्टीमेट मल्टिसाइटको क्षमताको फाइदा उठाउन सक्छन् जसले प्रत्येक नयाँ साइटका लागि यी थीमहरू स्वचालित रूपमा सक्रिय गर्छ।

एजेन्सी मूल्य निर्धारणका लागि सामान्य र लोकप्रिय प्लगइनहरूको प्रशस्त डिलहरू जस्तै, अल्टीमेट मल्टिसाइटको प्रयोगले एजेन्सीहरूलाई विद्यमान लगानीको फाइदा उठाउन अनुमति दिन्छ जसले एक साझा प्लेटफर्म प्रदान गर्दछ जहाँबाट प्लगइनहरू इन्स्टल गर्न, मर्मत गर्न र प्रयोग गर्न सकिन्छ।

बहुसंख्यक अवस्थामा, कन्फिगरेसन (configuration) को प्रयोग चाहिने हुन्छ र खुसीको कुरा अल्टीमेट मल्टिसाइट ले धेरै लोकप्रिय होस्टिङ प्रदायकहरूका लागि डोमेन मैपिंग र SSL प्रमाणपत्रहरूलाई Cloudflare र cPanel जस्ता सेवाहरूसँग आफ्ना एकीकरणहरूको साथ अविश्वसनीय रूपमा सजिलो बनाउँछ।

त्यसैले, यी मध्ये कुनै एक प्रदायकको फाइदा उठाउँदा वा अल्टीमेट मल्टिसाइटलाई Cloudflare पछाडि राख्दा डोमेन व्यवस्थापन र SSL प्रमाणपत्रहरू जस्ता पक्षहरू केही सहज (trivial) बन्न जान्छन्।

जुन एजेन्सीहरू साइटहरूको निर्माणमा कडा नियन्त्रण राख्न चाहन्छन्, तिनीहरू अल्टीमेट मल्टिसाइटको सुव्यवस्थित इन्टरफेस मार्फत साइटहरू बनाउन र ग्राहक तथा योजनाहरूसँग साइटहरू जोड्न सजिलो हुने कुराबाट खुसी हुनेछन्।

![Ultimate Multisite site management interface](/img/admin/sites-list.png)

प्रत्येक उत्पादनको आधारमा प्लगइन र थीममा कडा नियन्त्रण अल्टीमेट मल्टिसाइटका सहज इन्टरफेसहरू मार्फत कायम गरिन्छ, जसले नयाँ साइटका लागि इन्स्टेन्ट गरिएपछि प्लगइन र थीमहरू उपलब्ध वा लुकाउन सकिने तरिका र तिनीहरूको सक्रिय अवस्था पनि देखाउन अनुमति दिन्छ।

Themes وظيفے میں ملتا جلتا کام کرتے ہیں، جس کی وجہ سے سائٹ بناتے وقت مخصوص تھیمز کو فعال (activate) یا چھپانے کا اختیار ملتا ہے۔

![Product theme limitations interface](/img/config/product-themes.png)

Agencies Ultimate Multisite کے ساتھ سکون محسوس کریں گے کیونکہ یہ انہیں وہ کرنے دیتا ہے جو وہ سب سے اچھا کرتے ہیں - شاندار ویب سائٹس ڈیزائن کرنا۔

#### کیس 2: مخصوص فراہم کنندہ (Niche Provider) {#case-2-niche-provider}

ایک پرانا قول ہے کہ، "ایک چیز کرو اور اسے اچھی طرح کرو"۔ بہت سے ماہرین کے لیے اس کا مطلب ایک اکیلے بنیادی خیال کے گرد کوئی پروڈکٹ یا سروس بنانا ہوتا ہے۔

شاید آپ کسی کلبز کو ویب سائٹس کی تشہیر کرنے والے ایک شوقین گولڈر ہوں یا شاید کلنز کو ویب سائٹس فراہم کرنے والا ایک شوقین اسپورٹس گیمر ہوں۔ کیا یہ کسی ریسٹورنٹس کے لیے بکنگ سروس کی تشہیر کرنے والا کوئی فرد ہو سکتا ہے؟

بہت سے وجوہات کی بنا پر آپ کو ایک مشترکہ فریم ورک اور پلیٹ فارم پر مبنی خدمات فراہم کرنا چاہئیں گے۔ ہو سکتا ہے کہ آپ نے ضروری فنکشنلٹی فراہم کرنے کے لیے مخصوص پگ ان (bespoke plugins) ڈیزائن کیے ہوں یا یہ ایسا بھی ہو سکتا ہے کہ صنعت کے بہترین طریقے ڈیزائن کے لیے کسی قسم کے معیاری انداز کی ضرورت رکھتے ہیں۔

Ultimate Multisite کی ایک جدید خصوصیات میں ٹیمپلیٹ سائٹس کا استعمال ہے۔ ٹیمپلیٹ سائٹ وہ ہوتی ہے جہاں تھیم انسٹال اور فعال (activated) ہوتا ہے، ضروری پگ ان انسٹال اور فعال ہوتے ہیں، اور نمونے کے پوسٹس یا پیجز بنائے جاتے ہیں۔ جب کوئی کسٹمر ٹیمپلیٹ کی بنیاد پر نئی سائٹ بناتا ہے، تو ٹیمپلیٹ کا مواد اور سیٹنگز نئی بنائی گئی سائٹ میں نقل (copy) کر لی جاتی ہیں۔

مخصوص سائٹس اور سروس فراہم کرنے والے کے لیے یہ فوری طور پر تیار ایک ایسی سائٹ بنانے کی صلاحیت میں بے مثال فائدہ دیتا ہے جس میں کسٹم پگ ان اور ڈیزائن شامل ہوں۔ کسٹمر کو صرف سروس مکمل کرنے کے لیے سب سے کم ترین معلومات فراہم کرنے کی ضرورت ہوتی ہے۔

ضرورتیاں پر منحصر، دونوں میں سے _subdirectory_ یا _subdomain_ کی ترتیب آپ کے لیے موزوں ہو سکتی ہے، جس صورت میں آرکیٹیکچر کا انتخاب _subdirectories_ کے لیے سادہ SSL سرٹیفکیٹ یا _subdomains_ کے لیے وائلڈ کارڈ SSL سرٹیفکیٹ کے درمیان ہوگا۔

#### کیس 3: WordPress ویب ہوسٹنگ {#case-3-wordpress-web-hosting}

WordPress سائٹس کو ہوسٹ کرنے کے بہت سے طریقے ہیں لیکن یہ اتنی آسان نہیں ہے کہ آپ کسی کسٹمر کو پہلے سے انسٹال شدہ WordPress ورژن کے ساتھ ویب اسپیس فراہم کر دیں۔ ایسا اس لیے ہوتا ہے کیونکہ ایک بامعنی سروس فراہم کرنے کے لیے کئی فیصلوں اور غور و فکر کا مجموعہ درکار ہوتا ہے۔

Ultimate Multisite اس شعبے میں بہترین کارکردگی دکھاتا ہے کیونکہ یہ WordPress سائٹس کو ہوسٹ کرنے کے لیے ایک مکمل ٹرنکیٹ حل فراہم کرتا ہے۔ اس حل میں سبسکرپشن سروسز، ادائیگی وصولی، چیک آؤٹ فارمز، رعایت کے وؤچرز اور کسٹمر کمیونیکیشن فراہم کرنے کے بنیادی میکانزم شامل ہیں۔

WordPress Multisite کو صحیح طریقے سے انسٹال کرنے، ترتیب دینے اور برقرار رکھنے کے لیے درکار زیادہ تر اہم کام Ultimate Multisite کی طرف سے آسان بنایا جاتا ہے تاکہ نیٹ ورک ایڈمنسٹریٹرز کو صرف اپنے سروس یا مخصوص شعبے جیسے پروڈکٹ ٹियर्स، قیمتوں اور سروس آفرز سے متعلق پہلوؤں پر غور کرنا پڑے۔

Ultimate Multisite کے ساتھ انٹیگریٹ ہونا چاہتے ڈویلپرز کے لیے، یہ حل ایک مکمل RESTful API اور ایونٹ نوٹیفیکیشن کے لیے Webhooks بھی فراہم کرتا ہے۔

بہت سے بیرونی پلگインز اور لائسنسوں پر انحصار کے بغیر، Ultimate Multisite Wix، Squarespace، WordPress.com اور دیگر کی طرح ایک بھرپور فیچرز والا اور قابل موازنہ حل فراہم کرتا ہے۔

### آرکیٹیکچر کے غور و فکر {#architecture-considerations}

اگرچہ یہ کوئی مکمل گائیڈ نہیں ہے، لیکن درج ذیل چیزیں Ultimate Multisite انسٹالیشن کو سپورٹ کرنے کے لیے ٹیکنالوجیز کے درست انتخاب کی رہنمائی کر سکتی ہیں۔

#### شیئرڈ بمقابلہ ڈیڈیکیٹڈ ہوسٹنگ {#shared-vs-dedicated-hosting}

معذرت، سڀني هاستنگ فراهم ڪندڙن هڪجهڙا نه آهن ۽ ڪجهه عمل ۾ سخت سرور جي گهڻي کي استعمال ڪندا آهن. سست لاڳاپيل (Low-cost) فراهم ڪندڙ عام طور تي سرور جي گهڻي کي وڌائڻ سان آمدني حاصل ڪندا آهن. ان ڪري توهان جو Ultimate Multisite اسٽالڪي هڪ هميشه هڪ هميشه هجي ويندو آهي جتي هڪ ئي سرور تي ساهه سو سو حسابي موقعن پر ايندڙ هجرن جي تعداد ۾ شامل ٿي سگهي ٿو.

اگر فراهم ڪندڙ طرف کان مناسب حفاظت (safeguards) نه هجي، ته شئرڊ سرور تي موجود سائٽون 'شوردار همس' (noisy neighbour) جي مسئلي کي منهن ڏين ٿيون. يعني، هڪ ئي سرور تي موجوده ان حد تائين وسائل استعمال ڪرڻ واري ڪنهن سائيٽ جي ڪري ٻين سائيٽن کي بقي ٿيل وسائل لاءِ حڪومت ڪرڻو پوندو آهي. اڪثر اهو سائٽن جي صورت ۾ ظاهر ٿئي ٿو جيڪي سست هوندا آهن يا وقت تي جواب نه ڏين ٿا.

خود هڪ ويب هاستنگ فراهم ڪندڙ جي طور پر، اثرات جو عمل هيٺ ڏيکاري ٿو ته توهان جا گراهڪ خراب رفتار، نچيل پيچي رينڪ ۽ وڌيڪ بائنس ريت کي محسوس ڪندا آهن، جنهن نتيجي ۾ اهي ٻاهرين سرونٽيون ڳولڻ جي ڪري گراهڪان کانساني ٿي ويندا آهن.

خلاصو، سست ذات اڻ خوبصورت نه آهي.

Ultimate Multisite کي ڪجهه سٺين هاستنگ فراهم ڪندڙن سان کام ڪرڻ ۽ ان جي ماحول سان چڱيءَ طرح جوڙجڻ جهڙا ڪم ڪري ٿو ته هميشه مڪمل دامن ماپ (domain mapping) ۽ خودکار SSL جهڙا ڪمن کي فراهم ڪري سگهي ٿو. اهي فراهم ڪندڙين عملداري (performance) کي قدر ڏين ٿا ۽ شئرڊ هاستنگ کان وڌيڪ درجه جي خدمت پيش ڪندا آهن.

مطابقن، مطابقت وارن فراهم ڪندڙن جي فهرست ۽ هر هڪ لاءِ مڪمل ترتیب جو تعليمات حاصل ڪرڻ لاءِ مهرباني ڪري Compatible Providers جي داکومينٽيشن ڏسو.

#### عملداري جا خيال (Performance Considerations) {#performance-considerations}

Ultimate Multisite سست اطلاقي نه آهي، پر اهو ڏاڍو تيز آهي. اڃا به، اهو صرف ان بنيادي اطلاقي ۽ انفراسٽرڪچر جتنا سٺو ٿي سگهي ٿو، ۽ صرف ان جو استعمال ڪري سگهي ٿو جيڪو اسان کي رسائي آهي.

هي سوچيو: توهان Ultimate Multisite اسٽالڪي جي سرور ادميني (network administrator) آهيو جتي 100 سائٽون آهن. انهن ۾ ڪجهه سائٽون سٺي حالت ۾ آهن ۽ هر ڏينهن ڪيتريڪ ڪيترائي ويب سائيٽ جي بيمارن کي راغب ڪندا آهن.

यो अवस्था ठूलो स्केलमा फरक हुन्छ، जस्तै هڪ کان پنج سائيٽس لاءِ، پر لڳاتار مسئلا وڌڻ سان اسان کي اهو واضح ٿي ويندو.

گڏ نه رکڻ صورت ۾، هڪ اڪيلو Ultimate Multisite سائيٽ ان سائيٽس جي تمام گهرجيءَ جي درخواستون پوري ڪرڻ جو ذميوار بڻجي ويندي. اهڙيون درخواستون دينامڪ PHP سائيٽس يا اسٽيڪ اثاثن (assets) جهڙوڪ stylesheets، javascript يا ميڊيا فائلز لاءِ ٿي سگهن ٿيون. هڪ سائيٽ ہو يا هڪ سو سائيٽس، اهي ڪم بار بار ٿيڻ وارو، هڪجهڙا ۽ وقت ضايع ڪرڻ وارو بڻجي ويندا. جڏهن جواب هر درخواست لاءِ هڪ ئي اسٽيڪ معلومات هجي ته PHP فائل کي پروسيس ڪرڻ لاءِ CPU پاور ۽ memory استعمال ڪرڻ ضروري ناهي.

هيٺ هم، PHP يا HTML سائيٽ جي هڪ درخواست ان جي بدلي ۾ اسڪرپٽس، stylesheets ۽ image files لاءِ ڪيترن ئي بعد طلبون (requests) جنريٽ ڪري ٿي. اهي درخواستون direkt طور تي توهان جي Ultimate Multisite سرور کي ڇڪڻ وارين آهن.

هئي ته هي مسئلو کي سڌو حل ڪرڻ لاءِ سرور کي اپ گریڊ ڪرڻ آسان آهي، پر اهو ٻئي مسئلي جو حل نه ڪري ٿو - جنهن ۾ جغرافيي تاخیر (geographic latencies) شامل آهن. ان مسئلي کي صحيح طور تي حل ڪرڻ لاءِ ڪيترن ئي مختلف جڳehن واري سيلر سرورز جي ضرورت هجي.

ان سبب، گهڻا نٽ ورڪ اڊمنسٽريز اسٽيڪ سائيٽس جي درخواستن پوري ڪرڻ لاءِ فرءنڊ-ڊ اينڊ ڪيشن (front-end caching) حلن ۽ ڪنٽينٽ ديستريبيوشن نيٽ ورڪس (CDN) جو استعمال ڪن ٿا. اهي درخواستون پوري ڪرڻ ۽ سرور تک پهچڻ کان اڳ ئي اثاثن کي ڏيڻ سان پروسيسنگ ريسورس بچي ٿو، تاخیر ختم ٿيندي آهي، غير ضروري اپ گریڊز کان بچي ٿو ۽ ٽيڪنالاجي جي انویسٽمنٽ کي وڌيڪ ڪجهه بڻائي ٿو.

Ultimate Multisite ۾ هڪ نفيس Cloudflare add-on شامل آهي جيڪو نٽ ورڪ اڊمنسٽريز کي انهن جي installation کي Cloudflare پٺيان رکڻ ۽ ان جي caching صلاحيتن کان علاوه DNS هوست، SSL سرٽيفڪيٽس ۽ سيڪيورٽي ميڪانيزم استعمال ڪرڻ جي اجازت ڏئي ٿو.

#### بڪ اپس (Backups) {#backups}

توهان 50 ماڻهن کان بڪ اپس بابت مشورو طلب ڪري سگهو ٿا ۽ بڪ اپس اسٽريٽجي بابت 50 مختلف راءيون حاصل ڪري سگهو ٿا. جواب اهو آهي ته اهو ڪهڙي شيءِ تي منحصر آهي.

ڪهڙي कुッと منائي نه آهي ته بڪ اپس (backups) جي ضرورت آهي ۽ اهو تقریباً ناممڪن آهي ته اهڪا پراڤائڊر، خاص طور تي هڪ مانيجڊ سروس جو، ان کي منظم نه ڪندو. ان ڪري گهرجيءَ ڪارئونٽرز نتيجا هيٺ ڏنل سروس فراهم ڪرڻ ۽ سنڀالڻ لاءِ ڊيٽا نٽويڪ اڊمنسٽر (network administrator) جي طرف رخ ڪندا رهندا. جنهن کي نٽويڪ اڊمنسٽر سنڀالڻ جو رخ ڪري ٿو، اهو هڪ تمام گهڻي مختلف مسئلو آهي.

هي حصي لاءِ اسان اهو يقين ڪريون ته بڪ اپس ان وقت جي نظام جي حالت جو هڪ نقطي-در-وقت ڪاپي (point-in-time copy) آهي جڏهن بڪ اپس شروع ڪيو هو. سادو لفظن ۾، جيڪا به وقت تي بڪ اپس جي حالت آهي، اهو حالت بڪ اپس ۾ محفوظ ۽ قيد ڪري ويندي آهي.

هي سمجهڻ سان، هي بڪ اپس ڪيئن حاصل ڪجن ۽ توهان جي ماحول لاءِ ڇا بهتر آهي، اهو بالواسطه توهان جي گهرجيءن ۽ هلنڊنگ پراڤائڊر جي ان گهرجي کي پورو ڪرڻ جي صلاحيت تي منحصر ٿيندو. پر، سڀ کان وڌيڪ رائے ڏيندڙ کان وڌيڪ نه ڏيندڙ تائين، هيٺ ڏنل خيار ڪجهه رهنمائي فراهم ڪري سگهن ٿا.

#### اسناپٽس (Snapshots) {#snapshots}

اسناپٽس بڪ اپس لاءِ سنڌو حل آهن ڇاڪاڻ ته اهي آسان، سمجهڻ ۾ نه ٿيون آهن (جڏهن توهان بحال ڪرڻ چاهيو ته)، ۽ 'بس ڪم ڪري ويندا' آهن. پر ان لاءِ توهان کي پنهنجي پراڤائڊر جي مدد گهرجي آهي ۽ اهو اڪثر صرف تبديلي يا متبادل سروس (VPS) يا ان طرح جي شي لاءِ لاڳو ٿيندو. اسان جي ‘Compatible Providers’ جو ڪاڻي ۾ لکيل ڪيترن ئي پراڤائڊرن ۾ بڪ اپس آهن جن کي نٽويڪ اڊمنسٽر طرف کان ڪنهن به اضافي مداخلت يا غور وگر نه گهرجي ٿو.

پرانے بیک اپس فائلز ۽ ڈیٹا بیس کي ٽارگيت ڪن ٿا، جڏهن ته اسڪيٽ اسڪيپ (snapshot) پوري سڪڊ (disk) کي ٽارگيت ڪري ٿو. ان جو مطلب اهو آهي ته صرف سائيٽ جا ڊيٽا ئي اسڪيٽ ۾ محفوظ ٿيندا ناهن، پر اڪائينن نظامِ عمل (operating system) ۽ ڪيفيڪيشن به محفوظ ٿيندا. ان جي لاءِ انهن لاءِ هڪ خاص فائدو آهي ڇاڪاڻ ته سائيٽ جو خراب ٿيل حصو بدلڻ لاءِ اسڪيٽ کان تقریباً فوري طور تي نئين نظام کي شروع ڪري سگهجي ٿو. اها ئي ئي، فائلز واپس حاصل ڪرڻ لاءِ ڪير سان هيٺ ڏنل عمل صرف اسڪيٽ جي ايڊجي (image) کي هڪ ڊسڪ جيئن موجود حصو طور تي جوڙڻ سان گهرجي ته جيئن فائلز کي رسائي ۽ کاپی ڪري سگهجي وڃن.

اسڪيٽ کان حتي هتي هوستنگ پروائيڊر طرف کان اضافي قيمت لاڳو ٿي سگهي ٿي، پر اهو حادثات جي خلاف هڪ بيڪرينج پاليسي آهي.

#### ايڪسٽرنل اسڪرپٽس (External Scripts) {#external-scripts}

WordPress ۽ MySQL جي سيڙپڪاري کي بیک اپ ڪرڻ لاءِ ڪيترائي به ايڪسٽرنل اسڪرپٽس ۽ حل موجود آهن، جيڪي Ultimate Multisite لاءِ سٺا ڪم ڪندا ڇاڪاڻ ته اهو هڪ WordPress plugin آهي جيڪو WordPress جي فائل سسٽم ۽ ڊيٽابيس جو استعمال ڪري ٿو. ان ڪري، اها حل جيڪو WordPress سائيٽن کي بیک اپ ڪري ٿو، Ultimate Multisite جي ضرورتن کي پوري طرح ڍڪي سگهي ٿو.

اسان ڪنهن هڪ اسڪرپٽ کي ٻين کان بهتر نه ڪري سگهون ٿا پر اسان جو عام مشورو اهو آهي ته نتيجا مطلوب هجن ته لاءِ ڪيترائي به بیک اپ ۽ ريستور جا تجربا ڪندا، ۽ خاص طور تي جتي ڪنهن هيٺ ڏنل فرق وارو (differential backup) طريقو لاڳو ڪيو وڃي تڏهن "ڪير سان ڪير" رهڻ جو.

اها ڳالهه رکڻ ضروري آهي ته ان اسڪرپٽ جي چالو ٿيڻ دوران نظام جي بوجھ ۾ اضافو ٿيندو، جنهن کي لهو ڪرڻ گهرجي.

#### پلگجز (Plugins) {#plugins}

WordPress ۾ تقريب ڪنهن به مسئلي جو حل plugin سان نه ٿي سگهي، ۽ جيڪڏهن توهان ايڪسٽرنل اسڪرپٽس جو انتظام ڪرڻ لاءِ مناسب ناهين، ته هاڻي شاید هڪ plugin بهترين خيار هوندو.

Plugins جي مختلف آپشنز ۽ فيچرز رکندا آهن پر انهن جو ڪم عام طور تي هڪ ئي ٿيندو آهي، جنهن جو مقصد WordPress جون فائلون ۽ ڊيٽا بيس (database) کي ڪاپي ڪرڻ آهي. ان کان پوءِ ڪاميابيتون مختلف هونديون آهن؛ ڪجهه plugins بڪ اپس کي ٻاهرين خدمتن جهڙوڪ Google Drive يا Dropbox يا S3، Wasabi يا ٻين قسم جي ڪم جي گهرجي (object storage service) تي بھیجڻ جو طريقو رکندا آهن. وڌيڪ جامع plugins ڊيفيرنشيول بڪ اپس (differential backups) يا ڪنه قسم جي اسٽريٽي فراهم ڪندا آهن ته جيئن صرف ان ڊيٽا کي بڪ اپس ڪجي جيڪو تبديل ٿيو آهي، ۽ ٻاهرين گهرج جي لاڳوتھن کي بچائڻ لاءِ.

پلاجن چونڊندا هئي ته يقيني ڪريو ته اهو multisite aware آهي. اسان جو عمل چلڻ جي طبيعت سبب، جڏهن بڪ اپس چَلائي رهيو هوندو، توهان کي سرور تي ڪجهه موقت بوجھ (temporary load) محسوس ٿي سگهو ٿا تائين جڏهن عمل ختم نه ٿئي.

#### Domain ۽ SSL {#domain-and-ssl-1}

multisite _subdomain_ mode ۾ domain names جي بابت گهڻي ڳالهيون ڪرايا ويا آهن. نٽ ورڪ اڊمنٽرن لاءِ هڪ تقریباً عالمي حل آهي ته wildcard DNS entries جو استعمال ڪجي.

![Wildcard DNS entry configuration example](/img/config/settings-domain-mapping.png)

هي قسم جي DNS entry 'site1.domain.com' ۽ 'site2.domain.com' جهڙين subdomain کي IP address 1.2.3.4 تي ڪاميابن سان حل ڪري سگهي ٿي، جنهن Ultimate Multisite ۽ وڌيڪ WordPress Multisite جو استعمال '_subdomain_ mode ۾ سپورٽ ڪري ٿو.

هي HTTP لاءِ پوري طرح کام ڪري سگهي ٿي ڇاڪاڻ ته مقصد جو host HTTP headers کان ملي ٿو پر اڄ به ويب اتني سادي نه آهي ته HTTPS جي محفوظ عملن کي تقریباً لازمي سمجھيو وڃي.

خوش قسمتی ہے کہ SSL سرٹیفکیٹس کے لیے آسان آپشنز موجود ہیں۔ _subdirectory_ موڈ میں، ایک عام ڈومین سرٹیفکیٹ استعمال کی جا سکتی ہے۔ یہ ہوسٹنگ فراہم کرنے والوں سے آسانی سے اور مفت دستیاب ہوتے ہیں جو شاید فری Let'sEncrypt سروس یا کسی دوسرے ذریعہ کا استعمال کرتے ہوں، ورنہ اگر آپ سرٹیفکیٹ سائننگ ریکویسٹ جنریٹ کرنے کے قابل ہوں تو یہ اتھارٹیز سے تجارتی طور پر دستیاب ہوتے ہیں۔

_subdomain_ موڈ کے لیے، وائلڈ کارڈ SSL سرٹیفکیٹ کا استعمال ایک وائلڈ کارڈ ڈومین کے ساتھ بالکل بہترین طریقے سے جڑ جائے گا اور سرٹیفکیٹ کو رُوٹ ڈومین اور تمام _subdomains_ کے لیے اتھارٹی بنانے کی اجازت دے گا بغیر کسی اضافی ترتیب کے۔

تاہم، یہ نوٹ کرنا ضروری ہے کہ وائلڈ کارڈ SSL سرٹیفکیٹس کلاؤڈ فلیر جیسے سروسز کے ساتھ کام نہیں کر سکتے جب تک کہ آپ انٹرپرائز پلان پر نہ ہوں یا انٹری کو DNS صرف پر سیٹ نہ کریں جس صورت میں تمام کیشنگ اور آپٹیمائزیشن نظر انداز کر دی جاتی ہے۔

آؤٹ آف دی باکس Ultimate Multisite اس مسئلے کا حل فراہم کرتا ہے جو ورڈپریس ملٹی سائٹس کی ضروریات کے ساتھ ہمارے وسیع تجربے کو ظاہر کرتا ہے۔ اس سادہ ایڈ-آن کو فعال کرنے سے Ultimate Multisite آپ کے Cloudflare کریڈینشلز کا استعمال کرتے ہوئے نیٹ ورک سائٹس کے لیے Cloudflare میں DNS اندراجات خود شامل کرے گا اور ان کا موڈ 'proxied' پر سیٹ کر دے گا۔ اس طرح، جب بھی کوئی نیٹ ورک سب سائٹ بنائی جائے تو اسے SSL سمیت Cloudflare کے مکمل تحفظ اور فوائد حاصل ہوں گے۔

آپ کی Ultimate Multisite انسٹالیشن کی نوعیت اور مقصد کے لحاظ سے صارفین کو اپنے ڈومینز استعمال کرنے کی ضرورت ہو سکتی ہے۔ اس صورت میں، نیٹ ورک ایڈمنسٹریٹر کو دو مسائل حل کرنے کا ذمہ دار ٹھہرایا جاتا ہے۔ ایک، ڈومین نیم کی ہوسٹنگ اور دوسرا، ڈومین کے لیے SSL سرٹیفکیٹس۔

धेरै मानिस لاءِ، Cloudflare استعمال کرنا सजिलो آپشن ہے۔ کسٹمر کو بس اپنا ڈومین Cloudflare پر رکھنا ہے، ایک CNAME کو Ultimate Multisite کے روٹ ڈومین پر پوائنٹ کرنا ہے اور اپنی ڈومین کو Ultimate Multisite میں مپ کرنا ہے تاکہ اپنے کسٹم ڈومین نیم کا فائدہ اٹھانا شروع کر سکیں۔

اس کے علاوہ، ہمیں دوسرے حل تلاش کرنے کی ضرورت ہے جس کی وجہ سے Ultimate Multisite Compatible Providers کی ایک فہرست تجویز کرتا ہے۔ یہ اس لیے ہے کیونکہ DNS اور SSL سیٹ اپ کا عمل تھوڑا مشکل ہو سکتا ہے۔ تاہم، Ultimate Multisite کے ان فراہم کنندگان کے ساتھ انٹیگریشن کی وجہ سے پیچیدگی بہت کم ہو جاتی ہے اور طریقہ کار خود بخود (automated) ہوتا ہے۔

#### Plugins {#plugins-1}
