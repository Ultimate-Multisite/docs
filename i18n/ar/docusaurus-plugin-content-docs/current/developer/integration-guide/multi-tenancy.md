# تكامل تعدد المستأجرين

Ultimate Multisite: يغيّر Multi-Tenancy 1.2.0 عدة نقاط تماس للتكامل للمستأجرين السياديين، والتحقق من الترحيل، وأتمتة دورة حياة المستأجر.

## تدفق تمهيد المستأجر {#tenant-bootstrap-flow}

ينبغي للتكاملات التي تنشئ المستأجرين أو تعدّلهم اتباع هذا الترتيب:

1. حلّ سجل مستودع المستأجر ونموذج العزل.
2. إنشاء كاتب قاعدة بيانات المستأجر أو التحقق منه.
3. تمهيد مخطط المستأجر.
4. توفير مستخدمي المستأجر.
5. تسجيل توجيه المستأجر ومسارات نظام الملفات.
6. تشغيل التحقق من الترحيل قبل إتاحة المستأجر.

لا تفترض أن مستأجرًا سياديًا يمكنه إعادة استخدام اتصال قاعدة بيانات الشبكة. استخدم مستودع المستأجر وتجريدات الكاتب التي توفرها الإضافة.

## خطافات SSO وREST {#sso-and-rest-hooks}

يستخدم تسجيل الدخول التلقائي عديم الحالة للمستأجر رموزًا قصيرة العمر مع مطالبة غرض، وحماية JTI من إعادة التشغيل، وحدًا أقصى لانتهاء الصلاحية، وتثبيتًا للأصل. ينبغي للتكاملات التي تضيف أزرار تسجيل الدخول أو روابط الإدارة عن بُعد إنشاء زيارات المستأجر عبر تدفق SSO المدعوم بدلًا من إنشاء عناوين URL لتسجيل دخول المستأجر مباشرةً.

تتوفر أحداث تدقيق API من جهة الشبكة والملخصات اليومية لبوابات المستأجرين السياديين. استخدم تلك السجلات عند تصحيح أخطاء الأنظمة الخارجية التي تستدعي نقاط نهاية دورة حياة المستأجر.

## عناوين URL لإجراءات العميل السيادي {#sovereign-customer-action-urls}

يوجّه Ultimate Multisite v2.13.0 إجراءات عملاء المستأجرين السياديين إلى الموقع الرئيسي لتدفقات Account، والدفع، والفوترة، والفاتورة، والموقع، وتبديل القالب، وربط النطاقات. ينبغي للتكاملات التي تعرض روابط إدارة من جهة المستأجر توجيه تلك الإجراءات إلى لوحة عملاء الموقع الرئيسي وتضمين هدف عودة مُتحقق منه عندما ينبغي أن يتمكن المستخدم من الرجوع إلى المستأجر بعد إكمال الإجراء.

استخدم غلاف SSO الأساسي لروابط الإدارة عبر النطاقات:

```php
$url = wu_with_sso($main_site_customer_url);
```

يبقى عنوان URL المُنشأ قابلًا للترشيح عبر `wu_sso_url`، الذي يتلقى عنوان URL الخاص بـ SSO، والمستخدم الحالي، ومعرّف الموقع الهدف، وسياق إعادة التوجيه. يمكن للإضافات استخدام ذلك المرشح لإلحاق سياق خاص بالمزوّد أو لاستبدال عنوان URL للوسيط مع الحفاظ على تحقق Ultimate Multisite من الرمز.

لا تكرر حالة العضوية أو الفاتورة أو عنوان الفوترة أو القالب أو إدارة النطاق داخل المستأجر السيادي. تعامل مع لوحة تحكم المستأجر كمشغّل ومع لوحة عملاء الموقع الرئيسي كنظام السجل للإجراءات المُدارة.

## التحقق من الترحيل {#migration-verification}

بعد أن يغيّر تكامل ترحيل أو دورة حياة بيانات المستأجر، شغّل بوابات التحقق:

- `wp tenant verify-no-legacy --site=<site-id>` يؤكد أن المستأجر لم يعد يعتمد على مسارات قديمة من جهة الشبكة.
- `wp tenant verify-sovereign-push --site=<site-id>` يؤكد أن مهام الدفع السيادي يمكنها المعالجة والتفريغ.

ينبغي للتكاملات التعامل مع فشل التحقق كمانع للنشر وتجنب وضع علامة أن المستأجر نشط حتى يتم حل الفشل.

## حذف المستأجر {#tenant-deletion}

ينبغي لتدفقات الحذف استدعاء مسار التفكيك الخاص بالإضافة بحيث يتم تنظيف بيانات اعتماد قاعدة بيانات المستأجر. قد تزيل التكاملات الخارجية موارد المزوّد بعد نجاح التفكيك، لكنها ينبغي ألا تحذف قواعد بيانات المضيف أو المجلدات بينما لا يزال التحقق أو مهام الدفع غير المتزامنة قيد التشغيل.

## موجّه قاعدة البيانات المهمل {#deprecated-database-router}

تم استبدال `Database_Router` القديم بقالب إهمال. ينبغي للتكاملات الجديدة حلّ المستأجرين عبر موجّه الموقع الحالي وواجهات برمجة تطبيقات مستودع المستأجر بدلًا من الاعتماد على فئة الموجّه القديمة.
