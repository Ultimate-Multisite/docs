# سير عمل الترحيل السيادي (Sovereign Migration Workflow) {#sovereign-migration-workflow}

إصدار Ultimate Multisite: Multi-Tenancy 1.2.0 يضيف بوابات تحقق من عملية الترحيل لنقل موقع فرعي قياسي إلى بنية مستأجر سيادي (sovereign tenant infrastructure).

## قبل أن تبدأ {#before-you-start}

تأكد من توفر ما يلي لديك:

- نسخة احتياطية حالية للموقع المصدر.
- إضافة Multi-Tenancy مفعلة.
- قاعدة بيانات المستأجر الهدف، وجذر نظام الملفات، والنطاق جاهزان.
- ربط مضيف قاعدة البيانات الذي يتطابق مع بيئة الوجهة.
- إمكانية الوصول لتشغيل أوامر WP-CLI للشبكة.

## سير العمل الموصى به {#recommended-workflow}

1. قم بإعداد قاعدة بيانات المستأجر ونظام الملفات الهدف.
2. سجل أو حدث إعدادات عزل المستأجر (tenant isolation settings).
3. قم بتشغيل ترحيل المستأجر (tenant migration).
4. قم بتوفير مستخدمي المستأجر أو التحقق منهم.
5. قم بتفريغ مهام الترحيل غير المتزامنة (async migration jobs).
6. قم بتشغيل التحقق من الترحيل السيادي (sovereign migration verification).
7. قم بزيارة المستأجر باستخدام تسجيل الدخول الموحد (SSO).
8. قم بتغيير DNS أو التوجيه فقط بعد اجتياز عملية التحقق.

## بوابات التحقق {#verification-gates}

تتحقق عملية التحقق من عملية الترحيل من عدة زوايا:

- هل مخطط المستأجر موجود ويمكن كتابته بواسطة كاتب قاعدة بيانات المستأجر؟
- هل المضيف المحدد لقاعدة البيانات لديه صلاحيات (grants) صالحة؟
- هل مستخدمو المستأجر موجودون وهل يتطابقون مع العدد المتوقع لمستخدمي التثبيت السيادي؟
- هل يمكن تفريغ قائمة انتظار الدفع غير المتزامن بنجاح؟
- هل مسارات البيانات القديمة على جانب الشبكة لم تعد مطلوبة للمستأجر السيادي؟

عامل حالات فشل التحقق كعوائق قبل الإطلاق. قم بإصلاح مشكلة قاعدة البيانات أو المستخدم أو القائمة أو التوجيه التي تم الإبلاغ عنها، ثم أعد تشغيل عملية التحقق مرة أخرى قبل تعريض المستأجر للعملاء.

## أول زيارة إنتاجية {#first-production-visit}

بعد اجتياز عملية التحقق، استخدم **Visit (SSO)** من شاشة إدارة الموقع لأول زيارة إدارية. هذا يؤكد توجيه المستأجر، ومعالجة رمز SSO، وتثبيت المصدر (origin pinning)، وتوفير مستخدمات جانب المستأجر في خطوة واحدة مُتحكم بها.
