# മൾട്ടി-ടെനൻസി സംയോജനം {#multi-tenancy-integration}

Ultimate Multisite: Multi-Tenancy 1.2.0 സ്വതന്ത്ര ടെനന്റുകൾ, മൈഗ്രേഷൻ സ്ഥിരീകരണം, ടെനന്റ് ലൈഫ്‌സൈക്കിൾ ഓട്ടോമേഷൻ എന്നിവയ്ക്കായുള്ള നിരവധി സംയോജന ടച്ച്പോയിന്റുകൾ മാറ്റുന്നു.

## ടെനന്റ് ബൂട്ട്സ്ട്രാപ്പ് പ്രവാഹം {#tenant-bootstrap-flow}

ടെനന്റുകൾ സൃഷ്ടിക്കുകയോ മാറ്റുകയോ ചെയ്യുന്ന സംയോജനങ്ങൾ ഈ ക്രമം പാലിക്കണം:

1. ടെനന്റ് രജിസ്ട്രി രേഖയും ഐസൊലേഷൻ മാതൃകയും കണ്ടെത്തുക.
2. ടെനന്റ് ഡാറ്റാബേസ് റൈറ്റർ സൃഷ്ടിക്കുകയോ സ്ഥിരീകരിക്കുകയോ ചെയ്യുക.
3. ടെനന്റ് സ്കീമ ബൂട്ട്സ്ട്രാപ്പ് ചെയ്യുക.
4. ടെനന്റ് ഉപയോക്താക്കളെ പ്രൊവിഷൻ ചെയ്യുക.
5. ടെനന്റ് റൂട്ടിംഗും ഫയൽസിസ്റ്റം പാതകളും രജിസ്റ്റർ ചെയ്യുക.
6. ടെനന്റ് പുറത്തേക്ക് ലഭ്യമാക്കുന്നതിന് മുമ്പ് മൈഗ്രേഷൻ സ്ഥിരീകരണം പ്രവർത്തിപ്പിക്കുക.

ഒരു സ്വതന്ത്ര ടെനന്റിന് നെറ്റ്‌വർക്ക് ഡാറ്റാബേസ് കണക്ഷൻ വീണ്ടും ഉപയോഗിക്കാനാകുമെന്ന് കരുതരുത്. ആഡോൺ നൽകുന്ന ടെനന്റ് രജിസ്ട്രിയും റൈറ്റർ ആബ്സ്ട്രാക്ഷനുകളും ഉപയോഗിക്കുക.

## എസ്‌എസ്‌ഒയും REST ഹുക്കുകളും {#sso-and-rest-hooks}

സ്റ്റേറ്റ്‌ലെസ് ടെനന്റ് ഓട്ടോലോഗിൻ ഒരു പർപ്പസ് ക്ലെയിം, JTI റീപ്ലേ സംരക്ഷണം, കാലാവധി പരിധി, ഒറിജിൻ പിനിംഗ് എന്നിവയുള്ള കുറച്ചുകാലം മാത്രം സാധുവായ ടോക്കണുകൾ ഉപയോഗിക്കുന്നു. ലോഗിൻ ബട്ടണുകളോ റിമോട്ട് മാനേജ്മെന്റ് ലിങ്കുകളോ ചേർക്കുന്ന സംയോജനങ്ങൾ ടെനന്റ് ലോഗിൻ URL-കൾ നേരിട്ട് നിർമ്മിക്കുന്നതിന് പകരം പിന്തുണയ്ക്കുന്ന SSO പ്രവാഹത്തിലൂടെ ടെനന്റ് സന്ദർശനങ്ങൾ സൃഷ്ടിക്കണം.

സ്വതന്ത്ര ടെനന്റ് ഗേറ്റ്‌വേകൾക്ക് നെറ്റ്‌വർക്ക് വശത്തെ API ഓഡിറ്റ് ഇവന്റുകളും ദൈനംദിന സംഗ്രഹങ്ങളും ലഭ്യമാണ്. ടെനന്റ് ലൈഫ്‌സൈക്കിൾ endpoints വിളിക്കുന്ന ബാഹ്യ സിസ്റ്റങ്ങൾ ഡീബഗ് ചെയ്യുമ്പോൾ ആ ലോഗുകൾ ഉപയോഗിക്കുക.

## സ്വതന്ത്ര ഉപഭോക്തൃ ആക്ഷൻ URL-കൾ {#sovereign-customer-action-urls}

Ultimate Multisite v2.13.0 account, checkout, ബില്ലിംഗ്, ഇൻവോയ്സ്, സൈറ്റ്, ടെംപ്ലേറ്റ് മാറ്റൽ, ഡൊമെയ്ൻ മാപ്പിംഗ് പ്രവാഹങ്ങൾക്കായി സ്വതന്ത്ര-ടെനന്റ് ഉപഭോക്തൃ ആക്ഷനുകൾ പ്രധാന സൈറ്റിലേക്ക് തിരികെ റൂട്ട് ചെയ്യുന്നു. ടെനന്റ് വശത്തെ മാനേജ്മെന്റ് ലിങ്കുകൾ റെൻഡർ ചെയ്യുന്ന സംയോജനങ്ങൾ ആ ആക്ഷനുകൾ പ്രധാന-സൈറ്റ് ഉപഭോക്തൃ പാനലിലേക്കാണ് ചൂണ്ടിക്കാണിക്കേണ്ടത്. ആക്ഷൻ പൂർത്തിയാക്കിയതിന് ശേഷം ഉപയോക്താവിന് ടെനന്റിലേക്ക് മടങ്ങാൻ കഴിയേണ്ടതുണ്ടെങ്കിൽ സാധൂകരിച്ച റിട്ടേൺ ലക്ഷ്യവും ഉൾപ്പെടുത്തണം.

ക്രോസ്-ഡൊമെയ്ൻ മാനേജ്മെന്റ് ലിങ്കുകൾക്കായി കോർ SSO റാപ്പർ ഉപയോഗിക്കുക:

```php
$url = wu_with_sso($main_site_customer_url);
```

സൃഷ്ടിച്ച URL `wu_sso_url` വഴി ഇപ്പോഴും ഫിൽറ്റർ ചെയ്യാം. അതിന് SSO URL, നിലവിലെ ഉപയോക്താവ്, ലക്ഷ്യ സൈറ്റ് ID, റീഡയറക്ട് കോൺടെക്സ്റ്റ് എന്നിവ ലഭിക്കും. Ultimate Multisite-ന്റെ ടോക്കൺ വാലിഡേഷൻ നിലനിർത്തിക്കൊണ്ട് പ്രൊവൈഡർ-നിർദ്ദിഷ്ട കോൺടെക്സ്റ്റ് ചേർക്കാനോ ബ്രോക്കർ URL മാറ്റിസ്ഥാപിക്കാനോ ആഡോണുകൾക്ക് ആ ഫിൽറ്റർ ഉപയോഗിക്കാം.

മെമ്പർഷിപ്പ്, ഇൻവോയ്സ്, ബില്ലിംഗ് വിലാസം, ടെംപ്ലേറ്റ്, അല്ലെങ്കിൽ ഡൊമെയ്ൻ മാനേജ്മെന്റ് നില സ്വതന്ത്ര ടെനന്റിനുള്ളിൽ പകർന്നു സൂക്ഷിക്കരുത്. ടെനന്റ് Dashboard-നെ ലോഞ്ചറായും പ്രധാന-സൈറ്റ് ഉപഭോക്തൃ പാനലിനെ മാനേജ്ഡ് ആക്ഷനുകളുടെ ഔദ്യോഗിക റെക്കോർഡ് സിസ്റ്റമായും കാണുക.

## മൈഗ്രേഷൻ സ്ഥിരീകരണം {#migration-verification}

ഒരു മൈഗ്രേഷൻ അല്ലെങ്കിൽ ലൈഫ്‌സൈക്കിൾ സംയോജനം ടെനന്റ് ഡാറ്റ മാറ്റിയ ശേഷം, സ്ഥിരീകരണ ഗേറ്റുകൾ പ്രവർത്തിപ്പിക്കുക:

- `wp tenant verify-no-legacy --site=<site-id>` ടെനന്റ് ഇനി ലെഗസി നെറ്റ്‌വർക്ക്-വശ പാതകളിൽ ആശ്രയിക്കുന്നില്ലെന്ന് സ്ഥിരീകരിക്കുന്നു.
- `wp tenant verify-sovereign-push --site=<site-id>` സ്വതന്ത്ര പുഷ് ജോലികൾ പ്രോസസ് ചെയ്യാനും ഡ്രെയിൻ ചെയ്യാനും കഴിയുമെന്ന് സ്ഥിരീകരിക്കുന്നു.

പരാജയപ്പെട്ട സ്ഥിരീകരണം സംയോജനങ്ങൾ ഡിപ്ലോയ്മെന്റ് തടസ്സമായി കണക്കാക്കണം. പരാജയം പരിഹരിക്കുന്നതുവരെ ഒരു ടെനന്റിനെ ലൈവ് ആയി അടയാളപ്പെടുത്തുന്നത് ഒഴിവാക്കണം.

## ടെനന്റ് ഇല്ലാതാക്കൽ {#tenant-deletion}

ഇല്ലാതാക്കൽ പ്രവാഹങ്ങൾ ആഡോൺ ടിയർഡൗൺ പാത വിളിക്കണം, അതുവഴി ടെനന്റ് ഡാറ്റാബേസ് ക്രെഡൻഷ്യലുകൾ വൃത്തിയാക്കപ്പെടും. ടിയർഡൗൺ വിജയിച്ചതിന് ശേഷം ബാഹ്യ സംയോജനങ്ങൾക്ക് പ്രൊവൈഡർ വിഭവങ്ങൾ നീക്കാം. എന്നാൽ സ്ഥിരീകരണമോ അസിങ്ക് പുഷ് ജോലികളോ ഇപ്പോഴും പ്രവർത്തിക്കുന്നുണ്ടെങ്കിൽ ഹോസ്റ്റ് ഡാറ്റാബേസുകളോ ഫോൾഡറുകളോ ഇല്ലാതാക്കരുത്.

## കാലഹരണപ്പെട്ട ഡാറ്റാബേസ് റൂട്ടർ {#deprecated-database-router}

ലെഗസി `Database_Router` ഒരു ഡിപ്രിക്കേഷൻ സ്റ്റബ് ഉപയോഗിച്ച് മാറ്റിസ്ഥാപിച്ചിരിക്കുന്നു. പുതിയ സംയോജനങ്ങൾ പഴയ റൂട്ടർ ക്ലാസിനെ ആശ്രയിക്കുന്നതിന് പകരം നിലവിലെ സൈറ്റ് റൂട്ടറും ടെനന്റ് രജിസ്ട്രി API-കളും വഴി ടെനന്റുകളെ കണ്ടെത്തണം.
