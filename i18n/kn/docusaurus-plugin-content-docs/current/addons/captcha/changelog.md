# Captcha ಚೇಂಜ್‌ಲಾಗ್

Version: 1.5.0 - ಬಿಡುಗಡೆ ದಿನಾಂಕ: 2026-05-22
* ಹೊಸದಾಗಿ: ಹಾರ್ಡ್-ಸ್ಟಾಪ್ ರೇಟ್ ಲಿಮಿಟರ್ (Hard-stop rate limiter) — ಇದು captcha-ರಕ್ಷಿತ ಪುಟಗಳಲ್ಲಿ (wp-login / register / lost-password / comments, WooCommerce my-account / checkout / pay-for-order, Ultimate Multisite checkout / inline-login) ಬರುವ ಪ್ರತಿ GET ಮತ್ತು POST ವಿನಂತಿಯನ್ನು ಎಣಿಕೆ ಮಾಡುತ್ತದೆ. ಇದು HTTP 429, ಒಂದು `Retry-After` ಹೆಡರ್, ಮತ್ತು ಯಾದೃಚ್ಛಿಕ ಟಾರ್ಪಿಟ್ ಸ್ಲೀಪ್ (1–5 ಸೆಕೆಂಡುಗಳು, ಗರಿಷ್ಠ 15 ಸೆಕೆಂಡುಗಳಿಗೆ ಮಿತಿಗೊಳಿಸಲಾಗಿದೆ) ಯೊಂದಿಗೆ ಪ್ರತಿಕ್ರಿಯಿಸುತ್ತದೆ.
* ಹೊಸದಾಗಿ: ಟಾರ್ಪಿಟ್ ವಿಂಡೋವನ್ನು ಹೊಂದಿಸಲು `cap_rate_limit_tarpit_min` / `cap_rate_limit_tarpit_max` ಸೆಟ್ಟಿಂಗ್‌ಗಳು.
* ಹೊಸದಾಗಿ: ನಂಬಲರ್ಹ IP রেঞ্জಗಳನ್ನು ವಿನಾಯಿತಿ ನೀಡಲು `wu_cap_rate_limit_whitelist_ip` ಫಿಲ್ಟರ್.
* ಹೊಸದಾಗಿ: ಹಾರ್ಡ್-ಸ್ಟಾಪ್ ಪ್ರತಿಕ್ರಿಯೆ ಕಳುಹಿಸುವ ತಕ್ಷಣವೇ ಕಾರ್ಯನಿರ್ವಹಿಸುವ `wu_cap_rate_limit_will_block` ಆಕ್ಷನ್.
* ಹೊಸದಾಗಿ: ಸ್ಪೂಫ್-ನಿರೋಧಕ client-IP ಪತ್ತೆ. `Captcha_Core::get_client_ip()` (ರೇಟ್-ಲಿಮಿಟ್ ಬಕೆಟ್ ಕೀಗಳ ಮೂಲ, captcha siteverify `remoteip`, ಮತ್ತು ಸ್ಟ್ಯಾಟಿಸ್ಟಿಕ್ಸ್ IP ಹ್ಯಾಶ್‌ಗಳ ಮೂಲ) ಈಗ ಕಟ್ಟುನಿಟ್ಟಾದ ನಂಬಿಕೆ ಮಾದರಿಯನ್ನು ಜಾರಿಗೊಳಿಸುತ್ತದೆ: REMOTE_ADDR ಮೂಲ ಮಟ್ಟವಾಗಿದೆ, `CF-Connecting-IP` ಅನ್ನು ಕೇವಲ ತಕ್ಷಣದ ಪೀರ್ (immediate peer) ಪ್ರಸ್ತುತ Cloudflare IP রেঞ্জের ಒಳಗೆ ಇದ್ದಾಗ ಮಾತ್ರ ಗೌರವಿಸಲಾಗುತ್ತದೆ, ಮತ್ತು `X-Forwarded-For` ಅನ್ನು ಕೇವಲ ತಕ್ಷಣದ ಪೀರ್ ಆಡಮಿನ್-ಕನ್ಫಿಗರ್ ಮಾಡಿದ ನಂಬಲರ್ಹ-ಪ್ರಾಕ್ಸಿ ಪಟ್ಟಿಯಲ್ಲಿ ಇದ್ದಾಗ ಮಾತ್ರ ಗೌರವಿಸಲಾಗುತ್ತದೆ. ಇದು ಬಲದಿಂದ ಬಲಕ್ಕೆ (right-to-left) ನಡೆಯುತ್ತದೆ ಮತ್ತು ನಂಬಲರ್ಹ/CF ಹಂತಗಳನ್ನು ಬಿಟ್ಟು ಭೇಟಿ ನೀಡಿದ IP ಮೇಲೆ ನೆಲೆಗೊಳ್ಳುತ್ತದೆ.
* ಹೊಸದಾಗಿ: `cap_trust_cloudflare_headers` ಸೆಟ್ಟಿಂಗ್ (default OFF) — Cloudflare ಹಿಂದೆ ಇರುವಾಗ `CF-Connecting-IP` ನಂಬಿಕೆಗೆ ಆಯ್ಕೆ ಮಾಡಿಕೊಳ್ಳಬಹುದು. ಈ ಪ್ಲಗ್ಇನ್ ಒಂದು ಬಂಡಲ್ ಮಾಡಿದ Cloudflare CIDR ಸ್ನ್ಯಾಪ್‌ಶಾಟ್ ಅನ್ನು ಒದಗಿಸುತ್ತದೆ ಮತ್ತು ಈ ರಿಫ್ರೆಶ್ ಅನ್ನು ವಾರಕ್ಕೊಮ್ಮೆ wp-cron ಮೂಲಕ ಮಾಡುತ್ತದೆ. ರಿಫ್ರೆಶ್ ವಿಫಲವಾದರೆ ಬಂಡಲ್ ಮಾಡಿದ ಫಾಲ್ಬ್ಯಾಕ್ ಲಭ್ಯವಿದೆ.
* ಹೊಸದಾಗಿ: `cap_trusted_proxies` ಸೆಟ್ಟಿಂಗ್ — ನಿಮ್ಮ ಸ್ವಂತ ಫ್ರಂಟ್-ಲೈನ್ ಪ್ರಾಕ್ಸಿಗಳು / ಲೋಡ್-ಬ್ಯಾಲನ್ಸರ್‌ಗಳನ್ನು ಪಟ್ಟಿ ಮಾಡಲು CIDR ಅಥವಾ ನೇರ IPಗಳ (ಪ್ರತಿ ಲೈನ್‌ನಲ್ಲಿ ಒಂದು, `#` ಕಾಮೆಂಟ್‌ಗಳು ಅನುಮತಿಸಲಾಗಿದೆ) textarea. ಇದನ್ನು ಇಲ್ಲದಿದ್ದರೆ, ರೇಟ್ ಲಿಮಿಟರ್ ಅನ್ನು ಸಕ್ರಿಯಗೊಳಿಸಿದರೂ `X-Forwarded-For` ಅನ್ನು ನಿರ್ಲಕ್ಷಿಸಲಾಗುತ್ತದೆ.
* ಹೊಸದಾಗಿ: Cloudflare / ಪ್ರಾಕ್ಸಿ ಸ್ಥಿತಿಯ ಸ್ವಯಂಚಾಲಿತ ಪತ್ತೆ (auto-detection) ಮೊದಲ ಬಾರಿಗೆ ಸಕ್ರಿಯಗೊಳಿಸುವಾಗ, ಒಂದು-ಕ್ಲಿಕ್ "Apply detected settings" ಆಡಮಿನ್ ನೋಟಿಕೆ ಲಭ್ಯವಿದೆ. ಈ ಪ್ಲಗ್ಇನ್ ಎಂದಿಗೂ ನಿಮ್ಮ ಉಳಿಸಿದ ಮೌಲ್ಯಗಳನ್ನು ಮರೆಮಾಡುವುದಿಲ್ಲ; જો ನಂತರದ ಟ್ರಾಫಿಕ್ ನಿಮ್ಮ ಕಾನ್ಫಿಗರೇಶನ್ ವಾಸ್ತವಕ್ಕೆ ಹೊಂದಿಕೆಯಾಗದಿದ್ದರೆ (ಉದಾಹರಣೆಗೆ Cloudflare CIDR রেঞ্জಗಳನ್ನು ಬದಲಾಯಿಸಿದೆ ಮತ್ತು ನಿಮ್ಮ ಪ್ರಾಕ್ಸಿ CIDR ಈಗ ಹಳೆಯದಾಗಿದೆ), ಅಳಿಸಲು ಸಾಧ್ಯವಿಲ್ಲದ ಒಂದು ಹೊಂದಾಣಿಕೆಯಲ್ಲದ ನೋಟಿಕೆ (mismatch notice) ಶಿಫಾರಸು ಮಾಡಿದ ಅಪ್ಡೇಟ್ ಅನ್ನು ತೋರಿಸುತ್ತದೆ.
* ಸರಿಪಡಿಸಲಾಗಿದೆ: ಇನ್‌ವಿಜಿಬಲ್ ಮೋಡ್ (Invisible mode) ಈಗ `cap_security_level` ಅನ್ನು ಮೌನವಾಗಿ FAST ಗೆ ಕಡಿಮೆ ಮಾಡುವುದಿಲ್ಲ — ಆಡಮಿನ್ ಕಾನ್ಫಿಗರ್ ಮಾಡಿದ ಮಟ್ಟವನ್ನು ಗೌರವಿಸಲಾಗುತ್ತದೆ. ಹೆಚ್ಚು ಕಸ್ಟಮ್ ಲಾಗಿಕ್ ಬೇಕಾದ সাইಟ್‌ಗಳಿಗೆ ಹೊಸ `wu_cap_server_security_level` ಫಿಲ್ಟರ್ ಲಭ್ಯವಿದೆ.
* ಸರಿಪಡಿಸಲಾಗಿದೆ: ಸ್ಟ್ಯಾಟಿಸ್ಟಿಕ್ಸ್ `rate_limits_triggered` ಕೌಂಟರ್ ಈಗ ಪ್ರತಿ ಬ್ಲಾಕ್ ಆದಾಗ ಹೆಚ್ಚಾಗುತ್ತದೆ, ಕೇವಲ ಅಪರೂಪದ ಯಶಸ್ಸಿನ ನಂತರದ ಬ್ಯಾಕ್‌ಸ್ಟಾಪ್ ಮಾರ್ಗದಲ್ಲಿ ಮಾತ್ರವಲ್ಲ.
* ಸರಿಪಡಿಸಲಾಗಿದೆ: `Captcha_Core::get_client_ip()` ಈಗ ರೇಟ್ ಲಿಮಿಟರ್, captcha ಪроваಯರ್‌ಗಳು (reCAPTCHA + hCaptcha `siteverify`), ಮತ್ತು ಸ್ಟ್ಯಾಟಿಸ್ಟಿಕ್ಸ್‌ನಲ್ಲಿ ಭೇಟಿ ನೀಡಿದ IP ನಿಗಮಾವತಿಗೆ ಏಕೈಕ ಮೂಲದ ಸತ್ಯವಾಗಿದೆ — ಇದು ಒಂದು ಸ್ಪೂಫ್ ಮಾಡುವ ವೆಕ್ಟರ್ ಅನ್ನು ಮುಚ್ಚುತ್ತದೆ, ಅಲ್ಲಿ ನೇರ origin-server ವಿನಂತಿಗಳು ವಂಚಿತ `CF-Connecting-IP` ಹೆಡರ್ ಅನ್ನು ಹೊತ್ತಿದ್ದರೆ, ಅದು ನಿಜವಾದ ಪೀರ್ ಬದಲಿಗೆ ಸ್ಪೂಫ್ ಮಾಡಿದ IP ಬಕೆಟ್‌ನಲ್ಲಿ ಬರುತ್ತದೆ.
* ಸರಿಪಡಿಸಲಾಗಿದೆ: WooCommerce ಕ್ಲಾಸಿಕ್ ಚೆಕ್‌ಔಟ್ ರೇಟ್-ಲಿಮಿಟ್ ಗೇಟ್ ಈಗ `woocommerce_before_checkout_form` ಬದಲಿಗೆ `template_redirect` (priority 1) ನಲ್ಲಿ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ. ಈ ಫಾರ್ಮ್-ಮಟ್ಟದ ಹುಕ್ (hook) ಕಾರ್ಟ್ ಖಾಲಿಯಾದಾಗ ಎಂದಿಗೂ ಕಾರ್ಯನಿರ್ವಹಿಸುವುದಿಲ್ಲ, ಆದ್ದರಿಂದ ಉತ್ಪನ್ನವನ್ನು ಸೇರಿಸದ ಫ್ಲಡ್ ಟ್ರಾಫಿಕ್ ಸಂಪೂರ್ಣವಾಗಿ ಲಿಮಿಟರ್ ಅನ್ನು ಬೈಪಾಸ್ ಮಾಡುತ್ತಿತ್ತು.
* ಸರಿಪಡಿಸಲಾಗಿದೆ: WooCommerce pay-for-order ರೇಟ್-ಲಿಮಿಟ್ ಗೇಟ್ ಈಗ `woocommerce_before_pay_action` ಬದಲಿಗೆ `template_redirect` ನಲ್ಲಿ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ. ಕೊನೆಯದು ಕೇವಲ `wp_verify_nonce('woocommerce-pay')` ಯಶಸ್ವಿಯಾದ ನಂತರವೇ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ, ಇದರರ್ಥ ಅಧಿಕೃತವಲ್ಲದ ದಾಳಿಕೋರರು (ನಿಜವಾದ ಬೆದರಿಕೆ ಮಾದರಿ) ಎಂದಿಗೂ ಲಿಮಿಟರ್ ಅನ್ನು ಟ್ರಿಗ್গার ಮಾಡಲಿಲ್ಲ.
* ಸರಿಪಡಿಸಲಾಗಿದೆ: WooCommerce Store API (blocks) ಚೆಕ್‌ಔಟ್ ರೇಟ್-ಲಿಮಿಟ್ ಗೇಟ್ ಈಗ `woocommerce_store_api_checkout_update_order_from_request` ಬದಲಿಗೆ `rest_pre_dispatch` ನಲ್ಲಿ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ. ಕೊನೆಯದು ಕೇವಲ Store API ಕಾರ್ಟ್ ಮತ್ತು ಬಿಲ್ಲಿಂಗ್ ಫೀಲ್ಡ್‌ಗಳನ್ನು ಪರಿಶೀಲಿಸಿದ ನಂತರವೇ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ, ಆದ್ದರಿಂದ ಅಧಿಕೃತವಲ್ಲದ ಬೋಟ್‌ಗಳು ವ್ಯಾಲಿಡೇಟರ್‌ನಿಂದ 400 ಪಡೆದವು ಮತ್ತು ಎಂದಿಗೂ ಲಿಮಿಟರ್ ಅನ್ನು ಟ್ರಿಗ್গার ಮಾಡಲಿಲ್ಲ.
* ಸರಿಪಡಿಸಲಾಗಿದೆ: Ultimate Multisite inline-login ರೇಟ್-ಲಿಮಿಟ್ ಗೇಟ್ ಈಗ `wu_before_inline_login` ಬದಲಿಗೆ `wu_ajax_nopriv_wu_inline_login` priority 1 (ಮತ್ತು ಲಾಗ್-ಇನ್ ಮಾಡಿದ ಮಿರಾರ್) ನಲ್ಲಿ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ. ಕೊನೆಯದು ಕೇವಲ `check_ajax_referer('wu_checkout')` ಯಶಸ್ವಿಯಾದ ನಂತರವೇ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ, ಆದ್ದರಿಂದ ಮಾನ್ಯ wu_checkout nonce ಇಲ್ಲದ ಅಧಿಕೃತವಲ್ಲದ ಬೋಟ್‌ಗಳು 403 ಪಡೆದವು ಮತ್ತು ಎಂದಿಗೂ ಲಿಮಿಟರ್ ಅನ್ನು ಟ್ರಿಗ್গার ಮಾಡಲಿಲ್ಲ.
* ಸರಿಪಡಿಸಲಾಗಿದೆ: `Rate_Limiter::enforce()` ಈಗ `surface|ip` ಮೂಲಕ ಕೀ ಮಾಡಿದ ಒಂದು ಬಾರಿ-ಪ್ರತಿ-ವಿನಂತಿಗೆ (once-per-request) ಗಾರ್ಡ್ ಅನ್ನು ಅನ್ವಯಿಸುತ್ತದೆ, ಆದ್ದರಿಂದ ಮೇಲ್ಭಾಗದ ಹುಕ್‌ಗಳು (upstream hooks) ಪ್ರತಿ ರೆಂಡರ್‌ನಲ್ಲಿ ಎರಡು ಬಾರಿ ಕಾರ್ಯನಿರ್ವಹಿಸುವ (குறிப்பாக Ultimate Multisite ನಲ್ಲಿ `wu_setup_checkout`) ಸಂದರ್ಭದಲ್ಲಿ ಪರಿಣಾಮಕಾರಿ ರೇಟ್-ಲಿಮಿಟ್ ಮಿತಿಯನ್ನು ಅರ್ಧಕ್ಕೆ ಕಡಿತಗೊಳಿಸುವುದಿಲ್ಲ.
* ಸರಿಪಡಿಸಲಾಗಿದೆ: ರೇಟ್-ಲಿಮಿಟ್ ಸರ್ಫೇಸ್ ಗೇಟ್‌ಗಳು ಇನ್ನು ಮುಂದೆ `Captcha_Core::is_whitelisted()` (`wu_captcha_whitelisted` ಫಿಲ್ಟರ್) ಅನ್ನು ಪರಿಶೀಲಿಸುವುದಿಲ್ಲ. ಆ ಫಿಲ್ಟರ್ "captcha ಅನ್ನು ಈಗಾಗಲೇ ಬೇರೆ ಸರ್ಫೇಸ್‌ನಿಂದ ನಿರ್ವಹಿಸಲಾಗಿದೆ" ಎಂದು ಸೂಚಿಸುತ್ತದೆ ಮತ್ತು ಫ್ಲಡ್ ಪ್ರೊಟೆಕ್ಷನ್‌ಗೆ ಲಂಬವಾಗಿದೆ — WooCommerce ಇಂಟಿಗ್ರೇಷನ್ ಒಂದು Woo nonce ಇರುವಾಗ WordPress ಲಾಗಿನ್ captcha ಅನ್ನು ಸ್ಕಿಪ್ ಮಾಡಲು ಇದನ್ನು ಹುಕ್ ಮಾಡಿತ್ತು, ಇದು ರೇಟ್ ಕೌಂಟಿಂಗ್‌ನಲ್ಲಿ ಸೋರಿಕೆಯಾಯಿತು ಮತ್ತು Woo POST ಗಳನ್ನು ಲಿಮಿಟರ್ ಅನ್ನು ತಪ್ಪಿಸಲು ಬಿಟ್ಟಿತು. ರೇಟ್-ಲಿಮಿಟ್-ನಿರ್ದಿಷ್ಟ `wu_cap_rate_limit_whitelist_ip` ಫಿಲ್ಟರ್ ಮಾತ್ರ ಈಗ ಅನ್ವಯಿಸುವ ಬೈಪಾಸ್ ಆಗಿದೆ.

Version: 1.3.2 - ಬಿಡುಗಡೆ ದಿನಾಂಕ: 2026-01-27
* ಸರಿಪಡಿಸಲಾಗಿದೆ: Elementor ಅಥವಾ ಇತರ page builders ಬಳಸುವ ಚೆಕ್‌ಔಟ್ ಫಾರ್ಮ್‌ಗಳಲ್ಲಿ Cap widget ರೆಂಡರ್ ಆಗುತ್ತಿರಲಿಲ್ಲ.
