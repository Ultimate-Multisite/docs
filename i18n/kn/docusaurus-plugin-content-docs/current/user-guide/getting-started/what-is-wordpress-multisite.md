# WordPress Multisite ಎಂದರೇನು?

WordPress ತನ್ನ ಮೂಲ ವ್ಯವಸ್ಥೆಯಲ್ಲಿ 'Multisite' ಎಂಬ ವೈಶಿಷ್ಟ್ಯವನ್ನು ಒದಗಿಸುತ್ತದೆ, ಇದು 2010 ರಲ್ಲಿ WordPress 3.0 ಬಿಡುಗಡೆಯೊಂದಿಗೆ ಆರಂಭವಾಯಿತು. ಅಂದಿನಿಂದ ಹೊಸ ವೈಶಿಷ್ಟ್ಯಗಳನ್ನು ಸೇರಿಸಲು ಮತ್ತು ಭದ್ರತೆಯನ್ನು ಬಲಪಡಿಸಲು ಹಲವಾರು ಸುಧಾರಣೆಗಳನ್ನು ಮಾಡಲಾಗಿದೆ.

ಸರಳವಾಗಿ ಹೇಳುವುದಾದರೆ, WordPress multisite ಅನ್ನು ಈ ರೀತಿ ಅರ್ಥಮಾಡಿಕೊಳ್ಳಬಹುದು: ಒಂದು ವಿಶ್ವವಿದ್ಯಾಲಯವು WordPress ನ ಒಂದೇ ಸ್ಥಾಪನೆಯನ್ನು ನಿರ್ವಹಿಸುತ್ತದೆ, ಆದರೆ ಪ್ರತಿ ವಿಭಾಗವು ತಮ್ಮದೇ ಆದ WordPress ಸೈಟ್ ಅನ್ನು ನಿರ್ವಹಿಸುತ್ತದೆ.

## 

## WordPress Multisite ಎಂದರೆ ನಿಖರವಾಗಿ ಏನು?

Multisite ಎನ್ನುವುದು WordPress ನ ಒಂದು ವೈಶಿಷ್ಟ್ಯವಾಗಿದ್ದು, ಇದು ಬಹು ಸೈಟ್‌ಗಳಿಗೆ ಒಂದೇ WordPress ಸ್ಥಾಪನೆಯನ್ನು ಹಂಚಿಕೊಳ್ಳಲು ಅನುವು ಮಾಡಿಕೊಡುತ್ತದೆ. Multisite ಸಕ್ರಿಯಗೊಳಿಸಿದಾಗ, ಮೂಲ WordPress ಸೈಟ್ ಸಾಮಾನ್ಯವಾಗಿ **ಸೈಟ್‌ಗಳ ನೆಟ್‌ವರ್ಕ್** ಎಂದು ಕರೆಯಲ್ಪಡುವುದನ್ನು ಬೆಂಬಲಿಸಲು ಪರಿವರ್ತನೆಗೊಳ್ಳುತ್ತದೆ.

ಈ ನೆಟ್‌ವರ್ಕ್ ಫೈಲ್ ಸಿಸ್ಟಮ್ ಅನ್ನು ಹಂಚಿಕೊಳ್ಳುತ್ತದೆ (ಅಂದರೆ **plugins ಮತ್ತು themes ಕೂಡ ಹಂಚಿಕೆಯಾಗುತ್ತವೆ**), ಡೇಟಾಬೇಸ್, WordPress core ಫೈಲ್‌ಗಳು, wp-config.php, ಇತ್ಯಾದಿ.

ಇದರರ್ಥ WordPress, theme, ಮತ್ತು plugin ಅಪ್‌ಡೇಟ್‌ಗಳನ್ನು ನಿಮ್ಮ ಎಲ್ಲಾ ನೆಟ್‌ವರ್ಕ್ ಸೈಟ್‌ಗಳಿಗೆ ಒಂದೇ ಬಾರಿ ಮಾಡಬೇಕಾಗುತ್ತದೆ ಏಕೆಂದರೆ ಅವು ಫೈಲ್‌ಸಿಸ್ಟಮ್‌ನಲ್ಲಿ ಒಂದೇ ಫೈಲ್‌ಗಳನ್ನು ಹಂಚಿಕೊಳ್ಳುತ್ತವೆ.

ಈ ಅಂಶವು multisite ನ ಮುಖ್ಯ ಪ್ರಯೋಜನಗಳಲ್ಲಿ ಒಂದಾಗಿದೆ, ಏಕೆಂದರೆ ನಿಮ್ಮ ಗ್ರಾಹಕರ ಸೈಟ್‌ಗಳನ್ನು ನಿರ್ವಹಿಸಲು ಮಾಡಬೇಕಾದ ಕಾರ್ಯಗಳ ಸಂಖ್ಯೆಯನ್ನು ಅದೇ ಮಟ್ಟದಲ್ಲಿ ಇಟ್ಟುಕೊಂಡು ನಿಮ್ಮ ಸೈಟ್‌ಗಳ ಸಂಖ್ಯೆಯನ್ನು ಹೆಚ್ಚಿಸಲು ಇದು ನಿಮಗೆ ಅನುವು ಮಾಡಿಕೊಡುತ್ತದೆ.

## 

## Subdomain ಅಥವಾ Subdirectory?

WordPress multisite ಅನ್ನು ಚಲಾಯಿಸಲು ಎರಡು ವಿಧಾನಗಳಿವೆ – ಮತ್ತು ನಿಮ್ಮ ಸಾಮಾನ್ಯ WordPress ಸ್ಥಾಪನೆಯನ್ನು multisite ಆಗಿ ಪರಿವರ್ತಿಸುವಾಗ ನೀವು ಒಂದನ್ನು ಆಯ್ಕೆ ಮಾಡಬೇಕು:

**Subdomain:** ಉದಾ.: [site.domain.com](http://site.domain.com)

…ಅಥವಾ

**Subdirectory:** ಉದಾ.: [yourdomain.com/site](http://yourdomain.com/site)

ಪ್ರತಿ ವಿಧಾನಕ್ಕೂ ಅನುಕೂಲಗಳು ಮತ್ತು ಅನಾನುಕೂಲಗಳಿವೆ, ಈ ನಿರ್ಧಾರ ತೆಗೆದುಕೊಳ್ಳುವಾಗ ನೀವು ಅವುಗಳನ್ನು ಪರಿಗಣಿಸಬೇಕು.

ಆದಾಗ್ಯೂ, ಒಂದು ಮುಖ್ಯ ವಿಷಯವನ್ನು ಗಮನಿಸಬೇಕು: ಒಮ್ಮೆ ನೀವು ನಿರ್ಧಾರ ತೆಗೆದುಕೊಂಡ ನಂತರ, ನಿಮ್ಮ ನೆಟ್‌ವರ್ಕ್ ಅನ್ನು subdirectory ಯಿಂದ subdomain ಗೆ ಅಥವಾ ಪ್ರತಿಯಾಗಿ ಬದಲಾಯಿಸುವುದು ತುಂಬಾ ಕಷ್ಟ – ವಿಶೇಷವಾಗಿ ನೀವು ಈಗಾಗಲೇ ಕೆಲವು ಸೈಟ್‌ಗಳನ್ನು ರಚಿಸಿದ್ದರೆ.

ಆ ನಿರ್ಧಾರ ತೆಗೆದುಕೊಳ್ಳುವ ಮೊದಲು, ಇಲ್ಲಿ ಕೆಲವು ಅಂಶಗಳನ್ನು ಗಮನದಲ್ಲಿಡಿ:

**Subdirectory Mode** ಸ್ಥಾಪನೆ ಮತ್ತು ನಿರ್ವಹಣೆಯ ವಿಷಯದಲ್ಲಿ ಸುಲಭವಾದ ವಿಧಾನವಾಗಿದೆ. ಏಕೆಂದರೆ ಎಲ್ಲಾ ಸೈಟ್‌ಗಳು ಕೇವಲ ಮುಖ್ಯ domain ಗೆ ಲಗತ್ತಿಸಲಾದ paths ಆಗಿವೆ (ಉದಾ. [yourdomain.com/subsite](http://yourdomain.com/subsite)). ಇದರ ಪರಿಣಾಮವಾಗಿ, ನಿಮಗೆ ಮುಖ್ಯ domain ಗೆ **ಒಂದೇ SSL certificate** ಅಗತ್ಯವಿದೆ ಮತ್ತು ಅದು ಇಡೀ ನೆಟ್‌ವರ್ಕ್ ಅನ್ನು ಒಳಗೊಳ್ಳುತ್ತದೆ.

ಅದೇ ಸಮಯದಲ್ಲಿ, ಅದರ URL ರಚನೆಯಿಂದಾಗಿ, Google ಮತ್ತು ಇತರ ಹೆಚ್ಚಿನ search engines ನಿಮ್ಮ subdirectory-ಆಧಾರಿತ ನೆಟ್‌ವರ್ಕ್‌ನಲ್ಲಿರುವ ಎಲ್ಲಾ subsites ಅನ್ನು ಒಂದೇ ದೊಡ್ಡ ಸೈಟ್ ಎಂದು ಪರಿಗಣಿಸುತ್ತವೆ. ಇದರ ಪರಿಣಾಮವಾಗಿ, ನಿಮ್ಮ ಗ್ರಾಹಕರು subsites ಗೆ ಸೇರಿಸಿದ ವಿಷಯವು ನಿಮ್ಮ landing site ನ SEO ಕಾರ್ಯಕ್ಷಮತೆಯ ಮೇಲೆ ಪರಿಣಾಮ ಬೀರಬಹುದು. ಈ ಪ್ರಭಾವದ ಮಟ್ಟ ಚರ್ಚಾಸ್ಪದವಾಗಿದೆ ಮತ್ತು ಇಂತಹ ವ್ಯವಸ್ಥೆಯು SEO ಕಾರ್ಯಕ್ಷಮತೆಗೆ ಪ್ರಯೋಜನಕಾರಿಯಾಗಬಹುದು ಎಂಬ ವಾದವೂ ಇದೆ.

**Subdomain Mode** ಸ್ಥಾಪಿಸಲು ಸ್ವಲ್ಪ ಹೆಚ್ಚು ಸಂಕೀರ್ಣವಾಗಿದೆ, ಆದರೆ ಅದರ URL ರಚನೆ (ಉದಾ. [subsite.yournetwork.com](http://subsite.yournetwork.com)) ಸಾಮಾನ್ಯವಾಗಿ "ಹೆಚ್ಚು ವೃತ್ತಿಪರ" ಎಂದು ಕಾಣುತ್ತದೆ.

Subdomain mode ಸ್ಥಾಪಿಸುವಲ್ಲಿನ ಮುಖ್ಯ ಸವಾಲುಗಳಲ್ಲಿ ಒಂದು ಇಡೀ ನೆಟ್‌ವರ್ಕ್‌ಗೆ SSL ರಕ್ಷಣೆ (HTTPS) ಆಗಿದೆ. ಬ್ರೌಸರ್‌ಗಳು subdomains ಅನ್ನು ಪ್ರತ್ಯೇಕ ಘಟಕಗಳೆಂದು ಪರಿಗಣಿಸುತ್ತವೆ ಎಂಬ ಅಂಶಕ್ಕೆ ಇದು ಬರುತ್ತದೆ. ಇದರ ಪರಿಣಾಮವಾಗಿ, ನಿಮ್ಮ ನೆಟ್‌ವರ್ಕ್‌ನಲ್ಲಿರುವ ಪ್ರತಿ subdomain ಗೆ ವಿಭಿನ್ನ SSL certificate ಬೇಕಾಗುತ್ತದೆ, ಅಥವಾ **Wildcard SSL certificate** ಎಂಬ ವಿಶೇಷ ರೀತಿಯ certificate ಬೇಕಾಗುತ್ತದೆ. ಇತ್ತೀಚಿನ ವರ್ಷಗಳಲ್ಲಿ, hosting providers ಮತ್ತು panels SSL provisioning ವಿಷಯದಲ್ಲಿ ತಮ್ಮ ಸಾಮರ್ಥ್ಯವನ್ನು ಹೆಚ್ಚಿಸುತ್ತಿವೆ ಮತ್ತು ಕೆಲವು ಒಂದೇ ಕ್ಲಿಕ್‌ನಲ್ಲಿ wildcard certificates ಅನ್ನು ನೀಡುತ್ತವೆ, ಎರಡು ವಿಧಾನಗಳ ನಡುವಿನ ಸ್ಥಾಪನೆಯ ಸಂಕೀರ್ಣತೆಯ ಅಂತರವನ್ನು ಕಡಿಮೆ ಮಾಡುತ್ತವೆ.

Subdirectory mode ಗೆ ವ್ಯತಿರಿಕ್ತವಾಗಿ, subdomain-ಆಧಾರಿತ ನೆಟ್‌ವರ್ಕ್‌ನಲ್ಲಿರುವ subsites ಅನ್ನು search engines ಪ್ರತ್ಯೇಕ websites ಆಗಿ ಪರಿಗಣಿಸುತ್ತವೆ, ಅಂದರೆ ಒಂದು subsite ನಲ್ಲಿರುವ ವಿಷಯವು ಇತರ subsites ನ SEO ಕಾರ್ಯಕ್ಷಮತೆಯ ಮೇಲೆ ಯಾವುದೇ ಪ್ರಭಾವ ಬೀರುವುದಿಲ್ಲ.

## Super Admin

Single-site WordPress ಸ್ಥಾಪನೆಗಳು ಅನಿಯಮಿತ ಬಳಕೆದಾರರನ್ನು ಸೇರಿಸಲು ಮತ್ತು ಆ ಬಳಕೆದಾರರಿಗೆ ವಿಭಿನ್ನ ಅನುಮತಿಗಳೊಂದಿಗೆ ವಿಭಿನ್ನ user roles ನೀಡಲು ನಿಮಗೆ ಅನುಮತಿಸುತ್ತವೆ.

WordPress Multisite ನಲ್ಲಿ, ಹೊಸ ರೀತಿಯ ಬಳಕೆದಾರ ಅನ್‌ಲಾಕ್ ಆಗುತ್ತಾರೆ: **super admin** – ಮತ್ತು ಹೊಸ admin panel ಅನ್‌ಲಾಕ್ ಆಗುತ್ತದೆ: **network admin panel**.

ಹೆಸರೇ ಸೂಚಿಸುವಂತೆ, super admin ನೆಟ್‌ವರ್ಕ್ ಮೇಲೆ ಮಹಾಶಕ್ತಿಗಳನ್ನು ಹೊಂದಿದ್ದು, ಅದರ ಎಲ್ಲಾ subsites, plugins, themes, ಎಲ್ಲವನ್ನೂ ನಿರ್ವಹಿಸುವ ಸಾಮರ್ಥ್ಯ ಹೊಂದಿದ್ದಾರೆ!

ನಿಮ್ಮ single-site WordPress ಸ್ಥಾಪನೆಯನ್ನು multisite ಆಗಿ ಪರಿವರ್ತಿಸಿದ ನಂತರ, single site ನ ಮೂಲ admin ಸ್ವಯಂಚಾಲಿತವಾಗಿ super admin ಆಗಿ ಅಪ್‌ಗ್ರೇಡ್ ಆಗುತ್ತಾರೆ.

Plugins ಮತ್ತು themes ಅನ್ನು super admins ಮಾತ್ರ network admin panel ನಿಂದ install ಅಥವಾ uninstall ಮಾಡಬಹುದು. Subsite admins ನಂತರ ಆ plugins ಅಥವಾ themes ಅನ್ನು activate ಅಥವಾ deactivate ಮಾಡಲು ಆಯ್ಕೆ ಮಾಡಬಹುದು, super admin ಒಂದು plugin ಅನ್ನು network activate ಮಾಡದ ಹೊರತು, ಅದು ಎಲ್ಲಾ subsites ಗೆ ಯಾವಾಗಲೂ ಸಕ್ರಿಯವಾಗಿರುವಂತೆ ಮಾಡುತ್ತದೆ.

_ಟಿಪ್ಪಣಿ: ನೀವು ನೋಡುವಂತೆ, ಯಾರನ್ನಾದರೂ ನಿಮ್ಮ ನೆಟ್‌ವರ್ಕ್‌ಗೆ ಆಹ್ವಾನಿಸಿ ಅವರಿಗೆ super admin ಸ್ಥಾನಮಾನ ನೀಡುವುದು ಈ ಬಳಕೆದಾರರಿಗೆ ನಿಮ್ಮ ನೆಟ್‌ವರ್ಕ್ ಮೇಲೆ ಸಂಪೂರ್ಣ ನಿಯಂತ್ರಣ ನೀಡುತ್ತದೆ. ಉದಾಹರಣೆಗೆ, ಇತರ super admins ನಿಮ್ಮ super admin ಸ್ಥಾನಮಾನವನ್ನು ತೆಗೆದುಹಾಕಬಹುದು, ನಿಮ್ಮ ಸ್ವಂತ network admin panel ನಿಂದ ನಿಮ್ಮನ್ನು ಹೊರಗಿಡಬಹುದು. Ultimate Multisite ಗ್ರಾಹಕರಿಗೆ ಹೆಚ್ಚುವರಿ super admins ಏನು ಮಾಡಬಹುದು ಎಂಬುದರ ಮೇಲೆ ಸೂಕ್ಷ್ಮ ನಿಯಂತ್ರಣವನ್ನು ಹೊಂದಲು ಅನುಮತಿಸಲು, ನಮ್ಮಲ್ಲಿ Support Agents ಎಂಬ add-on ಇದೆ. ಈ add-on ಇನ್ನೊಂದು ರೀತಿಯ ಬಳಕೆದಾರರನ್ನು ರಚಿಸಲು ನಿಮಗೆ ಅನುವು ಮಾಡಿಕೊಡುತ್ತದೆ – agent – ನೆಟ್‌ವರ್ಕ್‌ನಲ್ಲಿ ತಮ್ಮ ಕಾರ್ಯಗಳನ್ನು ನಿರ್ವಹಿಸಲು ಅಗತ್ಯವಿರುವ ಅನುಮತಿಗಳೊಂದಿಗೆ ಮಾತ್ರ._

## Subsites ನಡುವೆ ಏನು ಹಂಚಿಕೆಯಾಗುತ್ತದೆ ಮತ್ತು ಏನು ಆಗುವುದಿಲ್ಲ

ನಾವು ಮೊದಲು ಹೇಳಿದಂತೆ, WordPress multisite ನ ಪ್ರಮುಖ ಪ್ರಯೋಜನಗಳಲ್ಲಿ ಒಂದು ಎಂದರೆ ಎಲ್ಲಾ subsites ಒಂದೇ configurations, core files, themes, plugins, WordPress core files, ಇತ್ಯಾದಿಗಳನ್ನು ಹಂಚಿಕೊಳ್ಳುತ್ತವೆ.

ಆದಾಗ್ಯೂ, ಪ್ರತಿ subsite ಆಧಾರದಲ್ಲಿ ಸುಂದರವಾಗಿ ಬೇರ್ಪಡಿಸಲಾದ ಅಂಶಗಳಿವೆ.

\- ಉದಾಹರಣೆಗೆ, ಪ್ರತಿ subsite ತನ್ನದೇ ಆದ uploads folder ಅನ್ನು ಪಡೆಯುತ್ತದೆ. ಪರಿಣಾಮವಾಗಿ, ಒಂದು ನಿರ್ದಿಷ್ಟ subsite ನ ಬಳಕೆದಾರರು ಅಪ್‌ಲೋಡ್ ಮಾಡಿದ ಫೈಲ್‌ಗಳನ್ನು ಇನ್ನೊಂದು subsite ನಲ್ಲಿ ಪ್ರವೇಶಿಸಲು ಸಾಧ್ಯವಿಲ್ಲ.

\- ಪ್ರತಿ subsite ತನ್ನದೇ ಆದ ಮೀಸಲಾದ admin panel ಅನ್ನು ಹೊಂದಿದೆ ಮತ್ತು super admin network active ಮಾಡದ ಹೊರತು plugins ಅಥವಾ themes ಅನ್ನು activate ಅಥವಾ deactivate ಮಾಡಬಹುದು.

\- ಹೆಚ್ಚಿನ database tables ಪ್ರತಿ subsite ಗೆ ರಚಿಸಲ್ಪಡುತ್ತವೆ, ಅಂದರೆ posts, comments, pages, settings, ಮತ್ತು ಹೆಚ್ಚಿನವು ಪ್ರತಿ subsite ಗೆ ಸೀಮಿತವಾಗಿರುತ್ತವೆ.

## WordPress Multisite ನಲ್ಲಿ ಬಳಕೆದಾರ ನಿರ್ವಹಣೆ

WordPress multisite ನಲ್ಲಿ ಒಂದು ಸೂಕ್ಷ್ಮ ವಿಷಯವೆಂದರೆ ಬಳಕೆದಾರ ನಿರ್ವಹಣೆ. WordPress user table ಎಲ್ಲಾ subsites ನಡುವೆ ಹಂಚಿಕೆಯಾಗುವ ಕೆಲವೇ ಕೆಲವು tables ಗಳಲ್ಲಿ ಒಂದಾಗಿದೆ.

ನಿಮ್ಮ ನೆಟ್‌ವರ್ಕ್‌ನಲ್ಲಿ ಏನು ನಿರ್ಮಿಸಲು ಯೋಜಿಸುತ್ತಿದ್ದೀರಿ ಎಂಬುದರ ಆಧಾರದ ಮೇಲೆ ಈ ವ್ಯವಸ್ಥೆಯು ಕೆಲವು ಸಮಸ್ಯೆಗಳನ್ನು ಉಂಟುಮಾಡಬಹುದು. ಕೆಳಗಿನ ಉದಾಹರಣೆಯು ಅತ್ಯಂತ ಒತ್ತುವ ಸಮಸ್ಯೆಯನ್ನು ವಿವರಿಸಲು ಸಹಾಯ ಮಾಡುತ್ತದೆ.

ಈ ಸನ್ನಿವೇಶವನ್ನು ಊಹಿಸಿ:

ನೀವು WordPress multisite ನೆಟ್‌ವರ್ಕ್ ರಚಿಸುತ್ತೀರಿ ಮತ್ತು e-commerce ಸ್ಟೋರ್ ಹೊಂದಲು ಬಯಸುವ ಜನರಿಗೆ ಮಾಸಿಕ ಶುಲ್ಕಕ್ಕೆ subsites ನೀಡಲು ಪ್ರಾರಂಭಿಸುತ್ತೀರಿ.

ನಿಮ್ಮ ಮೊದಲ ಪಾವತಿಸುವ ಗ್ರಾಹಕ – John ಬರುತ್ತಾರೆ. ನೀವು ನಿಮ್ಮ ನೆಟ್‌ವರ್ಕ್‌ನಲ್ಲಿ John ಗೆ ಒಂದು site ರಚಿಸುತ್ತೀರಿ, ಅಗತ್ಯವಿರುವ ಎಲ್ಲಾ plugins install ಮಾಡುತ್ತೀರಿ, ನಂತರ John ತಮ್ಮ store ನಿರ್ವಹಿಸಲು ಒಂದು user ರಚಿಸುತ್ತೀರಿ.

ನಂತರ ಎರಡನೇ ಗ್ರಾಹಕ – Alice ಬರುತ್ತಾರೆ. ನೀವು ಅವರಿಗೂ ಅದೇ ರೀತಿ ಮಾಡುತ್ತೀರಿ ಮತ್ತು ಈಗ ಅವರು ನಿಮ್ಮ ನೆಟ್‌ವರ್ಕ್‌ನಲ್ಲಿ ಒಂದು store ಹೊಂದಿದ್ದಾರೆ.

John ಮತ್ತು Alice ಇಬ್ಬರೂ ನಿಮ್ಮ ಗ್ರಾಹಕರು, ಆದರೆ ಅವರು ಪರಸ್ಪರ ಪರಿಚಯವಿಲ್ಲ. ಮುಖ್ಯವಾಗಿ, ಅವರಲ್ಲಿ ಒಬ್ಬರು ಇನ್ನೊಬ್ಬರ store website ಗೆ ಭೇಟಿ ನೀಡಿದರೆ, ಈ store ಅದೇ sites ನೆಟ್‌ವರ್ಕ್‌ನಲ್ಲಿ host ಆಗಿದೆ ಎಂದು ತಿಳಿಯಲು ಯಾವುದೇ ಮಾರ್ಗವಿಲ್ಲ.

ಒಂದು ದಿನ, John ಗೆ ಹೊಸ ಜೊತೆ ಶೂಗಳು ಬೇಕಾಗುತ್ತವೆ ಮತ್ತು Alice ನ store ನಲ್ಲಿ ಪರಿಪೂರ್ಣವಾದವುಗಳನ್ನು ಕಂಡುಕೊಳ್ಳುತ್ತಾರೆ. ಖರೀದಿಯನ್ನು ಪೂರ್ಣಗೊಳಿಸಲು ಪ್ರಯತ್ನಿಸಿದಾಗ, "email already in use" ದೋಷ ಸಂದೇಶ ಬರುತ್ತದೆ, ಇದು ವಿಚಿತ್ರವಾಗಿದೆ ಏಕೆಂದರೆ John ಇದು Alice ನ website ಗೆ ಮೊದಲ ಭೇಟಿ ಎಂದು 100% ಖಚಿತವಾಗಿದ್ದಾರೆ.

ಇಲ್ಲಿ ಏನಾಯಿತೆಂದರೆ John ನ user ಇಡೀ ನೆಟ್‌ವರ್ಕ್‌ನಾದ್ಯಂತ ಹಂಚಿಕೆಯಾಗಿದೆ, ಆದ್ದರಿಂದ Alice ನ site ನಲ್ಲಿ checkout ಮಾಡಲು account ರಚಿಸಲು ಪ್ರಯತ್ನಿಸಿದಾಗ, WordPress ಅದೇ email address ನೊಂದಿಗೆ user ಈಗಾಗಲೇ ಅಸ್ತಿತ್ವದಲ್ಲಿದೆ ಎಂದು ಪತ್ತೆಹಚ್ಚಿ ದೋಷ ತೋರಿಸುತ್ತದೆ.

_ಟಿಪ್ಪಣಿ: ನಿಮ್ಮ ಬಳಕೆಯ ಸಂದರ್ಭವನ್ನು ಅವಲಂಬಿಸಿ ಇದು ಎಷ್ಟು ಕೆಟ್ಟದಾಗಬಹುದು ಎಂಬುದನ್ನು ನಾವು ಅರಿತುಕೊಂಡಿದ್ದೇವೆ, ಆದ್ದರಿಂದ Ultimate Multisite ಅಸ್ತಿತ್ವದಲ್ಲಿರುವ user ಗಾಗಿ ಸಾಮಾನ್ಯ ಪರಿಶೀಲನೆಗಳನ್ನು ಬೈಪಾಸ್ ಮಾಡುವ ಆಯ್ಕೆಯನ್ನು ಹೊಂದಿದೆ, ಇದು ಅದೇ email address ಬಳಸಿ ಬಹು accounts ರಚಿಸಲು ಅನುಮತಿಸುತ್ತದೆ. ಪ್ರತಿ account ಒಂದು subsite ಗೆ ಬದ್ಧವಾಗಿದೆ, ಆದ್ದರಿಂದ ಘರ್ಷಣೆಯ ಅಪಾಯ ಕನಿಷ್ಠವಾಗಿರುತ್ತದೆ. ಮೇಲಿನ ಉದಾಹರಣೆಯಲ್ಲಿ, John ಗೆ ದೋಷ ಸಂದೇಶ ಬರುತ್ತಿರಲಿಲ್ಲ ಮತ್ತು ಸಮಸ್ಯೆಯಿಲ್ಲದೆ ಆ ಶೂಗಳನ್ನು ಖರೀದಿಸಲು ಸಾಧ್ಯವಾಗುತ್ತಿತ್ತು. ಈ ಆಯ್ಕೆಯನ್ನು Enable Multiple Accounts ಎಂದು ಕರೆಯಲಾಗುತ್ತದೆ, ಮತ್ತು Ultimate Multisite → Settings → Login & Registration ನಲ್ಲಿ ಸಕ್ರಿಯಗೊಳಿಸಬಹುದು._

User table ಹಂಚಿಕೆಯಾಗಿದ್ದರೂ, ಬಳಕೆದಾರರನ್ನು subsite admins ಅಥವಾ super admin ಮೂಲಕ subsites ಗೆ ಸೇರಿಸಬಹುದು ಮತ್ತು ತೆಗೆದುಹಾಕಬಹುದು, ಮತ್ತು ಅವರು ವಿಭಿನ್ನ subsites ನಲ್ಲಿ ವಿಭಿನ್ನ user roles ಹೊಂದಬಹುದು.

## ಕಾರ್ಯಕ್ಷಮತೆ ಪರಿಗಣನೆಗಳು

WordPress multisite ಬೆಂಬಲಿಸಬಹುದಾದ sites ಸಂಖ್ಯೆಯ ವಿಷಯದಲ್ಲಿ ನಿಜವಾಗಿಯೂ ಶಕ್ತಿಶಾಲಿಯಾಗಿದೆ. [WordPress.com](https://WordPress.com), Edublogs, ಮತ್ತು Campuspress ಎಲ್ಲವೂ multisite-ಆಧಾರಿತ ಸೇವೆಗಳಾಗಿವೆ ಮತ್ತು ಪ್ರತಿಯೊಂದೂ ಸಾವಿರಾರು sites ಅನ್ನು host ಮಾಡುತ್ತವೆ ಎಂಬ ಅಂಶದಿಂದ ಇದನ್ನು ಪರೀಕ್ಷಿಸಬಹುದು.

ಸಿದ್ಧಾಂತದಲ್ಲಿ ಒಂದೇ WordPress multisite ಸ್ಥಾಪನೆಯಲ್ಲಿ host ಮಾಡಬಹುದಾದ sites ಗೆ ಗರಿಷ್ಠ ಸಂಖ್ಯೆ ಇಲ್ಲದಿದ್ದರೂ, ಪ್ರಾಯೋಗಿಕವಾಗಿ ನೀವು ತೃಪ್ತಿಕರವಾಗಿ ಚಲಾಯಿಸಬಹುದಾದ sites ಸಂಖ್ಯೆಯು ಹಲವಾರು ವಿಭಿನ್ನ ಅಂಶಗಳ ಆಧಾರದ ಮೇಲೆ ವ್ಯಾಪಕವಾಗಿ ಬದಲಾಗಬಹುದು: sites ಎಷ್ಟು dynamic ಆಗಿವೆ, subsites ಗೆ ಯಾವ plugins ಲಭ್ಯವಿವೆ, ಇತ್ಯಾದಿ.

ಸಾಮಾನ್ಯ ನಿಯಮವಾಗಿ, ನಿಮ್ಮ ನೆಟ್‌ವರ್ಕ್ ಸರಳವಾಗಿದ್ದಷ್ಟೂ ಒಳ್ಳೆಯದು. ವಿಷಯವು ನಿಜವಾಗಿಯೂ dynamic ಅಲ್ಲದ sites ಗೆ ಆದ್ಯತೆ ನೀಡುವುದು (ಇದು ಅವುಗಳನ್ನು aggressive caching strategies ಗೆ ಉತ್ತಮ ಅಭ್ಯರ್ಥಿಗಳನ್ನಾಗಿ ಮಾಡುತ್ತದೆ) ಮತ್ತು plugin stack ಅನ್ನು ಸಾಧ್ಯವಾದಷ್ಟು ಹಗುರವಾಗಿ ಇಟ್ಟುಕೊಳ್ಳುವುದು (active plugins ಸಂಖ್ಯೆ ಕಡಿಮೆಯಾದಷ್ಟೂ ಒಳ್ಳೆಯದು) ನೀವು host ಮಾಡಬಹುದಾದ subsites ಸಂಖ್ಯೆಯನ್ನು ಗಣನೀಯವಾಗಿ ಹೆಚ್ಚಿಸಬಹುದು.

ಅತ್ಯುತ್ತಮ ಭಾಗವೆಂದರೆ ಇದೆಲ್ಲವೂ WordPress ಆಗಿರುವುದರಿಂದ, ಕಾರ್ಯಕ್ಷಮತೆ ಸುಧಾರಣೆಗಳಿಗಾಗಿ ನೀವು ಈಗಾಗಲೇ ತಿಳಿದಿರುವ ಮತ್ತು ಇಷ್ಟಪಡುವ ಅದೇ ಪರಿಕರಗಳು multisite ನೆಟ್‌ವರ್ಕ್‌ಗೂ ಕೆಲಸ ಮಾಡುತ್ತವೆ.

Multisite ಗೆ ಮುಖ್ಯ ಅಡಚಣೆ database ಆಗಿದೆ ಆದರೆ ಎಲ್ಲವೂ ಸರಿಯಾಗಿ ಹೊಂದಿಸಿದ್ದರೆ, ನೀವು ಅದರ ಬಗ್ಗೆ ಚಿಂತಿಸುವ ಮೊದಲು ಕೆಲವು ಸಾವಿರ sites ಆಗಬಹುದು. ಆಗಲೂ, ಆ ಹಂತದಲ್ಲಿ ಕ್ರಮೇಣ ಸೇರಿಸಬಹುದಾದ ಪರಿಹಾರಗಳಿವೆ (ಉದಾಹರಣೆಗೆ database sharding ಪರಿಹಾರಗಳು).
