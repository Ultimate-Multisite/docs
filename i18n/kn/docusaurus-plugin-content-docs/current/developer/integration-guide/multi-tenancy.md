# ಬಹು-ಬಾಡಿಗೆದಾರಿಕೆ ಏಕೀಕರಣ {#multi-tenancy-integration}

Ultimate Multisite: ಬಹು-ಬಾಡಿಗೆದಾರಿಕೆ 1.2.0 ಸ್ವಾಯತ್ತ ಬಾಡಿಗೆದಾರರು, ವಲಸೆ ಪರಿಶೀಲನೆ ಮತ್ತು ಬಾಡಿಗೆದಾರ ಜೀವನಚಕ್ರ ಸ್ವಯಂಚಾಲನೆಗಾಗಿ ಹಲವಾರು ಏಕೀಕರಣ ಸಂಪರ್ಕ ಬಿಂದುಗಳನ್ನು ಬದಲಿಸುತ್ತದೆ.

## ಬಾಡಿಗೆದಾರ ಆರಂಭಿಕ ಹರಿವು {#tenant-bootstrap-flow}

ಬಾಡಿಗೆದಾರರನ್ನು ರಚಿಸುವ ಅಥವಾ ಬದಲಿಸುವ ಏಕೀಕರಣಗಳು ಈ ಕ್ರಮವನ್ನು ಅನುಸರಿಸಬೇಕು:

1. ಬಾಡಿಗೆದಾರ ನೋಂದಣಿ ದಾಖಲೆಯನ್ನು ಮತ್ತು ಪ್ರತ್ಯೇಕತಾ ಮಾದರಿಯನ್ನು ನಿರ್ಧರಿಸಿ.
2. ಬಾಡಿಗೆದಾರ ಡೇಟಾಬೇಸ್ ಬರಹಗಾರನನ್ನು ರಚಿಸಿ ಅಥವಾ ಪರಿಶೀಲಿಸಿ.
3. ಬಾಡಿಗೆದಾರ ಸ್ಕೀಮಾವನ್ನು ಆರಂಭಿಸಿ.
4. ಬಾಡಿಗೆದಾರ ಬಳಕೆದಾರರನ್ನು ಒದಗಿಸಿ.
5. ಬಾಡಿಗೆದಾರ ರೌಟಿಂಗ್ ಮತ್ತು ಫೈಲ್‌ಸಿಸ್ಟಮ್ ಮಾರ್ಗಗಳನ್ನು ನೋಂದಾಯಿಸಿ.
6. ಬಾಡಿಗೆದಾರರನ್ನು ಬಹಿರಂಗಪಡಿಸುವ ಮೊದಲು ವಲಸೆ ಪರಿಶೀಲನೆ ನಡೆಸಿ.

ಸ್ವಾಯತ್ತ ಬಾಡಿಗೆದಾರನು ನೆಟ್‌ವರ್ಕ್ ಡೇಟಾಬೇಸ್ ಸಂಪರ್ಕವನ್ನು ಮರುಬಳಸಬಹುದು ಎಂದು ಊಹಿಸಬೇಡಿ. ಆಡ್‌ಆನ್ ಒದಗಿಸುವ ಬಾಡಿಗೆದಾರ ನೋಂದಣಿ ಮತ್ತು ಬರಹಗಾರ ಅಭ್ಯಾಸಗಳನ್ನು ಬಳಸಿ.

## SSO ಮತ್ತು REST hooks {#sso-and-rest-hooks}

ಸ್ಥಿತಿರಹಿತ ಬಾಡಿಗೆದಾರ ಸ್ವಯಂ ಲಾಗಿನ್ ಅಲ್ಪಾವಧಿಯ tokens ಅನ್ನು ಬಳಸುತ್ತದೆ. ಅವುಗಳಲ್ಲಿ purpose claim, JTI replay protection, expiry cap ಮತ್ತು origin pinning ಇರುತ್ತವೆ. ಲಾಗಿನ್ ಬಟನ್‌ಗಳು ಅಥವಾ ದೂರಸ್ಥ ನಿರ್ವಹಣಾ ಲಿಂಕ್‌ಗಳನ್ನು ಸೇರಿಸುವ ಏಕೀಕರಣಗಳು, ಬಾಡಿಗೆದಾರ ಲಾಗಿನ್ URLಗಳನ್ನು ನೇರವಾಗಿ ರಚಿಸುವ ಬದಲು, ಬೆಂಬಲಿತ SSO ಹರಿವಿನ ಮೂಲಕ ಬಾಡಿಗೆದಾರ ಭೇಟಿಗಳನ್ನು ರಚಿಸಬೇಕು.

ಸ್ವಾಯತ್ತ ಬಾಡಿಗೆದಾರ ಗೇಟ್‌ವೇಗಳಿಗಾಗಿ ನೆಟ್‌ವರ್ಕ್-ಬದಿಯ API ಪರಿಶೀಲನಾ ಘಟನೆಗಳು ಮತ್ತು ದೈನಂದಿನ ಸಾರಾಂಶಗಳು ಲಭ್ಯವಿವೆ. ಬಾಡಿಗೆದಾರ ಜೀವನಚಕ್ರ endpointಗಳನ್ನು ಕರೆ ಮಾಡುವ ಬಾಹ್ಯ ವ್ಯವಸ್ಥೆಗಳನ್ನು ದೋಷನಿವಾರಣೆ ಮಾಡುವಾಗ ಆ ದಾಖಲೆಗಳನ್ನು ಬಳಸಿ.

## ಸ್ವಾಯತ್ತ ಗ್ರಾಹಕ ಕ್ರಿಯೆ URLಗಳು {#sovereign-customer-action-urls}

Ultimate Multisite v2.13.0 ಸ್ವಾಯತ್ತ-ಬಾಡಿಗೆದಾರ ಗ್ರಾಹಕ ಕ್ರಿಯೆಗಳನ್ನು Account, ಚೆಕ್‌ಔಟ್, ಬಿಲ್ಲಿಂಗ್, ಇನ್‌ವಾಯ್ಸ್, ಸೈಟ್, ಟೆಂಪ್ಲೇಟ್ ಬದಲಾವಣೆ ಮತ್ತು ಡೊಮೇನ್ ಮ್ಯಾಪಿಂಗ್ ಹರಿವುಗಳಿಗಾಗಿ ಮುಖ್ಯ ಸೈಟ್‌ಗೆ ಮರಳಿ ಕಳುಹಿಸುತ್ತದೆ. ಬಾಡಿಗೆದಾರ-ಬದಿಯ ನಿರ್ವಹಣಾ ಲಿಂಕ್‌ಗಳನ್ನು ಪ್ರದರ್ಶಿಸುವ ಏಕೀಕರಣಗಳು ಆ ಕ್ರಿಯೆಗಳನ್ನು ಮುಖ್ಯ-ಸೈಟ್ ಗ್ರಾಹಕ ಫಲಕಕ್ಕೆ ಕಳುಹಿಸಬೇಕು. ಕ್ರಿಯೆ ಪೂರ್ಣಗೊಂಡ ನಂತರ ಬಳಕೆದಾರನು ಬಾಡಿಗೆದಾರನಿಗೆ ಮರಳಲು ಸಾಧ್ಯವಾಗಬೇಕಾದರೆ, ಪರಿಶೀಲಿತ ಮರಳುವ ಗುರಿಯನ್ನೂ ಸೇರಿಸಬೇಕು.

ಕ್ರಾಸ್-ಡೊಮೇನ್ ನಿರ್ವಹಣಾ ಲಿಂಕ್‌ಗಳಿಗೆ core SSO wrapper ಬಳಸಿ:

```php
$url = wu_with_sso($main_site_customer_url);
```

ರಚಿಸಲಾದ URL ಅನ್ನು `wu_sso_url` ಮೂಲಕ ಇನ್ನೂ ಫಿಲ್ಟರ್ ಮಾಡಬಹುದು. ಅದು SSO URL, ಪ್ರಸ್ತುತ ಬಳಕೆದಾರ, ಗುರಿ ಸೈಟ್ ID ಮತ್ತು ಮರುನಿರ್ದೇಶನ ಸಂದರ್ಭವನ್ನು ಪಡೆಯುತ್ತದೆ. Ultimate Multisite ನ token ಪರಿಶೀಲನೆಯನ್ನು ಉಳಿಸಿಕೊಂಡೇ, ಒದಗಿಸುವವರಿಗೆ-ನಿರ್ದಿಷ್ಟವಾದ ಸಂದರ್ಭವನ್ನು ಸೇರಿಸಲು ಅಥವಾ broker URL ಅನ್ನು ಬದಲಿಸಲು ಆಡ್‌ಆನ್‌ಗಳು ಆ ಫಿಲ್ಟರ್ ಅನ್ನು ಬಳಸಬಹುದು.

ಸದಸ್ಯತ್ವ, ಇನ್‌ವಾಯ್ಸ್, ಬಿಲ್ಲಿಂಗ್-ವಿಳಾಸ, ಟೆಂಪ್ಲೇಟ್ ಅಥವಾ ಡೊಮೇನ್-ನಿರ್ವಹಣಾ ಸ್ಥಿತಿಯನ್ನು ಸ್ವಾಯತ್ತ ಬಾಡಿಗೆದಾರನೊಳಗೆ ನಕಲಿಸಬೇಡಿ. ಬಾಡಿಗೆದಾರ Dashboard ಅನ್ನು ಆರಂಭಕವಾಗಿ, ಮುಖ್ಯ-ಸೈಟ್ ಗ್ರಾಹಕ ಫಲಕವನ್ನು ನಿರ್ವಹಿತ ಕ್ರಿಯೆಗಳ ದಾಖಲೆ ವ್ಯವಸ್ಥೆಯಾಗಿ ಪರಿಗಣಿಸಿ.

## ವಲಸೆ ಪರಿಶೀಲನೆ {#migration-verification}

ವಲಸೆ ಅಥವಾ ಜೀವನಚಕ್ರ ಏಕೀಕರಣವು ಬಾಡಿಗೆದಾರ ಡೇಟಾವನ್ನು ಬದಲಿಸಿದ ನಂತರ, ಪರಿಶೀಲನಾ ದ್ವಾರಗಳನ್ನು ಚಲಾಯಿಸಿ:

- `wp tenant verify-no-legacy --site=<site-id>` ಬಾಡಿಗೆದಾರನು ಇನ್ನು ಮುಂದೆ ಹಳೆಯ ನೆಟ್‌ವರ್ಕ್-ಬದಿಯ ಮಾರ್ಗಗಳ ಮೇಲೆ ಅವಲಂಬಿತವಾಗಿಲ್ಲ ಎಂಬುದನ್ನು ದೃಢಪಡಿಸುತ್ತದೆ.
- `wp tenant verify-sovereign-push --site=<site-id>` ಸ್ವಾಯತ್ತ push jobs ಪ್ರಕ್ರಿಯೆಗೊಳಿಸಿ ಖಾಲಿ ಮಾಡಬಹುದು ಎಂಬುದನ್ನು ದೃಢಪಡಿಸುತ್ತದೆ.

ಪರಿಶೀಲನೆ ವಿಫಲವಾದರೆ ಏಕೀಕರಣಗಳು ಅದನ್ನು ನಿಯೋಜನೆ ತಡೆಗೋಡೆಯಾಗಿ ಪರಿಗಣಿಸಬೇಕು. ವೈಫಲ್ಯ ಪರಿಹಾರವಾಗುವವರೆಗೆ ಬಾಡಿಗೆದಾರನನ್ನು live ಎಂದು ಗುರುತಿಸುವುದನ್ನು ತಪ್ಪಿಸಬೇಕು.

## ಬಾಡಿಗೆದಾರ ಅಳಿಸುವಿಕೆ {#tenant-deletion}

ಅಳಿಸುವಿಕೆ ಹರಿವುಗಳು ಆಡ್‌ಆನ್ teardown ಮಾರ್ಗವನ್ನು ಕರೆ ಮಾಡಬೇಕು. ಹೀಗೆ ಮಾಡಿದರೆ ಬಾಡಿಗೆದಾರ ಡೇಟಾಬೇಸ್ ರುಜುವಾತುಗಳು ಸ್ವಚ್ಛಗೊಳ್ಳುತ್ತವೆ. teardown ಯಶಸ್ವಿಯಾದ ನಂತರ ಬಾಹ್ಯ ಏಕೀಕರಣಗಳು ಒದಗಿಸುವವರ ಸಂಪನ್ಮೂಲಗಳನ್ನು ತೆಗೆದುಹಾಕಬಹುದು. ಆದರೆ ಪರಿಶೀಲನೆ ಅಥವಾ async push jobs ಇನ್ನೂ ನಡೆಯುತ್ತಿರುವಾಗ host ಡೇಟಾಬೇಸ್‌ಗಳು ಅಥವಾ ಫೋಲ್ಡರ್‌ಗಳನ್ನು ಅಳಿಸಬಾರದು.

## ಅಪ್ರಚಲಿತ ಡೇಟಾಬೇಸ್ ರೌಟರ್ {#deprecated-database-router}

ಹಳೆಯ `Database_Router` ಅನ್ನು ಅಪ್ರಚಲಿತ stub ನಿಂದ ಬದಲಿಸಲಾಗಿದೆ. ಹೊಸ ಏಕೀಕರಣಗಳು ಹಳೆಯ router class ಮೇಲೆ ಅವಲಂಬಿಸುವ ಬದಲು, ಪ್ರಸ್ತುತ ಸೈಟ್ ರೌಟರ್ ಮತ್ತು ಬಾಡಿಗೆದಾರ ನೋಂದಣಿ APIಗಳ ಮೂಲಕ ಬಾಡಿಗೆದಾರರನ್ನು ನಿರ್ಧರಿಸಬೇಕು.
