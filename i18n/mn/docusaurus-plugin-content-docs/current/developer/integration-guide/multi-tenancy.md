# Multi-Tenancy интеграц {#multi-tenancy-integration}

Ultimate Multisite: Multi-Tenancy 1.2.0 нь бие даасан tenant-ууд, шилжилтийн баталгаажуулалт, tenant-ийн амьдралын мөчлөгийн автоматжуулалтын хэд хэдэн интеграцын цэгийг өөрчилнө.

## Tenant эхлүүлэх урсгал {#tenant-bootstrap-flow}

Tenant үүсгэх эсвэл өөрчлөх интеграцууд дараах дарааллыг мөрдөх ёстой:

1. Tenant бүртгэлийн бичлэг болон тусгаарлалтын загварыг тодорхойлно.
2. Tenant өгөгдлийн сангийн бичигчийг үүсгэх эсвэл баталгаажуулна.
3. Tenant схемийг эхлүүлнэ.
4. Tenant хэрэглэгчдийг бэлтгэнэ.
5. Tenant чиглүүлэлт болон файлын системийн замуудыг бүртгэнэ.
6. Tenant-ийг ил болгохоос өмнө шилжилтийн баталгаажуулалтыг ажиллуулна.

Бие даасан tenant нь сүлжээний өгөгдлийн сангийн холболтыг дахин ашиглаж чадна гэж бүү таамагла. Addon-оос өгсөн tenant бүртгэл болон бичигчийн абстракцуудыг ашигла.

## SSO болон REST hook-ууд {#sso-and-rest-hooks}

Төлөв хадгалахгүй tenant autologin нь зорилгын claim, JTI дахин ашиглалтаас хамгаалалт, дуусах хугацааны хязгаар, гарал үүслийн бэхэлгээ бүхий богино настай token-ууд ашигладаг. Нэвтрэх товч эсвэл алсын удирдлагын холбоос нэмдэг интеграцууд tenant нэвтрэх URL-уудыг шууд байгуулахын оронд дэмжигдсэн SSO урсгалаар tenant зочлол үүсгэх ёстой.

Сүлжээний талын API аудитын үйл явдлууд болон өдөр тутмын хураангуй нь бие даасан tenant gateway-үүдэд боломжтой. Tenant амьдралын мөчлөгийн endpoint-уудыг дууддаг гадаад системүүдийг алдааг засахдаа эдгээр логийг ашигла.

## Бие даасан хэрэглэгчийн үйлдлийн URL-ууд {#sovereign-customer-action-urls}

Ultimate Multisite v2.13.0 нь бие даасан-tenant хэрэглэгчийн үйлдлүүдийг account, checkout, billing, invoice, site, template-солих, domain-mapping урсгалуудад зориулан үндсэн сайт руу буцаан чиглүүлдэг. Tenant талын удирдлагын холбоосуудыг харуулдаг интеграцууд эдгээр үйлдлийг үндсэн сайтын хэрэглэгчийн самбар руу зааж, хэрэглэгч үйлдлийг дуусгасны дараа tenant руу буцаж очих боломжтой байх ёстой үед баталгаажсан буцах зорилтыг оруулах ёстой.

Домэйн хоорондын удирдлагын холбоосуудад үндсэн SSO wrapper-ийг ашигла:

```php
$url = wu_with_sso($main_site_customer_url);
```

Үүсгэсэн URL нь `wu_sso_url`-ээр шүүгдэх боломжтой хэвээр байна; энэ нь SSO URL, одоогийн хэрэглэгч, зорилтот сайтын ID, redirect контекстийг хүлээн авдаг. Add-on-ууд тухайн filter-ийг ашиглан нийлүүлэгчид тусгай контекст нэмж эсвэл Ultimate Multisite-ийн token баталгаажуулалтыг хадгалан broker URL-ийг сольж болно.

Membership, invoice, billing-address, template, эсвэл domain-management төлөвийг бие даасан tenant дотор давхардуулж болохгүй. Tenant dashboard-ийг эхлүүлэгч, үндсэн сайтын хэрэглэгчийн самбарыг удирдагдах үйлдлүүдийн системийн албан ёсны бүртгэл гэж үз.

## Шилжилтийн баталгаажуулалт {#migration-verification}

Шилжилт эсвэл амьдралын мөчлөгийн интеграц tenant өгөгдлийг өөрчилсний дараа баталгаажуулалтын шалгуурыг ажиллуул:

- `wp tenant verify-no-legacy --site=<site-id>` нь tenant хуучин сүлжээний талын замуудаас цаашид хамаарахгүй болсныг баталгаажуулна.
- `wp tenant verify-sovereign-push --site=<site-id>` нь бие даасан push ажлууд боловсруулж, хоосорч чадна гэдгийг баталгаажуулна.

Интеграцууд баталгаажуулалт амжилтгүй болсныг байршуулалтыг хориглох шалтгаан гэж үзэж, алдаа шийдэгдэх хүртэл tenant-ийг live гэж тэмдэглэхээс зайлсхийх ёстой.

## Tenant устгах {#tenant-deletion}

Устгах урсгалууд tenant өгөгдлийн сангийн credentials-ийг цэвэрлэхийн тулд addon teardown замыг дуудах ёстой. Гадаад интеграцууд teardown амжилттай болсны дараа нийлүүлэгчийн нөөцүүдийг устгаж болно, гэхдээ баталгаажуулалт эсвэл async push ажлууд ажиллаж байх үед host өгөгдлийн сангууд эсвэл хавтсуудыг устгах ёсгүй.

## Хэрэглээнээс гарсан өгөгдлийн сангийн router {#deprecated-database-router}

Хуучин `Database_Router` нь deprecation stub-аар солигдсон. Шинэ интеграцууд хуучин router class-д найдахын оронд одоогийн сайтын router болон tenant бүртгэлийн API-уудаар tenant-уудыг тодорхойлох ёстой.
