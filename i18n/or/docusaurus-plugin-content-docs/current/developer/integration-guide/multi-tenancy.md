# ବହୁ-ଟେନାଣ୍ଟ ସମାକଳନ

Ultimate Multisite: ବହୁ-ଟେନାଣ୍ଟ 1.2.0 ସ୍ୱାଧୀନ ଟେନାଣ୍ଟ, ସ୍ଥାନାନ୍ତରଣ ଯାଞ୍ଚ, ଏବଂ ଟେନାଣ୍ଟ ଜୀବନଚକ୍ର ସ୍ୱୟଂଚାଳନ ପାଇଁ ଅନେକ ସମାକଳନ ସ୍ପର୍ଶବିନ୍ଦୁ ପରିବର୍ତ୍ତନ କରେ।

## ଟେନାଣ୍ଟ ବୁଟ୍‌ଷ୍ଟ୍ରାପ୍ ପ୍ରବାହ {#tenant-bootstrap-flow}

ଟେନାଣ୍ଟ ସୃଷ୍ଟି କିମ୍ବା ସଂଶୋଧନ କରୁଥିବା ସମାକଳନଗୁଡ଼ିକ ଏହି କ୍ରମ ଅନୁସରଣ କରିବା ଉଚିତ:

1. ଟେନାଣ୍ଟ ରେଜିଷ୍ଟ୍ରି ରେକର୍ଡ ଏବଂ ପୃଥକୀକରଣ ମଡେଲ୍ ସମାଧାନ କରନ୍ତୁ।
2. ଟେନାଣ୍ଟ ଡାଟାବେସ୍ ରାଇଟର୍ ସୃଷ୍ଟି କିମ୍ବା ଯାଞ୍ଚ କରନ୍ତୁ।
3. ଟେନାଣ୍ଟ ସ୍କିମା ବୁଟ୍‌ଷ୍ଟ୍ରାପ୍ କରନ୍ତୁ।
4. ଟେନାଣ୍ଟ ବ୍ୟବହାରକାରୀମାନଙ୍କୁ ପ୍ରାବଧାନ କରନ୍ତୁ।
5. ଟେନାଣ୍ଟ ରାଉଟିଙ୍ଗ୍ ଏବଂ ଫାଇଲ୍‌ସିଷ୍ଟମ୍ ପଥଗୁଡ଼ିକ ପଞ୍ଜିକରଣ କରନ୍ତୁ।
6. ଟେନାଣ୍ଟକୁ ପ୍ରକାଶ କରିବା ପୂର୍ବରୁ ସ୍ଥାନାନ୍ତରଣ ଯାଞ୍ଚ ଚଳାନ୍ତୁ।

କୌଣସି ସ୍ୱାଧୀନ ଟେନାଣ୍ଟ ନେଟୱାର୍କ ଡାଟାବେସ୍ ସଂଯୋଗ ପୁନଃବ୍ୟବହାର କରିପାରିବ ବୋଲି ଧାରଣା କରନ୍ତୁ ନାହିଁ। ଏଡ଼-ଅନ୍ ଦ୍ୱାରା ଦିଆଯାଇଥିବା ଟେନାଣ୍ଟ ରେଜିଷ୍ଟ୍ରି ଏବଂ ରାଇଟର୍ ଅବସ୍ତାପନା ବ୍ୟବହାର କରନ୍ତୁ।

## ଏସଏସଓ ଏବଂ REST ହୁକ୍ {#sso-and-rest-hooks}

ସ୍ଥିତିହୀନ ଟେନାଣ୍ଟ ସ୍ୱୟଂ-ଲଗଇନ୍ ଉଦ୍ଦେଶ୍ୟ ଦାବି, JTI ପୁନଃଚାଳନ ସୁରକ୍ଷା, ମିଆଦ ସୀମା, ଏବଂ ଉତ୍ସ ପିନିଙ୍ଗ୍ ସହିତ ସ୍ୱଳ୍ପକାଳୀନ ଟୋକେନ୍ ବ୍ୟବହାର କରେ। ଲଗଇନ୍ ବଟନ୍ କିମ୍ବା ଦୂରସ୍ଥ ପରିଚାଳନା ଲିଙ୍କ୍ ଯୋଡ଼ୁଥିବା ସମାକଳନଗୁଡ଼ିକ ଟେନାଣ୍ଟ ଲଗଇନ୍ URL ସିଧାସଳଖ ତିଆରି କରିବା ପରିବର୍ତ୍ତେ ସମର୍ଥିତ ଏସଏସଓ ପ୍ରବାହ ମାଧ୍ୟମରେ ଟେନାଣ୍ଟ ଭ୍ରମଣ ସୃଷ୍ଟି କରିବା ଉଚିତ।

ସ୍ୱାଧୀନ ଟେନାଣ୍ଟ ଗେଟୱେ ପାଇଁ ନେଟୱାର୍କ-ପାର୍ଶ୍ୱ API ଅଡିଟ୍ ଘଟଣା ଏବଂ ଦୈନିକ ସାରାଂଶ ଉପଲବ୍ଧ। ଟେନାଣ୍ଟ ଜୀବନଚକ୍ର ଏଣ୍ଡପଏଣ୍ଟକୁ ଡାକୁଥିବା ବାହ୍ୟ ପ୍ରଣାଳୀଗୁଡ଼ିକୁ ଡିବଗ୍ କରିବାବେଳେ ସେହି ଲଗ୍‌ଗୁଡ଼ିକ ବ୍ୟବହାର କରନ୍ତୁ।

## ସ୍ୱାଧୀନ ଗ୍ରାହକ କାର୍ଯ୍ୟ URL {#sovereign-customer-action-urls}

Ultimate Multisite v2.13.0 Account, ଚେକଆଉଟ୍, ବିଲିଂ, ଇନଭଏସ୍, ସାଇଟ, ଟେମ୍ପଲେଟ୍-ପରିବର୍ତ୍ତନ, ଏବଂ ଡୋମେନ୍-ମ୍ୟାପିଂ ପ୍ରବାହ ପାଇଁ ସ୍ୱାଧୀନ-ଟେନାଣ୍ଟ ଗ୍ରାହକ କାର୍ଯ୍ୟଗୁଡ଼ିକୁ ମୁଖ୍ୟ ସାଇଟକୁ ଫେରାଇ ରାଉଟ୍ କରେ। ଟେନାଣ୍ଟ-ପାର୍ଶ୍ୱ ପରିଚାଳନା ଲିଙ୍କ୍ ଦେଖାଉଥିବା ସମାକଳନଗୁଡ଼ିକ ସେହି କାର୍ଯ୍ୟଗୁଡ଼ିକୁ ମୁଖ୍ୟ-ସାଇଟ ଗ୍ରାହକ ପ୍ୟାନେଲ୍‌କୁ ନିର୍ଦ୍ଦେଶ କରିବା ଉଚିତ, ଏବଂ କାର୍ଯ୍ୟ ସମାପ୍ତ ପରେ ବ୍ୟବହାରକାରୀ ଟେନାଣ୍ଟକୁ ଫେରି ଯାଇପାରିବା ଉଚିତ ହେଲେ ଏକ ଯାଞ୍ଚିତ ଫେରା ଲକ୍ଷ୍ୟ ସମ୍ମିଳିତ କରିବା ଉଚିତ।

କ୍ରସ୍-ଡୋମେନ୍ ପରିଚାଳନା ଲିଙ୍କ୍ ପାଇଁ କୋର୍ ଏସଏସଓ ରାପର୍ ବ୍ୟବହାର କରନ୍ତୁ:

```php
$url = wu_with_sso($main_site_customer_url);
```

ସୃଷ୍ଟି ହୋଇଥିବା URL `wu_sso_url` ମାଧ୍ୟମରେ ଫିଲ୍ଟର୍ ହୋଇପାରେ, ଯାହା ଏସଏସଓ URL, ବର୍ତ୍ତମାନ ବ୍ୟବହାରକାରୀ, ଲକ୍ଷ୍ୟ ସାଇଟ ID, ଏବଂ ପୁନର୍ନିର୍ଦ୍ଦେଶ ପରିପ୍ରେକ୍ଷ୍ୟ ପାଏ। ଏଡ଼-ଅନ୍‌ଗୁଡ଼ିକ ସେହି ଫିଲ୍ଟର୍ ବ୍ୟବହାର କରି ପ୍ରଦାତା-ନିର୍ଦ୍ଦିଷ୍ଟ ପରିପ୍ରେକ୍ଷ୍ୟ ଯୋଡ଼ିପାରନ୍ତି କିମ୍ବା Ultimate Multisite ର ଟୋକେନ୍ ଯାଞ୍ଚକୁ ଅବିକୃତ ରଖି ବ୍ରୋକର୍ URL ପରିବର୍ତ୍ତନ କରିପାରନ୍ତି।

ସଦସ୍ୟତା, ଇନଭଏସ୍, ବିଲିଂ-ଠିକଣା, ଟେମ୍ପଲେଟ୍, କିମ୍ବା ଡୋମେନ୍-ପରିଚାଳନା ଅବସ୍ଥାକୁ ସ୍ୱାଧୀନ ଟେନାଣ୍ଟ ଭିତରେ ନକଲ କରନ୍ତୁ ନାହିଁ। ଟେନାଣ୍ଟ Dashboardକୁ ଆରମ୍ଭକାରୀ ଭାବେ ଏବଂ ମୁଖ୍ୟ-ସାଇଟ ଗ୍ରାହକ ପ୍ୟାନେଲ୍‌କୁ ପରିଚାଳିତ କାର୍ଯ୍ୟ ପାଇଁ ଅଧିକୃତ ରେକର୍ଡ ପ୍ରଣାଳୀ ଭାବେ ଦେଖନ୍ତୁ।

## ସ୍ଥାନାନ୍ତରଣ ଯାଞ୍ଚ {#migration-verification}

ସ୍ଥାନାନ୍ତରଣ ପରେ କିମ୍ବା ଜୀବନଚକ୍ର ସମାକଳନ ଟେନାଣ୍ଟ ତଥ୍ୟ ପରିବର୍ତ୍ତନ କଲେ, ଯାଞ୍ଚ ଗେଟ୍‌ଗୁଡ଼ିକ ଚଳାନ୍ତୁ:

- `wp tenant verify-no-legacy --site=<site-id>` ନିଶ୍ଚିତ କରେ ଯେ ଟେନାଣ୍ଟ ଆଉ ପୁରୁଣା ନେଟୱାର୍କ-ପାର୍ଶ୍ୱ ପଥଗୁଡ଼ିକ ଉପରେ ନିର୍ଭର କରୁନାହିଁ।
- `wp tenant verify-sovereign-push --site=<site-id>` ନିଶ୍ଚିତ କରେ ଯେ ସ୍ୱାଧୀନ ପୁଶ୍ କାମଗୁଡ଼ିକ ପ୍ରକ୍ରିୟାକରଣ ଏବଂ ଖାଲି କରିପାରେ।

ସମାକଳନଗୁଡ଼ିକ ବିଫଳ ଯାଞ୍ଚକୁ ଡିପ୍ଲୟମେଣ୍ଟ ଅବରୋଧକ ଭାବେ ଧରିବା ଉଚିତ ଏବଂ ବିଫଳତା ସମାଧାନ ନହେଉଅବଧି ଟେନାଣ୍ଟକୁ ଲାଇଭ୍ ବୋଲି ଚିହ୍ନିତ କରିବାରୁ ବଞ୍ଚିବା ଉଚିତ।

## ଟେନାଣ୍ଟ ବିଲୋପ {#tenant-deletion}

ବିଲୋପ ପ୍ରବାହଗୁଡ଼ିକ ଏଡ଼-ଅନ୍ ଟିୟର୍‌ଡାଉନ୍ ପଥକୁ ଡାକିବା ଉଚିତ, ଯାହାଦ୍ୱାରା ଟେନାଣ୍ଟ ଡାଟାବେସ୍ ପ୍ରମାଣପତ୍ର ସଫା ହୁଏ। ବାହ୍ୟ ସମାକଳନଗୁଡ଼ିକ ଟିୟର୍‌ଡାଉନ୍ ସଫଳ ହେବା ପରେ ପ୍ରଦାତା ସମ୍ପଦ ହଟାଇପାରନ୍ତି, କିନ୍ତୁ ଯାଞ୍ଚ କିମ୍ବା ଅସମକାଳୀନ ପୁଶ୍ କାମ ଚାଲୁଥିବା ବେଳେ ହୋଷ୍ଟ ଡାଟାବେସ୍ କିମ୍ବା ଫୋଲ୍ଡର୍ ବିଲୋପ କରିବା ଉଚିତ ନୁହେଁ।

## ଅପ୍ରଚଳିତ ଡାଟାବେସ୍ ରାଉଟର୍ {#deprecated-database-router}

ପୁରୁଣା `Database_Router`କୁ ଏକ ଅପ୍ରଚଳନ ଷ୍ଟବ୍ ଦ୍ୱାରା ବଦଳାଯାଇଛି। ନୂଆ ସମାକଳନଗୁଡ଼ିକ ପୁରୁଣା ରାଉଟର୍ କ୍ଲାସ୍ ଉପରେ ନିର୍ଭର କରିବା ପରିବର୍ତ୍ତେ ବର୍ତ୍ତମାନ ସାଇଟ ରାଉଟର୍ ଏବଂ ଟେନାଣ୍ଟ ରେଜିଷ୍ଟ୍ରି API ମାଧ୍ୟମରେ ଟେନାଣ୍ଟ ସମାଧାନ କରିବା ଉଚିତ।
