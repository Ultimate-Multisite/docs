# ການເຊື່ອມປະສານແບບ Multi-Tenancy

Ultimate Multisite: Multi-Tenancy 1.2.0 ປ່ຽນຈຸດເຊື່ອມປະສານຫຼາຍຈຸດສຳລັບຜູ້ເຊົ່າແບບອະທິປະໄຕ, ການກວດຢືນຢັນການຍ້າຍຂໍ້ມູນ, ແລະການອັດຕະໂນມັດວົງຈອນຊີວິດຜູ້ເຊົ່າ.

## ຂະບວນການເລີ່ມຕັ້ງຄ່າຜູ້ເຊົ່າ

ການເຊື່ອມປະສານທີ່ສ້າງ ຫຼື ແກ້ໄຂຜູ້ເຊົ່າ ຄວນເຮັດຕາມລຳດັບນີ້:

1. ລະບຸລະບຽນບັນທຶກຜູ້ເຊົ່າ ແລະ ໂມເດວການແຍກໂດດດ່ຽວ.
2. ສ້າງ ຫຼື ກວດຢືນຢັນຕົວຂຽນຖານຂໍ້ມູນຂອງຜູ້ເຊົ່າ.
3. ເລີ່ມຕັ້ງຄ່າໂຄງສ້າງຂໍ້ມູນຂອງຜູ້ເຊົ່າ.
4. ຈັດສັນຜູ້ໃຊ້ຂອງຜູ້ເຊົ່າ.
5. ລົງທະບຽນການນຳທາງຜູ້ເຊົ່າ ແລະ ເສັ້ນທາງລະບົບໄຟລ໌.
6. ເຮັດການກວດຢືນຢັນການຍ້າຍຂໍ້ມູນກ່ອນເປີດໃຫ້ເຂົ້າເຖິງຜູ້ເຊົ່າ.

ຢ່າສົມມຸດວ່າຜູ້ເຊົ່າແບບອະທິປະໄຕສາມາດໃຊ້ການເຊື່ອມຕໍ່ຖານຂໍ້ມູນເຄືອຂ່າຍຮ່ວມໄດ້. ໃຫ້ໃຊ້ລະບຽນຜູ້ເຊົ່າ ແລະ ຊັ້ນນາມມະທຳຂອງຕົວຂຽນທີ່ສ່ວນເສີມຈັດໃຫ້.

## SSO ແລະ REST hooks

ການເຂົ້າລະບົບອັດຕະໂນມັດຜູ້ເຊົ່າແບບບໍ່ເກັບສະຖານະໃຊ້ token ອາຍຸສັ້ນທີ່ມີການອ້າງຈຸດປະສົງ, ການປ້ອງກັນການຫຼິ້ນຊ້ຳ JTI, ຂອບເຂດໝົດອາຍຸ, ແລະ ການປັກໝຸດຕົ້ນທາງ. ການເຊື່ອມປະສານທີ່ເພີ່ມປຸ່ມເຂົ້າລະບົບ ຫຼື ລິ້ງການຈັດການລະຍະໄກ ຄວນສ້າງການເຂົ້າຢ້ຽມຜູ້ເຊົ່າຜ່ານຂະບວນການ SSO ທີ່ຮອງຮັບ ແທນການສ້າງ URL ເຂົ້າລະບົບຜູ້ເຊົ່າໂດຍກົງ.

ເຫດການກວດສອບ API ຝັ່ງເຄືອຂ່າຍ ແລະ ສະຫຼຸບປະຈຳວັນ ມີໃຫ້ສຳລັບປະຕູທາງເຂົ້າຜູ້ເຊົ່າແບບອະທິປະໄຕ. ໃຊ້ບັນທຶກເຫຼົ່ານັ້ນເມື່ອດີບັກລະບົບພາຍນອກທີ່ເອີ້ນ endpoint ວົງຈອນຊີວິດຜູ້ເຊົ່າ.

## URL ການກະທຳຂອງລູກຄ້າແບບອະທິປະໄຕ

Ultimate Multisite v2.13.0 ນຳທາງການກະທຳຂອງລູກຄ້າຜູ້ເຊົ່າແບບອະທິປະໄຕກັບໄປຫາເວັບໄຊຫຼັກ ສຳລັບຂະບວນການບັນຊີ, ການຊຳລະເງິນ, ການເກັບເງິນ, ໃບແຈ້ງໜີ້, ເວັບໄຊ, ການສະຫຼັບແມ່ແບບ, ແລະ ການຈັບຄູ່ໂດເມນ. ການເຊື່ອມປະສານທີ່ສະແດງລິ້ງການຈັດການຝັ່ງຜູ້ເຊົ່າ ຄວນຊີ້ການກະທຳເຫຼົ່ານັ້ນໄປທີ່ແຜງລູກຄ້າຂອງເວັບໄຊຫຼັກ ແລະ ລວມເປົ້າໝາຍກັບຄືນທີ່ຜ່ານການກວດຢືນຢັນເມື່ອຜູ້ໃຊ້ຄວນສາມາດນຳທາງກັບໄປຫາຜູ້ເຊົ່າຫຼັງຈາກເຮັດການກະທຳສຳເລັດ.

ໃຊ້ wrapper SSO ຫຼັກສຳລັບລິ້ງການຈັດການຂ້າມໂດເມນ:

```php
$url = wu_with_sso($main_site_customer_url);
```

URL ທີ່ສ້າງຂຶ້ນຍັງຄົງສາມາດຖືກກັ່ນຕອງຜ່ານ `wu_sso_url`, ເຊິ່ງຮັບ URL SSO, ຜູ້ໃຊ້ປັດຈຸບັນ, ID ເວັບໄຊເປົ້າໝາຍ, ແລະ ບໍລິບົດການປ່ຽນທາງ. ສ່ວນເສີມສາມາດໃຊ້ຕົວກັ່ນຕອງນັ້ນເພື່ອແນບບໍລິບົດສະເພາະຂອງຜູ້ໃຫ້ບໍລິການ ຫຼື ເພື່ອແທນທີ່ broker URL ໂດຍຍັງຮັກສາການກວດຢືນຢັນ token ຂອງ Ultimate Multisite.

ຢ່າເຮັດຊ້ຳສະຖານະສະມາຊິກ, ໃບແຈ້ງໜີ້, ທີ່ຢູ່ເກັບເງິນ, ແມ່ແບບ, ຫຼື ການຈັດການໂດເມນ ພາຍໃນຜູ້ເຊົ່າແບບອະທິປະໄຕ. ໃຫ້ຖືແຜງຄວບຄຸມຜູ້ເຊົ່າເປັນຕົວເປີດໃຊ້ ແລະ ແຜງລູກຄ້າເວັບໄຊຫຼັກເປັນລະບົບບັນທຶກຫຼັກສຳລັບການກະທຳທີ່ຖືກຈັດການ.

## ການກວດຢືນຢັນການຍ້າຍຂໍ້ມູນ

ຫຼັງຈາກການຍ້າຍຂໍ້ມູນ ຫຼື ການເຊື່ອມປະສານວົງຈອນຊີວິດປ່ຽນຂໍ້ມູນຜູ້ເຊົ່າ, ໃຫ້ເຮັດດ່ານການກວດຢືນຢັນ:

- `wp tenant verify-no-legacy --site=<site-id>` ຢືນຢັນວ່າຜູ້ເຊົ່າບໍ່ພຶ່ງພາເສັ້ນທາງຝັ່ງເຄືອຂ່າຍແບບເກົ່າອີກຕໍ່ໄປ.
- `wp tenant verify-sovereign-push --site=<site-id>` ຢືນຢັນວ່າງານ push ແບບອະທິປະໄຕສາມາດປະມວນຜົນ ແລະ ລະບາຍຄິວໄດ້.

ການເຊື່ອມປະສານຄວນຖືວ່າການກວດຢືນຢັນທີ່ລົ້ມເຫຼວເປັນຕົວຂັດຂວາງການປ່ອຍໃຊ້ງານ ແລະ ຫຼີກລ່ຽງການໝາຍຜູ້ເຊົ່າວ່າພ້ອມໃຊ້ງານ ຈົນກວ່າຈະແກ້ໄຂຄວາມລົ້ມເຫຼວນັ້ນແລ້ວ.

## ການລຶບຜູ້ເຊົ່າ

ຂະບວນການລຶບຄວນເອີ້ນເສັ້ນທາງ teardown ຂອງສ່ວນເສີມ ເພື່ອໃຫ້ຂໍ້ມູນປະຈຳຕົວຖານຂໍ້ມູນຜູ້ເຊົ່າຖືກລ້າງ. ການເຊື່ອມປະສານພາຍນອກອາດລຶບຊັບພະຍາກອນຜູ້ໃຫ້ບໍລິການຫຼັງຈາກ teardown ສຳເລັດ, ແຕ່ບໍ່ຄວນລຶບຖານຂໍ້ມູນ ຫຼື ໂຟນເດີຂອງໂຮສ ໃນຂະນະທີ່ການກວດຢືນຢັນ ຫຼື ງານ push ແບບອະຊິງໂຄຣນັດຍັງກຳລັງເຮັດວຽກຢູ່.

## ຕົວນຳທາງຖານຂໍ້ມູນທີ່ເຊົາໃຊ້ແລ້ວ

`Database_Router` ແບບເກົ່າໄດ້ຖືກແທນທີ່ດ້ວຍ stub ການເຊົາໃຊ້. ການເຊື່ອມປະສານໃໝ່ຄວນລະບຸຜູ້ເຊົ່າຜ່ານ API ຕົວນຳທາງເວັບໄຊປັດຈຸບັນ ແລະ ລະບຽນຜູ້ເຊົ່າ ແທນການພຶ່ງພາຄລາສຕົວນຳທາງເກົ່າ.
