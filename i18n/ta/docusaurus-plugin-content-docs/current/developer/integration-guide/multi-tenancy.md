# Multi-Tenancy ஒருங்கிணைப்பு

Ultimate Multisite: Multi-Tenancy 1.2.0, சுயாட்சியுள்ள குத்தகையாளர்களுக்கான ஒருங்கிணைப்பு தொடுப்புகள், இடமாற்றச் சரிபார்ப்பு, மற்றும் குத்தகையாளர் வாழ்க்கைச்சுழற்சி தானியக்கம் ஆகியவற்றில் பல மாற்றங்களைச் செய்கிறது.

## குத்தகையாளர் தொடக்க ஓட்டம் {#tenant-bootstrap-flow}

குத்தகையாளர்களை உருவாக்கும் அல்லது மாற்றும் ஒருங்கிணைப்புகள் இந்த வரிசையைப் பின்பற்ற வேண்டும்:

1. குத்தகையாளர் பதிவகப் பதிவையும் தனிமைப்படுத்தல் மாதிரியையும் தீர்மானிக்கவும்.
2. குத்தகையாளர் தரவுத்தள எழுத்தாளரை உருவாக்கவும் அல்லது சரிபார்க்கவும்.
3. குத்தகையாளர் schema-வை தொடக்கநிலைப்படுத்தவும்.
4. குத்தகையாளர் பயனர்களை provision செய்யவும்.
5. குத்தகையாளர் routing மற்றும் கோப்பு முறைமைப் பாதைகளைப் பதிவு செய்யவும்.
6. குத்தகையாளரை வெளிப்படுத்தும் முன் இடமாற்றச் சரிபார்ப்பை இயக்கவும்.

சுயாட்சியுள்ள குத்தகையாளர் பிணையத் தரவுத்தள இணைப்பை மீண்டும் பயன்படுத்த முடியும் என்று கருத வேண்டாம். சேர்க்கை வழங்கும் குத்தகையாளர் பதிவகம் மற்றும் எழுத்தாளர் abstraction-களைப் பயன்படுத்தவும்.

## SSO மற்றும் REST hooks {#sso-and-rest-hooks}

நிலையற்ற குத்தகையாளர் தானியங்கு உள்நுழைவு, purpose claim, JTI replay பாதுகாப்பு, காலாவதி வரம்பு, மற்றும் origin pinning உடன் குறுகிய ஆயுள் கொண்ட token-களைப் பயன்படுத்துகிறது. உள்நுழைவு பொத்தான்கள் அல்லது தொலைநிலை மேலாண்மை இணைப்புகளைச் சேர்க்கும் ஒருங்கிணைப்புகள், குத்தகையாளர் உள்நுழைவு URL-களை நேரடியாக உருவாக்குவதற்குப் பதிலாக, ஆதரிக்கப்படும் SSO ஓட்டத்தின் மூலம் குத்தகையாளர் வருகைகளை உருவாக்க வேண்டும்.

சுயாட்சியுள்ள குத்தகையாளர் நுழைவாயில்களுக்கு பிணையப் பக்க API தணிக்கை நிகழ்வுகள் மற்றும் தினசரி சுருக்கங்கள் கிடைக்கின்றன. குத்தகையாளர் வாழ்க்கைச்சுழற்சி endpoint-களை அழைக்கும் வெளிப்புற அமைப்புகளை debug செய்யும்போது அந்த பதிவுகளைப் பயன்படுத்தவும்.

## சுயாட்சியுள்ள வாடிக்கையாளர் செயல் URL-கள் {#sovereign-customer-action-urls}

Ultimate Multisite v2.13.0, கணக்கு, செலுத்தல், கட்டணம், விலைப்பட்டியல், தளம், வார்ப்புரு மாற்றம், மற்றும் domain-mapping ஓட்டங்களுக்காக சுயாட்சியுள்ள குத்தகையாளர் வாடிக்கையாளர் செயல்களை முதன்மைத் தளத்துக்கு மீண்டும் route செய்கிறது. குத்தகையாளர் பக்க மேலாண்மை இணைப்புகளை render செய்யும் ஒருங்கிணைப்புகள், அந்தச் செயல்களை முதன்மைத் தள வாடிக்கையாளர் panel-க்கு சுட்டிக்காட்ட வேண்டும். செயலை முடித்த பிறகு பயனர் மீண்டும் குத்தகையாளரிடம் செல்ல வேண்டியிருந்தால், சரிபார்க்கப்பட்ட திரும்பும் இலக்கையும் சேர்க்க வேண்டும்.

Cross-domain மேலாண்மை இணைப்புகளுக்கு core SSO wrapper-ஐப் பயன்படுத்தவும்:

```php
$url = wu_with_sso($main_site_customer_url);
```

உருவாக்கப்பட்ட URL, `wu_sso_url` மூலம் தொடர்ந்து filter செய்யக்கூடியதாக இருக்கும். அது SSO URL, தற்போதைய பயனர், இலக்கு தள ID, மற்றும் redirect சூழலைப் பெறுகிறது. Ultimate Multisite-இன் token சரிபார்ப்பை பாதுகாத்தபடி, provider-க்கு உரிய சூழலைச் சேர்க்க அல்லது broker URL-ஐ மாற்ற சேர்க்கைகள் அந்த filter-ஐப் பயன்படுத்தலாம்.

உறுப்பினர் நிலை, விலைப்பட்டியல், கட்டண முகவரி, வார்ப்புரு, அல்லது domain-management நிலையை சுயாட்சியுள்ள குத்தகையாளருக்குள் நகலெடுக்க வேண்டாம். குத்தகையாளர் Dashboard-ஐ தொடக்கியாகவும், முதன்மைத் தள வாடிக்கையாளர் panel-ஐ நிர்வகிக்கப்படும் செயல்களுக்கான அதிகாரப்பூர்வ பதிவாகவும் கருதவும்.

## இடமாற்றச் சரிபார்ப்பு {#migration-verification}

இடமாற்றம் அல்லது வாழ்க்கைச்சுழற்சி ஒருங்கிணைப்பு குத்தகையாளர் தரவை மாற்றிய பிறகு, சரிபார்ப்பு வாயில்களை இயக்கவும்:

- `wp tenant verify-no-legacy --site=<site-id>` குத்தகையாளர் இனி பழைய பிணையப் பக்கப் பாதைகளில் சார்ந்திருக்கவில்லை என்பதை உறுதிப்படுத்துகிறது.
- `wp tenant verify-sovereign-push --site=<site-id>` சுயாட்சியுள்ள push பணிகள் செயலாக்கி காலியாக்க முடியும் என்பதை உறுதிப்படுத்துகிறது.

சரிபார்ப்பு தோல்வியை ஒருங்கிணைப்புகள் deployment தடையாகக் கருத வேண்டும். தோல்வி தீர்க்கப்படும் வரை குத்தகையாளரை live எனக் குறிக்காமல் இருக்க வேண்டும்.

## குத்தகையாளர் நீக்கம் {#tenant-deletion}

நீக்க ஓட்டங்கள் சேர்க்கையின் teardown பாதையை அழைக்க வேண்டும், இதனால் குத்தகையாளர் தரவுத்தள credentials சுத்தம் செய்யப்படும். Teardown வெற்றியடைந்த பிறகு வெளிப்புற ஒருங்கிணைப்புகள் provider வளங்களை அகற்றலாம். ஆனால் சரிபார்ப்பு அல்லது async push பணிகள் இன்னும் இயங்கிக்கொண்டிருக்கும்போது host தரவுத்தளங்கள் அல்லது கோப்புறைகளை நீக்கக் கூடாது.

## Deprecated தரவுத்தள router {#deprecated-database-router}

பழைய `Database_Router` deprecation stub-ஆல் மாற்றப்பட்டுள்ளது. புதிய ஒருங்கிணைப்புகள் பழைய router class மீது சார்ந்திருக்காமல், தற்போதைய தள router மற்றும் குத்தகையாளர் பதிவக API-கள் மூலம் குத்தகையாளர்களைத் தீர்மானிக்க வேண்டும்.
