# இறையாண்மை இடம்பெயர்வு சரிபார்ப்பு

Ultimate Multisite: Multi-Tenancy 1.2.0 இல் இறையாண்மை டெண்டன்ட் (sovereign tenant) இடம்பெயர்ச்சிக்கான WP-CLI சரிபார்ப்பு கட்டளைகள் உள்ளன. ஒரு டெண்டன்ட் இடம்பெயர்வு, SSO வருகை அல்லது தனிமைப்படுத்தப்பட்ட நிறுவல் எதிர்பார்த்தபடி செயல்படவில்லை என்றால் அவற்றைப் பயன்படுத்தவும்.

## இயக்க வேண்டிய கட்டளைகள்

நெட்வொர்க் WordPress நிறுவலில் இருந்து சரிபார்ப்பைச் செய்யவும்:

```bash
wp tenant verify-no-legacy --site=<site-id>
wp tenant verify-sovereign-push --site=<site-id>
```

நீங்கள் இடம்பெயர்க்கும் டெண்டன்ட் (tenant) ஐப் பொறுத்து Site ID ஐப் பயன்படுத்தவும். முதல் கட்டளை, அந்த டெண்டன்ட் இனி பழைய நெட்வொர்க் தரவைச் சார்ந்து இல்லை என்பதைச் சரிபார்க்கிறது. இரண்டாவது கட்டளை, இறையாண்மை புஷ் வேலைகளை (sovereign push jobs) செயலாக்க மற்றும் வடிகட்ட (drain) முடியும் என்பதைச் சரிபார்க்கிறது.

## பொதுவான தோல்விகள்

### தரவுத்தள அனுமதிகள் ஹோஸ்டுடன் பொருந்தவில்லை

சரிபார்ப்பு கிராந்துகள் (grant) அல்லது escritor-பயனர் (writer-user) தோல்விகளைப் பற்றிக் கூறினால், கட்டமைக்கப்பட்ட தரவுத்தள ஹோஸ்டைச் சரிபார்க்கவும். `localhost`, `127.0.0.1` மற்றும் ஒரு கண்டெய்னர் சேவை பெயர் ஆகியவை வெவ்வேறு MySQL கிராந்த ஹோஸ்ட்கள் ஆகும். டெண்டன்ட் ஹோஸ்ட் பிணைப்பை (host binding) அல்லது தரவுத்தள அனுமதிகளைப் புதுப்பித்து, பின்னர் சரிபார்ப்பை மீண்டும் இயக்கவும்.

### Bedrock அல்லது உள்ளூர் நிறுவல்கள் இணைக்க முடியவில்லை

Bedrock மற்றும் உள்ளூர் சாக்கெட் நிறுவல்கள் தரவுத்தளத்தை `localhost` ஆகக் காட்டலாம், ஆனால்运行时 ஒரு சாதாரண முகவரியின் (normalized address) மூலம் இணைக்கிறது. பதிப்பு 1.2.0 ஒரே இயந்திர ஹோஸ்ட் சரங்களைச் சீராக்குகிறது, ஆனால் தனிப்பயன் ஹோஸ்ட் மாற்றங்கள் (custom host overrides) இன்னும் தரவுத்தள அனுமதிகளுடன் முரண்படலாம்.

### அசிங்க் புஷ் வரிசை வடிகட்டப்படவில்லை (does not drain)

`verify-sovereign-push` முடிவடையவில்லை என்றால், Action Scheduler அல்லது கட்டமைக்கப்பட்ட அசிங்க் ரன்னரைச் சரிபார்க்கவும். மீண்டும் முயற்சிக்கவோ அல்லது நிராகரிக்கவோ பாதுகாப்பாக இருப்பதை உறுதிசெய்த பின்னரே தோல்வியுற்ற வேலைகளை அழிக்கவும்.

### டெண்டன்ட் பயனர் எண்ணிக்கை தவறாக உள்ளது

இடம்பெயர்வு இறையாண்மை டெண்டனுக்குப் பயனர்களை வழங்க வேண்டும். எதிர்பார்க்கப்படும் நிறுவப்பட்ட பயனர் (install user) காணப்படவில்லை என்றால், SSO-வை மீண்டும் முயற்சிக்க (retry) முன் பயனர் வழங்குதல் படிநிலையை மீண்டும் இயக்கவும்.

### SSO வருகை நிராகரிக்கப்படுகிறது

நிலை இல்லாமல் உள்ள வாடகை தானியங்கி உள்நுழைவு (Stateless tenant autologin) செயல்பட, வாடகை டொமைன் (tenant domain), originate pin, டோக்கன் நோக்கம் (token purpose), nonce மற்றும் காலாவதியாகும் நேரம் (expiry) ஆகியவை சரியாகப் பொருந்த வேண்டும். SSO வருகையை உருவாக்கிய பிறகு, வாடகை URL சரியானது என்பதை உறுதிசெய்து உள்நுழைவு முயற்சிக்கவும்.

## மீண்டும் முயற்சி செய்ய வேண்டிய நேரம்

ஒவ்வொரு உள்கட்டமைப்பு மாற்றத்திற்குப் பிறகும் சரிபார்ப்பை மீண்டும் செய்யவும். அனைத்து சரிபார்ப்பு சோதனைகளும் வெற்றிகரமாக முடிந்தால் மட்டுமே உற்பத்தி (production) போக்குவரத்தை மாற்றுவது, மூல தரவை நீக்குவது அல்லது இடம்பெயர்வு (migration) தகவல்களை நீக்குவது போன்ற செயல்களைச் செய்ய வேண்டாம்.
