# درس ۱۲: د کاروبار چلول

یو پلېټفارم هغه پروژه نه ده چې تاسو یې پای ته رسوئ -- دا یو کاروبار دی چې تاسو یې چلوئ. دا درس د FitSite د مدیریت ورځني عملیات رانغاړي: ملاتړ، بلېنګ، ساتنه، او د پېرېدونکو خوښ ساتل.

## موږ چېرته پاتې وو

FitSite ژوندی دی او پېرېدونکي نوم‌لیکنه کوي. اوس تاسو اړتیا لرئ عملیات په پایدار ډول وچلوئ.

## ورځني عملیات

### څارنه

دا هره ورځ وګورئ (یا خبرتیاوې تنظیم کړئ):

- **Uptime**: ایا پلېټفارم ته لاسرسی شته؟ د uptime څارنې خدمت وکاروئ.
- **نوې نوم‌لیکنې**: نن څو نوي پېرېدونکو نوم‌لیکنه وکړه؟
- **ناکامې تادیې**: ایا داسې د تادیې ناکامۍ شته چې پاملرنې ته اړتیا لري؟
- **د ملاتړ غوښتنې**: ایا د پېرېدونکو داسې پوښتنې شته چې ځواب نه دي ورکړل شوي؟

### د پېرېدونکو ملاتړ

ستاسو د ځانګړې برخې تمرکز دلته یوه ګټه ده. ځکه ستاسو ټول پېرېدونکي د فټنس سټوډیوګانې دي، تاسو به هماغه پوښتنې بیا بیا ووینئ:

**عامې پوښتنې چې تاسو به یې ترلاسه کوئ:**

- "زه څنګه د خپلو ټولګیو مهالویش تازه کړم؟"
- "ایا زه د خپل سایټ رنګونه بدلولی شم؟"
- "زه څنګه خپل سایټ ته نوی روزونکی اضافه کړم؟"
- "زما ډومېن کار نه کوي"
- "زه څنګه خپل پلان لغوه/لوړ کړم؟"

خپله د پوهې زېرمه (چې په درس ۸ کې پیل شوې) د همدې تکراري پوښتنو پر بنسټ جوړه کړئ. هر د ملاتړ ټکټ چې د پوهې زېرمې مقاله کېدای شوای، د دې نښه ده چې هماغه مقاله ولیکئ.

### د ملاتړ کچې

لکه څنګه چې وده کوئ، ملاتړ د پلان له مخې جوړ کړئ:

| پلان | د ملاتړ کچه | د ځواب وخت |
|------|--------------|---------------|
| پیل | د پوهې زېرمه + برېښنالیک | ۴۸ ساعته |
| وده | د برېښنالیک ملاتړ | ۲۴ ساعته |
| مسلکي | لومړیتوب لرونکی برېښنالیک + د پیل کولو زنګ | ۴ ساعته |

د [Support Tickets اضافه](/addons/support-tickets) کولی شي په پلېټفارم کې د ملاتړ غوښتنو مدیریت کې مرسته وکړي.

## د بلېنګ عملیات

### تکراري تادیې

Ultimate Multisite د recurring billing چارې ستاسو د payment gateway له لارې په اتومات ډول سمبالوي. ستاسو دنده دا ده چې دا موارد وڅارئ:

- **ناکامې تادیې**: ژر تعقیب یې کړئ. ډېری ناکامۍ د ختمو شویو کارتونو له امله وي، نه قصدي لغوه‌کول.
- **Dunning**: د خپل payment gateway له لارې اتومات د بیا هڅې منطق تنظیم کړئ (Stripe دا ښه سمبالوي)
- **د لغوه‌کولو غوښتنې**: پوه شئ چې پېرېدونکي ولې ځي. هر لغوه‌کول feedback دی.

### د غړیتوبونو مدیریت

**Ultimate Multisite > غړیتوبونه** ته لاړ شئ تر څو:

- ټولې فعالې ګډونونه وګورئ
- د لوړولو او ښکته کولو غوښتنې سمبال کړئ
- د اړتیا پر مهال بېرته‌ورکړې پروسس کړئ
- د ازمایښتي مودو پای مدیریت کړئ

د بشپړ ماخذ لپاره [د غړیتوبونو مدیریت](/user-guide/administration/managing-memberships) وګورئ.

### انوایس کول

ډاډ ترلاسه کړئ چې د هرې تادیې لپاره انوایسونه سم تولیدېږي. پېرېدونکي ښايي د کاروباري لګښت راپور ورکولو لپاره انوایسونو ته اړتیا ولري. [د تادیاتو او انوایسونو مدیریت](/user-guide/administration/managing-payments-and-invoices) وګورئ.

## د پلېټفارم ساتنه

### د WordPress او plugin تازه‌کول

د شبکې د مدیر په توګه، تاسو د دې مسؤل یاست:

- **د WordPress core تازه‌کول**: مخکې له دې چې په production کې یې پلي کړئ، په staging سایټ کې یې وازمویئ
- **د plugin تازه‌کول**: همداسې -- لومړی یې وازمویئ، بیا یې په ټوله شبکه کې پلي کړئ
- **د theme تازه‌کول**: تایید کړئ چې templates لا هم د theme له تازه‌کولو وروسته سم ښکاري
- **د Ultimate Multisite تازه‌کول**: changelog تعقیب کړئ او له تازه‌کولو مخکې یې وازمویئ

:::warning Never Update in Production Without Testing
ستاسو په شبکه کې هر پېرېدونکي سایټ ته مات تازه‌کول اغېز رسوي. تل تازه‌کول لومړی د خپلې شبکې په staging کاپي کې وازمویئ.
:::

### امنیت

- ټول software تازه وساتئ
- د admin حسابونو لپاره پیاوړي پاسورډونه او two-factor authentication وکاروئ
- شکمن فعالیت وڅارئ
- د امنیتي پېښو لپاره یو پلان ولرئ

### کړنه

لکه څنګه چې ستاسو شبکه وده کوي، وڅارئ:

- **د پاڼې د پورته کېدو وختونه**: ایا د پېرېدونکو سایټونه چټک دي؟
- **د سرور د سرچینو کارونه**: CPU، حافظه، د ډیسک ځای
- **د ډیټابېس کړنه**: لویې شبکې د وخت په تېرېدو د ډیټابېس ښه‌والي ته اړتیا لري

که مخکې مو نه وي کړي، د caching (page cache، object cache) او CDN پلي کول په پام کې ونیسئ. د [Cloudflare ادغام](/user-guide/host-integrations/cloudflare) د دې ډېره برخه سمبالوي.

## د پېرېدونکي د ژوندپړاو مدیریت

### د پرېښودو کمول

پرېښودل د هغو پېرېدونکو سلنه ده چې هره میاشت لغوه کوي. د ګډون پر بنسټ کاروبار لپاره، د پرېښودو کمول د نوو پېرېدونکو د ترلاسه کولو په څېر مهم دي.

**عام دلیلونه چې د فټنس سټوډیو پېرېدونکي پرېږدي:**

- هغوی ونه شو کړای پوه شي چې پلېټفارم څنګه وکاروي → د پیل کولو بهیر ښه کړئ
- سایټ کافي ښه نه ښکارېده → templates ښه کړئ
- هغوی ارزښت ونه لید → ځانګړتیاوې یا اړیکه ښه کړئ
- هغوی ارزانه بدیل وموند → خپل د ځانګړې برخې ارزښت پیاوړی کړئ
- د هغوی کاروبار وتړل شو → نه مخنیوی کېدونکی دی، خو دا جلا تعقیب کړئ

### د لوړولو هڅونه

هغه پېرېدونکي چې په پیل پلان کې دي او بریالي روان دي، باید لوړولو ته وهڅول شي:

- کله چې د پلان حدونو ته ورسېږي (سایټونه، زېرمه)، د لوړولو اشارې وښایئ
- هدفمند برېښنالیکونه واستوئ چې د وده پلان هغه ځانګړتیاوې روښانه کوي چې هغوی ترې ګټه اخیستلی شي
- وښایئ چې د وده/مسلکي پېرېدونکو څه جوړ کړي دي

### د بېرته‌راګرځولو کمپاینونه

کله چې پېرېدونکی لغوه کوي:

1. پوښتنه وکړئ ولې (د وتلو سروې یا برېښنالیک)
2. که امکان ولري، د هغوی اندېښنه حل کړئ
3. د بېرته راتګ لپاره تخفیف وړاندې کړئ (له لغوه‌کولو ۳۰-۶۰ ورځې وروسته)

## اوونیز او میاشتني معمولونه

### اوونیز

- نوې نوم‌لیکنې او لغوه‌کول وڅېړئ
- ټولو پرانیستو د ملاتړ ټکټونو ته ځواب ورکړئ
- د پلېټفارم کړنه او uptime وګورئ
- هرې ناکامې تادیې ته کتنه وکړئ

### میاشتنی

- کلیدي شمېرې وڅېړئ (MRR، پرېښودل، نوي پېرېدونکي، لوړونې)
- د WordPress او plugin تازه‌کول پلي کړئ (د staging ازموینې وروسته)
- د ملاتړ د بڼو پر بنسټ د پوهې زېرمه وڅېړئ او تازه یې کړئ
- پېرېدونکو ته خبرپاڼه یا تازه‌معلومات واستوئ (نوې ځانګړتیاوې، سپارښتنې، د فټنس صنعت خبرونه)

### ربع‌وار

- د سیالانو او د پېرېدونکو د feedback پر وړاندې بیې وڅېړئ
- د template ډیزاینونه وارزوئ -- ایا نوي کولو ته اړتیا لري؟
- د hosting ظرفیت وارزوئ -- ایا د پراخېدو اړتیا لرئ؟
- د فعالېدو د معلوماتو پر بنسټ د پیل کولو بهیر وڅېړئ او ښه یې کړئ

## هغه څه چې موږ په دې درس کې جوړ کړل

- **د ورځني څارنې کړنلارې** د uptime، نوم‌لیکنو، تادیاتو، او ملاتړ لپاره
- **د پوړیز ملاتړ جوړښت** چې د پلان له کچو سره برابر دی
- **د Billing عملیات** د dunning او cancellation اداره کولو په ګډون
- **د ساتنې کړنلارې** د تازه‌کارۍ، امنیت، او فعالیت لپاره
- **د churn کمولو ستراتیژۍ** چې د فټنس ځانګړې برخې ته ځانګړې دي
- **اونیزې، میاشتني، او درې‌میاشتنۍ عملیاتي کړنلارې**
