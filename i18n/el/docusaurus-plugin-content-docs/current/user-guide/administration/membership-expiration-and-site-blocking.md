# Λήξη Μητρώου και Αποκλεισμός Site

Αυτό το εγχειρίδιο εξηγεί πώς το Ultimate Multisite διαχειρίζεται τη λήξη των μητρώων, το τέλος των δοκιμαστικών περιόδων και τον αποκλεισμό του frontend του site. Περιγράφει τον κύκλο ζωής ενός μητρώου, από ενεργό έως λήξει, τις ρυθμίσεις που ελέγχουν αν τα sites αποκλείονται, και τι πρέπει να ελέγξετε όταν τα sites παραμένουν προσβάσιμα μετά τη λήξη ενός μητρώου.

## Κύκλος Ζωής του Μητρώου

Κάθε μητρώο στο Ultimate Multisite έχει μία από τις ακόλουθες καταστάσεις:

:::note Τα μητρώα δωρεάν είναι για πάντα
Τα μητρώα δωρεάν δεν λήγουν αυτόματα. Το Ultimate Multisite τα αντιμετωπίζει ως πρόσβαση για πάντα, οπότε δεν περιλαμβάνονται στους ελέγχους λήξης εκτός αν ένας διαχειριστής αλλάξει την κατάστασή τους ή μετακινήσει τον πελάτη σε ένα άλλο προϊόν.
:::

| Κατάσταση | Σημασία |
|---|---|
| **Pending** | Αναμένεται η πρώτη πληρωμή ή η επαλήθευση μέσω email |
| **Trialing** | Ενεργή δοκιμαστική περίοδος, δεν έχει συλλεχθεί ακόμη πληρωμή |
| **Active** | Πληρωμένο και τρέχον |
| **On Hold** | Αναμένεται η πληρωμή ανανέωσης (δημιουργήθηκε τιμολόγιο, αναμένεται πληρωμή) |
| **Expired** | Έπεσε η ημερομηνία λήξης και η περίοδος χάριτος χωρίς ανανέωση |
| **Cancelled** | Ακυρωμένο ρητά από τον πελάτη ή τον διαχειριστή |

### Πώς τα Μητρώα Μεταβαίνουν σε Λήξη

Το Ultimate Multisite εκτελεί έναν έλεγχο στο background **κάθε ώρα** για να αναζητήσει μητρώα που πρέπει να σημειωθούν ως λήξει. Αυτός ο έλεγχος χρησιμοποιεί το [Action Scheduler](https://actionscheduler.org/) (και όχι απευθείας το WP-Cron) και εκτελείται ως η προγραμματισμένη ενέργεια `wu_membership_check`.

Ο έλεγχος λήξης έχει μια **ενσωματωμένη περίοδο χάριτος 3 ημερών** εξ ορισμού. Ένα μητρώο δεν θα σημειωθεί ως `expired` μέχρι να περάσουν 3 ημέρες από την `date_expiration` του. Αυτό δίνει στους πελάτες χρόνο να ολοκληρώσουν μια καθυστερημένη πληρωμή πριν αλλάξει η κατάστασή τους.

:::info
Η περίοδος χάριτος λήξης των 3 ημερών είναι ξεχωριστή από τη ρύθμιση της Περιόδου Χάριτος Αποκλεισμού Frontend που περιγράφεται παρακάτω. Η περίοδος χάριτος λήξης ελέγχει πότε **αλλάζει η κατάσταση** από ενεργό/σε αναστολή σε λήξει. Η περίοδος χάριτος αποκλεισμού frontend ελέγχει πότε **αποκλείεται το site** αφού η κατάσταση έχει ήδη αλλάξει.
:::

#### Μητρώα με Αυτόματη και Χωρίς Αυτόματη Ανανέωση

Αυτή η διάκριση είναι κρίσιμη για την κατανόηση του συμπεριφορικού μοτίβου λήξης:

- **Μητρώα χωρίς αυτόματη ανανέωση** (`auto_renew = false`): Το ώρα-ανά-ώρα cron job διαχειρίζεται τον πλήρη κύκλο ζωής — δημιουργεί πληρωμές ανανέωσης, μεταβαίνει το μητρώο σε `on-hold` και τελικά το σημειώνει ως λήξει αν δεν παραληφθεί πληρωμή.

- **Μητρώα με αυτόματη ανανέωση** (`auto_renew = true`): Ο έλεγχος λήξης cron **παρακάμπτει αυτά εντελώς**. Αναμένεται ότι το payment gateway (Stripe, PayPal κ.λπ.) θα ενημερώσει το Ultimate Multisite μέσω webhooks όταν μια συνδρομή αποτύχει ή ακυρωθεί. Εάν δεν παραληφθεί το webhook — λόγω λανθαστικά ρυθμισμένου endpoint, διακοπής του gateway, ή συνδρομής που ακυρώθηκε εκτός του συστήματος — το μητρώο μπορεί να παραμείνει `active` απεριόριστα, ακόμα και μετά την πάροδο της ημερομηνίας λήξης.

### Πώς Τελειώνουν οι Δοκιμαστικές Περιόδους

Όταν τελειώνει η δοκιμαστική περίοδος ενός μητρώου, το σύστημα:

1. Δημιουργεί μια αναμενόμενη πληρωμή ανανέωσης με το πλήρες ποσό της συνδρομής
2. Μεταβαίνει την κατάσταση του μητρώου από `trialing` σε `on-hold`
3. Στέλνει ένα email ειδοποίησης πληρωμής ανανέωσης στον πελάτη

Αυτή η διαδικασία εκτελείται στο ίδιο ώρα-ανά-ώρα πρόγραμμα με τον κανονικό έλεγχο λήξης, αλλά **μόνο για τα μητρώα χωρίς αυτόματη ανανέωση**. Για τις δοκιμαστικές περιόδους με αυτόματη ανανέωση, το payment gateway διαχειρίζεται τη μετάβαση από δοκιμαστική σε πληρωμένη συνδρομή.

## Αποκλεισμός Frontend Πρόσβασης

Εξ ορισμού, όταν ένα μητρώο λήγει ή πάει σε αναστολή, **περιορίζεται μόνο το dashboard του wp-admin**. Το public frontend του site παραμένει προσβάσιμο στους επισκέπτες. Για να αποκλείσετε επίσης την δημόσια πρόσβαση, πρέπει να ενεργοποιήσετε τη ρύθμιση **Block Frontend Access**.

### Ρύθμιση της Ρύθμισης

Μεταβείτε στο **Ultimate Multisite > Settings > Memberships** και ενεργοποιήστε το **Block Frontend Access**.

![Block Frontend Access settings showing the toggle enabled, a grace period of 7 days, and a Frontend Block Page selection](/img/config/settings-membership-block-frontend.png)

Ακολουθεί μια πλήρης προβολή της σελίδας ρυθμίσεων μητρώου:

![Membership settings full page](/img/config/settings-membership-full.png)

Τρεις σχετικές ρυθμίσεις ελέγχουν αυτή τη συμπεριφορά:

| Setting | Description | Default |
|---|---|---|
| **Block Frontend Access** | Κύριος διακόπτης. Όταν είναι ενεργοποιημένος, αποκλείει το δημόσιο frontend των sites του network όταν το μητρώο τους δεν είναι πλέον ενεργό. | Off |
| **Frontend Block Grace Period** | Αριθμός ημερών που πρέπει να περιμένετε αφού το μητρώο γίνει μη ενεργό πριν τον αποκλεισμό. Ρυθμίστε σε `0` για άμεσο αποκλεισμό. | 0 |
| **Frontend Block Page** | Μια σελίδα στο κύριο site για να ανακατευθύνετε τους επισκέπτες όταν το site αποκλειστεί. Εάν δεν έχει ρυθμιστεί, οι επισκέπτες βλέπουν ένα γενικό μήνυμα "Site not available". | None |

### Τι Βλέπουν οι Επισκέπτες Όταν Αποκλείεται ένα Site

Όταν ο αποκλεισμός του frontend είναι ενεργός, οι επισκέπτες του site θα:

1. **Επανακατευθυνθούν** στη σελίδα που επιλέξατε στο **Frontend Block Page** (εάν είναι ρυθμισμένο), ή
2. **Να δουν ένα προεπιλεγμένο μήνυμα**: "This site is not available at the moment." με έναν σύνδεσμο στη σελίδα σύνδεσης για τον διαχειριστή του site.

Οι διαχειριστές των sites μπορούν ακόμα να συνδεθούν — η σελίδα σύνδεσης δεν αποκλείεται ποτέ.

### Τι Αποκλείεται και Πότε

Η συμπεριφορά του αποκλεισμού εξαρτάται από την κατάσταση του μητρώου:

| Membership Status | Frontend Blocked? | Grace Period Applied? |
|---|---|---|
| Active | No | -- |
| Trialing | **No** (βλ. σημείωση παρακάτω) | -- |
| On Hold | Θεωρείται ενεργό -- δεν αποκλείεται | -- |
| Expired | **Yes**, αν το Block Frontend Access είναι ενεργό | Yes |
| Cancelled | **Yes**, πάντα (ανεξάρτητα από τη ρύθμιση) | **No** -- αποκλείεται αμέσως |
| Pending | Δεν αποκλείεται μέσω του ελέγχου μητρώου | -- |

:::warning Τα μητρώα σε δοκιμαστική περίοδο δεν αποκλείονται
Ακόμα κι αν έχει τελειώσει μια δοκιμαστική περίοδος, ένα μητρώο με κατάσταση `trialing` **δεν** θα αποκλειστεί στο frontend. Η δοκιμαστική περίοδος πρέπει πρώτα να μεταβεί σε μια άλλη κατάσταση (συνήθως `on-hold` μέσω του cron job, και μετά `expired` αν δεν πληρωθεί). Εάν βλέπετε μητρώα σε δοκιμαστική περίοδο που δεν έχουν μεταβεί, ελέγξτε την ενότητα απο₂)

:::note Τα μητρώα που ακυρώθηκαν παρακάμπτουν την περίοδο χάριτος
Τα μητρώα που ακυρώθηκαν αποκλείονται πάντα μόλις περάσει η ημερομηνία λήξης, ανεξάρτητα από το αν είναι ενεργό το Block Frontend Access. Η Περίοδος Χάριτος Αποκλεισμού Frontend **δεν** εφαρμόζεται στα ακυρωμένα μητρώα.
:::

## Αντιμετώπιση Προβλημάτων: Sites που Παραμένουν Προσβάσιμα Μετά τη Λήξη

Εάν τα sites παραμένουν δημόσια προσβάσιμα μετά τη λήξη ενός μητρώου, περάστε από τους ακόλουθους ελέγχους με σειρά:

### 1. Επιβεβαιώστε ότι η Ρύθμιση Block Frontend Access Είναι Ενεργή

Μεταβείτε στο **Ultimate Multisite > Settings > Memberships** και επιβεβαιώστε ότι ο διακόπτης **Block Frontend Access** είναι ενεργός. Αυτή η ρύθμιση είναι **απενεργοποιημένη εξ ορισμού**, πράγμα που σημαίνει ότι μόνο το wp-admin περιορίζεται όταν ένα μητρώο γίνεται μη ενεργό.

### 2. Ελέγξτε την Περίοδο Χάριτος Αποκλεισμού Frontend

Στην ίδια σελίδα ρυθμίσεων, ελέγξτε την τιμή του **Frontend Block Grace Period**. Εάν αυτή είναι ρυθμισμένη σε 7 ημέρες, για παράδειγμα, το frontend δεν θα αποκλειστεί μέχρι να περάσουν 7 ημέρες από την ημερομηνία λήξης του μητρώου — ακόμα κι αν η κατάσταση του μητρώου είναι ήδη `expired`.

Ρυθμίστε το σε `0` εάν θέλετε άμεσο αποκλεισμό αφού το μητρώο γίνει μη ενεργό.

### 3. Επιβεβαιώστε ότι Η Κατάσταση του Μητρώου Έχει Πραγματικά Αλλάξει

Μεταβείτε στο **Ultimate Multisite > Memberships** και ελέγξτε την κατάσταση του επηρεαζόμενου μητρώου. Εάν ακόμα εμφανίζει `active` παρόλο που έχει περάσει η ημερομηνία λήξης, η μετάβαση της κατάστασης δεν έχει πραγματοποιηθεί. Συνηθισμένες αιτίες:

- **Το μητρώο είναι με αυτόματη ανανέωση**: Ελέγξτε το πεδίο `auto_renew` στη σελίδα επεξεργασίας του μητρώου. Εάν η αυτόματη ανανέωση είναι ενεργή, το cron job λήξης παρακάμπτει αυτό το μητρώο — βασίζεται στο payment gateway για να αναφέρει την αποτυχία. Ελέγξτε το dashboard του gateway σας (Stripe, PayPal) για να επιβεβαιώσετε ότι η κατάσταση της συνδρομής ταιριάζει με αυτό που εμφανίζει το Ultimate Multisite.

- **Το cron job δεν έχει εκτελεστεί**: Βλ. το επόμενο βήμα.

### 4. Επιβεβαιώστε ότι το Action Scheduler Τρέχει

Το Ultimate Multisite χρησιμοποιεί το Action Scheduler για τα cron jobs του. Μεταβείτε στο **Tools > Scheduled Actions** στο network admin και αναζητήστε:

- **`wu_membership_check`** — Αυτό πρέπει να εμφανίζεται ως μια επαναλαμβανόμενη ενέργεια που τρέχει κάθε ώρα. Εάν λείπει, οι ελέγχους μητρώων δεν προγραμματίζονται.
- **`wu_async_mark_membership_as_expired`** — Αυτές είναι μεμονωμένες εργασίες για να σημειώσουν συγκεκριμένα μητρώα ως λήξει. Εάν βλέπετε αποτυχημένες ενέργειες εδώ, ενδέχεται να περιέχουν μηνύματα σφάλματος που εξηγούν γιατί.

Εάν βλέπετε κολλημένες ή αποτυχημένες ενέργειες, μπορεί να έχετε πρόβλημα με το Action Scheduler. Συνηθισμένες αιτίες:

- Το `DISABLE_WP_CRON` είναι ρυθμισμένο σε `true` στο `wp-config.php` χωρίς αντικατάστατο cron σε επίπεδο συστήματος
- Χαμηλή κίνηση στο site — το WP-Cron τρέχει μόνο όταν κάποιος επισκέπτεται το site

Για να διασφαλίσετε αξιόπιστη εκτέλεση cron, ρυθμίστε ένα system cron job:

```bash
# Run every 5 minutes via wget
*/5 * * * * wget -q -O /dev/null "https://your-network-url.com/wp-cron.php?doing_wp_cron"

# Or via WP-CLI
*/5 * * * * cd /path/to/wordpress && wp cron event run --due-now --url=https://your-network-url.com
```

### 5. Έλεγχος Προβλημάτων Webhook Gateway (Μητρώα με Αυτόματη Ανανέωση)

Εάν το μητρώο είναι με αυτόματη ανανέωση και η συνδρομή του gateway έχει ακυρωθεί ή έχει αποτύχει, αλλά το Ultimate Multisite το εμφανίζει ακόμα ως `active`:

- **Stripe**: Μεταβείτε στο Stripe Dashboard > Customers και ελέγξτε την κατάσταση της συνδρομής. Στη συνέχεια, επιβεβαιώστε ότι το endpoint του webhook είναι ενεργό κάτω από Developers > Webhooks. Το endpoint πρέπει να δείχνει στο site σας και να δείχνει επιτυχημένες παραδόσεις.
- **PayPal**: Ελέγξτε την κατάσταση της συνδρομής στον λογαριασμό σας PayPal business και επιβεβαιώστε την παράδοση IPN/webhook.

Εάν το gateway δείχνει ότι η συνδρομή έχει ακυρωθεί, αλλά το Ultimate Multisite όχι, το μήνυμα webhook πιθανότατα χάθηκε. Μπορείτε να αλλάξετε χειροκίνητα την κατάσταση του μητρώου στο **Ultimate Multisite > Memberships > [Edit Membership]**.

### 6. Έλεγχος της Περιόδου Χάριτος Λήξης (Επίπεδο Cron)

Ο έλεγχος cron έχει τη δική του περίοδο χάριτος (εξ ορισμού: 3 ημέρες) πριν σημειώσει ένα μητρώο ως λήξει. Αυτό είναι ξεχωριστό από τη περίοδο χάριτος αποκλεισμού frontend. Ο συνολικός χρόνος πριν το site αποκλειστεί μπορεί να είναι:

**Περίοδος χάριτος λήξης (3 ημέρες)** + **Περίοδος χάριτος αποκλεισμού frontend (η ρύθమిσή σας)** = Συνολική καθυστέρηση

Για παράδειγμα, με τις προεπιλεγμένες ρυθμίσεις και μια περίοδο χάριτος frontend 7 ημερών, μπορεί να περάσουν έως και 10 ημέρες μετά την `date_expiration` πριν το site αποκλειστεί πραγματικά.

### 7. Χειροκίνητη Λήξη ενός Μητρώου

Εάν χρειάζεται να αποκλείσετε αμέσως ένα site χωρίς να περιμένετε τον κύκλο του cron, μπορείτε να αλλάξετε χειροκίνητα την κατάσταση του μητρώου:

1. Μεταβείτε στο **Ultimate Multisite > Memberships**
2. Κάντε κλικ στο επηρεαζόμενο μητρώο
3. Αλλάξτε την κατάσταση σε **Expired** ή **Cancelled**
4. Κάντε κλικ στο **Save**

Ο αποκλεισμός του frontend θα τίθεται σε εφαρμογή στην επόμενη φόρτωση σελίδας (υπό την περίοδο χάριτος αποκλεισμού frontend για τα λήξευσα μητρώα, ή αμέσως για τα ακυρωμένα).
