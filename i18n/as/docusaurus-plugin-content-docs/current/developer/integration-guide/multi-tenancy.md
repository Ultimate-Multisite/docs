# Multi-Tenancy সংহতি

Ultimate Multisite: Multi-Tenancy 1.2.0-এ স্বাধীন টেনেণ্ট, মাইগ্ৰেশ্যন যাচাই, আৰু টেনেণ্ট জীৱনচক্ৰ স্বয়ংক্ৰিয়কৰণৰ বাবে কেইবাটাও সংহতি স্পৰ্শবিন্দু সলনি কৰা হৈছে।

## টেনেণ্ট বুটষ্ট্ৰেপ প্ৰবাহ

টেনেণ্ট সৃষ্টি বা সলনি কৰা সংহতিসমূহে এই ক্ৰম অনুসৰণ কৰিব লাগে:

1. টেনেণ্ট ৰেজিষ্ট্ৰি ৰেকৰ্ড আৰু পৃথকীকৰণ মডেল সমাধান কৰক।
2. টেনেণ্ট ডাটাবেছ লেখক সৃষ্টি বা যাচাই কৰক।
3. টেনেণ্ট স্কিমা বুটষ্ট্ৰেপ কৰক।
4. টেনেণ্ট ব্যৱহাৰকাৰী প্ৰভিজন কৰক।
5. টেনেণ্ট ৰাউটিং আৰু ফাইলচিষ্টেম পথ পঞ্জীয়ন কৰক।
6. টেনেণ্ট প্ৰকাশ কৰাৰ আগতে মাইগ্ৰেশ্যন যাচাই চলাওক।

এটা স্বাধীন টেনেণ্টে নেটৱৰ্ক ডাটাবেছ সংযোগ পুনৰ ব্যৱহাৰ কৰিব পাৰে বুলি ধৰি নল’ব। addon-এ প্ৰদান কৰা টেনেণ্ট ৰেজিষ্ট্ৰি আৰু লেখক বিমূৰ্তকৰণ ব্যৱহাৰ কৰক।

## SSO আৰু REST হুকসমূহ

ষ্টেটলেছ টেনেণ্ট অট’লগইনে purpose claim, JTI পুনৰপ্লে সুৰক্ষা, মেয়াদ-উকলাৰ সৰ্বোচ্চ সীমা, আৰু origin pinning থকা অল্প-মেয়াদী টোকেন ব্যৱহাৰ কৰে। লগইন বুটাম বা দূৰৱৰ্তী পৰিচালনা লিংক যোগ কৰা সংহতিসমূহে টেনেণ্ট লগইন URL প্ৰত্যক্ষভাৱে নিৰ্মাণ কৰাৰ সলনি সমৰ্থিত SSO প্ৰবাহৰ জৰিয়তে টেনেণ্ট ভিজিট সৃষ্টি কৰিব লাগে।

স্বাধীন টেনেণ্ট গেটৱেৰ বাবে নেটৱৰ্ক-পক্ষ API অডিট ঘটনা আৰু দৈনিক সাৰাংশ উপলব্ধ। টেনেণ্ট জীৱনচক্ৰ endpoint কল কৰা বাহ্যিক চিষ্টেমসমূহ ডিবাগ কৰাৰ সময়ত সেই লগসমূহ ব্যৱহাৰ কৰক।

## স্বাধীন গ্ৰাহক কাৰ্য URLসমূহ

Ultimate Multisite v2.13.0-এ account, checkout, billing, invoice, site, টেমপ্লেট-সলনি, আৰু domain-mapping প্ৰবাহৰ বাবে স্বাধীন-টেনেণ্ট গ্ৰাহক কাৰ্যসমূহ মুখ্য ছাইটলৈ ঘূৰাই ৰাউট কৰে। টেনেণ্ট-পক্ষ পৰিচালনা লিংক ৰেণ্ডাৰ কৰা সংহতিসমূহে সেই কাৰ্যসমূহ মুখ্য-ছাইট গ্ৰাহক পেনেললৈ নিৰ্দেশ কৰিব লাগে আৰু কাৰ্য সম্পূৰ্ণ কৰাৰ পিছত ব্যৱহাৰকাৰীয়ে টেনেণ্টলৈ উভতি যাব পাৰিব লাগে যেতিয়া, এটা যাচাইকৃত উভতি-যোৱাৰ লক্ষ্য অন্তৰ্ভুক্ত কৰিব লাগে।

ক্ৰছ-ড’মেইন পৰিচালনা লিংকৰ বাবে মূল SSO wrapper ব্যৱহাৰ কৰক:

```php
$url = wu_with_sso($main_site_customer_url);
```

উৎপন্ন URL `wu_sso_url`-ৰ জৰিয়তে ফিল্টাৰ কৰিব পাৰি, যিয়ে SSO URL, বৰ্তমান ব্যৱহাৰকাৰী, লক্ষ্য ছাইট ID, আৰু redirect প্ৰসংগ গ্ৰহণ কৰে। Add-onসমূহে প্ৰদানকাৰী-নিৰ্দিষ্ট প্ৰসংগ সংযোজন কৰিবলৈ বা Ultimate Multisite-ৰ টোকেন যাচাই অক্ষুণ্ণ ৰাখি broker URL সলনি কৰিবলৈ সেই ফিল্টাৰ ব্যৱহাৰ কৰিব পাৰে।

স্বাধীন টেনেণ্টৰ ভিতৰত membership, invoice, billing-address, template, বা domain-management অৱস্থা নকল নকৰিব। টেনেণ্ট Dashboard-ক launcher হিচাপে আৰু মুখ্য-ছাইট গ্ৰাহক পেনেলক পৰিচালিত কাৰ্যসমূহৰ system of record হিচাপে গণ্য কৰক।

## মাইগ্ৰেশ্যন যাচাই

মাইগ্ৰেশ্যন বা জীৱনচক্ৰ সংহতিয়ে টেনেণ্ট ডাটা সলনি কৰাৰ পিছত, যাচাই গেটসমূহ চলাওক:

- `wp tenant verify-no-legacy --site=<site-id>`-এ নিশ্চিত কৰে যে টেনেণ্টে আৰু legacy নেটৱৰ্ক-পক্ষ পথৰ ওপৰত নিৰ্ভৰ নকৰে।
- `wp tenant verify-sovereign-push --site=<site-id>`-এ নিশ্চিত কৰে যে স্বাধীন push কামসমূহ প্ৰক্ৰিয়াকৰণ আৰু drain কৰিব পাৰি।

সংহতিসমূহে ব্যৰ্থ যাচাইক deployment blocker হিচাপে গণ্য কৰিব লাগে আৰু ব্যৰ্থতা সমাধান নোহোৱালৈকে টেনেণ্ট live হিচাপে চিহ্নিত কৰাৰ পৰা বিৰত থাকিব লাগে।

## টেনেণ্ট বিলোপ

বিলোপ প্ৰবাহসমূহে addon teardown পথ কল কৰিব লাগে যাতে টেনেণ্ট ডাটাবেছ credentialsসমূহ পৰিষ্কাৰ কৰা হয়। বাহ্যিক সংহতিসমূহে teardown সফল হোৱাৰ পিছত প্ৰদানকাৰী সম্পদ আঁতৰাব পাৰে, কিন্তু যাচাই বা async push কামসমূহ এতিয়াও চলি থকা অৱস্থাত host ডাটাবেছ বা folder মচিব নালাগে।

## Deprecated ডাটাবেছ ৰাউটাৰ

legacy `Database_Router`-ক deprecation stub-ৰে সলনি কৰা হৈছে। নতুন সংহতিসমূহে পুৰণি router class-ৰ ওপৰত নিৰ্ভৰ কৰাৰ সলনি বৰ্তমান ছাইট ৰাউটাৰ আৰু টেনেণ্ট ৰেজিষ্ট্ৰি APIসমূহৰ জৰিয়তে টেনেণ্ট সমাধান কৰিব লাগে।
