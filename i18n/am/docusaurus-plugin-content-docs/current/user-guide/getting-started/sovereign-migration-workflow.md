# souverain Migration Workflow

Ultimate Multisite: Multi-Tenancy 1.2.0 ஆனது ஒரு நிலையான துணைதளத்தை (standard subsite) இறையாண்மை வாடிக்கையாளர் கட்டமைப்பிற்கு (sovereign tenant infrastructure) மாற்றுவதற்கான இடம்பெயர்வு சரிபார்ப்பு வாயில்களைச் (migration verification gates) சேர்க்கிறது.

## நீங்கள் தொடங்குவதற்கு முன் {#before-you-start}

உங்களிடம் உள்ளதை உறுதிப்படுத்தவும்:

- மூல தளத்தின் தற்போதைய காப்புப்பிரதி (backup).
- மல்டி-டென்டன்சி அட்-ஆட்-இன் (Multi-Tenancy addon) செயலில் உள்ளது.
- இலக்கு வாடிக்கையாளர் தரவுத்தளம் (destination tenant database), கோப்பு முறைமை வேர் (filesystem root), மற்றும் டொமைன் தயாராக உள்ளன.
- இலக்கு சூழலுக்குப் பொருந்தக்கூடிய ஒரு தரவுத்தள ஹோஸ்ட் பிணைப்பு (database host binding).
- நெட்வொர்க்கிற்கு WP-CLI கட்டளைகளை இயக்க அணுகல்.

## பரிந்துரைக்கப்பட்ட பணிப்பாய்வு (Recommended workflow) {#recommended-workflow}

1. இலக்கு வாடிக்கையாளர் தரவுத்தளம் மற்றும் கோப்பு முறைமையை தயார் செய்யவும்.
2. வாடிக்கையாளர் தனிமைப்படுத்தல் அமைப்புகளைப் பதிவு செய்யவும் அல்லது புதுப்பிக்கவும்.
3. வாடிக்கையாளர் இடம்பெயர்வு (tenant migration) இயக்கவும்.
4. வாடிக்கையாளர் பயனர்களை வழங்கவும் அல்லது சரிபார்க்கவும்.
5. ஒத்திசைவற்ற இடம்பெயர்வு வேலைகளை (async migration jobs) வெளியேற்றவும் (drain).
6. இறையாண்மை இடம்பெயர்வு சரிபார்ப்பை இயக்கவும்.
7. SSO உடன் வாடிக்கையாளரை அணுகவும் (Visit with SSO).
8. சரிபார்ப்பு வெற்றி பெற்ற பின்னரே DNS அல்லது ரூட்டிங்கை மாற்றவும்.

## சரிபார்ப்பு வாயில்கள் (Verification gates) {#verification-gates}

சரிபார்ப்பு பணிப்பாய்வு பல கோணங்களில் இடம்பெயர்ப்பைச் சரிபார்க்கிறது:

- வாடிக்கையாளர் ஸ்கீமா (schema) இருக்க வேண்டும் மற்றும் வாடிக்கையாளர் தரவுத்தள எழுத்தாளரால் எழுதப்பட வேண்டும்.
- கட்டமைக்கப்பட்ட தரவுத்தள ஹோஸ்டுக்கு சரியான அனுமதிகள் (grants) இருக்க வேண்டும்.
- வாடிக்கையாளர் பயனர்கள் இருக்க வேண்டும் மற்றும் எதிர்பார்க்கப்படும் இறையாண்மை நிறுவப்பட்ட பயனர் எண்ணிக்கையுடன் பொருந்த வேண்டும்.
- ஒத்திசைவற்ற பஷ் வரிசையை (async push queue) வெற்றிகரமாக வெளியேற்ற முடியும்.
- இறையாண்மை வாடிக்கையாளருக்கு இனி பழைய நெட்வொர்க் பக்க தரவுப் பாதைகள் தேவையில்லை.

சரிபார்ப்பு தோல்விகளை வெளியீட்டிற்கு முன் தடுக்கக்கூடிய தடைகளாகக் (pre-launch blockers) கருதுங்கள். புகாரளிக்கப்பட்ட தரவுத்தளம், பயனர், வரிசை அல்லது ரூட்டிங் சிக்கலை சரிசெய்த பிறகு, வாடிக்கையாளருக்கு வெளிப்படுத்துவதற்கு முன்பு மீண்டும் சரிபார்க்கவும்.

## முதல் உற்பத்தி வருகை (First production visit) {#first-production-visit}

சரிபார்ப்பு வெற்றி பெற்ற பிறகு, முதல் நிர்வாக வருகைக்காக தள மேலாண்மை திரையில் **அணுகுதல் (Visit (SSO))** ஐப் பயன்படுத்தவும். இது ஒரு கட்டுப்படுத்தப்பட்ட படிநிலையில் வாடிக்கையாளர் ரூட்டிங், SSO டோக்கன் கையாளுதல், தோற்றம் பிணைப்பு (origin pinning), மற்றும் வாடிக்கையாளர் பக்க பயனர் வழங்குதலை உறுதி செய்கிறது.
