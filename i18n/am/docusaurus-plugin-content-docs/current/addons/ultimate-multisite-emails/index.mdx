# ኢሜይሎች (Emails)

የUltimate Multisite: Emails addon የኔትወርክ ባለቤቶች ከድረ-ገጾች እና አባላትነት (memberships) ጋር ተያይዘው የደንበኞችን የኢሜይል አካውንት እንዲያዘጋጁ እና እንዲያስተዳድሩ ያስችላል። የ1.1.0 ስሪት የደንበኞችን የኢሜይል አካውንት አስተዳደር በደንበኞች በኩል፣ የwebmail single sign-on፣ የCyberPanel forwarder ድጋፍ እና በቼክአውት ጊዜ የተሻሻለ አቅርቦት አቅምን ይጨምራል።

## ዋና ዋና ባህሪያት (Key Features)

* **የደንበኛ የኢሜይል አካውንት widget**: ደንበኞች የድጋፍ ቲኬት ሳይከፍሉ በቀጥታ ከየአካውንት ገጻቸው የኢሜይል አካውንት መፍጠር እና ማስተዳደር ይችላሉ።
* **Webmail SSO እና magic links**: ተቋራጩ የሚደግፍ ከሆነ ደንበኞች በቀጥታ ከአካውንት ገጽ የwebmail አገልግሎት መግባት ይችላሉ።
* **የይለፍ ቃል ለውጥ**: ደንበኞች እና አስተዳዳሪዎች የኢሜይል አካውንት አስተዳደር widget በመጠቀም የሜይልቦክስ የይለፍ ቃላቸውን ሊያዘምኑ ይችላሉ።
* **በቼክአውት ጊዜ አቅርቦት**: የደንበኛ የኢሜይል አካውንት መፍጠር በቼክአውት ሂደት ውስጥ ሊከናወን ይችላል። ይህም የገንዘብ ክፍያ ሂደት ከተጠናቀቀ በኋላ ሜይልቦክሱ ዝግጁ እንዲሆን ያደርጋል።
* **የCyberPanel forwarders**: በCyberPanel የተደገፉ ኔትወርኮች የኢሜይል forwarders መፍጠር እና ማስተዳደር ይችላሉ።
* **በአቅራቢው ላይ የተመሰረቱ ቅንብሮች**: የቅንብሮች ገጽ አስተዳዳሪዎች ሊጠቀሙባቸው የሚችሉ አማራጮችን ብቻ እንዲያዩ በማድረግ አቅራቢዎችን ያጣራል (filters)።

## መስፈርቶች (Requirements)

* Ultimate Multisite 2.0.0 ወይም ከዚያ በላይ
* WordPress Multisite መጫን
* ቢያንስ አንድ የተዋቀረ የኢሜይል አቅራቢ ውህደት (provider integration)
* የሜይልቦክስ ለመፍጠር፣ የይለፍ ቃሎችን ለመቀየር እና የwebmail መዳረሻ ሊንኮችን ለማውጣት ስልጣን ያላቸው የአቅራቢ API መለያዎች

## መጫን (Installation)

1. የፕልጋይኑ ፋይሎችን ወደ `/wp-content/plugins/ultimate-multisite-emails` ይጫኑ።
2. በWordPress network admin ውስጥ ከ**Plugins** አማካይነት የaddonን በኔትወርክ ደረጃ ያንቁ።
3. ወደ **Ultimate Multisite** > **Settings** ይሂዱ እና የaddon የኢሜይል ቅንብሮችን ይክፈቱ።
4. ደንበኞች እንዲጠቀሙባቸው የሚፈልጓቸውን የኢሜይል አቅራቢ ውህደቶች ያንቁ።
5. ቅንብሮችን ያስቀምጡ (Save settings)፣ ከዚያም በሙከራ ደንበኛ ወይም በstaging site ላይ አካውንት መፍጠርን ይሞክሩ።

## የደንበኛ አካውንት የኢሜይል አስተዳደር (Customer Account Email Management)

የ1.1.0 ስሪት ለደንበኛ አካውንት ገጽ የኢሜይል አስተዳደር widget ይጨምራል። ደንበኞች ለየራሳቸው ድረ-ገጽ የሜይልቦክስ መፍጠር እና ጥገና በራሳቸው እንዲያከናውኑ (self-service) በሚፈልጉበት ጊዜ ይጠቀሙ።

ከአካውንት ገጹ ላይ ደንበኞች የሚከተሉትን ማድረግ ይችላሉ፦

1. ከአሁኑ የአባላትነት ስምምነታቸው ወይም ድረ-ገጻቸው ጋር የተገናኙ የኢሜይል አካውንቶችን ማየት።
2. ዕቅዳቸው ተጨማሪ አካውንቶችን የሚፈቅድ ከሆነ አዲስ ሜይልቦክስ መፍጠር።
3. ያለውን የሜይልቦክስ የይለፍ ቃል መቀየር።
4. በSSO ወይም magic-link login በኩል የሚደገፉ የwebmail አቅራቢዎችን መክፈት።

ይህ widget የUltimate Multisite account-page currents systemን ስለሚጠቀም፣ የሚደረጉ ድርጊቶች ደንበኛው እያየበት ባለው አካውንት ላይ ብቻ የተገደቡ ይሆናሉ። ይህ ደንበኞች በሌላ የአባላትነት ስምምነት ወይም ድረ-ገጽ የሆኑ የኢሜይል አካውንቶችን እንዳይቆጣጠሩ ይከላከላል።

## Webmail SSO እና Magic-Link Login

Webmail SSO ደንበኞች ከአካውንት ገጽ ላይ በቀጥታ **Open webmail** የሚል እርምጃ ይሰጣል። በንቁ አቅራቢው የሚደገፍ ከሆነ፣ addon አጭር ጊዜ የሚቆይ የመዳረሻ URL ይጠይቃል እና ደንበኛውን የሜይልቦክስ መለያ ዝርዝሮችን እንዲገልጽ ሳይጠይቅ ወደ webmail ይመራዋል።

የአቅራቢዎች ባህሪ ሊለያይ ይችላል፦

* ተፈጥሯዊ SSO APIs ያላቸው አቅራቢዎች ቀጥተኛ የተረጋገጠ የwebmail ክፍለ ጊዜ መፍጠር ይችላሉ።
* magic-link ድጋፍ ያላቸው አቅራቢዎች ጊዜ የተወሰነ የlogin ሊንክ መፍጠር ይችላሉ።
* SSO ወይም magic-link ድጋፍ የሌላቸው አቅራቢዎች የሜይልቦክስ ዝርዝሮችን እና የይለፍ ቃል አስተዳደር ድርጊቶችን ማሳየት አለባቸው፣ ነገር ግን አንድ-በ-ጫን (one-click) webmail ቁልፍ አይሰጡም።

የተፈለገውን API መለያዎች ካዘጋጁበት አቅራቢ ላይ ብቻ የwebmail መዳረሻን ያንቁ። ድርጊቱ ለአቅራቢው ከሌለ፣ በመጀመሪያ አቅራቢው ንቁ መሆኑን፣ ተዋቅሮ መሆኑን እና የwebmail login ሊንኮችን እንደሚደግፍ ያረጋግጡ።

## በቼክአውት ጊዜ አቅርቦት (Checkout Provisioning)

addon የደንበኛ የኢሜይል አካውንቶችን በቼክአውት ሂደት አካል መፍጠር ይችላል። ይህ ዕቅዱ በነባሪነት የሜይልቦክስን የሚያካትት ከሆነ ወይም የቼክአውት ቅጽ የኢሜይል አካውንት ዝርዝሮችን የሚሰበስብ ከሆነ ጠቃሚ ነው።

የሚመከር አተገባበር (Recommended rollout):

1. የቼክአውት አቅርቦትን ከመንቃትዎ በፊት አቅራቢውን እና የዕቅድ ገደቦችን ያዘጋጁ።
2. በstaging product ላይ ግዢ በማድረግ እና ከቼክአውት በኋላ ሜይልቦክሱ መፈጠሩን በማረጋገጥ ይሞክሩ።
3. አዲሱ ሜይልቦክስ በኢሜይል widget ውስጥ መታየቱን ለማረጋገጥ የደንበኛ አካውንት ገጽን ይገምግሙ።
4. የሂደቱን ለproduction products ከመንቃትዎ በፊት የይለፍ ቃል ለውጥ እና webmail ድርጊቶችን ያረጋግጡ።

አቅርቦት ቢከሽፍም፣ የደንበኛው አካውንት እና የአባላትነት ስምምነት ሊኖሩ ይችላሉ፣ ነገር ግን ሜይልቦክሱ የአስተዳዳሪ ክትትል ይፈልጋል። እንደገና ከመሞከርዎ በፊት የUltimate Multisite activity logs እና የአቅራቢ logs ይፈትሹ።

## የCyberPanel የኢሜይል forwarders

ለCyberPanel የተደገፉ ኔትወርኮች፣ የ1.1.0 ስሪት የኢሜይል forwarder ድጋፍ ይጨምራል። ደንበኞች ሙሉ ሜይልቦክስ ሳይፈጥሩ እንደ `info@customer-domain.test` ባሉ አድራሻዎች መልእክት ወደ ሌላ የኢንቦክስ እንዲላክ የሚያስፈልጋቸው ጊዜ forwarders ይጠቀሙ።

forwarders ከመስጠትዎ በፊት፣ የCyberPanel ውህደት (integration) ተዋቅሮ መሆኑን እና መድረስ እንደሚችል ያረጋግጡ። የሚያስፈልጉትን host, username, password እና package constants ለማግኘት [CyberPanel Integration](../../user-guide/administration/cyberpanel-integration) መመሪያውን ይመልከቱ።

## በተደጋጋሚ የሚጠየቁ ጥያቄዎች (Frequently Asked Questions)

## ደንበኞች የራሳቸውን የኢሜይል አካውንት መፍጠር ይችላሉ?

አዎ። የደንበኛ አካውንት widget ሲበራ እና ዕቅዱ የኢሜይል አካውንቶችን የሚፈቅድ ከሆነ፣ ደንበኞች የራሳቸውን ሜይልቦክስ ከአካውንት ገጽ መፍጠር እና ማስተዳደር ይችላሉ።

## እያንዳንዱ አቅራቢ webmail SSOን ይደግፋል?

አይ። የSSO እና magic-link ድጋፍ የሚወሰነው በንቁ የኢሜይል አቅራቢ ላይ ነው። የመዳረሻ-ሊንክ API የሌላቸው አቅራቢዎች አሁንም የሜይልቦክስ መፍጠር እና የይለፍ ቃል ለውጥን መደገፍ ይችላሉ።

## አስተዳዳሪዎች ለደንበኞች የኢሜይል አካውንት መፍጠር ይችላሉ?

አዎ። አስተዳዳሪዎች የአካውንቱን ሲፈጥሩ የመጀመሪያውን የይለፍ ቃልን ማዘጋጀትን ጨምሮ ከአስተዳዳሪው በይነገጽ የኢሜይል አካውንቶችን መፍጠር ይችላሉ።

## forwarders ከሜይልቦክስ ጋር እኩል ናቸው?

አይ። የሜይልቦክስ መልእክት የሚያስቀምጥ እና የlogin መለያዎች አሉት። forwarder መልእክትን ወደ ሌላ አድራሻ የሚያመላክል (redirect) እና የተለየ የኢንቦክስ ቦታ አይሰጥም።
