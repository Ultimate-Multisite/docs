# 会員資格の失効とサイトのブロック {#membership-expiration-and-site-blocking}

本ガイドでは、Ultimate Multisiteが会員資格の失効、トライアル期間の終了、およびフロントエンドのサイトブロックをどのように処理するかを説明します。アクティブな状態から失効するまでの会員資格のライフサイクル、サイトがブロックされるかどうかを制御する設定、そして会員資格が失効した後もサイトにアクセスできる場合に確認すべき点について解説します。

## 会員資格のステータスライフサイクル {#membership-status-lifecycle}

Ultimate Multisiteのすべての会員資格は、以下のいずれかのステータスを持っています。

:::note 無料会員資格は永続的です
無料会員資格は自動的に失効しません。Ultimate Multisiteはこれらを永続的なアクセスとして扱い、管理者がステータスを変更したり、顧客を別の製品に移行させたりしない限り、失効チェックの対象にはなりません。
:::

| ステータス | 意味 |
|---|---|
| **Pending** | 初回支払いまたはメール認証待ち |
| **Trialing** | アクティブなトライアル期間。まだ支払いは行われていません。 |
| **Active** | 有料で有効な状態 |
| **On Hold** | 更新支払いが保留中（請求書が作成され、支払いを待っている状態） |
| **Expired** | 更新されず、失効日および猶予期間を過ぎた状態 |
| **Cancelled** | 顧客または管理者が明示的にキャンセルした状態 |

### 会員資格が失効する仕組み {#how-memberships-transition-to-expired}

Ultimate Multisiteは、失効すべき会員資格がないかを確認するバックグラウンドチェックを**毎時間**実行します。このチェックは、[Action Scheduler](https://actionscheduler.org/)（WP-Cronを直接使用するわけではありません）を使用して、`wu_membership_check`というスケジュールされたアクションとして実行されます。

失効チェックには、デフォルトで**3日間の組み込み猶予期間**があります。会員資格は、`date_expiration`を過ぎてから3日間が経過するまで、`expired`としてマークされません。これにより、顧客がステータスが変更される前に支払い遅延を解消する時間を与えます。

:::info
3日間の失効猶予期間は、後述するフロントエンドブロック猶予期間とは別物です。失効猶予期間は、ステータスがアクティブ/保留から失効に**変更されるタイミング**を制御します。一方、フロントエンドブロック猶予期間は、ステータスがすでに変更された後、サイトが**ブロックされるタイミング**を制御します。
:::

#### 自動更新と非自動更新の会員資格 {#auto-renewing-vs-non-auto-renewing-memberships}

この区別は、失効の挙動を理解する上で非常に重要です。

- **非自動更新の会員資格** (`auto_renew = false`)：毎時間のcronジョブがライフサイクル全体を処理します。更新支払いを作成し、会員資格を`on-hold`に移行させ、最終的に支払いがない場合は`expired`としてマークします。

- **自動更新の会員資格** (`auto_renew = true`)：cronによる失効チェックは**これらを完全にスキップします**。支払いゲートウェイ（Stripe、PayPalなど）は、サブスクリプションが失敗またはキャンセルされた際に、ウェブフックを介してUltimate Multisiteに通知することが期待されています。ウェブフックが受信されない場合（エンドポイントの設定ミス、ゲートウェイの障害、またはシステム外でのサブスクリプションキャンセルなど）、会員資格は失効日を過ぎても無期限に`active`のまま残る可能性があります。

### トライアルの終了方法 {#how-trials-end}

トライアル期間が終了する際、システムは以下の処理を行います。

1. フルサブスクリプション額の保留更新支払いを作成します。
2. 会員資格のステータスを`trialing`から`on-hold`に移行させます。
3. 顧客に更新支払い通知メールを送信します。

このプロセスは、通常の失効チェックと同じ毎時間のスケジュールで実行されますが、**非自動更新の会員資格のみ**に適用されます。自動更新のトライアルの場合、支払いゲートウェイがトライアルから有料サブスクリプションへの移行を処理します。

## フロントエンドアクセスをブロックする {#block-frontend-access}

デフォルトでは、会員資格が失効したり保留になったりしても、**wp-adminダッシュボードのみが制限されます**。サイトのパブリックなフロントエンドは訪問者に対してアクセス可能です。パブリックアクセスもブロックするには、**フロントエンドアクセスをブロック**設定を有効にする必要があります。

### 設定の構成 {#configuring-the-setting}

**Ultimate Multisite > Settings > Memberships** に移動し、**Block Frontend Access** を有効にします。

![Block Frontend Access settings showing the toggle enabled, a grace period of 7 days, and a Frontend Block Page selection](/img/config/settings-membership-block-frontend.png)

会員資格設定ページの全体像は以下の通りです。

![Membership settings full page](/img/config/settings-membership-full.png)

この動作を制御する関連設定が3つあります。

| 設定 | 説明 | デフォルト |
|---|---|---|
| **Block Frontend Access** | マスター切り替え。有効にすると、会員資格がアクティブでなくなった際に、ネットワークサイトのパブリックフロントエンドをブロックします。 | オフ |
| **Frontend Block Grace Period** | 非アクティブになってからブロックするまでの待機日数。即時ブロックしたい場合は `0` に設定します。 | 0 |
| **Frontend Block Page** | サイトがブロックされた際に訪問者をリダイレクトするメインサイト上のページ。設定しない場合、訪問者には一般的な「サイトは現在ご利用いただけません」というメッセージが表示されます。 | なし |

### サイトがブロックされた際に訪問者が見るもの {#what-visitors-see-when-a-site-is-blocked}

フロントエンドアクセスがブロックされると、サイトの訪問者は以下のいずれかになります。

1. **リダイレクト**：**Frontend Block Page** で選択したページにリダイレクトされます（設定されている場合）。
2. **デフォルトメッセージの表示**：「このサイトは現在ご利用いただけません。」というメッセージと、サイト管理者向けのログインページへのリンクが表示されます。

サイト管理者は引き続きログインできます。ログインページは決してブロックされません。

### 何が、いつブロックされるか {#what-gets-blocked-and-when}

ブロックの挙動は、会員資格のステータスによって異なります。

| 会員資格ステータス | フロントエンドブロック？ | 猶予期間が適用される？ |
|---|---|---|
| Active | いいえ | -- |
| Trialing | **いいえ**（下記の注意を参照） | -- |
| On Hold | アクティブと見なされるためブロックされない | -- |
| Expired | **はい**（Block Frontend Accessがオンの場合） | はい |
| Cancelled | **はい**（設定に関わらず常に） | **いいえ** — 即時ブロック |
| Pending | 会員資格チェックによるブロックは行われない | -- |

:::warning トライアル中の会員資格はブロックされません
トライアル期間が終了した場合でも、`trialing`ステータスの会員資格はフロントエンドで**ブロックされません**。トライアルは、まず別のステータス（通常はcronジョブ経由で`on-hold`、その後支払いがない場合は`expired`）に移行する必要があります。トライアル中の会員資格が移行していない場合は、下記のトラブルシューティングセクションを確認してください。
:::

:::note キャンセルされた会員資格は猶予期間をスキップします
キャンセルされた会員資格は、Block Frontend Accessが有効になっているかどうかに関わらず、失効日を過ぎると常にブロックされます。フロントエンドブロック猶予期間は、キャンセルされた会員資格には**適用されません**。
:::

## トラブルシューティング：失効後もサイトにアクセスできる場合 {#troubleshooting-sites-remaining-accessible-after-expiration}

会員資格が失効した後もサイトが一般公開されている場合は、以下のチェックを順番に実行してください。

### 1. Block Frontend Access設定が有効になっているか確認する {#1-verify-the-block-frontend-access-setting-is-enabled}

**Ultimate Multisite > Settings > Memberships** に移動し、**Block Frontend Access** トグルがオンになっていることを確認してください。この設定は**デフォルトでオフ**であるため、会員資格が非アクティブになった場合、wp-adminのみが制限されます。

### 2. Frontend Block Grace Periodを確認する {#2-check-the-frontend-block-grace-period}

同じ設定ページで、**Frontend Block Grace Period** の値を確認してください。これが例えば7日に設定されている場合、会員資格の失効日から7日間はフロントエンドはブロックされません。会員資格のステータスがすでに`expired`であっても、この猶予期間が適用されます。

即時ブロックしたい場合は、これを `0` に設定してください。

### 3. 会員資格のステータスが実際に変更されているか確認する {#3-confirm-the-membership-status-has-actually-changed}

**Ultimate Multisite > Memberships** に移動し、該当する会員資格のステータスを確認してください。失効日を過ぎているにもかかわらず、まだ`active`と表示されている場合、ステータスの移行が発生していません。一般的な原因は以下の通りです。

- **会員資格が自動更新されている**: 会員資格編集ページで`auto_renew`フィールドを確認してください。自動更新が有効な場合、失効cronはこの会員資格をスキップします。支払いゲートウェイが失敗を報告することに依存します。ゲートウェイダッシュボード（Stripe、PayPal）を確認し、サブスクリプションの状態がUltimate Multisiteの表示と一致しているか確認してください。

- **cronジョブが実行されていない**: 次のステップを参照してください。

### 4. Action Schedulerが実行されているか確認する {#4-verify-action-scheduler-is-running}

Ultimate Multisiteは、cronジョブにAction Schedulerを使用します。ネットワーク管理画面の **Tools > Scheduled Actions** に移動し、以下を探してください。

- **`wu_membership_check`** — これは毎時間実行される定期的なアクションとして表示されるはずです。これが存在しない場合、会員資格チェックがスケジュールされていません。
- **`wu_async_mark_membership_as_expired`** — これらは特定の会員資格を失効としてマークするための個別のタスクです。ここに失敗したアクションがある場合、なぜ失敗したのかを説明するエラーメッセージが含まれている可能性があります。

スタックしている、または失敗したアクションがある場合、Action Schedulerに問題がある可能性があります。一般的な原因は以下の通りです。

- `wp-config.php`で`DISABLE_WP_CRON`が`true`に設定されているが、システムレベルのcronによる代替処理がない。
- サイトのトラフィックが少ない（WP-Cronは誰かがサイトを訪問したときにのみ実行されるため）。

信頼性の高いcron実行を保証するには、システムcronジョブを設定してください。

```bash
# wgetを使って5分ごとに実行
*/5 * * * * wget -q -O /dev/null "https://your-network-url.com/wp-cron.php?doing_wp_cron"

# またはWP-CLI経由で実行
*/5 * * * * cd /path/to/wordpress && wp cron event run --due-now --url=https://your-network-url.com
```

### 5. ゲートウェイウェブフックの問題を確認する（自動更新の会員資格） {#5-check-for-gateway-webhook-issues-auto-renewing-memberships}

会員資格が自動更新されており、ゲートウェイのサブスクリプションがキャンセルされたか失敗したにもかかわらず、Ultimate Multisiteがまだ`active`と表示している場合：

- **Stripe**: Stripeダッシュボード > Customersに移動し、サブスクリプションの状態を確認します。次に、Developers > Webhooksでウェブフックエンドポイントがアクティブであることを確認してください。エンドポイントはサイトを指し、成功した配信が表示されている必要があります。
- **PayPal**: PayPalビジネスアカウントでサブスクリプションの状態を確認し、IPN/ウェブフックの配信を確認してください。

ゲートウェイがサブスクリプションをキャンセルしたと示しているが、Ultimate Multisiteがそうではない場合、ウェブフック通知が失われた可能性が高いです。**Ultimate Multisite > Memberships > [会員資格を編集]** で手動で会員資格のステータスを変更できます。

### 6. 失効猶予期間を確認する（Cronレベル） {#6-check-the-expiration-grace-period-cron-level}

cronチェックには、会員資格を失効としてマークする前に独自の猶予期間（デフォルト：3日間）があります。これはフロントエンドブロック猶予期間とは別物です。サイトがブロックされるまでの合計時間は以下のようになります。

**失効猶予期間（3日間）** + **フロントエンドブロック猶予期間（設定値）** = 合計遅延日数

例えば、デフォルト設定と7日間のフロントエンド猶予期間の場合、`date_expiration`から実際にサイトがブロックされるまでに最大10日かかる可能性があります。

### 7. 会員資格を手動で失効させる {#7-manually-expire-a-membership}

cronサイクルを待たずに、すぐにサイトをブロックする必要がある場合は、手動で会員資格のステータスを変更できます。

1. **Ultimate Multisite > Memberships** に移動します。
2. 該当の会員資格をクリックします。
3. ステータスを**Expired** または **Cancelled** に変更します。
4. **Save** をクリックします。

フロントエンドブロックは、次のページロード時に適用されます（失効会員資格の場合はフロントエンドブロック猶予期間に従い、キャンセル会員資格の場合は即座に適用されます）。

## まとめ {#summary}

失効日からサイトブロックまでの完全なタイムライン：

```text
date_expirationが経過する
       |
       v
  [3日間のcron猶予期間]     <-- 会員資格はまだactive/on-holdと表示される
       |
       v
  cronが会員資格を「expired」としてマークする
       |
       v
  [フロントエンドブロック猶予期間]  <-- Settings > Membershipsで設定
       |
       v
  サイトのフロントエンドがブロックされる
```

キャンセルされた会員資格の場合、プロセスはより短縮されます。

```text
  会員資格がキャンセルされる
       |
       v
  date_expirationが経過する（猶予期間なし）
       |
       v
  サイトのフロントエンドが即座にブロックされる
```

## 開発者向けリファレンス {#developer-reference}

以下のフックとフィルターを使用すると、失効およびブロックの動作をカスタマイズできます。

| Hook/Filter | 説明 |
|---|---|
| `wu_membership_grace_period_days` | 会員資格を失効としてマークする前の猶予期間の日数をフィルターします（デフォルト：3）。 |
| `wu_schedule_membership_check_interval` | 会員資格チェックの間隔をフィルターします（デフォルト：1時間）。 |
| `wu_membership_renewal_days_before_expiring` | 失効前に更新支払いを作成する日数をフィルターします（デフォルト：3）。 |
| `wu_blocked_site_reactivation_url` | サイトがブロックされた際にカスタムの再アクティブ化URLを提供するためのフィルターです。 |
| `wu_membership_is_active` | 会員資格がアクティブと見なされるかどうかをフィルターします。 |
| `wu_membership_expired_check_query_params` | 失効した会員資格を見つけるために使用されるクエリパラメータをフィルターします。 |
| `wu_membership_trial_check_query_params` | 失効したトライアルを見つけるために使用されるクエリパラメータをフィルターします。 |
