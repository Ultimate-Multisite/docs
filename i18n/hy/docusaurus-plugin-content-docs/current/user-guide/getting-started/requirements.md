# Ultimate Multisite လိုအပ်ချက်များ {#ultimate-multisite-requirements}

Ultimate Multisite ဝယ်ယူဖို့ မလုပ်ခင် ကိုယ့်ကိုယ်ကို မေးသင့်တဲ့ ပထမဆုံးအရာကတော့ ကျွန်တော်တို့ရဲ့ plugin ကို အသုံးပြုဖို့ ဘာတွေလိုအပ်လဲဆိုတာပဲ။ ကံကောင်းစွာနဲ့ပဲ၊ Ultimate Multisite ကို run ဖို့အတွက် အခြေခံ လိုအပ်ချက် အနည်းငယ်ပဲ ရှိပါတယ်- **domain name** တစ်ခု၊ ကောင်းမွန်တဲ့ **hosting provider** နဲ့ WordPress **Multisite** တည်ဆောက်မှုတစ်ခုပေါ့။ ဒါတွေတင် လုံလောက်ပါတယ်!

## Domain Name {#domain-name}

ပထမဆုံးအနေနဲ့ domain name အကြောင်း ပြောကြည့်ရအောင်။

Domain name ကို ရွေးချယ်တဲ့အခါ သင့်ရဲ့ အထူးနယ်ပယ် (niche) က ဘာလဲ၊ Ultimate Multisite နဲ့ ဘာတွေ ဖန်တီးသွားမှာလဲဆိုတာကို စဉ်းစားထားသင့်တယ်။ အေဂျင်စီတစ်ခုလား? ဆိုင်လို website တွေ ရောင်းချမလား? ဒါမှမဟုတ် WordPress web hosting လား? Ultimate Multisite နဲ့ မည်သည့်အရာမဆို ဖြစ်နိုင်ပါတယ်။

ကျွန်တော်တို့ရဲ့ plugin က subscription ဝန်ဆောင်မှုများ၊ ငွေပေးချေမှု စုဆောင်းခြင်း၊ checkout forms များ၊ discount vouchers နဲ့ customer ဆက်သွယ်ရေးတွေကို ပံ့ပိုးပေးတယ်။ WordPress Multisite ကို install လုပ်ဖို့၊ configure လုပ်ဖို့နဲ့ ထိန်းသိမ်းဖို့ လိုအပ်တဲ့ အလုပ်တွေ အများကြီးကို Ultimate Multisite က အလွယ်တကူ လုပ်ဆောင်နိုင်အောင် ကူညီပေးပါတယ်။

သင်ရွေးချယ်တဲ့ နယ်ပယ်တစ်ခုခုကို website တွေကို ဝန်ဆောင်မှုအဖြစ် ရောင်းချနေသရွေ့ ကျွန်တော်တို့ plugin ကို အသုံးပြုပြီး စီမံခန့်ခွဲနိုင်ပါတယ်။ သင့်ရဲ့ business plan နဲ့ ကိုက်ညီတဲ့ domain name တစ်ခုကို ရွေးချယ်ဖို့ မမေ့ပါနဲ့။

## Hosting Provider {#hosting-provider}

သင့်နယ်ပယ်နဲ့ လုပ်ငန်းကို ကိုယ်စားပြုတဲ့ ဆွဲဆောင်မှုရှိတဲ့ domain name တစ်ခု ရပြီဆိုရင် Ultimate Multisite နဲ့ သင့် network ကို host လုပ်ဖို့ ကောင်းမွန်တဲ့ web hosting ကုမ္ပဏီတစ်ခုကို ရွေးချယ်ဖို့ လိုအပ်ပါလိမ့်မယ်။ စျေးကွက်ထဲမှာ ကောင်းတဲ့ ရွေးချယ်စရာတွေ ရှိပါတယ်။ သင် ဘာတွေ တည်ဆောက်ဖို့ စီစဉ်ထားသလဲဆိုတာပေါ် မူတည်ပြီး Ultimate Multisite နဲ့ အကောင်းဆုံး ပေါင်းစပ်နိုင်မယ့် hosting provider တွေ ရှိနိုင်ပါတယ်။

Ultimate Multisite-ը կարող է աշխատել ցանկացած հոස්տինգի պրովայդերի հետ։ Միակ տարբերությունը այն է, որ որոշ հոස්տինգ ընկերություններ, ինչպիսիք են _WPMU DEV Hosting_ , _WP Engine_ , _Closte_ , _Cloudways_ , _Gridpane_ և _Runcloud_ , ունեն ավելի խորը ինտեգրում է Դոմենային քարտեզագրման (domain mapping) համար։ Այլ հոස්տինգների դեպքում կարող եք անհրաժեշտ լինել սահմանել հատուկ ադամնային տիրույթները ձեռքով։ Նաև լավ գաղափար է ընտրել այն հոස්տինգ պրովայդեր, որոնք աջակցում են AutoSSL-ի (վอัตโนատար SSL)։ _Closte_ , _Runcloud_ և _cPanel_-ը աջակցում են դրան։

Ինչ վերաբերում է սերվերի բնութագրերին, կոնկրետ բանաձև չկա, քանի որ այն կախված է առաջարկվող հնարավորությունների և ցանցի չափից։ Կիսված հոස්տինգը կարող է լավ սկիզբ լինել, բայց երբ ձեր ցանցը սկսի մեծանալ, ամենայն հավանականությամբ անհրաժեշտ կլինեք հատուկ VPS։

## WordPress Multisite {#wordpress-multisite}

Այժմ եկեք մի փոքր խոսենք WordPress Multisite-ի տեղադրման մասին։

WordPress Multisite-ը թույլ է տալիս մեկ ինստալացիայի մեջ ունենալ կայքերի ցանց։ Սա ներկառուցված հնարավորություն է, բայց սովորաբար չի աշխատում։ Դուք պետք է այս հնարավորությունը ակտիվացնեք ձեր WordPress-ի և վեբ հոස්տինգի կառավարման պանելում։

WordPress-ը Multisite-ով ակտիվացված լինելը **Ultimate Multisite-ի հիմքն է**։ Քանի որ Ultimate Multisite-ի հիմնական օգտագործումը օգնելու է օգտվողներին կայքերը վերահսկել և դրանք սերվիսային տեսքով վաճառել, դա պետք է անել WordPress-ի Multisite-ով ակտիվացված։

Մենք ունենք **խորը հոդված**, որում մանրամասն բացատրվում է այն քայլերը, որոնք անհրաժեշտ են ձեր WordPress-ի ինստալացիայի վրա Multisite-ի հնարավորությունը ակտիվացնելու համար։

WordPress-ում Multisite-ը ակտիվացր્યા après, ժամանակն է Ultimate Multisite-ը տեղադրելու։ Ultimate Multisite-ը ճիշտ տեղադրելու համար պետք է ունենաք.

  * WordPress: v5.3+ (Խորհուրդ է տրվում՝ վերջին կայուն տարբերակը)

  * PHP: 7.4.x (աջակցություն > 8.0-ի համար շուտով գալիս է)

  * MySQL: v5+ (Խորհուրդ է տրվում՝ 5.6, եթե ձեր հոස්տինգ պրովայդերում չունեք 8.0 տարբերակը)

Սրանք Ultimate Multisite-ի համար միակ ծրագրային պահանջներն են։

Հիշեք, որ Ultimate Multisite-ը կարող է աշխատել PHP 8.0-ով, բայց մենք խորհուրդ ենք տալիս այն գործարկել PHP 7.4.x-ով։

Նաև, մենք խորհուրդ ենք տալիս այն տեղադրել **հիմնական Դոմեիններում**, այլ ոչ թե ենթադիր Դոմեիններում կամ ենթադիր դիրքերում (subdomains or subdirectories)։ Այն կարող է աշխատել ենթադիր դոմեինների/ենթադիր դիրքերի վրա, բայց հնարավոր է առաջանան որոշ սխալներ։
