# Сдача ў срок ўступства і блокаванне сайта

Гутэ, што Ultimate Multisite робіць з адносінам да сдання ў срок ўступства, заканчэннем прабных перыядаў і блокаваннем доступу да сайта з фронтэнда. Тут разгледжаецца жыцключакл ўступства ад актыўнага да сдачнага, налады, якія вызначаюць, ці блокуецца сайт, і што праверыць, калі сайт застаецца даступным пасля сдання ў срок ўступства.

## Жыцключакл статусу ўступства

Кажнае ўступства ў Ultimate Multisite мае адзін з наступных статусаў:

:::note Уступства бясплатна — на ўвекі час
Бясплатныя ўступства не сдыходзяць з курсу аўтаматычна. Ultimate Multisite тратуе іх як доступ на ўвекі час, таму яны не ўключаюцца ў праверкі сдання ў срок, калі адміністратар не змяніць іх статус або не перанесці кліента на іншы прадукт.
:::

| Статус | Значэнне |
|---|---|
| **Pending** | Чакае першага плацമെннем або праверкай па электроннай пошце |
| **Trialing** | Актыўны прабны перыяд, яшчэ не збіраецца плацмент |
| **Active** | Аплатана і актыўны |
| **On Hold** | Чакае плацമെнне на пановленне (сфармавана паведамленне, чакае плацമെнне) |
| **Expired** | Мінула дата сдання ў срок і перадыўны перыяд без пановлення |
| **Cancelled** | Яўна адменёна кліентам або адміністратарам |

### Як ўступства пераходзяць у статус "Expired"

Ultimate Multisite праводзіць фоновы надзор **кожны гадзіну**, шукаючы ўступства, якія павінны быць адзначэнні як сдачнага. Гэтая праверка выкарыстоўвае [Action Scheduler](https://actionscheduler.org/) (а не прамым WP-Cron) і праходзіць як распярэджаная дзеянне `wu_membership_check`.

Праверка сдання ў срок мае па ўстанкаванні **перадыўны перыяд у 3 дні**. Уступства не будуць адзначэнні як `expired`, пакуль не прайдуць 3 дні пасля іх `date_expiration`. Гэта дае кліентам час на вырашэнне запазмастэйшага плацമെнне, перш чым змяніцца іх статус.

:::info
Перадыўны перыяд сдання ў срок (3 дні) адрозніваецца ад налады "Frontend Block Grace Period", апісаных ніжэй. Перадыўны перыяд сдання ў срок вызначае, калі **зменіцца статус** з актыўнага/на захаванні на сдачнага. Налады "Frontend Block Grace Period" вызначае, калі **заблокуецца сайт** пасля таго, як статус ужо змяніўся.
:::

#### Уступства з аўтаматычным пановленнем і без яго

Гэтая адрозненне крытычна для разумення паводзінаў сдання ў срок:

- **Уступства без аўтаматычнага пановлення** (`auto_renew = false`): Гадзінны cron job адпраўляе ўвесь жыцключакл — ён стварае паведамленні на пановленне, пераводзіць ўступства ў статус on-hold, і ў кан<$алі, адзначае іх як сдачнага, калі не атрымаецца плацмент.

- **Уступства з аўтаматычным пановленнем** (`auto_renew = true`): Праверка сдання ў срок cron **паўгоджае іх**. Очакваецца, што шлюз плацമെнтоў (Stripe, PayPal і г.д.) паведаміць Ultimate Multisite праз webhooks, калі падпіска не ўдалася або адменёна. Калі webhook не атрымаецца — з-за непадрабязна настроінага кінтпоінта, выключэння шлюзу або падпіскі, адменёнай па-за сістэмай — ўступства могуць застацца `active` на ўвекі час, нават пасля прайścia даты сдання ў срок.

### Як заканчваецца прабны перыяд

Калі прабны перыяд ўступства заканчваецца, сістэма:

1. Стварае паведамленне на пановленне з поўнай сумай падпіскі
2. Пераводзіць статус ўступства з `trialing` на `on-hold`
3. Адпраўляе паведамленне на пановленне плацമെннем кліенту

Гэты працэс праходзіць па гадзінным распярэджанні, як і звычайная праверка сдання ў срок, але ** толькі для ўступства, якія не маюць аўтаматычнага пановлення**. Для прабных ўступстваў з аўтаматычным пановленнем гэты працэс адпраўляе шлюз плацമെнтоў ад прабнага да аплатанага ўступства.

## Блокаванне доступу да фронтэнда

Па ўстанкаванні, калі ўступства сдыходзяць з курсу або пераходзяць на захаванне, **обмежавана толькі панель кіравання wp-admin**. Публічны фронтэнд сайта застаецца даступным для наведвальнікаў. Калі вы таксама хочаце заблакаваць публічны доступ, вам трэба актываваць налады **Block Frontend Access**.

### Налады канфігурацыі

Праййдзіце на **Ultimate Multisite > Settings > Memberships** і актывуйце **Block Frontend Access**.

![Block Frontend Access settings showing the toggle enabled, a grace period of 7 days, and a Frontend Block Page selection](/img/config/settings-membership-block-frontend.png)

Вось поўны прагляд старонкі налад:

![Membership settings full page](/img/config/settings-membership-full.png)

Тры адносівая налады вызначаюць гэтую паводзінасць:

| Налада | Апісан | Па ўстанкаванні |
|---|---|---|
| **Block Frontend Access** | Голоўны перамычка. Калі актыўвана, блокуе публічны фронтэнд сеткавых сайтаў, калі іх ўступства больш не актыўныя. | Off |
| **Frontend Block Grace Period** | Колькасць дзён, якія трэба пачакаць пасля таго, калі ўступства стане неактыўным, перш чым заблакаваць. Уставіць на `0` для неадкладнага блокавання. | 0 |
| **Frontend Block Page** | Сторонка на галоўным сайце, на якую перанаправляюцца наведвальнікі, калі сайт заблакаваны. Калі не ўстаноўлена, наведвальнікі бачэць агульны паведамленне "Site not available". | None |

### Што бачэць наведвальнікі, калі сайт заблакаваны

Калі доступ да фронтэнда заблакаваны, наведвальнікі на сайт або:

1. **Будуць перанаправлены** на сторонку, якую вы выбралі ў **Frontend Block Page** (калі гэта настроена), або
2. **Убачаць паслядоўнае паведамленне**: "This site is not available at the moment." з спасылкай на сторонку ўваходу для адміністратара сайта.

Адміністратары сайта ўсё яшчэ могуць увайсці — сторонка ўваходу ніколі не блокуецца.

### Што і калі блокуецца

Паводзінасць блокавання залежыць ад статусу ўступства:

| Статус ўступства | Блокаваны фронтэнд? | Выкарыстоўваецца перадыўны перыяд? |
|---|---|---|
| Active | Ні | -- |
| Trialing | **Ня** (глядзіце пазначку ніжэй) | -- |
| On Hold | Спрымаецца актыўным — не блокуецца | -- |
| Expired | **Так**, калі Block Frontend Access актыўваны | Так |
| Cancelled | **Так**, заўсёды (незалечна ад налад) | **Ня** — блокуецца неадкладна |
| Pending | Не блокуецца праз праверку ўступства | -- |

:::warning Уступства Trialing не блокуюцца
Нават калі прабны перыяд скінчыўся, ўступства са статусам `trialing` **не будуць** заблакаваны на фронтэндзе. Прабны перыяд павінен першыя пераходзіць у іншы статус (тыпа `on-hold` праз cron job, а потым `expired`, калі няма плацമെнне). Калі вы бачыце ўступства `trialing`, якія не пераходзілі, праверце секцыю выяўлення праблем.
:::

:::note Уступства Cancelled падрабязна праходзяць мінуўшы перадыўны перыяд
Уступства Cancelled заўсёды блокуюцца, калі прайшло сдання ў срок, незалежна ад таго, ці актыўваны Block Frontend Access. Налады Frontend Block Grace Period **не** праходзяць на ўступствах Cancelled.
:::

## Выяўленне праблем: Сайты, якія застаюцца даступнымі пасля сдання ў срок

Калі сайты застаюцца публічна даступнымі пасля сдання ў срок ўступства, прайце гэтыя праверкі па раскладзе:

### 1. Праверце, што налады Block Frontend Access актыўваны

Праййдзіце на **Ultimate Multisite > Settings > Memberships** і пацверце, што перамычка **Block Frontend Access** актыўвана. Гэтая налада **не актыўвана па ўстанкаванні**, што азначае, што блокуецца толькі wp-admin, калі ўступства становяцца неактыўнымі.

### 2. Праверце Frontend Block Grace Period

На той жа старонцы налад праверце значэнне **Frontend Block Grace Period**. Калі яно ўстаноўлена на 7 дзён, напрыклад, фронтэнд не будзе заблакаваны, пакуль не прайдуць 7 дзён пасля даты сдання ў срок ўступства — нават калі статус ўступства ўжо `expired`.

Уставіце гэта на `0`, калі вы хочаце неадкладнае блокаванне пасля таго, калі ўступства стане неактыўным.

### 3. Пацверце, што статус ўступства сапраўды змяніўся

Праййдзіце на **Ultimate Multisite > Memberships** і праверце статус азначанага ўступства. Калі яно ўсё яшчэ паказвае `active` нягледзячы на тое, што прайшла дата сдання ў срок, пераход статусу не адбыўся. Частыя прычыны:

- **Уступства з аўтаматычным пановленнем**: Праверце поле `auto_renew` на старонцы рэдактавання ўступства. Калі аўтаматычнае пановленне актыўна, cron сдання ў срок паўгоджае гэта ўступства — яно залежыць ад шлюзу плацമെнтоў, каб паведаміць пра неўдачу. Праверце панель кіравання шлюзу (Stripe, PayPal), каб пацверыць, што статус падпіскі адпавядае таму, што паказвае Ultimate Multisite.

- **Cron job не прайшоў**: Гэта паказвае наступны крок.

### 4. Праверце, што Action Scheduler працуе

Ultimate Multisite выкарыстоўвае Action Scheduler для сваіх cron jobs. Праййдзіце на **Tools > Scheduled Actions** у адмініструванні сеткі і знайце:

- **`wu_membership_check`** — Гэта павінна быць паказвацца як паўторнае дзеянне, якое праходзіць кожны гадзіну. Калі яго няма, праверкі ўступства не распярэджоўваюцца.
- **`wu_async_mark_membership_as_expired`** — Гэтыя асобныя заданні для адзначэння канкрэтных ўступстваў як сдачнага. Калі вы бачыце невыкананыя дзеянні тут, яны могуць 함ערваць паведамленні, якія паказваюць, чаму.

Калі вы бачыце застрэленыя або невыкананыя дзеянні, у вас можа быць праблема з Action Scheduler. Частыя прычыны:

- **`DISABLE_WP_CRON` ўстаноўлена на `true`** у `wp-config.php` без заменіўшыга cron на ўзроўні сістэмы
- **Низкі трафік сайта** — WP-Cron праходзіць толькі, калі хтосьці набываецца на сайце

Каб забяспечыць надзейнае выкананне cron, наладзьце cron на ўзроўні сістэмы:

```bash
# Праходзіць кожныя 5 хвілін праз wget
*/5 * * * * wget -q -O /dev/null "https://your-network-url.com/wp-cron.php?doing_wp_cron"

# Або праз WP-CLI
*/5 * * * * cd /path/to/wordpress && wp cron event run --due-now --url=https://your-network-url.com
```

### 5. Праверка праблем з webhooks шлюзу (Уступства з аўтаматычным пановленнем)

Калі ўступства з аўтаматычным пановленнем і шлюз падпіскі быў адменён або не ўдаўся, але Ultimate Multisite ўсё яшчэ паказвае яго як `active`:

- **Stripe**: Праййдзіце на Stripe Dashboard > Customers і праверце статус падпіскі. Пасля гэтага праверце, ці актыўны кінтпоінт webhook у Developers > Webhooks. Кінтпоінт павінен паказваць на ваш сайт і паказваць паспяховае дастаўленне.
- **PayPal**: Праверце статус падпіскі ў вашым бізнес-сатвай PayPal і праверце дастаўку IPN/webhook.

Калі шлюз паказвае падпіску як адменённую, але Ultimate Multisite — ні, верагодна, што паведамленне webhook было страчана. Вы можаце на мануляцыі змяніць статус ўступства ў **Ultimate Multisite > Memberships > [Edit Membership]**.

### 6. Праверка перадыўнага перыяду сдання ў срок (На ўзроўні Cron)

Cron праверы мае свой перадыўны перыяд (па ўстанкаванні: 3 дні) перш чым адзначыць ўступства як сдачнага. Гэта адрозніваецца ад налады перадыўнага перыяду фронтэнда. Суммарны час, па якім сайт будзе заблакаваны:

**Перадыўны перыяд сдання ў срок (3 дні)** + **Перадыўны перыяд блокавання фронтэнда (ваша налада)** = Суммарная замарозка

Напрыклад, з па ўстанкаванні па ўстанкаванні і перадыўным перыядам фронтэнда ў 7 дзён, можа прайсці да 10 дзён пасля `date_expiration`, перш чым сайт будзе сапраўды заблакаваны.

### 7. Мануляўнае сдання ў срок ўступства

Калі вам трэба неадкладна заблакаваць сайт, не чакаючы цыклу cron, вы можаце мануляўна змяніць статус ўступства:

1. Праййдзіце на **Ultimate Multisite > Memberships**
2. Націсніце на азначанае ўступства
3. Змяніце статус на **Expired** або **Cancelled**
4. Націсніце **Save**

Блокаванне фронтэнда ўступае ў сілу на наступнай загрузцы старонкі (залежна ад перадыўнага перыяду фронтэнда для сдачнага ўступства, або неадкладна для адменёнага ўступства).

## Рэзюмэ

Поўны часовы разклад ад даты сдання ў срок да блокавання сайта:

```text
date_expiration праходзіць
       |
       v
  [3-дзённы перадыўны перыяд cron]     <-- ўступства ўсё яшчэ паказваецца як актыўныя/на захаванні
       |
       v
  Cron адзначае ўступства як "expired"
       |
       v
  [Frontend Block Grace Period]  <-- настроена ў Settings > Memberships
       |
       v
  Фронтэнд сайта заблакаваны
```

Для ўступства Cancelled шлях больш кароткі:

```text
  Уступства адменены
       |
       v
  date_expiration праходзіць (без перадыўнага перыяду)
       |
       v
  Фронтэнд сайта заблакаваны неадкладна
```

## Рэцэнтні рэфараванне для праграмістаў

Наступныя хукі і фільтры дазваляюць вам наладзіць паводзінасць сдання ў срок і блокавання:

| Hook/Filter | Description |
|---|---|
| `wu_membership_grace_period_days` | Фільтруе колькасць дзён перадыўнага перыяду папярэдні сдання ў срок ўступства (па ўстанкаванні: 3) |
| `wu_schedule_membership_check_interval` | Фільтруе перыяд паміж праверкамі ўступства (па ўстанкаванні: 1 гадзіна) |
| `wu_membership_renewal_days_before_expiring` | Фільтруе, колькі дзён папярэдні сдання ў срок ўступства, каб стварыць паведамленне на пановленне (па ўстанкаванні: 3) |
| `wu_blocked_site_reactivation_url` | Фільтруе, каб прадастаўляць карыстальніцкі URL для актывацыі, калі сайт заблакаваны |
| `wu_membership_is_active` | Фільтруе, ці ўступства спасываецца як актыўнае |
| `wu_membership_expired_check_query_params` | Фільтруе параметры zapўту, выкарыстоўваныя для выяўлення сдачнага ўступства |
| `wu_membership_trial_check_query_params` | Фільтруе параметры zapўту, выкарыстоўваныя для выяўлення сдачнага прабнага ўступства |
