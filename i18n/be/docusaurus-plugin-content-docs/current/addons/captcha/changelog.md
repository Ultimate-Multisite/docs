# Captcha Changelog

Version: 1.5.0 - Released on 2026-05-22
* New: Hard-stop rate limiter — лічыць кожны GET і POST на спагадленых captcha-паверхнях (wp-login / register / lost-password / comments, WooCommerce my-account / checkout / pay-for-order, Ultimate Multisite checkout / inline-login) і адказвае HTTP 429, запіс `Retry-After` і рэалізоваваную "tarpit sleep" (1–5с, з максімум 15с).
* New: Налады `cap_rate_limit_tarpit_min` / `cap_rate_limit_tarpit_max` для рэгулявання "tarpit" вікна.
* New: Фільтр `wu_cap_rate_limit_whitelist_ip` для выключэння даверных IP-дыпазонаў.
* New: Дзеянне `wu_cap_rate_limit_will_block`, якое спраўляецца неадкладна перад адпраўленнем адказу hard-stop.
* New: Выяўленне client-IP, вытрымлівае ўплывы (spoof-resistant). `Captcha_Core::get_client_ip()` (адлік правды для ключаў бакета rate-limit, captcha siteverify `remoteip` і хешаў IP для статыстыкі) цяпер выконвае строгую мадэль даверу: REMOTE_ADDR — гэта база, `CF-Connecting-IP` выкарыстоўваецца толькі тады, калі непасрэдны peer знаходзіцца ў бягуNC Cloudflare IP-дыпазоне, а `X-Forwarded-For` выкарыстоўваецца толькі тады, калі непасрэдны peer знаходзіцца ў спісе даверных-праксі, настроеным адмініватэрам, з праходжаннем справа налева, якое пажаецца даверныя/CF-хапы, перш чым сканцэнтруецца на IP наведвальніка.
* New: Налада `cap_trust_cloudflare_headers` (па спічатку) — магчыма ўключыць давер да `CF-Connecting-IP`, калі вы за ад Cloudflare. Плагін пастаўляе ўпакоўлены знімок Cloudflare CIDR і абнаўляе яго штогадзённа праз wp-cron з ўпакоўленым запасным варыянтам, калі абнаўленне не ўдакладняецца.
* New: Налада `cap_trusted_proxies` — тэкставая прастора для CIDR або проста IP-адрэсаў (па адным на радку, дазваляецца каментары ў `#`), якія пералічваюць вашы ўласныя фронтавыя праксі / load-balancers. Без гэтага `X-Forwarded-For` ігнаруецца нават калі ўключаны rate limiter.
* New: Першае ўкрэджанне аўтаматычнага выяўлення верагоднага стану Cloudflare / праксі з адмініватэрам-паведамленнем "Задніе выяўленыя налады" ў адзін клік. Плагін ніколі не перапісвае вашы захаваныя значэнні; калі наступны трафік паказвае, што ваша канфігурацыя больш не адпавядае рэальнасці (напрыклад, Cloudflare змяніў дыпазоны CIDR, а ваш праксі CIDR цяпер застарэлы), паказваецца неадключаванае паведамленне непаўнаднасці з рэкамендаваным абнаўленнем.
* Fixed: Падрабязны рэжым больш не пакіднага таго, што паніжае `cap_security_level` да FAST — вызначаны адмініватэрам ўзровень выкарыстоўваецца. Даступны новы фільтр `wu_cap_server_security_level` для сайтаў, якія жадаюць спецыфічнай лагікі.
* Fixed: Лічыльнік статыстыкі `rate_limits_triggered` цяпер павялічваецца па кожным блоку, а не толькі на рэдкім шляху "запасным" пасля паспяховасці.
* Fixed: `Captcha_Core::get_client_ip()` цяпер з'яўляецца адным крынікам правды для апісання IP наведвальніка па ўсім zakresie rate limitera, captcha-провайдэраў (reCAPTCHA + hCaptcha `siteverify`) і статыстыкі — зачыняючы вектар ўплыву, дзе непасрэдныя запыты з крынішча-сервера, якія пераносяць падроблены `CF-Connecting-IP` заголовок, маглі быць бакетаваныя па падробленым IP, а не па рэальным peer.
* Fixed: Rate-limit gate для класічнага checkout WooCommerce цяпер спраўляецца на `template_redirect` (прымарынкі 1) замест `woocommerce_before_checkout_form`. Хук на ўзроўні формы ніколі не спраўляецца, калі кошык пуст, таму трафік, які ніколі не дадаў прадукт, праходзіў мінуўшы rate limiter.
* Fixed: Rate-limit gate для pay-for-order WooCommerce цяпер спраўляецца на `template_redirect` замест `woocommerce_before_pay_action`. Последні спраўляецца толькі пасля паспяховасці `wp_verify_nonce('woocommerce-pay')`, што азначае, што неаўтарызаваныя атакавальнікі (сапраўдная мадэль пагрозы) ніколі не актывавалі rate limiter.
* Fixed: Rate-limit gate для checkout WooCommerce Store API (blocks) цяпер спраўляецца на `rest_pre_dispatch` замест `woocommerce_store_api_checkout_update_order_from_request`. Последні спраўляецца толькі пасля таго, калі Store API праверыў кошык і поля выстаўвання рахунку, таму неаўтарызаваныя боты атрымлівалі 400 ад валідатара і ніколі не спраўлялі rate limiter.
* Fixed: Rate-limit gate для inline-login Ultimate Multisite цяпер спраўляецца на `wu_ajax_nopriv_wu_inline_login` прымарынкі 1 (і адлюстраваным для ўвайшоўшых) замест `wu_before_inline_login`. Последні спраўляецца толькі пасля паспяховасці `check_ajax_referer('wu_checkout')`, таму неаўтарызаваныя боты без валіднага wu_checkout nonce атрымлівалі 403 і ніколі не спраўлялі rate limiter.
* Fixed: `Rate_Limiter::enforce()` цяпер праеўно ўжывае абарону "адзін раз на запыт", ключаваны па `surface|ip`, таму ўжоўныя хукі, якія спраўляюцца два разы на рэндер (напрыклад, `wu_setup_checkout` у Ultimate Multisite), больш не паказваюць паловіну эфектыўнага парага rate limitera.
* Fixed: Rate-limit surface gates больш не кансультаваўваюць `Captcha_Core::is_whitelisted()` (фільтр `wu_captcha_whitelisted`). Гэты фільтр паказвае "captcha ўжо апрацавана іншай паверхнёй" і ортыгоналёчны да абароны ад патопу — інтэграцыя WooCommerce хукавала яго, каб праскочыць captcha ўвайта на WordPress, калі быў прысутні капчыка-nonce, што пранікна малоджанню лічбы rate і дазволіла POST-запытам Woo адступаць ад rate limitera. Для rate limitera спецыфічны фільтр `wu_cap_rate_limit_whitelist_ip` — гэта адзіны праход, які цяпер дзейнічае.
