# სუვერენული ტენანტების ოპერაციები

Ultimate Multisite: Multi-Tenancy 1.2.0 აკლია ოპერაციული ინსტრუმენტები სუვერენული ტენანტებისთვის: ქვესაიტები, რომლებიც მუშაობენ საკუთარ მონაცემთა ბაზასთან, ფაილის სისტემის ძირეულ დირექტორიაში და მარშრუტირების კონტექსტში, თუმცა მაინც ხელმისაწვდომია ქსელის ადმინისტრატორისთვის.

გამოიყენეთ ეს გვერდი იზოლირებული მომხმარებლის საიტების მართვისას, დისტანციური საიტების გადაცემებისას ან მიგრაციებისას, რომლებიც სტანდარტულ ქვესაიტს სუვერენული ინფრასტრუქტურაში გადაჰყავს.

## რა იცვლება ადმინისტრატორებისთვის

- **ステートლესი ტენანტის ავტომატური შესვლა** — ქსელის ადმინისტრატორებს შეუძლიათ სუვერენული ტენანტის მონახულება იმის გარეშე, რომ დამოკიდებულონ ხანგრძლივად არსებულ საერთო სესია მდგომარეობაზე. SSO ტოკენი არის მიზნობრივი (purpose-scoped), წარმოშობის მიხედვით შეზღუდული (origin-pinned), განმეორებისგან დაცული (replay-protected) და მოკლე ვადით გაცემული.
- **სუვერენული ცნობიერ მარშრუტირება** — ძველი იზოლირებული ქსელები და სუვერენული ტენანტები წყვეტენ ერთსა და იმავე საიტის მარშრუტის გზით, რაც ამცირებს ძველ და ახალ იზოლირებულ ინსტალაციებს შორის ბBootstrap განსხვავებებს.
- **დადასტურებული მიგრაციის მდგომარეობა** — მიგრაციის ვერიფიკაცია აკონტროლებს მომხმარებლის პროვიზიციას, მონაცემთა ბაზის წერის უფლებებს, რიგების تخلیერის სტატუსს და ძველი ცხრილების არარსებობას მანამ, სანამ ტენანტი სრულად დამუშავებულად იქნევა.
- **უფრო უსაფრთხო დეპლოი (teardown)** — სუვერენული დეპლოის ახლა temizებს ტენანტის საკრედინაციო მონაცემებს, რათა წაშლილი ტენანტები არ დატოვონ ძველი ბაზის წვდომა.

## სუვერენული ტენანტის მონახულება

1. გახსენით **Network Admin > Ultimate Multisite > Sites**.
2. აირჩიეთ სუვერენული ტენანტი.
3. ხელმისაწვდომობის შემთხვევაში გამოიყენეთ **Visit (SSO)**, ნაცვლად პაროლების კოპირებისა და დროებითი ადმინისტრატორის ანგარიშების შექმნის.

ვიზიტის პროცესი ქმნის მოკლევადიან შესვლის ტოკენს ამ ტენანტისთვის და აფიქსირებს SSO-ს მოვლენას ტენანტის 監査 (audit trail) აღწერილობაში. თუ ღილაკი არ მუშაობს, შეამოწმეთ, რომ ტენანტის დომენი წყვეტს მოსალოდნელ ინსტალაციას და რომ ტენანტს შეუძლია მიაღწიოს ქსელის მხარეს არსებულ SSO-პუნქტს.

## დისტანციური საიტების ოპერაციების ჩამონათვალი

სუვერენული ან დისტანციური ტენანტის შეცSebelum, დაადასტურეთ:

- მეinqurent domain მიუთითებს ჰოსტზე, რომელიც ფლობს მეinqurent ფაილსისტემას.
- მეinqurent მონაცემთა ბაზის ჰოსტი ემთხვევა კონფიგურირებულ ჰოსტის ბაინდინგს იმ ინსტალაციისთვის.
- მიგრაციის გადამოწმების ბრძანებები წარმატებით დასრულდა მეinqurentისთვის.
- ასინქრო მიგრაციის kolejები تخلیდებიან DNS-ის ან საკუთრების ცვლილებებისBefore.
- მეinqurent ადმინისტრატორის მომხმარებელი არის შექმნილი მიგრაციის დროს და შეუძლია შესვლა SSO-ს საშუალებით.

## სუვერენული მეinqurent-ების წაშლა

სუვერენული მეinqurent-ის წაშლა დამანგრეველია. ჯერ დაადასტურეთ ბექაპისა და ექსპორტის სტატუსი, შემდეგ კი წაშალეთ საიტის მართვის ეკრანიდან. ვერსია 1.2.0-ის დეტეშოუნის პროცესი აშლის მეinqurent მონაცემთა ბაზის საკრედინტალებს გაწმენდის ნაწილად, თუმცა ადმინისტრატორებმა მაინც უნდა დაადასტურონ, რომ ჰოსტის დონის მონაცემთა მომხმარებლები და საქაღალდეები აღარ არსებობს, როდესაც გარე ჰოსტინგ პანელებს იყენებთ.

:::warning
არ წაშალოთ სუვერენული მეinqurent-ი მიგრაციის გადამოწმების განხორციელებისას ან ასინქრო push ამოცანების დალაგებისას. დაელოდეთ გადამოწმების დასრულებას, რომ დეტეშოუნმა არ წაშალოს საკრედინტალები, რომლებიც საჭიროა დაუყოვნებლივ შესასრულებელი ამოცანებისთვის.
:::
