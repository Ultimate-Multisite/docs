# 멤버십 만료 및 사이트 차단

이 가이드는 Ultimate Multisite가 멤버십 만료, 체험 기간 종료, 그리고 프론트엔드 사이트 차단을 처리하는 방법을 설명합니다. 활성 상태에서 만료 상태에 이르기까지 멤버십의 전체 생명주기, 사이트 차단 여부를 제어하는 설정, 그리고 멤버십 만료 후에도 사이트가 접근 가능한 상태로 남아 있을 때 확인해야 할 사항들을 다룹니다.

## 멤버십 상태 생명주기

Ultimate Multisite의 모든 멤버십은 다음 상태 중 하나를 가집니다.

:::note 무료 멤버십은 평생 유효합니다
무료 멤버십은 자동으로 만료되지 않습니다. Ultimate Multisite는 이를 평생 접근 권한으로 간주하므로, 관리자가 상태를 변경하거나 고객을 다른 제품으로 이동시키지 않는 한 만료 확인 대상에 포함되지 않습니다.
:::

| 상태 | 의미 |
|---|---|
| **Pending** | 첫 결제 또는 이메일 인증 대기 중 |
| **Trialing** | 활성 체험 기간, 아직 결제된 금액 없음 |
| **Active** | 결제 완료 및 현재 유효함 |
| **On Hold** | 갱신 결제 대기 중 (인보이스 생성, 결제 대기) |
| **Expired** | 만료일 및 유예 기간이 지나도 갱신하지 않은 경우 |
| **Cancelled** | 고객 또는 관리자에 의해 명시적으로 취소된 경우 |

### 멤버십이 만료되는 과정

Ultimate Multisite는 멤버십이 만료되어야 하는지 확인하는 백그라운드 검사를 **매시간** 실행합니다. 이 검사는 [Action Scheduler](https://actionscheduler.org/) (WP-Cron을 직접 사용하지 않음)를 사용하며 `wu_membership_check` 예약된 액션으로 실행됩니다.

만료 확인에는 기본적으로 **3일의 내장 유예 기간**이 있습니다. 멤버십은 `date_expiration`이 지난 후 3일이 지나기 전까지는 `expired`로 표시되지 않습니다. 이는 고객이 상태가 변경되기 전에 연체 결제를 완료할 시간을 주기 위함입니다.

:::info
3일의 만료 유예 기간은 아래 설명된 프론트엔드 차단 유예 기간 설정과는 별개입니다. 만료 유예 기간은 **상태가** 활성/보류에서 만료로 **변경되는 시점**을 제어합니다. 프론트엔드 차단 유예 기간은 상태가 이미 변경된 후 **사이트가 차단되는 시점**을 제어합니다.
:::

#### 자동 갱신 vs. 비자동 갱신 멤버십

이 구분은 만료 동작을 이해하는 데 매우 중요합니다.

- **비자동 갱신 멤버십** (`auto_renew = false`): 시간별 크론 작업이 전체 생명주기를 처리합니다. 갱신 결제를 생성하고, 멤버십을 `on-hold`로 전환하며, 결국 결제가 이루어지지 않으면 만료로 표시합니다.

- **자동 갱신 멤버십** (`auto_renew = true`): 크론 만료 확인은 **이 멤버십을 완전히 건너뜁니다**. 결제 게이트웨이(Stripe, PayPal 등)는 구독 실패 또는 취소 시 웹훅을 통해 Ultimate Multisite에 알리는 것이 예상됩니다. 웹훅을 받지 못하는 경우(잘못 구성된 엔드포인트, 게이트웨이 장애 또는 시스템 외부에서 취소된 구독의 경우), 멤버십은 만료일이 지났음에도 무기한 `active` 상태로 남아 있을 수 있습니다.

### 체험 기간 종료 과정

체험 중인 멤버십의 체험 기간이 끝나면 시스템은 다음 작업을 수행합니다.

1. 전체 구독 금액으로 보류 갱신 결제를 생성합니다.
2. 멤버십 상태를 `trialing`에서 `on-hold`로 전환합니다.
3. 고객에게 갱신 결제 알림 이메일을 보냅니다.

이 과정은 일반 만료 확인과 동일한 시간별 스케줄로 실행되지만, **비자동 갱신 멤버십에 대해서만** 적용됩니다. 자동 갱신 체험의 경우, 결제 게이트웨이가 체험 기간에서 유료 구독으로의 전환을 처리합니다.

## 프론트엔드 접근 차단

기본적으로 멤버십이 만료되거나 보류 상태가 되면, **wp-admin 대시보드만 제한됩니다**. 사이트의 공개 프론트엔드는 방문자에게 계속 접근 가능합니다. 공개 접근까지 차단하려면 **프론트엔드 접근 차단** 설정을 활성화해야 합니다.

### 설정 구성하기

**Ultimate Multisite > 설정 > 멤버십**으로 이동하여 **프론트엔드 접근 차단(Block Frontend Access)**을 활성화합니다.

![Block Frontend Access settings showing the toggle enabled, a grace period of 7 days, and a Frontend Block Page selection](/img/config/settings-membership-block-frontend.png)

다음은 멤버십 설정 페이지의 전체 보기입니다.

![Membership settings full page](/img/config/settings-membership-full.png)

이 동작을 제어하는 세 가지 관련 설정이 있습니다.

| 설정 | 설명 | 기본값 |
|---|---|---|
| **Block Frontend Access** | 마스터 토글. 활성화하면 멤버십이 더 이상 활성 상태가 아닐 때 네트워크 사이트의 공개 프론트엔드를 차단합니다. | 비활성화 |
| **Frontend Block Grace Period** | 멤버십이 비활성 상태가 된 후 차단하기까지 기다릴 일수입니다. 즉시 차단하려면 `0`으로 설정합니다. | 0 |
| **Frontend Block Page** | 사이트가 차단되었을 때 방문자를 리디렉션할 메인 사이트의 페이지입니다. 설정하지 않으면 방문자는 일반적인 "사이트를 이용할 수 없습니다" 메시지를 보게 됩니다. | 없음 |

### 사이트가 차단되었을 때 방문자가 보는 것

프론트엔드 접근이 차단되면, 사이트 방문자는 다음 중 하나를 보게 됩니다.

1. **리디렉션**: **프론트엔드 차단 페이지**에서 선택한 페이지로 리디렉션되거나,
2. **기본 메시지**: "현재 이 사이트는 이용할 수 없습니다."라는 메시지와 사이트 관리자 로그인 페이지로 연결되는 링크를 보게 됩니다.

사이트 관리자는 여전히 로그인할 수 있습니다. 로그인 페이지는 절대 차단되지 않습니다.

### 무엇이, 언제 차단되는가

차단 동작은 멤버십 상태에 따라 달라집니다.

| 멤버십 상태 | 프론트엔드 차단 여부? | 유예 기간 적용? |
|---|---|---|
| Active | 아니요 | -- |
| Trialing | **아니요** (아래 참고) | -- |
| On Hold | 활성으로 간주됨 -- 차단되지 않음 | -- |
| Expired | **예**, 프론트엔드 접근 차단이 켜져 있는 경우 | 예 |
| Cancelled | **예**, 항상 (설정 무관) | **아니요** -- 즉시 차단 |
| Pending | 멤버십 검사를 통해 차단되지 않음 | -- |

:::warning 체험 중인 멤버십은 차단되지 않습니다
체험 기간이 끝났더라도, `trialing` 상태의 멤버십은 프론트엔드에서 **차단되지 않습니다**. 체험 기간은 먼저 다른 상태(일반적으로 크론 작업을 통해 `on-hold`, 이후 미결제 시 `expired`)로 전환되어야 합니다. 체험 중인 멤버십이 전환되지 않은 것을 발견하면 아래 문제 해결 섹션을 확인하세요.
:::

:::note 취소된 멤버십은 유예 기간을 건너뜁니다
취소된 멤버십은 프론트엔드 접근 차단이 활성화되어 있는지 여부와 관계없이, 만료일이 지나면 항상 차단됩니다. 프론트엔드 차단 유예 기간은 취소된 멤버십에 **적용되지 않습니다**.
:::

## 문제 해결: 만료 후에도 사이트가 접근 가능한 경우

멤버십이 만료된 후에도 사이트가 공개적으로 접근 가능하다면, 다음 확인 사항들을 순서대로 점검해 보세요.

### 1. 프론트엔드 접근 차단 설정이 활성화되었는지 확인

**Ultimate Multisite > 설정 > 멤버십**으로 이동하여 **프론트엔드 접근 차단** 토글이 켜져 있는지 확인합니다. 이 설정은 **기본적으로 비활성화**되어 있어, 멤버십이 비활성 상태가 될 때 wp-admin만 제한됩니다.

### 2. 프론트엔드 차단 유예 기간 확인

같은 설정 페이지에서 **프론트엔드 차단 유예 기간** 값을 확인합니다. 예를 들어, 이 값이 7일로 설정되어 있다면, 멤버십 상태가 이미 `expired`일지라도 프론트엔드는 7일이 지나기 전까지 차단되지 않습니다.

즉시 차단하려면 이 값을 `0`으로 설정하세요.

### 3. 멤버십 상태가 실제로 변경되었는지 확인

**Ultimate Multisite > 멤버십**으로 이동하여 영향을 받는 멤버십의 상태를 확인합니다. 만료일이 지났음에도 여전히 `active`로 표시된다면, 상태 전환이 발생하지 않은 것입니다. 일반적인 원인:

- **멤버십이 자동 갱신되는 경우**: 멤버십 편집 페이지에서 `auto_renew` 필드를 확인하세요. 자동 갱신이 활성화되어 있으면, 만료 크론 작업은 이 멤버십을 건너뜁니다. 결제 게이트웨이가 실패를 보고하는 것에 의존합니다. 게이트웨이 대시보드(Stripe, PayPal)를 확인하여 구독 상태가 Ultimate Multisite가 보여주는 상태와 일치하는지 확인하세요.

- **크론 작업이 실행되지 않은 경우**: 다음 단계를 확인하세요.

### 4. Action Scheduler가 실행되는지 확인

Ultimate Multisite는 크론 작업에 Action Scheduler를 사용합니다. 네트워크 관리자에서 **도구 > 예약된 작업(Scheduled Actions)**으로 이동하여 다음을 찾습니다.

- **`wu_membership_check`** -- 이 항목은 매시간 실행되는 반복 작업으로 나타나야 합니다. 이 항목이 없으면 멤버십 검사가 예약되지 않은 것입니다.
- **`wu_async_mark_membership_as_expired`** -- 이들은 특정 멤버십을 만료로 표시하는 개별 작업입니다. 여기서 실패한 작업을 보면 오류 메시지가 포함되어 이유를 설명할 수 있습니다.

멈추거나 실패한 작업이 보인다면 Action Scheduler에 문제가 있을 수 있습니다. 일반적인 원인:

- `wp-config.php`에 `DISABLE_WP_CRON`이 `true`로 설정되어 있고 시스템 수준의 크론 대체 기능이 없는 경우
- 사이트 트래픽이 낮은 경우 -- WP-Cron은 누군가 사이트를 방문할 때만 실행됩니다.

신뢰할 수 있는 크론 실행을 보장하려면 시스템 크론 작업을 설정하세요.

```bash
# wget을 사용하여 5분마다 실행
*/5 * * * * wget -q -O /dev/null "https://your-network-url.com/wp-cron.php?doing_wp_cron"

# 또는 WP-CLI를 사용하여 실행
*/5 * * * * cd /path/to/wordpress && wp cron event run --due-now --url=https://your-network-url.com
```

### 5. 게이트웨이 웹훅 문제 확인 (자동 갱신 멤버십)

멤버십이 자동 갱신되고 게이트웨이 구독이 취소되었거나 실패했지만, Ultimate Multisite가 여전히 `active`로 표시하는 경우:

- **Stripe**: Stripe 대시보드 > 고객으로 이동하여 구독 상태를 확인합니다. 그런 다음 개발자 > 웹훅에서 웹훅 엔드포인트가 활성화되어 있는지 확인합니다. 엔드포인트는 사이트를 가리켜야 하며 성공적인 전송 기록이 보여야 합니다.
- **PayPal**: PayPal 비즈니스 계정에서 구독 상태를 확인하고 IPN/웹훅 전송을 확인합니다.

게이트웨이가 구독을 취소했다고 보여주지만 Ultimate Multisite가 그렇지 않은 경우, 웹훅 알림이 손실되었을 가능성이 높습니다. **Ultimate Multisite > 멤버십 > [멤버십 편집]**에서 멤버십 상태를 수동으로 변경할 수 있습니다.

### 6. 만료 유예 기간 확인 (크론 레벨)

크론 검사 자체에도 멤버십을 만료로 표시하기 전까지의 유예 기간(기본값: 3일)이 있습니다. 이는 프론트엔드 차단 유예 기간과는 별개입니다. 사이트가 실제로 차단되기까지의 총 시간은 다음과 같습니다.

**만료 유예 기간 (3일)** + **프론트엔드 차단 유예 기간 (설정 값)** = 총 지연 시간

예를 들어, 기본 설정과 7일의 프론트엔드 유예 기간을 사용한다면, `date_expiration` 이후 최대 10일이 지나야 사이트가 실제로 차단될 수 있습니다.

### 7. 멤버십 수동 만료

크론 주기를 기다리지 않고 사이트를 즉시 차단해야 하는 경우, 멤버십 상태를 수동으로 변경할 수 있습니다.

1. **Ultimate Multisite > 멤버십**으로 이동합니다.
2. 영향을 받는 멤버십을 클릭합니다.
3. 상태를 **Expired** 또는 **Cancelled**로 변경합니다.
4. **저장**을 클릭합니다.

프론트엔드 차단은 다음 페이지 로드 시 적용됩니다 (만료된 멤버십의 경우 프론트엔드 차단 유예 기간을 따르거나, 취소된 멤버십의 경우 즉시 적용됩니다).

## 요약

만료일로부터 사이트 차단까지의 전체 타임라인:

```text
date_expiration 경과
       |
       v
  [3일 크론 유예 기간]     <-- 멤버십은 여전히 활성/보류 상태로 표시됨
       |
       v
  크론이 멤버십을 "만료"로 표시함
       |
       v
  [프론트엔드 차단 유예 기간]  <-- Settings > Memberships에 설정됨
       |
       v
  사이트 프론트엔드 차단됨
```

취소된 멤버십의 경로는 더 짧습니다:

```text
  멤버십 취소
       |
       v
  date_expiration 경과 (유예 기간 없음)
       |
       v
  사이트 프론트엔드 즉시 차단됨
```

## 개발자 참고 자료

다음 훅과 필터는 만료 및 차단 동작을 사용자 정의할 수 있도록 합니다.

| Hook/Filter | 설명 |
|---|---|
| `wu_membership_grace_period_days` | 멤버십을 만료로 표시하기 전 유예 기간 일수를 필터링합니다 (기본값: 3) |
| `wu_schedule_membership_check_interval` | 멤버십 검사 간격(간격)을 필터링합니다 (기본값: 1시간) |
| `wu_membership_renewal_days_before_expiring` | 만료일로부터 며칠 전에 갱신 결제를 생성할지 필터링합니다 (기본값: 3) |
| `wu_blocked_site_reactivation_url` | 사이트가 차단되었을 때 사용자 지정 재활성화 URL을 제공하도록 필터링합니다 |
| `wu_membership_is_active` | 멤버십이 활성으로 간주되는지 여부를 필터링합니다 |
| `wu_membership_expired_check_query_params` | 만료된 멤버십을 찾는 데 사용되는 쿼리 매개변수를 필터링합니다 |
| `wu_membership_trial_check_query_params` | 만료된 체험 기간을 찾는 데 사용되는 쿼리 매개변수를 필터링합니다 |
