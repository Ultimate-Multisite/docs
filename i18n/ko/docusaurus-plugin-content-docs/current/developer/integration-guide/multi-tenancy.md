# 멀티테넌시 통합

Ultimate Multisite: 멀티테넌시 1.2.0은 주권형 테넌트, 마이그레이션 검증, 테넌트 수명주기 자동화를 위한 여러 통합 접점을 변경합니다.

## 테넌트 부트스트랩 흐름

테넌트를 생성하거나 수정하는 통합은 다음 순서를 따라야 합니다.

1. 테넌트 레지스트리 레코드와 격리 모델을 확인합니다.
2. 테넌트 데이터베이스 작성자를 생성하거나 검증합니다.
3. 테넌트 스키마를 부트스트랩합니다.
4. 테넌트 사용자를 프로비저닝합니다.
5. 테넌트 라우팅 및 파일 시스템 경로를 등록합니다.
6. 테넌트를 노출하기 전에 마이그레이션 검증을 실행합니다.

주권형 테넌트가 네트워크 데이터베이스 연결을 재사용할 수 있다고 가정하지 마세요. 애드온에서 제공하는 테넌트 레지스트리와 작성자 추상화를 사용하세요.

## SSO 및 REST 훅

상태 비저장 테넌트 자동 로그인은 목적 클레임, JTI 재생 보호, 만료 상한, 출처 고정이 포함된 단기 토큰을 사용합니다. 로그인 버튼이나 원격 관리 링크를 추가하는 통합은 테넌트 로그인 URL을 직접 구성하는 대신 지원되는 SSO 흐름을 통해 테넌트 방문을 생성해야 합니다.

네트워크 측 API 감사 이벤트와 일일 요약은 주권형 테넌트 게이트웨이에 사용할 수 있습니다. 테넌트 수명주기 엔드포인트를 호출하는 외부 시스템을 디버깅할 때 해당 로그를 사용하세요.

## 주권형 고객 작업 URL

Ultimate Multisite v2.13.0은 주권형 테넌트 고객 작업을 Account, checkout, 청구, 송장, 사이트, 템플릿 전환, 도메인 매핑 흐름을 위해 메인 사이트로 다시 라우팅합니다. 테넌트 측 관리 링크를 렌더링하는 통합은 해당 작업이 메인 사이트 고객 패널을 가리키도록 해야 하며, 사용자가 작업을 완료한 후 테넌트로 다시 이동할 수 있어야 하는 경우 검증된 반환 대상을 포함해야 합니다.

교차 도메인 관리 링크에는 핵심 SSO 래퍼를 사용하세요.

```php
$url = wu_with_sso($main_site_customer_url);
```

생성된 URL은 `wu_sso_url`을 통해 계속 필터링할 수 있으며, 이 필터는 SSO URL, 현재 사용자, 대상 사이트 ID, 리디렉션 컨텍스트를 받습니다. 애드온은 이 필터를 사용해 공급자별 컨텍스트를 추가하거나, Ultimate Multisite의 토큰 검증을 유지하면서 브로커 URL을 대체할 수 있습니다.

주권형 테넌트 내부에 멤버십, 송장, 청구 주소, 템플릿 또는 도메인 관리 상태를 중복하지 마세요. 테넌트 Dashboard는 실행기로, 메인 사이트 고객 패널은 관리 작업의 기준 시스템으로 취급하세요.

## 마이그레이션 검증

마이그레이션 또는 수명주기 통합이 테넌트 데이터를 변경한 후에는 검증 게이트를 실행하세요.

- `wp tenant verify-no-legacy --site=<site-id>`는 테넌트가 더 이상 레거시 네트워크 측 경로에 의존하지 않음을 확인합니다.
- `wp tenant verify-sovereign-push --site=<site-id>`는 주권형 푸시 작업이 처리 및 비우기를 수행할 수 있음을 확인합니다.

통합은 검증 실패를 배포 차단 요인으로 취급해야 하며, 실패가 해결될 때까지 테넌트를 라이브로 표시하지 않아야 합니다.

## 테넌트 삭제

삭제 흐름은 테넌트 데이터베이스 자격 증명이 정리되도록 애드온 해체 경로를 호출해야 합니다. 외부 통합은 해체가 성공한 후 공급자 리소스를 제거할 수 있지만, 검증 또는 비동기 푸시 작업이 아직 실행 중인 동안 호스트 데이터베이스나 폴더를 삭제해서는 안 됩니다.

## 사용 중단된 데이터베이스 라우터

레거시 `Database_Router`는 사용 중단 스텁으로 대체되었습니다. 새 통합은 이전 라우터 클래스에 의존하는 대신 현재 사이트 라우터와 테넌트 레지스트리 API를 통해 테넌트를 확인해야 합니다.
