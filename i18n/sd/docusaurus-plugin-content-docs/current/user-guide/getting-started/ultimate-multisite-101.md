# Ultimate Multisite 101

Ultimate Multisite هڪ ووردپرس ملٽيسايت (Multisite) پلگئن آهي جيڪو توهان کي ڪلائين کي WaaS يا ويب سائٽس ايز ا سروس (Websites as a Service) ڏيڻ جي اجازت ڏئي ٿو. ان ۾ گهريان ۽ اسان سکون ٿا ته Ultimate Multisite توهان جي ڪاروبار ۽ ڪلائين کي ڪيئن مدد ڪري سگهي ٿو، پر شروع ڪرڻ کان اڳ، اسان کي ڪجهه بنيادي علم حاصل ڪرڻ گهرجي.

## ووردپرس ملٽيسايت (The WordPress Multisite) {#the-wordpress-multisite}

هڪ سڀني ماڻهن کي عام طور تي جوڙيل ووردپرس انٽالين (installation) بابت واقف آهي. توهان يا ته پنهنجي هاوسنگ پروائيڊر جي ڪنٽرول پینل ذريعي اهو ٺاهيو ٿا، يا جيڪڏهن توهان جِتني باهيون آهيو، ته هڪ نئين ويب سرور ۽ ڈیٹا بيسٽ (database) کي سٽ اپ ڪريو، کور فائلون ڈاؤنلوڊ ڪريو ۽ انٽالين پراجڪت شروع ڪريو.

هي دنيا ۾ مليارات ووردپرس سائٽن لاءِ ڪم ڪندو آهي پر ايجنسي يا هاوسنگ پروائيڊر جي نقطي نظر کان، اسان هڪ منٽ حجم بابت بحث ڪيون ٿا.

جڏهن توهان هڪ ووردپرس سائيٽ يا هيٺ هڪ سو سائٽ خود بخود ڪنٽرول پینل ذريعي ٺاهڻ جو حصو آهي ته جلدي ئي ان جي انتظام (management) تي اچڻ کان سواءِ سمجهجي ويندا آهن. جيڪڏهن اهو منظم نه هجي، ته توهان مالوير (malware) لاءِ هڪ اهم مقصد بڻجي ويندا آهيو. انتظام ڪرڻ کا مطلب ڪوشش ۽ وسيلن جو استعمال آهي ۽ جيتوڻي ووردپرس سائٽن جي انتظام ۽ عمل کي آسان بڻائڻ ۾ مددگار ٻاهر ئي ٽولز ۽ پلگئن موجود آهن، پر ان واقعيت جو مطلب آهي ته ڪلائين طرفان ادمنিষ্টڪ رسائي (administrative access) برقرار رکڻ ڪري اهڙا ڪوشون آساني سان شکست ملي سگهن ٿيون.

پنهنجي بنيادي حصي ۾، ووردپرس هڪ خاص خصوصيت ڏئي ٿو جنهن جو نالو 'Multisite' آهي، جيڪا 2010 ۾ ووردپرس 3.0 جي لانچ کان شروع ٿي ٿي. ان کان پوءِ اسان کي نئين خصوصيتون شامل ڪرڻ ۽ سيڪيورٽي کي مضبوط ڪرڻ لاءِ ڪيترائي موجوده revision (تبديل هلڻ) حاصل ٿيا آهن.

اصل ۾، هڪ ووردپرس ملٽيسايت کي هي طرح سوچيو جا سگهو ٿا: هڪ اڪاديمي ادارو ووردپرس جو هڪ واحد انٽالين برقرار رکي ٿو پر هر فیکلٽي پنهنجي ووردپرس سائٽ برقرار رکي ٿو.

### نیٹ ورک (The Network) {#the-network}

WordPress جي لحاظ کان، هڪ ملٽي سائيٽ نڪتون آهي جتي ڪجهه subsites کي هڪ ئي dashboard کان منظم ڪري سگهجي ٿو. جيتوڻيڪ hosting providers جو وچ ۾ ملٽي سائيٽ نڪتون کڻڻ ۾ فرق پوي ٿو، پر ان جو نتيجو عام طور تي wp-config.php فائل ۾ ڪجهه اضافي هدايتون هونديون آهن ته جيئن WordPress کي خبر ٿي سگهي ته اهو هن خاص mode ۾ کام ڪري رهيو آهي.

ملٽي سائيٽ نڪتون ۽ Stand-alone WordPress installation جو ڪيترن ئي واضح فرق آهن جن تي اسين مختصر طور تي بحث ڪنداسين.

#### Subdomain vs. Subdirectory (سبڊومين مقابل سبڊيڪٽري) {#subdomain-vs-subdirectory}

ان جو هڪ اهم قدم جيڪو توهان کي وٺڻو پوندو اهو آهي ته ملٽي سائيٽ installation ڪهڙي _subdirectories_ يا _subdomains_ سان کام ڪندي ٿي. Ultimate Multisite ان ٻنهي چونڊن سان برابر ئي ڪم ڪري سگهي ٿو پر انهن ٻنهي configuration جي وچ ۾ ڪجهه architectural فرق آهن.

_subdirectory_ configuration ۾، network sites مائين دامين نام تي بنيادي پاتھ (path) وراثت ۾ حاصل ڪندا آهن. مثال طور، 'site1' نالو ڏنل network site جو پورو URL https://domain.com/site1 هوندو. _subdomain_ configuration ۾، network site جو پنهنجو خود هڪ _subdomain_ مائين دامين نام کان حاصل ڪندو. تنهن ڪري 'site1' نالو ڏنل site جو پورو URL https://site1.domain.com/.

جيتوڻيڪ ٻئي انتخاب صحيح آهن، پر _subdomains_ جو استعمال ڪجهه فائدن سان گڏ آهي ۽ ان جي architecture ۾ وڌيڪ سوچ ۽ پلاننگ جي ضرورت ٿيندي آهي.

DNS جي لحاظ کان، _subdirectories_ جو استعمال ڪيو وڌيڪ ساهل چيلينجي آهي. جهڙوڪ نتيجي واري سائٽس (network sites) صرف ماءُ (parent path) جا деца هوندا آهن، ان لاءِ صرف هڪ هميشه (single domain name entry) ضرورت هوندو آهي جو بنيادي ڊومين نام لاءِ موجود هجي. _subdomains_ جي معاملن ۾ چيلينجي ٿي ٿي وڌيڪ گهڻي جڳهه تي، جنهن لاءِ يا ته هر نتيجي موقعي لاءِ هڪ الگ CNAME entry ضرورت هوندي آهي يا DNS records ۾ wildcard (*) entry.

SSL ۽ SSL certificates جي ڇڏڻ ۽ استعمال جو هڪ ٻيو اهم حصو آهي. _subdirectory_ configuration ۾، هڪ هميشه (single domain) certificate استعمال ڪري سگهجي ٿو جڏهن ته نتيجي سائٽس صرف بنيادي ڊومين نام جا رستو هوندا آهن. تنهن ڪري domain.com لاءِ جيڪا certificate هجي، اهو https://domain.com/site1، https://domain.com/site2 ۽ ان طرح جيئن آهي ان لاءِ SSL مناسب فراهم ڪندي.

_subdomain_ configuration ۾، wildcard SSL certificate جو استعمال هي سڀ کان عام روي آهي. هن قسم جي SSL certificate هڪ ڊومين ۽ ان جا _subdomains_ کي encryption فراهم ڪري ٿي. تنهن ڪري، هڪ wildcard SSL certificate https://site1.domain.com، https://site2.domain.com ۽ https://domain.com خود لاءِ encryption فراهم ڪندي.

جيتوڻيڪ ٻيا خيار موجود آهن، پر اه عام طور تي محدود دائرے ۽ استعمال ۾ هوندا آهن ۽ مناسبت جي لحاظ کان ان لاءِ اضافي configuration ۽ غور و عمل جي ضرورت هوندي آهي.

#### Plugins and Themes {#plugins-and-themes}

WordPress جو فائدو هي آهي ته اهو ڪجهه ڇڏي ويندو آهي، اڳتي ڪجهه صارف جي نقطي نظر کان به. جيڪڏهن هڪ stand-alone WordPress installation ۾ سائيٽءَءَ جي ادمن (site administrator) هڪ خراب plugin نصب ڪري ٿو يا پنهنجي installation کي اپ ٹو ڈیٹ رکڻ ۾ ناڪام ٿئي ٿو، ته ان جو واحد متاثر ۽ نقصان خود ذات آهي. پر جيڪڏهن ڪا site administrator multisite installation تي خراب plugin نصب ڪري ٿو، ته اهو هر نتيجي موقعي جي هڪ متاثر بڻائي ڇڏيندو آهي.

اس ڪري ان سبب جي ڪري جڏهن توهان ان کي ملٽي سائيٽ ورڊپريس طور configure ڪندا آهيو، ته وائسٽ اسٽمينيٽرن (site administrators) کان پليجنز ۽ ٿيمز نصب ڪرڻ جو صلاحيت ختم ٿي ويندي آهي ۽ ان صلاحيت کي نئين ڊيڪٽر يا 'سپر ڊيڪٽر' جي رول ۾ منتقل ڪيو ويندو آهي. هي خاص رول اهو ٿي سگهي ٿو ته هو فوري ڪري سگهي ٿو ته ملٽي سائيٽن جا اسٽمينيٽر پنهنجيแดش بورڊ ۾ پليجنز جو منيو ڏسڻ يا رسائي کي قبول ڪرڻ جي اجازت ڏئي، ۽ جيڪڏهن هي اجازت ڪجي، ته اهو صلاحيت _پليجنز_ کي _فعال_ ڪرڻ يا _غیر فعال_ ڪرڻ تک محدود ٿيندي آهي.

هي حد تائين، ملٽي سائيٽ ڊيڪٽر ملٽي سائيٽ ۾ پليجنز ۽ ٿيمز نصب ڪرڻ جي ذميواري رکي ٿو ۽ انن طريقن کي استعمال ڪرڻ لاءِ ملٽي سائيٽن کي اجازت ڏيڻ لاءِ صلاحيتون تفويض ڪري ٿو. اسٽمينيٽر پنهنجي سائيٽ ۾ نه نصب ڪندڙ يا ان کي ماني نه ڏني پليجنز ۽ ٿيمز تک رسائي حاصل نه ڪري سگهن ٿا.

#### صارف ۽ ڊيڪٽر (Users and Administrators) {#users-and-administrators}

ملٽي سائيٽ ورڊپريس ۾، تمام ملٽي سائيٽون هڪ ئي ڊيٽابيز کي شيئر ڪن ٿيون ۽ ان ڪري تمام استعمال، رول ۽ صلاحيتون شيئر ڪن ٿيون. ان کي سمجهڻ جو هي بهترين طريقو آهي ته تمام استعمال ملٽي سائيٽ جي ممبر آهن نه ڪنهن خاص سائيٽ جا.

هي سمجھ ڏئي ٿي ته صارف کي ٺهڻ جي اجازت ڏيڻ مناسب نه ٿي سگهي، ۽ ان ڪري ورڊپريس ملٽي سائيٽ اسٽمينيٽرن کان هي صلاحيت ختم ڪري ڇڏيندي آهي ۽ ان کي ملٽي سائيٽ ڊيڪٽر جي رول ۾ منتقل ڪري ڇڏيندي آهي. ان طرح ملٽي سائيٽ ڊيڪٽر هڪ سائيٽ اسٽمينيٽر کي ضروري صلاحيتون تفويض ڪري سگهي ٿو ته جيئن هو پنهنجي سائيٽ لاءِ صارف جو حساب bana سگهن.

<!-- Screenshot unavailable: WordPress Multisite network user management interface -->

اوپر واري بيان کي ٻيهر دہرائيندڙ، جڏهن ته استعمال ڪندڙ حساب ان سائيٽ سان تعلق رکڻ کان سواءِ، هي درحقیقت ملٽي سائيٽ جي رول ۾ مقرر ٿيل آهن ۽ ان ڪري تمام ملٽي سائيٽن لاءِ انهمڪندي (unique) هجڻ گهرجي. ڪجهه صورتحالون آهن جتي هن سبب صارف جا نام register ڪرڻ لاءِ موجود نه هوندا آهن.

گر پريئسٽن ۾، جيتوڻيڪ اهو ڪنهن به ڪمپني جي سسٽم ۾ نئين نئين تصور جو نه هجڻ، پر صارف جي رجيشن ۽ تصديق لاءِ هي هڪ ئي ذريعو عام طور تي ان لڳاتار کي سمجهڻ لاءِ ڏکيو تصور آهي جڏهن ماڻهن کي وڏن WordPress installations سان واقفيت آهي جتي صارف جي انتظام ٿورو آسان هوندو آهي.

#### Media (ڊائري) {#media}

جتي ڪنهن نڪرٽ سائيٽون ۾ هڪ ئي ڊيٽا بيس استعمال ڪندا آهن، ته اهي ميڊيا فائلن لاءِ فزڪل سسٽم تي مختلف پٿر (paths) برقرار رکندا آهن.

معياري WordPress جو جڳهه (wp-content/uploads) وڌيڪ رهندي آهي؛ پر ان جو پاتھ ان نڪرٽ سائيٽ جي خاص ID کي ظاهر ڪرڻ لاءِ بدلجي ويندي آهي. اها يعني ته ڪنه نڪرٽ سائيٽ جي ميڊيا فائلون wp-contents/uploads/site/[id] جي صورت ۾ نظر اچن ٿيون.

#### Permalinks (پيرملينکس) {#permalinks}

هڪ وقت پهرين اسان اهو ٻڌايو هو ته _subdomain_ کان _subdirectory_ کان بهتر ڪجهه فائدن آهن ۽ اهي هيٺ ڏنل آهن: پٿر (paths).

_subdirectory_ جي صورت ۾، مٿي سائيٽ (پهرين سائيٽ جو جيڪا نڪرٽ قائم ٿيڻ وقت بنائي ويندي آهي) ۽ نڪرٽ سبي سائيٽون کي دومين نام کان لڳو رهيو هڪ ئي پاتھ شیئر ڪرڻو پوندو. ان ۾ گهڻيون تڪرار (conflicts) ٿيڻ جي امڪاني هجي.

پوسٽس لاءِ، نڪرٽ سائيٽن سان تصادم رکڻ جو روڪڻ لاءِ مٿي سائيٽ تي هڪ لازمي /blog/ پاتھ شامل ڪيو ويندو آهي. ان جو مطلب اهو آهي ته خوبصورت پيرملينکس جهڙوڪ ‘Post name’ کي domain.name/blog/post-name/ جي صورت ۾ ڏيکاريو ويندو.

<!-- Screenshot unavailable: WordPress permalink settings showing /blog/ path in subdirectory configuration -->

_subdomain_ جي صورت ۾، هي عمل ضروري نه آهي ڇاڪاڻ ته هر نڪرٽ سائيٽ کي پورو domain separation جو فائدو ٿيندو آهي ۽ ان کي هڪ ئي پاتھ تي انھناري نه رهڻ گهرجي. اهي جنهن انهن جي _subdomain_ جي بنياد تي پنهنجي خاص پٿر (paths) برقرار رکندا آهن.

#### Static Pages (اسٽيڪ پيڄا) {#static-pages}

_subdirectory_ configuration ۾، سٽيڪ پيجس (static pages) جي نالي چونڊ ۾ تضاد (naming conflicts) جو امکان گهڻو هوندو آهي ڇا ته مين سائٹ ۽ ڪم ڪندڙ سائٽون هڪ ئي پاتھ (path) استعمال ڪن ٿيون.

ان کي روڪڻ لاءِ، WordPress واري هڪ طريقو فراهم ڪري ٿو جنهن سان توهان ڪجهه خاص سائٽ جي نالا بلڪي اسٽر (blacklist) ڪري سگهو ٿا ته جيئن اهي مين سائٽ جي نالا کان تضاد نه کڻن. عام طور تي، ورڪنگ اڊمن (network administrator) مين سائٽ جا روٽ پاتھس داخل ڪندو آهي.

<!-- Screenshot unavailable: WordPress network settings showing banned site names to prevent conflicts -->

_subdomain_ configuration ۾، سٽيڪ نالا کان تضاد جي امڪاني کي _subdomain_ پاران ختم ڪري ڇڏيو وڃي ٿو ڇاڪاڻ ته اهو نتيجي سائٽ لاءِ مخصوص آهي ۽ مين سائٽ سان ڪنه به طرح جو تعلق نه رکي ٿو.

### رجسٽر ڪرڻ (Registration) {#registration}

WordPress Multisite جي ورڪنگ گهرن ۾، نئين ۽ موجودہ استعمال ڪندڙن کي سائٽون کڻڻ جي اجازت ڏيڻ لاءِ ڪجهه نئين صارف رجسٽريشن جا خيار آهن.

<!-- Screenshot unavailable: WordPress Multisite network settings showing registration options -->

ان جهڙيءَ طرح، اڪيلو (stand-alone) WordPress installation جي برعڪس ۾، ورڪنگ سائٽون ان عام خيارن کي برقرار نه رکين جن سان صارف جي رجسٽريشن کڻي سگهجي ٿو يا ان رجسٽريشن کي رولن (roles) سان جوڙائي سگهجي ٿو.

<!-- Screenshot unavailable: WordPress standalone site user registration settings showing limited options -->

جڏهن صارف جون حسابون (user accounts) کڻيون وينديون، اهي حساب ورڪنگ niv (network level) تي جنريٽ ٿين ٿا. تنهن ڪري، ڪنه هڪ خاص سائٽ سان تعلق رکڻ جي بدران، اهي سڄي نتيجي نظام (network) جو حصو بڻجن ٿا. ان ۾ ڪجهه خاص فائدن ۽ نقصان پڻ آهن.

مثال کے طور پر، فرض کریں کہ آپ کا WordPress Multisite نیوز اور معلومات کے کاروبار میں ہے۔ آپ اس multisite کو قائم کریں گے اور پھر فنانس، ٹیکنالوجی، تفریح اور دیگر دلچسپیوں کے مختلف شعبوں کے لیے نیٹ ورک سائٹس بنائیں گے جبکہ پلے ان اور تھیمز پر مکمل کنٹرول برقرار رکھیں گے۔ ہر نیٹ ورک سائٹ کا اپنے نیٹ ورک سائٹ کی شکل و صورت اور صارف کے تجربے پر کسٹم پوسٹ ٹائپس یا عام پوسٹ کیٹیگریز سے کہیں زیادہ کنٹرول ہوگا۔

اس حد تک، جب کوئی صارف لاگ ان کرتا ہے تو وہ نیٹ ورک میں لاگ ان ہوتا ہے اور بالآخر ایک ہموار تجربہ فراہم کرنے کے لیے ہر نیٹ ورک سائٹ میں بھی لاگ ان ہوتا ہے۔ اگر آپ کی نئی سائٹ سبسکرپشن پر مبنی تھی تو یہ بہترین حل اور نتیجہ ہوگا۔

اگر تاہم، multisite کا مقصد مختلف نیٹ ورک سائٹس پیش کرنا ہو جن کا ایک دوسرے سے کوئی تعلق نہ ہو، تو تقریباً ہمیشہ ایسا ہی ہوتا ہے کہ صارف کے کرداروں (user roles) کو تبدیل کرنے کے لیے بیرونی یا اضافی پلے ان کی ضرورت پڑتی ہے۔

### ڈومین اور SSL {#domain-and-ssl}

آئیے ایک WordPress Multisite انسٹالیشن کے بارے میں بات کرتے ہیں جو ہماری توجہ سے تقریباً چھپ جاتا ہے - Wordpress.com۔ یہ بالآخر ایک WordPress multisite کا سب سے وسیع مثال ہے اور اس کی اپنی مقصد پورا کرنے کے لیے کسٹمائز ہونے اور ڈھلنے کی وسیع صلاحیتوں کو ظاہر کرتا ہے۔

آج کل جدید انٹرنیٹ پر SSL کا استعمال تقریباً لازمی ہو گیا ہے اور WordPress multisites کے نیٹ ورک ایڈمنز کو جلد ہی ان چیلنجز کا سامنا کرنا پڑتا ہے۔

_subdomain_ کنفیگریشن میں سائٹس کو روٹ ڈومین نیم کی بنیاد پر بنایا جاتا ہے۔ اس طرح، 'site1' نامی سائٹ 'site1.domain.com' کے طور پر بنائی جائے گی۔ وائلڈ کارڈ SSL سرٹیفکیٹ کا استعمال کرتے ہوئے، ایک نیٹ ورک ایڈمن اس چیلنج کا کامیابی سے حل کر سکتا ہے اور نیٹ ورک کے لیے SSL انکرپشن کی صلاحیت فراہم کر سکتا ہے۔

WordPress Multisite واريڈ (Multisite) وچ ایک ایسا فنکشن ہوندا اے جیہڑا نیٹ ورک سائٹس نوں کسٹم ڈومین نیمز یا نیٹ ورک دے روٹ ڈومین توں مختلف ڈومین نیمز سان جڑنے دی اجازت دیندا اے۔

نیٹ ورک ایڈمنسٹریٹرز لئی، یہ ڈومین نیم کنفیگریشن تے SSL سرٹیفکیٹ دے جاری کرن تے دیکھ بھال وچ اضافی پیچیت پیش کردا اے۔

اس حد تک کہ جب WordPress Multisite ایہ سہولت فراہم ڪريتا اے کہ [www.anotherdomain.com](http://www.anotherdomain.com) نوں 'site1' توں جوڑیا جا سکدا اے، تب نیٹ ورک ایڈمنسٹریٹر دے کول DNS entries تے SSL سرٹیفکیٹ دی بیرونی مینجمنٹ تے انھاں لاگو کرن دا چیلنج رہندا اے۔

## Ultimate Multisite {#ultimate-multisite}

ایک اسٹینڈ الون WordPress انسٹالیشن تے Multisite انسٹالیشن دے درمیان فرق سمجھ کے، آئیے دیکھتے ہیں کہ Ultimate Multisite Websites as a Service (WaaS) فراہم کرنے لئی کس طرح سب سے بہترین ہتھیار ہے۔

### تعارف {#introduction}

Ultimate Multisite آپ دا وہ سوئس آرمی نائف اے جیہڑا Website as a Service (WaaS) بنانے وچ مدد کردا اے۔ Wix.com، Squarespace، WordPress.com تے پھر اپنا سروس اپنے پاس رکھنے دے بارے وچ سوچو۔

اندرونی طور تے Ultimate Multisite WordPress Multisite دا استعمال کرتا ہے پر ایسا طریقے نال کہ یہ نیٹ ورک ایڈمنسٹریٹرز جو multisite انسٹالیشن دے مسائل نال سامنا کردے نیں ان سارے چیلنجز حل کر کے، بیک وقت صلاحیتاں بڑھاتا اے جیہڑے بہت سارے مختلف استعمال دے حالات (use cases) دی حمایت کرن لئی اجازت دیندا اے۔

اگلے حصوں وچ ہم کچھ عام استعمال دے حالات تے ان حالات دی حمایت کرن لئی درکار غور و فکر دیکھاں گے۔

### استعمال دے حالات (Use Cases) {#use-cases}

#### کیس 1: ایک ایجنسی {#case-1-an-agency}

عام طور تے کسی ایجنسی دے بنیادی ہنر ویب سائٹس ڈیزائن کرنا ہوندا اے جیہڑے ان دے ہوسٹنگ یا مارکیٹنگ نوں اضافی سروسز دے طور تے لسٹ کرن جیسی چیزاں ہوندی اے۔

एजेन्सीहरूका लागि, Ultimate Multisite ले एउटै प्लेटफर्ममा धेरै वेबसाइट होस्ट र व्यवस्थापन गर्ने क्षमताले अविश्वसनीय मूल्य प्रस्ताव दिन्छ। विशेष गरी ती एजेन्सीहरूको लागि जसले GeneratePress, Astra, OceanWP वा अन्य जस्ता निश्चित थीममा आफ्नो डिजाइन मानकीकरण गर्छन्, तिनीहरू Ultimate Multisite को क्षमताको फाइदा उठाउन सक्छन् जसले प्रत्येक नयाँ साइटका लागि यी थीमहरू स्वचालित रूपमा सक्रिय गर्न मद्दत गर्दछ।

एजेन्सी मूल्य निर्धारणका लागि सामान्य र लोकप्रिय प्लगइनहरूको प्रशस्त डिलहरूसँगै, Ultimate Multisite को प्रयोगले एजेन्सीहरूलाई विद्यमान लगानीको फाइदा उठाउन अनुमति दिन्छ जसद्वारा प्लगइनहरू इन्स्टल गर्ने, मर्मत गर्ने र उपयोग गर्ने साझा प्लेटफर्म प्रदान गर्न सकिन्छ।

बहुसंख्यक अवस्थामा, कन्फिगरेसनको आवश्यकता हुनेछ र सौभाग्यवश Ultimate Multisite ले धेरै लोकप्रिय होस्टिंग प्रदायकहरूका लागि डोमेन मैपिंग र SSL प्रमाणपत्रहरूलाई Cloudflare र cPanel जस्ता सेवाहरूसँग आफ्ना एकीकरणहरूको साथ अविश्वसनीय रूपमा सजिलो बनाउँछ।

त्यसैले, यी मध्ये कुनै एक प्रदायकको फाइदा उठाउँदा वा Ultimate Multisite लाई Cloudflare पछाडि राख्दा डोमेन व्यवस्थापन र SSL प्रमाणपत्रहरू जस्ता पक्षहरू केही सहज हुन्छन्।

जुन एजेन्सीहरू साइटहरूको निर्माणमा कडा नियन्त्रण राख्न चाहन्छन्, तिनीहरूले Ultimate Multisite को सुव्यवस्थित इन्टरफेस मार्फत साइटहरू बनाउन र ग्राहक तथा योजनाहरूसँग साइटहरू जोड्न सक्ने सजिलोपनको सराहना गर्नेछन्।

![Ultimate Multisite site management interface](/img/admin/sites-list.png)

प्रत्येक उत्पादनको आधारमा प्लगइन र थीममा कडा नियन्त्रण Ultimate Multisite को सहज इन्टरफेस मार्फत कायम गरिन्छ, जसले नयाँ साइटका लागि इन्स्टेन्ट गरिएमा प्लगइन र थीमहरू उपलब्ध वा लुकाउन सकिन्छ साथै तिनीहरूको सक्रिय अवस्था पनि।

![Product plugin limitations interface](/img/config/product-plugins.png)

Themes وِشے (themes) ناهيئن ڪجهه مشابہ ڪم ڪن ٿا، جنهن سان خاص themes کي سائيٽ جي ٺهڻ دوران فعال يا ٻاهر ڪرڻ جي اجازت آهي.

![Product theme limitations interface](/img/config/product-themes.png)

Agencies کي Ultimate Multisite استعمال ڪرڻ ۾ سکون ملندو، اها انهيءَ جو ڪري سگهن جيڪو هو سب تائين بهتر ڪري ٿا - شاندار ويب سائٽس डिजाइन ڪرڻ.

#### ڪيس 2: Niche Provider (خاص جي فراهمي) {#case-2-niche-provider}

ڪجهه پراني قول آهي ته، "هڪ ڪم ڪيو ۽ اهو خوبصورت ڪيو". ڪيترن ئي ماهرين لاءِ ان جو مطلب هڪ اڪيلو بنيادي خيال يا سرڪاري کي گهرائي ڏيڻ سان ڪنهن پروڊڪٽ يا سروس کي ٺاهڻ آهي.

شڪريت، توهان شاید ڪلابن کي ويب سائٽس پيش ڪريون ٿا ۽ هڪ شوقيه گولفار آهيو يا توهان ڪلانز لاءِ ويب سائٽس فراهم ڪرڻ ۾ شوقين اسپورٽس گيمر رهيا. ڪنه رستوران لاءِ بوڪنگ سروس پيش ڪرڻ واري فرد؟

ڪيترن ئي وجوهن پر توهان کي عام فريم ورڪ ۽ platform تي بنيادي خدمات پيش ڪرڻ چاهي ٿو. اهو اها شيءِ هج sakiyo ته توهان ضروري ڪم ڪندڙ صلاحيتون فراهم ڪرڻ لاءِ bespoke plugins جو تصميم يا ان ۾ سرمایہ کاری ڪئي آهي، يا اهو اها صورت هجي ته صنعت جا بهتر عمل (industry best practices) کي تصميم ۽ انداز جي ڪنهن هيٺ ڏنل طريقو جي ضرورت آهي.

Ultimate Multisite جي نئين خصوصيتن مان هڪ قالبائي سائٽس (template sites) جو استعمال آهي. قالبائي سائٽ هو اها جتي theme نصب ٿي وڃي ۽ فعال ڪجي، ضروري plugins نصب ٿين ۽ فعال ٿين، ۽ نموني پوسٽ يا ايڪسائٽس ٺاهيون وڃن. جڏهن ڪنه ڪلائين قالب (template) تي بنيادي نئين سائٽ ٺاهي ٿو، ته قالب جي مواد ۽ ترتيئون نئين ٺهي واري سائٽ ۾ كوبي ٿين.

خاص سائٽس ۽ خدمتن جي فراهمي لاءِ، اهو خاص plugins ۽ تصميم سان سڌي تائين قائم ٿيڻ جهڙي هڪ سائٽ instant ٺاهڻ جي صلاحيت ۾ هڪ غير متوازن فائدو ڏئي ٿو. ڪلائي کي صرف خدمت پورو ڪرڻ لاءِ تمام گهٽ ان پيٽ (minimal input) فراهم ڪرڻو آهي.

आवश्यकताओं کے لحاظ سے، دونوں میں سے _subdirectory_ یا _subdomain_ کی ترتیب مناسب ہو سکتی ہے، جس صورت میں آرکیٹیکچر کے انتخاب کا معاملہ _subdirectories_ کے لیے ایک سادہ SSL سرٹیفکیٹ یا _subdomains_ کے لیے وائلڈ کارڈ SSL سرٹیفکیٹ کے درمیان ہوگا۔

#### کیس 3: ورڈپریس ویب ہوسٹنگ {#case-3-wordpress-web-hosting}

ورڈپریس سائٹس کو ہوسٹ کرنے کے بہت سے طریقے ہیں لیکن یہ اتنی آسان نہیں ہے کہ کسی کسٹمر کو پہلے سے انسٹال شدہ ورڈپریس ورژن کے ساتھ ویب اسپیس فراہم کر دی جائے۔ ایسا اس لیے ہوتا ہے کیونکہ ایک بامعنی سروس فراہم کرنے کے لیے کئی فیصلوں اور غور و فکر کا مجموعہ ہونا ضروری ہے۔

Ultimate Multisite اس شعبے میں بہترین کارکردگی دکھاتا ہے کیونکہ یہ ورڈپریس سائٹس کو ہوسٹ کرنے کے لیے ایک مکمل ٹرنکیٹ حل فراہم کرتا ہے۔ اس حل میں سبسکرپشن سروسز، ادائیگی وصولی، چیک آؤٹ فارمز، رعایت کے وؤچرز اور کسٹمر کمیونیکیشن فراہم کرنے کے بنیادی میکانزم شامل ہیں۔

ورڈپریس ملٹی سائٹ کو صحیح طریقے سے انسٹال کرنے، ترتیب دینے اور برقرار رکھنے کے لیے درکار زیادہ تر اہم کام Ultimate Multisite کی طرف سے آسان بنایا جاتا ہے تاکہ نیٹ ورک ایڈمنسٹریٹرز کو صرف اپنے سروس یا مخصوص شعبے جیسے پروڈکٹ ٹियर्स، قیمتوں اور سروس کی پیشکش سے متعلق پہلوؤں پر غور کرنا پڑے۔

Ultimate Multisite کے ساتھ انٹیگریٹ ہونا چاہتے ڈویلپرز کے لیے، یہ حل ایک مکمل RESTful API اور ایونٹ نوٹیفیکیشنز کے لیے Webhooks بھی فراہم کرتا ہے۔

بہت سے بیرونی پلگ انز اور لائسنسوں پر انحصار کے بغیر، Ultimate Multisite Wix، Squarespace، WordPress.com اور دیگر کی طرح ایک بھرپور فیچرز والا اور قابلِ موازنہ حل فراہم کرتا ہے۔

### آرکیٹیکچر کے غور و فکر {#architecture-considerations}

اگرچہ یہ مکمل گائیڈ نہیں ہے، لیکن Ultimate Multisite انسٹالیشن کو سپورٹ کرنے کے لیے ٹیکنالوجیز کے درست انتخاب کے لیے درج ذیل چیزیں رہنمائی کر سکتی ہیں۔

#### شیئرڈ بمقابلہ ڈیڈیکیٹڈ ہوسٹنگ {#shared-vs-dedicated-hosting}

بدقسمتي، تمام ہوسٹنگ فراہم کرنے والے سب برابر نہیں ہوتے اور کچھ انتہائی سرور ڈینسٹی پر کام کرتے ہیں۔ کم قیمت والے فراہم کنندگان عام طور پر سرور کی زیادہ سے زیادہ صلاحیت استعمال کر کے پیسے کما کر رکھتے ہیں۔ اس طرح آپ کا Ultimate Multisite انسٹالیشن صرف ایک ایسے چند سو سائٹس میں سے ایک ہو سکتا ہے جو اسی سرور پر چل رہے ہوں۔

اگر فراہم کنندہ کی طرف سے مناسب حفاظتی اقدامات نہ کیے جائیں تو شیئرڈ سرور پر موجود سائٹس کو 'شور والی پڑوسی' (noisy neighbour) کا مسئلہ درپیش ہوتا ہے۔ یعنی، اگر ایک ہی سرور پر کوئی سائٹ اتنے وسائل استعمال کر رہی ہو کہ باقی سائٹس کو بچا ہوا حصہ حاصل کرنے کے لیے مقابلہ کرنا پڑے۔ اکثر یہ اس طرح ظاہر ہوتا ہے کہ سائٹس سست ہوتی ہیں یا بروقت جواب نہیں دیتی ہیں۔

خود ویب ہوسٹنگ فراہم کنندہ ہونے کے ناطے، اثرات کا بہاؤ ایسا ہوگا کہ آپ کے کسٹمر خراب رفتار، کم پیج رینک اور زیادہ بائس ریٹ کا تجربہ کریں گے، جس کی وجہ سے وہ کہیں اور سروسیں تلاش کرتے ہوئے کسٹمر چور ہو جاتے ہیں۔

مختصراً، سستا ہونے کا مطلب اچھا ہونا نہیں ہے۔

Ultimate Multisite کو کئی اچھے ہوسٹنگ فراہم کرنے والوں کے ساتھ کام کرنے کے لیے جانا جاتا ہے اور یہ ڈومین میپنگ (domain mapping) اور خودکار SSL جیسے فنکشنز فراہم کرنے کے لیے ان کے ماحول کے ساتھ اچھی طرح سے інтеریٹ ہوتا ہے۔ یہ فراہم کنندگان کارکردگی کی قدر کرتے ہیں اور شیئرڈ ہوسٹنگ سے اعلیٰ درجے کی سروس فراہم کرتے ہیں۔

موجودہ مطابقت رکھنے والے فراہم کنندگان کی فہرست اور ہر ایک کے لیے مکمل سیٹ اپ ہدایات کے لیے براہ کرم Compatible Providers کے دستاویزات چیک کریں۔

#### کارکردگی کے لحاظ سے غور و فکر (Performance Considerations) {#performance-considerations}

Ultimate Multisite کوئی سست ایپلیکیشن نہیں ہے، بلکہ یہ حیرت انگیز طور پر تیز ہے۔ تاہم، یہ صرف اس قدر اچھا کام کرے گا جتنا بنیادی ایپلیکیشن اور انفراسٹرکچر اسے فراہم کرتا ہے اور صرف وہی استعمال کر سکے گا جو اسے دستیاب ہے۔

اس بات پر غور کریں: آپ کے پاس 100 سائٹس کی ایک Ultimate Multisite انسٹالیشن کا نیٹ ورک ایڈمن ہونے کا کام ہے۔ ان میں سے کچھ سائٹس اچھی کارکردگی دکھا رہی ہیں اور ہر روز وزٹرز کو اپنی طرف متوجہ کر رہی ہیں۔

Ultimate Multisite میں ایک جدید Cloudflare add-on شامل ہے جو نیٹ ورک ایڈمنسٹریٹرز کو اپنے انسٹالیشنز کو Cloudflare کے پیچھے رکھنے اور نہ صرف اس کی کیشنگ صلاحیتوں بلکہ DNS ہوسٹنگ، SSL سرٹیفکیٹس اور سیکیورٹی میکانزمز کا بھی استعمال کرنے کے قابل بناتا ہے۔

#### بیک اپس (Backups) {#backups}

آپ 50 لوگوں سے بیک اپس کے بارے میں مشورہ لے سکتے ہیں اور بیک اپ حکمت عملیوں پر 50 مختلف رائے حاصل کر سکتے ہیں۔ جواب یہ ہے کہ یہ اس پر منحصر کرتا ہے۔

ڪو شي فيه نه مناظر آهي ته بڪ اپس (backups) جي ضرورت آهي ۽ اهو تقریباً ناممڪن آهي ته ان کي پراؤ바이ڊر، خاص طور تي هڪ مانيجڊ سروس جو جيڪو يقيني طور تي فراهم ڪري ٿو، سنڀال ڏئي ٿو. ان ڪري ڪسٹمرز اسانٽيئر (network administrator) کان ان سروس کي ڏيندڙ ۽ سنڀالڻ وارو رهندا. جنهن جي اسانٽيئر طرف روي ٿا، اهو هڪ تمام مختلف مسئلو آهي.

هي حصي لاءِ، اچو ته هي يقين ڪريون ته بڪ اپس انهيء وقت جي سسٽم جي حالت جو هڪ نقطي-در-وقت ڪاپي آهي جڏهن بڪ اپس شروع ڪيو هو. سادو لفظن ۾، جيڪا به سسٽم جي حالت بڪ اپس جي وقت تي موجود هجي، ان حالت کي محفوظ ڪري ورتو ۽ بڪ اپس ۾ بند ڪري ڇڏيندا آهيون.

هي سمجھڻ سان، هي بڪ اپس ڪيئن حاصل ڪجن ۽ توهان جي ماحول لاءِ جو بهتر آهي اهو توهان جي ضرورتن ۽ هلٽنگ پراؤ바이ڊر جي انهن ضرورتن کي پورو ڪرڻ جي صلاحيت تي گهري طور انحصار ڪري ٿو. پر، سڀ کان وڌيڪ رائے ڏيندڙ کان وڌيڪ نه ڏيندڙ تائين، هيٺ ڏنل خيار ڪجهه رهنمائي فراهم ڪندا.

#### اسناپٽس (Snapshots) {#snapshots}

اسناپٽس بڪ اپس لاءِ سنڌو حل آهن ڇاڪاڻ ته اهي ساهل، سمجهڻ ۾ آسان (جڏهن توهان ريستور ڪرڻ چاهيو ته نه)، ۽ 'بس کام ڪندا' آهن. پر ان لاءِ توهان کي پنهنجي پراؤ바이ڊر جي مدد گهرجي ۽ اهو اڪثر صرف تب علق آهي جڏهن توهان جي پاسي VPS (Virtual Private Server) يا ان طرح جو شي موجود هجي. اسان جي ‘Compatible Providers’ документаشن ۾ لکيل ڪيترن ئي پراؤ바이ڊرن ۾ بڪ اپس آهن جن کي اسانٽيئر طرف کان ڪنهن وڌيڪ مداخلت يا غور ڪرڻ جي ضرورت نه آهي.

جديہ بیک اپس فائلز ۽ ڈیٹا بیس کي ٽارگيت ڪن ٿا، جڏهن ته اسڪيٽ اسڪيپ (snapshot) پوري ڊسڪ کي ٽارگيت ڪري ٿو. ان جو مطلب اهو آهي ته نه صرف سائيٽءَ جا ڊيٽا اسڪيٽ ۾ محفوظ ٿيندا آهن پر اها ايڪسپيريٽنگ سسٽم ۽ ڪنfiguration به محفوظ ٿيندي آهي. انهيءَ لاءِ ڪيترن ئي ماڻهن لاءِ هي هڪ خاص فائدو آهي ڇاڪاڻ ته اسڪيٽ کان تقریباً فوري طور تي نئين سسٽم کي جنم ڏئي سگهجي ٿو ۽ خراب ٿيل ايڪسيسن جي جاءِ تي بدلو ڪرڻ لاءِ عمل ۾ آڻي سگهجي ٿو. ان طرح، فائلون واپس حاصل ڪرڻ لاءِ ڪورپريشن پنهنجي موجود هجڻ وارن ايڪسيسن کي ڊسڪ طور جوڳي ڪرڻ سان صرف اسڪيٽ ايميج کي جوڙڻ جي ضرورت آهي ته جيئن فائلن تک رسائي ۽ کاپی ڪري سگهجي وينديون.

هستيلاري فراهم ڪندڙ سان اسڪيٽ کان اضافي لاڳ qnt ٿي سگهن ٿا پر اهو حادثات جي خلاف هڪ انشورنس پولي آهي.

#### ايڪسٽرنل اسڪرپٽس (External Scripts) {#external-scripts}

WordPress ۽ MySQL ريسورسز کي بیک اپ ڪرڻ لاءِ ڪيترائي ايڪسٽرنل اسڪرپٽس ۽ حل موجود آهن ۽ اهي Ultimate Multisite لاءِ سٺي ڪم ڪندا ته جيئن اهو WordPress جو هڪ plugin آهي جيڪو WordPress جي فائل سسٽم ۽ ڊيٽابيس جو استعمال ڪري ٿو. ان ڪري، اها حل جيڪا WordPress سائيٽن کي بیک اپ ڪندي هجي، هو Ultimate Multisite جي ضرورتن کي مناسب طور تي ڍڪي سگهي ٿي.

اسان ڪنهن هڪ اسڪرپٽ کي ٻئي کان بهتر نه ڪري سگهون ٿا پر اسان جو عام مشورو اهو آهي ته نتيجا مطلوب آهن يا "ڪهڙو به يقيني رهيو" لاءِ ڪيترائي ئي بیک اپ ۽ ريستور جا تجربا چالو ڪريو، ۽ خاص طور تي جتي ڪنهن به قسم جي ڊيفيرنشيئل (differential) بیک اپ اسٽرٽجي لاڳو ٿئي ٿي ان اسڪرپٽ کي مسلسل جائزو وٺڻ سان.

اها رکڻ ضروري آهي ته اهي اسڪرپٽس چالو دوران سسٽم جي لاد (load) ۾ اضافو ڪن ٿا جنهن کي لهو ڪرڻ گهرجي.

#### پليجنز (Plugins) {#plugins}

WordPress ۾ تقريبن ڪو مسئلہ نه آهي جيڪو هڪ plugin سان حل نه ڪري سگهجي، ۽ جيڪڏهن توهان ايڪسٽرنل اسڪرپٽس جو انتظام ڪرڻ لاءِ مناسب نه آهيو ته هاڻي شاید هڪ plugin بهترين خيار هوندو.

Plugins جي مختلف خيارن ۽ خصوصيتن ۾ مختلف هوندا آهن پر انهن جو بنيادي ڪم هڪ ئي آهي، جيڪو WordPress جون فائلون ۽ ڊيٽا ب aisay ڪتابي ڪرڻ آهي. ان کان پوءِ ڪجهه plugins جا ڪم مختلف ٿين ٿا؛ ڪجهه plugins ب aisay external services جهڙوڪ Google Drive يا Dropbox تائين backup شيڪرن ٿا، يا S3، Wasabi يا ٻين ڪنهن به compatible object storage service تائين. جتي وڌيڪ comprehensive plugins هوندا آهن، انهن ۾ differential backups يا ڪجهه طرح جي strategy هوندي آهي ته صرف ان ڊيٽا کي backup ڪجي جيڪو بدلا هوندو آهي، جنهن سان external storage جي لاڳوتيهن بچائي سگهجي ٿو.

پلاجن چونڊندا وانگر، اهو يقيني ڪريو ته هو multisite aware آهي. ان جي ڪم ڪرڻ جي طبيعت سبب، backup دوران توهان server تي temporary load کي توقع ڪري سگهو ٿا تائين جڏهن عمل پورو نه ٿئي.

#### Domain ۽ SSL {#domain-and-ssl-1}

multisite _subdomain_ mode ۾ domain names بابت وڌيڪ بحث ڪئي چڪي آهي. network administrators لاءِ هڪ تقریباً عام حل wildcard DNS entries استعمال ڪرڻ آهي.

![Wildcard DNS entry configuration example](/img/config/settings-domain-mapping.png)

هي DNS entry 'site1.domain.com' ۽ 'site2.domain.com' جهڙن subdomains کي IP address 1.2.3.4 تائين ڪامياب طور تي resolve ڪندي، جنهن سان Ultimate Multisite ۽ _subdomain_ mode استعمال ڪندڙ WordPress Multisite لاءِ سپورٽ فراهم ڪري ٿو.

هي HTTP لاءِ پوري طرح کام ڪري سگهي ٿو ڇاڪاڻ ته target host HTTP headers کان لِيو ويندو آهي، پر اڄ ڏينهن وڌيڪ web اهو نه آهي جنهن secure HTTPS transactions تقریباً لازمي آهن.

خوش قسمتی سے SSL سرٹیفکیٹ کے لیے آسان آپشنز موجود ہیں۔ _subdirectory_ موڈ میں، آپ ایک عام ڈومین سرٹیفکیٹ استعمال کر سکتے ہیں۔ یہ ہوسٹنگ فراہم کرنے والوں کی طرف سے آسانی سے اور مفت دستیاب ہوتے ہیں جو شاید فری Let'sEncrypt سروس یا کسی دوسرے ذریعہ کا استعمال کرتے ہوں، ورنہ اگر آپ سرٹیفکیٹ سائننگ ریکویسٹ جنریٹ کرنے کے قابل ہوں تو یہ اتھارٹیز سے تجارتی طور پر دستیاب ہوتے ہیں۔

_subdomain_ موڈ کے لیے، وائلڈ کارڈ SSL سرٹیفکیٹ کا استعمال ایک وائلڈ کارڈ ڈومین کے ساتھ بالکل بہترین طریقے سے جڑ جاتا ہے اور سرٹیفکیٹ کو بنیادی ڈومین اور تمام _subdomains_ کے لیے بغیر کسی اضافی ترتیب کے اتھارٹی بنانے کی اجازت دیتا ہے۔

تاہم، یہ نوٹ کرنا ضروری ہے کہ وائلڈ کارڈ SSL سرٹیفکیٹس Cloudflare جیسی سروسز کے ساتھ کام نہیں کر سکتے جب تک کہ آپ کے پاس انٹرپرائز پلان نہ ہو یا آپ انٹری کو DNS صرف پر سیٹ نہ کریں جس صورت میں تمام کیشنگ اور آپٹیمائزیشن نظر انداز کر دی جاتی ہے۔

Out-of-the-box Ultimate Multisite اس مسئلے کا حل فراہم کرتا ہے جو WordPress multisites کی ضروریات کے ساتھ ہمارے وسیع تجربے کو ظاہر کرتا ہے۔ اس سادہ add-on کو فعال کرنے سے Ultimate Multisite آپ کے Cloudflare credentials کا استعمال کرکے نیٹ ورک سائٹس کے لیے خود DNS entries شامل کرے گا اور ان کا موڈ 'proxied' پر سیٹ کر دے گا۔ اس طرح، جب بھی کوئی نیٹ ورک سب سائٹ بنائی جائے تو اسے SSL سمیت Cloudflare کے مکمل تحفظ اور فوائد حاصل ہوں گے۔

آپ کی Ultimate Multisite installation کی نوعیت اور مقصد کے لحاظ سے صارفین کو اپنی ڈومینز استعمال کرنے کی ضرورت پڑ سکتی ہے۔ اس صورت میں، نیٹ ورک ایڈمنسٹریٹر کو دو مسائل حل کرنے کا ذمہ دار ہوتا ہے۔ ایک، ڈومین نیم کی ہوسٹنگ اور دوسرا، ڈومین کے لیے SSL سرٹیفکیٹس۔

धेरै मानिसن لاءِ، Cloudflare استعمال کرنا آسان آپشن ہے۔ کسٹمروں کو بس اپنا ڈومین Cloudflare پر رکھنا ہے، ایک CNAME کو Ultimate Multisite کے روٹ ڈومین پر پوائنٹ کرنا ہے اور اپنی ڈومین کو Ultimate Multisite میں مپ کرنا ہے تاکہ اپنے کسٹم ڈومین نیم کا فائدہ اٹھانا شروع کر سکیں۔
