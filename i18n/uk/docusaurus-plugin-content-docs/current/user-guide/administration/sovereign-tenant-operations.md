# Операції суверенних орендарів (Sovereign Tenant Operations)

Ultimate Multisite версія 1.2.0 додає інструменти для роботи з суверенними орендарями: це підсайти, які працюють зі своєю власною базою даних, коренем файлової системи та контекстом маршрутизації, але залишаються видимими для мережевого адміністратора.

Використовуйте цю сторінку при керуванні ізольованими клієнтськими сайтами, передачі управління віддаленими сайтами або міграціях, які переміщують стандартний підсайт у суверенну інфраструктуру.

## Що змінюється для адміністраторів

- **Безстатевий автологін орендаря (Stateless tenant autologin)** — Мережеві адміністратори можуть відвідати суверенного орендаря без покладання на довготривалий спільний стан сесії. Токен SSO має обмежений обсяг, прив'язку до джерела, захист від повторного використання та короткий термін дії.
- **Суверенна маршрутизація (Sovereign-aware routing)** — Старі ізольовані мережі та суверенні орендарі вирішуються через той самий шлях маршрутизатора сайту, що зменшує відмінності при запуску між старими та новими ізольованими інсталяціями.
- **Перевірений стан міграції (Verified migration state)** — Перевірка міграції перевіряє наявність користувачів, дозволів запису бази даних, статусу вивантаження черги (queue drain status) та відсутність старих таблиць перед тим, як орендар буде вважатися завершеним.
- **Безпечне вимкнення (Safer teardown)** — Тепер при вимкненні суверенного орендаря облікові дані видаляються чисто, щоб видалені орендарі не залишали за собою застарілий доступ до бази даних.

## Відвідування суверенного орендаря

1. Відкрийте **Network Admin > Ultimate Multisite > Sites**.
2. Виберіть суверенного орендаря.
3. Використовуйте **Visit (SSO)**, коли це можливо, замість копіювання паролів або створення тимчасових облікових записів адміністратора.

Процес відвідування створює короткоживу токен доступу для цього орендаря та фіксує подію SSO в аудиторному журналі орендаря. Якщо кнопка не працює, перевірте, чи вирішується домен орендаря на очікувану інсталяцію і чи може орендар отримати доступ до кінцевої точки SSO на стороні мережі.

## Видалення суверенних сайтів

Видалення суверенного сайту є деструктивним процесом. Спочатку підтвердіть статус резервної копії та експорту, а потім видаліть його зі екрана керування сайтом. Процес виведення з ладу (teardown) версії 1.2.0 видаляє облікові дані бази даних сайту як частину очищення, але адміністратори все одно повинні перевірити, що на рівні хостингу видалено користувач бази даних та папки, якщо ви використовуєте зовнішні панелі керування.

:::warning
Не видаляйте суверенного сайту під час виконання перевірки міграції або поки в черзі стоять асинхронні завдання push. Зачекайте завершення перевірки, щоб процес виведення з ладу не видалив облікові дані, необхідні для очікуючих завдань.
:::
