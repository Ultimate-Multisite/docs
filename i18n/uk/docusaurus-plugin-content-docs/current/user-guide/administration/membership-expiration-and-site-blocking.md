# Істечення членства та блокування сайту {#membership-expiration-and-site-blocking}

Цей посібник пояснює, як Ultimate Multisite обробляє істечення членства, закінчення пробних періодів та блокування сайту на фронтенді. Він охоплює життєвий цикл членства — від активного до простроченого, налаштування, які контролюють, чи блокувати сайти, та що перевірити, якщо сайти залишаються доступними після закінчення членства.

## Життєвий цикл статусу членства {#membership-status-lifecycle}

Кожне членство в Ultimate Multisite має один із таких статусів:

:::note Безкоштовне членство є постійним
Безкоштовне членство не закінчується автоматично. Ultimate Multisite розглядає його як постійний доступ, тому воно не включається в перевірки істечення, якщо адміністратор не змінює його статус або не переводить клієнта на інший продукт.
:::

| Статус | Значення |
|---|---|
| **Pending** | Очікує першого платежу або підтвердження електронною поштою |
| **Trialing** | Активний пробний період, плата ще не стягнута |
| **Active** | Оплачено та чинний |
| **On Hold** | Очікує платежу за поновлення (створено рахунок-фактуру, очікує оплати) |
| **Expired** | Минув термін дії та період благодаті без поновлення |
| **Cancelled** | Явно скасовано клієнтом або адміністратором |

### Як членства переходять у статус "Expired" {#how-memberships-transition-to-expired}

Ultimate Multisite виконує фонову перевірку **щогодини**, яка шукає членства, які мають бути позначені як прострочені. Ця перевірка використовує [Action Scheduler](https://actionscheduler.org/) (а не WP-Cron напряму) і виконується як запланована дія `wu_membership_check`.

Перевірка істечення має **вбудований період благодаті тривалістю 3 дні** за замовчуванням. Членство не буде позначено як `expired` доки не минуть 3 дні після його `date_expiration`. Це дає клієнтам час здійснити запізнілий платіж, перш ніж зміниться їхній статус.

:::info
Період благодаті істечення у 3 дні відрізняється від налаштування "Frontend Block Grace Period", описаного нижче. Період благодаті істечення контролює, коли **змінюється статус** з активного/on-hold на expired. Період благодаті блокування фронтенду контролює, коли **блокується сайт** після того, як статус уже змінився.
:::

#### Членства з автоматичним поновленням vs. без автоматичного поновлення {#auto-renewing-vs-non-auto-renewing-memberships}

Це розрізнення є критично важливим для розуміння поведінки істечення:

- **Членства без автоматичного поновлення** (`auto_renew = false`): Погодинний cron-завдання обробляє весь життєвий цикл — воно створює платежі за поновлення, переводить членство у статус on-hold, а потім зрештою позначає його як expired, якщо платіж не отримано.

- **Членства з автоматичним поновленням** (`auto_renew = true`): Cron-перевірка істечення **повністю їх ігнорує**. Очікується, що платіжний шлюз (Stripe, PayPal тощо) повідомить Ultimate Multisite через вебхуки, коли підписка не вдасться або буде скасована. Якщо вебхук не отримано — через неправильно налаштовану кінцеву точку, збій шлюзу або скасування підписки поза системою — членство може залишатися `active` нескінченно, навіть після того, як минує дата істечення.

### Як закінчуються пробні періоди {#how-trials-end}

Коли пробний період членства закінчується, система:

1. Створює очікуваний платіж за поновлення з повною сумою підписки
2. Переводить статус членства з `trialing` на `on-hold`
3. Надсилає електронне повідомлення про платіж за поновлення клієнту

Цей процес виконується за тим самим погодинним графіком, що й звичайна перевірка істечення, але **тільки для членств без автоматичного поновлення**. Для пробних періодів з автоматичним поновленням перехід від пробного періоду до оплаченої підписки обробляє платіжний шлюз.

## Блокування доступу до фронтенду {#block-frontend-access}

За замовчуванням, коли членство закінчується або переходить у статус on hold, **обмежується лише панель wp-admin**. Публічний фронтенд сайту залишається доступним для відвідувачів. Щоб також заблокувати публічний доступ, ви повинні увімкнути налаштування **Block Frontend Access**.

### Налаштування параметрів {#configuring-the-setting}

Перейдіть до **Ultimate Multisite > Settings > Memberships** та увімкніть **Block Frontend Access**.

![Block Frontend Access settings showing the toggle enabled, a grace period of 7 days, and a Frontend Block Page selection](/img/config/settings-membership-block-frontend.png)

Ось повний вигляд сторінки налаштувань членства:

![Membership settings full page](/img/config/settings-membership-full.png)

Три пов'язані налаштування контролюють цю поведінку:

| Налаштування | Опис | Значення за замовчуванням |
|---|---|---|
| **Block Frontend Access** | Головльний перемикач. При увімкненні блокує публічний фронтенд мережевих сайтів, коли їхнє членство більше не активне. | Вимкнено |
| **Frontend Block Grace Period** | Кількість днів очікування після того, як членство стає неактивним, перш ніж блокувати. Встановіть `0` для негайного блокування. | 0 |
| **Frontend Block Page** | Сторінка на головному сайті, на яку перенаправляти відвідувачів, коли сайт заблоковано. Якщо не встановлено, відвідувачі бачать загальне повідомлення "Site not available". | None |

### Що бачать відвідувачі, коли сайт заблоковано {#what-visitors-see-when-a-site-is-blocked}

Коли доступ до фронтенду заблоковано, відвідувачі сайту або:

1. **Будуть перенаправлені** на сторінку, яку ви вибрали в **Frontend Block Page** (якщо налаштовано), або
2. **Побачать стандартне повідомлення**: "This site is not available at the moment." з посиланням на сторінку входу для адміністратора сайту.

Адміністратори сайтів все одно можуть увійти — сторінка входу ніколи не блокується.

### Що блокується і коли {#what-gets-blocked-and-when}

Поведінка блокування залежить від статусу членства:

| Статус членства | Блокується фронтенд? | Застосовується період благодаті? |
|---|---|---|
| Active | Ні | -- |
| Trialing | **Ні** (див. примітку нижче) | -- |
| On Hold | Вважається активним — не блокується | -- |
| Expired | **Так**, якщо Block Frontend Access увімкнено | Так |
| Cancelled | **Так**, завжди (незалежно від налаштувань) | **Ні** — блокується негайно |
| Pending | Не блокується через перевірку членства | -- |

:::warning Членства Trialing не блокуються
Навіть якщо пробний період закінчився, членство зі статусом `trialing` **не** буде заблоковано на фронтенді. Спочатку пробний період має перейти в інший статус (зазвичай `on-hold` через cron-завдання, а потім `expired`, якщо не сплачено). Якщо ви бачите членства зі статусом trialing, які не перейшли, перевірте розділ усунення несправностей нижче.
:::

:::note Членства Cancelled ігнорують період благодаті
Членства, скасовані, завжди блокуються після того, як минула дата істечення, незалежно від того, чи увімкнено Block Frontend Access. Frontend Block Grace Period **не** застосовується до скасованих членств.
:::

## Усунення несправностей: Сайти, які залишаються доступними після істечення {#troubleshooting-sites-remaining-accessible-after-expiration}

Якщо сайти залишаються публічно доступними після закінчення членства, виконайте наступні перевірки у порядку:

### 1. Перевірте, чи увімкнено налаштування Block Frontend Access {#1-verify-the-block-frontend-access-setting-is-enabled}

Перейдіть до **Ultimate Multisite > Settings > Memberships** та підтвердіть, що перемикач **Block Frontend Access** увімкнено. Це налаштування **вимкнено за замовчуванням**, що означає, що при неактивності членства обмежується лише wp-admin.

### 2. Перевірте Frontend Block Grace Period {#2-check-the-frontend-block-grace-period}

На тій самій сторінці налаштувань перевірте значення **Frontend Block Grace Period**. Якщо воно встановлено, наприклад, на 7 днів, фронтенд не буде заблоковано доки не мине 7 днів після дати істечення членства — навіть якщо статус членства вже `expired`.

Встановіть це значення на `0`, якщо ви хочете негайного блокування після того, як членство стане неактивним.

### 3. Підтвердіть, що статус членства справді змінився {#3-confirm-the-membership-status-has-actually-changed}

Перейдіть до **Ultimate Multisite > Memberships** та перевірте статус зачепленого членства. Якщо воно все ще показує `active`, незважаючи на те, що минула дата істечення, перехід статусу не відбувся. Поширені причини:

- **Членство з автоматичним поновленням**: Перевірте поле `auto_renew` на сторінці редагування членства. Якщо автоматичне поновлення увімкнено, cron-завдання істечення ігнорує це членство — воно покладається на платіжний шлюз для повідомлення про збій. Перевірте панель керування шлюзом (Stripe, PayPal), щоб підтвердити, що статус підписки відповідає тому, що показує Ultimate Multisite.

- **Cron-завдання не спрацювало**: Дивіться наступний крок.

### 4. Перевірте, чи працює Action Scheduler {#4-verify-action-scheduler-is-running}

Ultimate Multisite використовує Action Scheduler для своїх cron-завдань. Перейдіть до **Tools > Scheduled Actions** у мережевому адміні та знайдіть:

- **`wu_membership_check`** — Це має відображатися як повторювана дія, що виконується кожну годину. Якщо її немає, перевірки членства не планується.
- **`wu_async_mark_membership_as_expired`** — Це індивідуальні завдання для позначення конкретних членств як прострочених. Якщо ви бачите тут збої, вони можуть містити повідомлення про помилки, які пояснюють причину.

Якщо ви бачите завислі або збої дії, у вас може бути проблема з Action Scheduler. Поширені причини:

- **`DISABLE_WP_CRON` встановлено як `true`** у `wp-config.php` без системної заміни cron
- **Низький трафік сайту** — WP-Cron спрацьовує лише тоді, коли хтось відвідує сайт

Щоб забезпечити надійне виконання cron, налаштуйте системне cron-завдання:

```bash
# Run every 5 minutes via wget
*/5 * * * * wget -q -O /dev/null "https://your-network-url.com/wp-cron.php?doing_wp_cron"

# Or via WP-CLI
*/5 * * * * cd /path/to/wordpress && wp cron event run --due-now --url=https://your-network-url.com
```

### 5. Перевірте проблеми з вебхуками шлюзу (Членства з автоматичним поновленням) {#5-check-for-gateway-webhook-issues-auto-renewing-memberships}

Якщо членство з автоматичним поновленням, і підписка шлюзу була скасована або не вдалася, але Ultimate Multisite все ще показує його як `active`:

- **Stripe**: Перейдіть до Stripe Dashboard > Customers та перевірте статус підписки. Потім перевірте, чи активна кінцева точка вебхука в Developers > Webhooks. Кінцева точка повинна вказувати на ваш сайт і показувати успішні доставки.
- **PayPal**: Перевірте статус підписки у вашому бізнес-облікаунті PayPal та переконайтеся, що відбувається доставка IPN/вебхука.

Якщо шлюз показує, що підписка скасована, але Ultimate Multisite — ні, ймовірно, повідомлення вебхука було втрачено. Ви можете вручну змінити статус членства в **Ultimate Multisite > Memberships > [Edit Membership]**.

### 6. Перевірте період благодаті істечення (Рівень Cron) {#6-check-the-expiration-grace-period-cron-level}

Cron-перевірка має власний період благодаті (за замовчуванням: 3 дні) перед позначенням членства як простроченого. Це окремо від періоду благодаті блокування фронтенду. Загальний час до блокування сайту може становити:

**Період благодаті істечення (3 дні)** + **Період благодаті блокування фронтенду (ваше налаштування)** = Загальна затримка

Наприклад, за замовчуванням і з періодом благодаті фронтенду у 7 днів, може знадобитися до 10 днів після `date_expiration`, перш ніж сайт буде фактично заблоковано.

### 7. Вручну істечити членство {#7-manually-expire-a-membership}

Якщо вам потрібно негайно заблокувати сайт, не чекаючи циклу cron, ви можете вручну змінити статус членства:

1. Перейдіть до **Ultimate Multisite > Memberships**
2. Натисніть на зачеплене членство
3. Змініть статус на **Expired** або **Cancelled**
4. Натисніть **Save**

Блокування фронтенду набуде чинності при наступному завантаженні сторінки (за умови застосування Frontend Block Grace Period для прострочених членств, або негайно для скасованих членств).

## Підсумок {#summary}

Повний часовий інтервал від дати істечення до блокування сайту:

```text
date_expiration passes
       |
       v
  [3-day cron grace period]     <-- membership still shows as active/on-hold
       |
       v
  Cron marks membership as "expired"
       |
       v
  [Frontend Block Grace Period]  <-- configured in Settings > Memberships
       |
       v
  Site frontend is blocked
```

Для скасованих членств шлях коротший:

```text
  Membership cancelled
       |
       v
  date_expiration passes (no grace period)
       |
       v
  Site frontend is blocked immediately
```

## Довідник для розробників {#developer-reference}

Наступні хуки та фільтри дозволяють вам налаштувати поведінку істечення та блокування:

| Hook/Filter | Description |
|---|---|
| `wu_membership_grace_period_days` | Filter the number of grace period days before marking a membership as expired (default: 3) |
| `wu_schedule_membership_check_interval` | Filter the interval between membership checks (default: 1 hour) |
| `wu_membership_renewal_days_before_expiring` | Filter how many days before expiration to create a renewal payment (default: 3) |
| `wu_blocked_site_reactivation_url` | Filter to provide a custom reactivation URL when a site is blocked |
| `wu_membership_is_active` | Filter whether a membership is considered active |
| `wu_membership_expired_check_query_params` | Filter the query parameters used to find expired memberships |
| `wu_membership_trial_check_query_params` | Filter the query parameters used to find expired trials |
