# Участие Срока Действия Членства и Блокировка Сайта

Этот раздел объясняет, как работает автоматическое прекращение членства (membership expiration) и что происходит с сайтом, если у пользователя истекает срок действия его подписки.

## Прекращение Срока Действия Членства (Membership Expiration) {#membership-status-lifecycle}

Когда срок действия членства пользователя заканчивается, это означает, что он больше не имеет доступа к функциям, которые требуют активной подписки.

**Что происходит:**
1. **Автоматическое прекращение доступа:** Пользователь теряет доступ ко всем премиум-функциям сайта.
2. **Уведомление (Опционально):** Вы можете настроить уведомления для пользователя или администратора о приближении окончания срока действия подписки.

В WordPress и WooCommerce это обычно настраивается через плагины управления членством, которые используют стандартные хуки (hooks) для проверки статуса пользователя при попытке доступа к определенным страницам или функциям.

## Блокировка Сайта При Истечении Подписки {#how-memberships-transition-to-expired}

Если у пользователя истек срок действия его подписки, система должна автоматически блокировать доступ к контенту, который он не может себе позволить.

**Как это настраивается:**
Вам нужно убедиться, что ваш плагин членства правильно настроен для проверки статуса пользователя перед отображением страниц или функций. Если пользователь пытается зайти на страницу, которая требует активного членства (например, страница с премиум-контентом), система должна проверить его статус.

**Пример логики (Концептуально):**
Представьте, что у вас есть функция `is_member()`. Когда вы вызываете эту функцию для определенной страницы:

```php
if (!is_user_logged_in()) {
    // Пользователь не вошел в систему
    return false;
}

if ( ! current_user_can( 'premium_content' ) ) {
    // Пользователь вошел, но у него нет премиум-доступа. Блокируем доступ.
    wp_redirect( get_permalink( 10 ) ); // Перенаправляем на главную страницу или страницу оплаты
    exit;
}

// Если дошли сюда, значит у пользователя есть право доступа.
```

**Важные моменты:**
* **Проверка в реальном времени:** Проверки должны происходить каждый раз, когда пользователь пытается получить доступ к защищенному ресурсу.
* **Перенаправление:** Вместо простого показа ошибки 403 (Forbidden), лучше перенаправлять пользователя на страницу оплаты или страницу с информацией о том, как он может продлить подписку.

## Управление Процессом Продления Подписки {#auto-renewing-vs-non-auto-renewing-memberships}

Чтобы избежать проблем с блокировкой, важно иметь четкий процесс продления членства.

1. **Уведомление:** Отправляйте пользователю напоминание за несколько дней до окончания срока действия его подписки.
2. **Автоматическое продление (если настроено):** Если вы используете платежные шлюзы вроде Stripe, убедитесь, что настройки WooCommerce или вашего плагина позволяют автоматически продлевать подписку после успешной оплаты.

Если пользователь оплатил продление, система должна обновить его статус в базе данных, и доступ к контенту будет восстановлен до следующей даты истечения срока действия.

Бу гайд Ultimate Multisite-тың мыйзам саклау (membership) сортын аяклы булмалар, сынау хуҗалыкның аярылашы һәм фронтенд сайтыны блоклау турында түшмән. Бу мыйзам саклауны активдан аяклыка чекип ясау циклын, сайтлар блокланасы өчен контроль итүче параметрләрне һәм мыйзам саклау сортын аярыла咗нан соң сайтлар каккында булмаган арада ничек юксарга кирәлелек турында сөйләп китер.

# WordPress Documentation Translation

## Getting Started with Ultimate Multisite {#configuring-the-setting}

WordPress-те Ultimate Multisite-у белән якын таныштыру. Бу платформаны ничек ике барыдан артык сайт (multisite) үзеңдә куллану можна атыйкча түләүчеләрне күрсәтү өчен ясаган система.

### What is Ultimate Multisite? {#what-visitors-see-when-a-site-is-blocked}

Ultimate Multisite – бу WordPress-те артык сайтлар نديرгәндә хәзерге эң күп исеме булган мощный плагин (plugin). Бу плагин белән сиздер авылдагы бирлешmiş сайтларны (например, `site1.com` һәм `site2.com`) бир кагынчылап ясагыз, бик тез вакыт артык сайтка китерә аласыз.

### Installation Steps {#what-gets-blocked-and-when}

Ultimate Multisite-у кургама өчен азырлык шартлар: WordPress-те кургаган жана PHP-ни ике барыдан артык сайтга (multisite) да уздырып кургагансыз.

1. **Плагинны таңлау:** WordPress администратор панелендә "Plugins" бөлүмүнө кирип, Ultimate Multisite плагинне тандагыз һәм "Install Now" (Изилдә) батырмасын сызыңыз.
2. **Плагинны ислеу:** Плагин орнотулганыч, аны актив кылыгыз.
3. **Настройка:** Плагинны актив кылып килген сораулар аркылуу аның настройкаларына кирип, сайтларыңызды конфигурациялагыз. Бу жерде сиз сайтыгызның структурасын һәм ике барыдан артык сайтты ничек ике бирергә (site network) кирергә керермелеклерин аныклайсыз.

### Setting Up Your First Network {#troubleshooting-sites-remaining-accessible-after-expiration}

Бир сетьны кургау – булган эң маанилүү кадам. Бу процесс сиз сайтыгызның барлык структурасын аныктама турындагы планнарны ике бирергә (site network) кирергә керермелеклерин атыйкча түләүчеләрне күрсәтү өчен кирәк.

*   **Network ID:** Сиз сетьыңызга уникаль ID-гә (ID) сургулагыз. Бу ID сиз сайтыгызның барлык бирешmiş сайтларына ишенимсез китермелеклерин аныктамада булган маанилүү элемент.
*   **Site Creation:** Бирнеч сайтка артык кургап, аларны сетьнең бөлүгү катары киргизсеңиз, Ultimate Multisite сизгә оңай интерфейс аркылуу сайтларды башкарууга изилдәргә мөмкинлек берет.

### Managing Subdomains and Domains {#1-verify-the-block-frontend-access-setting-is-enabled}

Ultimate Multisite-у кулланунун эң күп маанилүү функциясы – бул домендерне (domain) һәм поддомендерне (subdomain) ясамлап башкару. Бу белән сиз бир кагынчылап сайтка китермелеклерин атыйкча түләүчеләрне күрсәтә аласыз.

*   **Subdomains:** Сиз сетьнең артык сайтлары өчен поддомендерны (мисалы, `blog.site1.com` һәм `shop.site2.com`) автоматлап ясарга аласыз. Бу сайтка китермелеклерин атыйкча түләүчеләрне күрсәтү өчен бик ыңалыша.
*   **Domain Mapping:** Домендерне сетьнең конкрет сайтларына (например, `www.site1.com` һәм `www.site2.com`) ике бирергә кирермелеклерин атыйкча түләүчеләрне күрсәтү өчен маанилүү.

### Security and Maintenance Tips {#2-check-the-frontend-block-grace-period}

Сайтыңызны коррупциядан сактап, үз эчендә саклау – булган эң кадимдәге маселе. Ultimate Multisite-у куллануда төмөр басыклыкка китермелеклерин атыйкча түләүчеләрне күрсәтү өчен:

1.  **Regular Updates:** WordPress, PHP һәм плагинны да үзеңизди дайыма шилчендесеңиз, коррупциядан сактап кала аласыз.
2.  **User Roles:** Сайтлар арасында ким нинди укучы (user role) булып исеетмелеклерин атыйкча түләүчеләрне күрсәтү өчен ролларны турыштырып таңлаугыз.

### Troubleshooting Common Issues {#3-confirm-the-membership-status-has-actually-changed}

Көп кезекте кабылаられる мәсьәгатьләр бар, әмма олар көчле решение белән çözүлмәс.

*   **Database Connection Errors:** Бу көчле исеме булган SQL серверының ачык буласын һәм WordPress-тең базасы белән турыштыру параметрлары (database connection parameters) туры вып елеклегене текшерегез.
*   **Plugin Conflicts:** Көп плагин кулланырганда конфликтылар ясалачак. Бу вакытта азырлык режимдә (deactivate mode) плагинларды бир-бир сынап, кайсы плагин проблеманы ясаганын аныктагыз.

### Участки как переходят в статус "Истекшие" (Expired) {#4-verify-action-scheduler-is-running}

Ultimate Multisite саат마다 фонында проверка жүргүзә, анда мәйлисезлекләрнің сорыйлы якынча булганларын "сорыйлар" дип белгиләргә тиеш бар дигән ясаучылыкны күче карай. Бу проверка [Action Scheduler](https://actionscheduler.org/) (төсбәрдә WP-Cronдан башка) һәм `wu_membership_check` булган әрефәле аракында жүрә.

Сырткы проверкаде де умолчанию **3 günlük kerlik döwür (grace period)** бар. Уештык (membership) `date_expiration`-дан 3 gün sonra `expired` деп белгиленмеде, уештыкның статусы өзгөрәчәк. Бу мылкыйларга кече төлөмдү аяклырга вакыт берергә мөмкинлекли.

:::info
3 günlük süreli uzatma dönemi, aşağıda açıklanan Frontend Block Grace Period ayarından ayrıdır. Süreli uzatma dönemi, durumun aktif/beklemede halden sona ermesine ne zaman karar verildiğini kontrol eder. Frontend block grace period ise durum değişmiş olduktan sonra sitenin ne zaman engelleneceğini kontrol eder.

#### Автоматически продлеваемая и неавтоматически продлеваемая подписки {#5-check-for-gateway-webhook-issues-auto-renewing-memberships}

Bu farikalar sonlanma (expiration) davranışını anlamak için çok önemlidir:

**Автоматик биргән булмаган мәнсезлекләр** (`auto_renew = false`): Саатлык cron эшчәге бүгенге циклнең толуын күрсәтә — ул биргән төлөмләрдә икенче цикльни яса, мәнсезлекнең "баракада" (on-hold) вазыйстай күченә һәм төлөм булмаган гамәлгә ашырга мисалы булганда соң карарлар.

**Автоматически продлеваемые подписки** (`auto_renew = true`): Проверка срока действия по расписанию (cron) **полностью пропускает эти подписки**. Ожидается, что платежный шлюз (Stripe, PayPal и т.д.) уведомит Ultimate Multisite через вебхуки, когда подписка не удастся оплатить или будет отменена. Если вебхук не получен — из-за неправильно настроенной конечной точки, сбоя в работе платежного шлюза или отмены подписки вне системы — членство может остаться `active` (активным) неопределенно долго, даже после того как дата истечения срока действия прошла.

### Испыталыш как завершается {#6-check-the-expiration-grace-period-cron-level}

Когда пробный период членства заканчивается, система:

1. Бәлешнең торуы өчен яңа төлөм мәсьәләсен, яшьлек мәсьәләсе буенча торуны "on-hold" (торуда) статусына өчрү белән yaratмаLAP калат.
2. Мембуанең статуссын `trialing` (яшьлек)дан `on-hold` (торуда)га өчрү.
3. Клиентгә яңа төлөм турында мәгълүматлы электрон почта хаты göndөрә.

Берәме бул процесс обычный срок кайдалап чыгу проверка белән генә бирле, әмма **фақат автомат кайдалап чыгышы булмаган мәнсезлекләр өчен**. Автомат кайдалап чыгылган сынаулар өчен, төлөм ворота (payment gateway) сынаудан öтеп ачылырга (paid subscription) көчле тәртиптә өтүне башкара.

## Блоктың фронтендга кирүү {#7-manually-expire-a-membership}

Бул бөлүмде сиз WordPress сайтыңыздагы блоктарды (блокларны) ка कशी көрүнүп, аларды алдыңкы бетте кантип көрсөтө аласыңар тууралуу маалымат берилет.

**Негизги идея:** Блоктар — бул сайттын контентин куруучу элементтер. Алар сиздин дизайнды жана мазмуныңызды аныктайт. Фронтендга кирүү дегенде, аларды колдонуучу (пользователь) көрүп, өз ишиңизди аткара ала турганыңызды баса белгилейбыз.

### Блоктардын көрсөтүлүшү {#summary}

Блоктарды кантип көрсөтө турушу керектигине көңүл буруу керек. Бул көбүнчө сиздин темаңыз (theme) же колдонгон блок редактору аркылуу аныкталат.

1. **Блокту тандау:** Сиз редактордун ичинде (например, Gutenberg блокунда) каалаган блокту тандайсыз.
2. **Настройки (Настройка):** Блоктун өзүнө тиешелүү параметрлерди (стиль, фон, текст сыяктуу) өзгөртө аласыз. Бул блоктун кандай көрүнүшү аткарыла тургандыгын аныктайт.
3. **Фронтендте көз keliши:** Сиздин сайттын дизайн жана макеттери (layout) бул блоктардын ордуна жайгаштырылат.

### Техникалык түшүндүрмө (Для разработчиков и продвинутых пользователей) {#developer-reference}

Эгер сиз код менен аракет кылсаңыз, блоктор PHP же JavaScript аркылуу фронтендге жөнөтүлөт жана CSS стилдары менен көрсөтүлөт.

*   **Блоктун структурасы:** Ар бир блок белгилүү бир HTML теги (element) түрүнө окшош болот.
*   **Hook'тар:** WordPress-те бул блоктарды өзгөртүү же кошуу үчүн "hook"тар колдонулат. Булар сиздин кодду атайын жерге кирип, блоктордун ишин контроллоого мүмкүндүк берет.

### Маанилүү кеңештер

*   **Консистентность (Канстанция):** Бардык блоктарды бирдей стилде колдонуу маанилүү. Бул сайттын профессионалдуу жана тартипталган көрүнүшү үчүн керек.
*   **Производительность:** Каармандуу блокторду колдонуңуз, анткени алар сайтыңыздын жогорку жылдамдыгына (performance) таасир этет.

Бул маалымат сизге блоктарды фронтендте кантип туура көрсөтүү жөнүндө жалпы түшүнүк берди деп үмүттөнөйм!

Башта, яңарылыклар (membership) сонунарында бирергә вакыт келгенса исә **фақ версионый (wp-admin dashboard)** сыгыртылады. Сайтның публик фронтенды (visitorlar өчен) ачык кала инде. Публик сыгыртылды да блоктау өчен **Block Frontend Access** настройкасын ясауңиз керек.

| **Frontend Блок Страница** | Сайтın bloklanması durumunda ziyaretçileri yönlendirmek için ana sitedeki bir sayfadır. Ayarlanmamışsa, ziyaretçiler genel bir "Site mevcut değil" mesajı görürler. | None |

### Site Bloklandığında Ziyaretçilerin Göreceği Şeyler

Когда доступ к фронтенду заблокирован, посетители сайта либо:

1. **Frontend Block Page**-де seçilmiş sayfaya yönləndirin (əgər konfiqurasiya edilmişdirsə), və ya
2. **Default mesajı görünür**: "Bu sayt hazırda istifadəyə açıq deyil." yazısı ilə sayt administratorunun giriş səhifəsinə link verilir.

Сайт администраторлары всё равно giriş yapabilirler -- giriş sayfası asla engellenmez.

### Ne Engellenir ve Ne Zaman?

Engelleme davranışı üyelik durumuna bağlıdır:

| Участие в членстве | Фронтенд заблокирован? | Применен льготный период? |
|---|---|---|
| Активный | Нет | -- |
| Пробный период | **Нет** (см. примечание ниже) | -- |
| В ожидании | Считается активным — не заблокирован | -- |

| Süresi dolmuş | **Иә**, Блоктың фронтенд сиясәге күченсе | Иә |
| Сүреп кагылган | **Иә**, дарылган (саптыардан тәэсирсиз) | **Йо** -- дереу блоклаецца |
| Берилмәгән | Уялылык текчасы аркылуу блокламай җибәрмесез | -- |

:::warning Trialing üyelikləri bloklanmıyor
Trial dövri bitmiş olsa bile, `trialing` statusuna sahip bir üyelik ön yüzde **bloklanmaz**. Deneme süresinin önce başka bir duruma (genellikle cron işi aracılığıyla `on-hold` durumuna, sonra ödenmemişse `expired` durumuna) geçmesi gerekir. Eğer geçiş yapmamış deneme üyeleri görüyorsanız, lütfen aşağıdaki sorun giderme bölümünü kontrol edin.

## Усуту (Troubleshooting): Сайтлар Сыйлап Барылгасны После Яхшылык Калганнары

Уатырны (membership) соң билек сайтлар халыкка ачык калган болса, төмөнкү тексерулерди бул чаралар менен баштайын:

### 1. Блоктын "Frontend Access" (Үзүлүк Кабул Илэтин) Тезерле Ачык Болганын Текшерүү

(This section is a direct translation of the heading, keeping technical terms where appropriate.)

**Ultimate Multisite > Settings > Memberships**-ге гүңүлдиңиз жана **Block Frontend Access** туташтыруучу (toggle) өчүрүлгөн экенин текшерип алыңыз. Бул параметр **көрсөтүлбөстөн өчүрүлгөн** болуп саналат, бул мүшенныен иштетүүдө активдүүлүгү токтогондо гана wp-admin'ге чектөө коюлат дегенди билдирет.

### 2. Фронтенд Блогының Сүтәлеме Дәгарына Кәкич

Бул бөлүмде сиз фронтенд блогыңардын (frontend blocks) узактысын текшерегез. Бул узактылык, яғни "grace period", блоктарды өз орнына туру үчүн берилген уақыт аралыгын аңлатат.

Эмне үчүн бул маанилӣ?
Көбидди бultzу (например, новый плагин или тема) орнотуудан кийин, система жаңы блокта кандайдыр бир каталар же күтүлбэгче эмес өзгөрүүлөр болушу мүмкүн. Бул узактылык учурунда сиз бул блоктардын туура иштеп жатканын текшере аласыз жана аларды оңдоо үчүн убакыт аласыз.

**Текшерүү жолу:**
1. **Блоктарды орнотуу:** Жаңы блоктору WordPress-те активдештириңиз.
2. **Узактылыкты күтүңүз:** Система белгилүү бир убакытка чейин (бұл узактылык аралыгын даректерден таба аласыз) блоктарды тексергенге убакыт берет.
3. **Каталарды аныктоо:** Эгерде каталар болса, олар бул период ичинде ачык болот.

Эгер сиз белгилүү бир блок үчүн узактылыкты өзүңүз өзгөртүп койсоңуз (конкреттәри документацияда берилген), система дагы тез иштеп калат.

Аynı настройка sayfasına gidip **Frontend Block Grace Period** değerini kontrol edin. Eğer bu değer örneğin 7 gün olarak ayarlanmışsa, frontend üyeliğin süresinin dolmasından sonra bile (üyelik durumu zaten `expired` olsa bile) 7 gün sonra bloke edilmeyecektir.

Мемлекетлик статусы инактивациядан кейин дереу блоклама салынса, булымытчыны `0`-га белгилең.

### 3. Мемлекетлик статусы шөкәмен өзгәрүени тасбатлаңыз

**Ultimate Multisite > Memberships**-ге gidin we etikan edilen üyelikning holatini tekshiring. Agar son kullanma sanasidan keyin ham hali `active` (faol) ko'rsatsa, holat o'zgarishi sodir bo'lmagan demakdir. Odatda sabablari:

**Участие автоматный продлевается**: Учтите поле `auto_renew` на странице редактирования членства. Если автопродление включено, планировщик сброса времени истечения срока действия пропускает это членство — оно полагается на платежный шлюз для сообщения о неудаче. Проверьте панель управления вашим шлюзом (Stripe, PayPal), чтобы убедиться, что статус подписки совпадает с тем, что показывает Ultimate Multisite.

**Cron işi (cron job) çalışmadı**: Bir sonraki adıma bakın.

### 4. Action Scheduler'ın Çalışıp Çalışmadığını Kontrol Edin

Ultimate Multisite, cron işleri için Action Scheduler kullanır. Ağ yöneticisinde **Tools > Scheduled Actions** (Araçlar > Planlanmış Eylemler) bölümüne gidin ve şunları arayın:

**`wu_membership_check`** -- Бул авылчылыкны тек саат сайын дип даруммалы әйтоучу булышы. Бу бул функция яки операция ялмыйча булса, авылчылыкны графикләшмичә кала.

**`wu_async_mark_membership_as_expired`** -- Булар конкрет мәртебәләрне сортып укып (expired) белгиләү өчен ясалган аңчык тапшырулардир. Аларның ичинде күчәй булган операцияларны карагыз, олар сый-сыйлап ниге күчәй калганларын түштергә мөмкин булыткан ошибка мәгълүматлары бар ди.

Если вы видите зависшие или неудачные действия, у вас может быть проблема с Action Scheduler. Распространенные причины:

**`DISABLE_WP_CRON` нь `wp-config.php`-де `true` деп қойылса да, жүйе деcron орнына жоқ болса**
**Төмен сайт трафигі** -- WP-Cron тек біреу сайтқа кездескенде ғана іске қосылады

Сыпас cron-ның надежно жұмыс істеуін қамтамасыз ету үшін жүйе cron жұмысын орнатыңыз:

```bash
# Hər 5 dəfə wget vasitəsinə görə işlətmək üçün
*/5 * * * * wget -q -O /dev/null "https://your-network-url.com/wp-cron.php?doing_wp_cron"

# Və ya WP-CLI vasitəsinə görə
*/5 * * * * cd /path/to/wordpress && wp cron event run --due-now --url=https://your-network-url.com
```

### 5. Gateway Webhook Sorunlarını Kontrol Edin (Автоматически yenilenen üyelikler için)

Əgər üyelik avtomatik yenilənirsə və gateway abunası ləğv edilmişdirsə və ya baş tutmamışdır, amma Ultimate Multisite onu hələ də `active` (aktiv) kimi göstərirsə:

**Stripe**: Stripe paneli (Dashboard) > Müşteriler (Customers) bölümüne gidin ve abonelik durumunu kontrol edin. Ardından, Geliştiriciler (Developers) > Webhook'lar (Webhooks) altında webhook uç noktasının aktif olup olmadığını doğrulayın. Uç nokta sitenize işaret etmeli ve başarılı teslimatları göstermelidir.

**PayPal**: Подписка статусын ваша PayPal бизнес хатында яшаунып чыгыгыз һәм IPN/webhook жеттерин текчагыз.

Эгерде шлюз (gateway) абонентский статус "отменен" покажет, но Ultimate Multisite этого не показывает, то уведомление через webhook, скорее всего, потерялось. Вы можете вручную изменить статус членства в **Ultimate Multisite > Memberships > [Edit Membership]**.

### 6. Срок действия льготного периода (уровня Cron)

Здесь мы посмотрим, как проверить срок действия льготного периода для вашего сайта на уровне Cron. Это важно, чтобы убедиться, что все настройки работают правильно и вы не пропустили важные события.

Для этого вам нужно добавить специальный код в ваш файл `functions.php` вашей темы или в собственный плагин. Этот код будет запускаться по расписанию (Cron).

Вот пример того, как это можно сделать:

```php
/**
 * Проверка льготного периода на уровне Cron
 */
function check_grace_period() {
    // Здесь вы можете добавить свою логику проверки.
    // Например, проверить дату окончания какого-либо действия.

    $current_time = time();
    $expiration_date = strtotime('tomorrow 10:00:00'); // Пример даты завтра в 10:00

    if ($current_time >= $expiration_date) {
        // Если текущее время достигло или превысило дату окончания,
        // вы можете выполнить нужные действия.
        error_log('Льготный период истек!');
        // Здесь можно добавить код для уведомления администратора или сброса настроек.
    } else {
        // Льготный период еще действует.
        error_log('Льготный период активен.');
    }
}

// Запуск функции по расписанию (Cron)
if ( ! wp_next_scheduled( 'my_grace_period_check_hook' ) ) {
    wp_schedule_event( time(), 'hourly', 'my_grace_period_check_hook' );
}
```

**Что здесь происходит?**

1.  **`function check_grace_period()`**: Мы создаем функцию, которая содержит всю логику проверки. Вы можете вставить сюда любую проверку, которую считаете нужной (например, сравнение текущей даты с датой окончания скидки или льготного периода).
2.  **`$current_time = time();`**: Получаем текущее время сервера.
3.  **`$expiration_date = strtotime('tomorrow 10:00:00');`**: Это пример того, как можно рассчитать дату окончания какого-либо события. Вы можете заменить эту строку на свою сложную логику.
4.  **Условие `if ($current_time >= $expiration_date)`**: Проверяем, прошло ли время окончания льготного периода. Если да, мы выполняем код внутри этого блока (например, логируем ошибку или сбрасываем настройки).
5.  **`wp_schedule_event(...)`**: Это самая важная часть для работы на уровне Cron. Она говорит WordPress: "Запускай функцию `my_grace_period_check_hook` каждый час". Вы можете изменить `'hourly'` на другое расписание, например, `'daily'`, если вам нужно проверять что-то раз в день.

Помните, что настройка Cron требует осторожности. Если вы ошибетесь с датами или логикой, это может привести к нежелательным последствиям. Всегда делайте резервные копии перед внесением изменений!

Крон проверканің өз уаты бар (дефолт: 3 күн), бұл мүшенни салынған кезде сорғылы деп белгілеуден алдын ала. Бұл фронтенд блогының уатынан ayrıdır. Сайт блокираланымчалық мән:

**Сырыныр кемиш мертибеси (3 gün) + Фронтенд блогының сырыныр кемиш мертибесең (сиз табиий настройкагыз) = Исемлеген gecikme**

### 7. Участнилыкны вруvar (руман) etmek

Məsələn, standart parametrlarda we 7 günlük ön səhifə müddeti (grace period) ilə, sayt gerçəyin bloklanmasından əvvəl `date_expiration`dan sonra ən çox 10 gün çəkə bilər.

Сайтыны крон циклын үтү без күтмәй яиндиргән блоклау өчен, мәчетнең статусын вруаәл өзгөртә аласыз:

1. **Ultimate Multisite > Memberships**-e gidin
2. АFFECTED (ситуациядан эскирде) üyeleşmәnlikkеге басыңыз
3. Статусын **Expired** (Эскирде) ýa **Cancelled** (Aýyaryldy) olarak değiştirin
4. **Save** (Saqlama) butonuna basın

Frontend bloku следующий страница yükləndikən təsir edir (sona çatmış üzvlər üçün Frontend Bloku ləğv müddəti və ya ləğv edilmiş üzvlər üçün dərhal).

       v
  [Frontend Block Grace Period (Ön Yüz Bloklama Süresi)] <-- Ayarlar > Memberships'te yapılandırılır
       |
       v
  Sitenin ön yüzü engellenir

Беркитләрне үзесезлекле мәгълүматлар өчен, юл кыскара:

```text
  Membership cancelled
       |
       v
  date_expiration passes (no grace period)
       |
       v
  Site frontend is blocked immediately
```

## Дәмивачләр өчен массив

Беркитләр үзесезлекле вакытта, юл кыскара булып калат:

```text
  Membership cancelled
       |
       v
  date_expiration passes (no grace period)
       |
       v
  Site frontend is blocked immediately
```

Арасында, эти хуки (hooks) һәм фильтрләр (filters) белән сортын һәм блоклау тәртибен özгә-özә ясауга исәле:

| Хук/Филтер | Видимиләр |
|---|---|
| `wu_membership_grace_period_days` | Уештыруны сортын күченең күнгәче днине фильтрлау (дефолт: 3) |

| `wu_schedule_membership_check_interval` | Участиениң текселеме аралыгын фильтрлейт (дефолт: 1 саат) |
| `wu_membership_renewal_days_before_expiring` | Түбәлемине қабырғадан қан күн бұрын жаңарту төлемді жасауды фильтрлейт (дефолт: 3) |

| `wu_blocked_site_reactivation_url` | Сайт блокирласа, кастомный перезапуск (активация) URL-ы беру үчүн фильтр |
| `wu_membership_is_active` | Уени актив (активдөр) деп эсептәми дип фильтрләмә |

| `wu_membership_expired_check_query_params` | Сүреп калган (expired) мыйзамдар үчүн колдонулган сурогу параметрлерин фильтрлеу |
| `wu_membership_trial_check_query_params` | Сүреп калган (expired) сынамы үчүн колдонулган сурогу параметрлерин фильтрлеу |
