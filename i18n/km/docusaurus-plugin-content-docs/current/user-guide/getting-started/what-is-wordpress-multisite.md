# WordPress Multisite ជាអ្វី?

នៅក្នុងផ្នែកស្នូល (core) របស់វា, WordPress មានមុខងារមួយហៅថា ‘Multisite’ ដែលមានប្រភពដើមត្រលប់ទៅឆ្នាំ ២០១០ នៅពេលដែល WordPress 3.0 បើកដំណើរការ។ ចាប់តាំងពីពេលនោះមក វាបានទទួលការកែសម្រួលជាច្រើន aiming ដើម្បីណែនាំមុខងារថ្មីៗ និងពង្រឹងសុវត្ថិភាព។

និយាយឱ្យងាយយល់, WordPress multisite អាចត្រូវបានគិតថាដូចនេះ៖ សាកលវិទ្យាល័យមួយរក្សាទុកការដំឡើង WordPress តែមួយ ប៉ុន្តែបណ្តា​មហាវិទ្យាល័យនីមួយៗរក្សាទុកគេហទំព័រ WordPress របស់ខ្លួនផ្ទាល់។

##

## តើ WordPress Multisite ជាអ្វីពិតប្រាកដ? {#what-exactly-is-wordpress-multisite}

Multisite គឺជាមុខងារមួយនៅក្នុង WordPress ដែលអនុញ្ញាតឱ្យគេហទំព័រច្រើនអាចប្រើប្រាស់ការដំឡើង WordPress តែមួយបាន។ នៅពេលដែល multisite ត្រូវបានបើកដំណើរការ, គេហទំព័រ WordPress ដើមនឹងត្រូវបានបំប្លែងដើម្បីគាំទ្រអ្វីដែលគេហៅថា **បណ្តាញនៃគេហទំព័រ (network of sites)**។

បណ្តាញនេះចែករំលែកប្រព័ន្ធឯកសារ (meaning **plugins និង themes ក៏ត្រូវបានចែករំលែកដែរ**), ម៉ាស៊ីនមូលដ្ឋានទិន្នន័យ (database), កំណត់ត្រា WordPress core files, wp-config.php ជាដើម។

នេះមានន័យថា ការធ្វើបច្ចុប្បន្នភាព WordPress, theme និង plugin គឺត្រូវធ្វើតែម្តងសម្រាប់គេហទំព័រទាំងអស់នៅក្នុងបណ្តាញរបស់អ្នក ព្រោះពួកវាចែករំលែកឯកសារដូចគ្នាលើប្រព័ន្ធឯកសារ (filesystem)។

ការពិតនេះគឺជាអត្ថប្រយោជន៍សំខាន់មួយនៃ multisite, ព្រោះវាអនុញ្ញាតឱ្យអ្នកពង្រីកចំនួនគេហទំព័រដែលអ្នកគ្រប់គ្រង ខណៈពេលដែលរក្សាបាននូវចំនួនភារកិច្ចដែលអ្នកត្រូវធ្វើដើម្បីថែរក្សាក្រុមហ៊ុនរបស់អ្នក។

##

## Subdomain ឬ Subdirectory? {#subdomain-or-subdirectory}

មានរបៀបដំណើរការ WordPress multisite ចំនួនពីរ – ហើយអ្នកត្រូវជ្រើសរើសមួយនៅពេលបំប្លែងការដំឡើង WordPress ធម្មតាទៅជាការដំឡើង multisite៖

**Subdomain:** ឧទាហរណ៍៖ [site.domain.com](http://site.domain.com)

…ឬ

**Subdirectory:** ឧទាហរណ៍៖ [yourdomain.com/site](http://yourdomain.com/site)

របៀបនីមួយៗមានអត្ថប្រយោជន៍ និងគុណវិបត្តិដែលអ្នកត្រូវពិចារណា នៅពេលធ្វើការសម្រេចចិត្តនេះ។

ចំណុចមួយដែលសំខាន់ត្រូវកត់សម្គាល់គឺ៖ នៅពេលអ្នកបានសម្រេចចិត្តហើយ ការផ្លាស់ប្តូរពី subdirectory ទៅ subdomain ឬផ្ទុយមកវិញគឺពិបាកណាស់ ជាពិសេសប្រសិនបើអ្នកធ្លាប់បង្កើតគេហទំព័រមួយចំនួនរួចហើយ។

មុននឹងធ្វើការសម្រេចចិត្តនោះ មានចំណុចមួយចំនួនដែលត្រូវគិតទុកជាមុន៖

**Subdirectory Mode (របៀបប្រើ subdirectory)** គឺជារបៀបដែលងាយស្រួលបំផុតក្នុងការរៀបចំ និងថែទាំ។ វាមានលក្ខណៈបែបនេះព្រោះគេហទំព័រទាំងអស់គ្រាន់តែជាផ្លូវ (paths) ដែលភ្ជាប់ទៅនឹងដែនចម្បងប៉ុណ្ណោះ (ឧទាហរណ៍៖ [yourdomain.com/subsite](http://yourdomain.com/subsite))។ ជាលទ្ធផល អ្នកត្រូវការ **SSL certificate មួយ** ប៉ុណ្ណោះសម្រាប់ដែនចម្បង ហើយវាអាចគ្របដណ្តប់លើបណ្តាញទាំងមូលបាន។

ក្នុងពេលជាមួយគ្នានេះ ដោយសាររចនាសម្ព័ន្ធ URL របស់វា Google និងម៉ាស៊ីនស្វែងរកផ្សេងទៀតភាគច្រើននឹងចាត់ទុកគេហទំព័រទាំងអស់នៅក្នុងបណ្តាញប្រភេទ subdirectory របស់អ្នកថាជាគេហទំព័រដ៏ធំមួយ។ ជាលទ្ធផល เนื้อหาដែលអតិថិជនរបស់អ្នកបន្ថែមទៅក្នុង subsites អាចប៉ះពាល់ដល់ប្រសិទ្ធភាព SEO នៃ landing site របស់អ្នកបាន ឧទាហរណ៍។ កម្រិតនៃការប៉ះពាល់នេះគឺអាចពិភាក្សាគ្នាបាន ហើយមានការលើកឡើងថា ការរៀបចំបែបនេះអាចផ្តល់អត្ថប្រយោជន៍ដល់ប្រសិទ្ធភាព SEO ផងដែរ។

**Subdomain Mode (របៀបប្រើ subdomain)** មានភាពស្មុគស្មាញបន្តិចក្នុងការរៀបចំ ប៉ុន្តែរចនាសម្ព័ន្ធ URL របស់វា (ឧទាហរណ៍៖ [subsite.yournetwork.com](http://subsite.yournetwork.com)) ជាទូទៅត្រូវបានគេយល់ថាមើលទៅ "កាន់តែប្រកបវិជ្ជាជីវៈ"។

បញ្ហាធំមួយក្នុងការរៀបចំរបៀប subdomain mode គឺការគ្របដណ្តប់ SSL (HTTPS) សម្រាប់បណ្តាញទាំងមូល។ វាជាការពិតដែលថា browser មើលឃើញ subdomain ជាអង្គភាពដែលនៅដាច់ពីគ្នា។ ជាលទ្ធផល អ្នកនឹងត្រូវការវិញ្ញាបនបត្រ SSL ផ្សេងទៀតសម្រាប់ subdomain នីមួយៗនៅលើបណ្តាញរបស់អ្នក ឬវិញ្ញាបនបត្រពិសេសមួយដែលហៅថា **Wildcard SSL certificate**។ ក្នុងប៉ុន្មានឆ្នាំចុងក្រោយនេះ អ្នកផ្តល់សេវាផ្ទុក (hosting providers) និង panel កំពុងបង្កើនសមត្ថភាពរបស់ពួកគេក្នុងការផ្តល់ SSL ហើយខ្លះផ្តល់វិញ្ញាបនបត្រ wildcard ដោយគ្រាន់តែចុចប៊ូតុងមួយ ដែលជួយបិទគម្លាតរវាងពីររបៀបនេះក្នុងលក្ខខណ្ឌនៃភាពស្មុគស្មាញនៃការកំណត់រចនាសម្ព័ន្ធ។

ផ្ទុយពី subdirectory mode, subsites នៅលើបណ្តាញផ្អែកលើ subdomain ត្រូវបានម៉ាស៊ីនស្វែងរក (search engines) មើលឃើញថាជាគេហទំព័រដាច់ដោយឡែកពីគ្នា ដែលមានន័យថាខ្លឹមសារដែលមាននៅលើ subsite មួយមិនប៉ះពាល់ដល់ដំណើរការ SEO របស់ subsites ផ្សេងទៀតទាល់តែសោះ។

## The Super Admin {#the-super-admin}

ការដំឡើង WordPress នៅលើគេហទំព័រតែមួយ (Single-site) អនុញ្ញាតឱ្យអ្នកបន្ថែមអ្នកប្រើប្រាស់បានគ្មានដែនកំណត់ និងផ្តល់តួនាទីអ្នកប្រើប្រាស់ខុសៗគ្នាជាមួយនឹងสิทธิ์ផ្សេងៗគ្នា។

នៅក្នុង WordPress Multisite ប្រភេទអ្នកប្រើប្រាស់ថ្មីមួយត្រូវបានដោះសោ៖ **the super admin** – ហើយ panel 管理ថ្មីមួយត្រូវបានដោះសោ៖ **the network admin panel**។

ដូចឈ្មោះដែលបានបង្ហាញដែរ super admin មានអំណាចពិសេសលើបណ្តាញ ដោយអាចគ្រប់គ្រង subsites, plugins, themes និងអ្វីៗទាំងអស់របស់វា!

នៅពេលអ្នកប្តូរការដំឡើង WordPress នៅលើគេហទំព័រតែមួយរបស់អ្នកទៅជា multisite អ្នកគ្រប់គ្រងដើមនៃគេហទំព័រតែមួយនោះនឹងត្រូវបានតម្លើងជា super admin ដោយស្វ័យប្រវត្តិ។

Plugins និង themes អាចត្រូវបានដំឡើង ឬលុបចេញពី network admin panel ដោយ super admins ប៉ុណ្ណោះ។ បន្ទាប់មក subsite admins អាចជ្រើសរើសបើកដំណើរការ (activate) ឬបិទដំណើរការ (deactivate) plugins ឬ themes ទាំងនោះ លុះត្រាតែ super admin របស់ network បើកដំណើរការ plugin ដែលបង្ខំឱ្យវាដំណើរការសម្រាប់ subsites ទាំងអស់ជានិច្ច។

_ចំណាំ៖ ដូចដែលអ្នកបានឃើញហើយ ការអញ្ជើញនរណាម្នាក់ចូលទៅក្នុងបណ្តាញរបស់អ្នក និងផ្តល់ឋានៈជា super admin នឹងផ្តល់ឱ្យអ្នកប្រើប្រាស់នោះនូវការគ្រប់គ្រងពេញលេញលើបណ្តាញរបស់អ្នក។ ជាឧទាហរណ៍ អ្នកគ្រប់គ្រង super admin ផ្សេងទៀតក៏អាចលុបចោលឋានៈ super admin របស់អ្នកបាន ដែលមានន័យថាអ្នកនឹងត្រូវបានបិទផ្លូវមិនឱ្យចូលទៅកាន់ផ្ទាំងគ្រប់គ្រង (admin panel) បណ្តាញរបស់អ្នកផ្ទាល់។ ដើម្បីអនុញ្ញាតឱ្យអតិថិជន Ultimate Multisite មានការគ្រប់គ្រងលម្អិតលើអ្វីដែល super admin បន្ថែមអាចធ្វើបាន យើងមាន add-on មួយឈ្មោះថា Support Agents។ add-on នេះអនុញ្ញាតឱ្យអ្នកបង្កើតប្រភេទអ្នកប្រើប្រាស់មួយទៀត – អ្នកជំនាញ (agent) ដែលមានតែសិទ្ធិដែលពួកគេត្រូវការដើម្បីបំពេញភារកិច្ចរបស់ពួកគេនៅលើបណ្តាញប៉ុណ្ណោះ។_

## អ្វីខ្លះដែលចែករំលែកគ្នារវាង subsites និងអ្វីដែលមិនចែករំលែកគ្នា {#what-is-shared-among-subsites-and-what-is-not}

ដូចដែលយើងបានលើកឡើងពីមុន អត្ថប្រយោជន៍សំខាន់មួយនៃ WordPress multisite គឺថា subsites ទាំងអស់កំពុងប្រើការកំណត់រចនាសម្ព័ន្ធ (configurations) បណ្ដាឯកសារស្នូល (core files) ប្រធានបទ (themes) ភ្នាក់ងារបន្ថែម (plugins) និងឯកសារស្នូលរបស់ WordPress គឺដូចគ្នា។

ទោះជាយ៉ាងណាក៏ដោយ មានធាតុមួយចំនួនដែលត្រូវបានកំណត់វិសាលភាព (scoped) ជាលក្ខណៈបុគ្គលសម្រាប់ subsite នីមួយៗ។

- ឧទាហរណ៍៖ subsite នីមួយៗទទួលបានโฟลเดอร์បង្ហោះ (uploads folder) ផ្ទាល់ខ្លួន។ ជាលទ្ធផល ការបង្ហោះដែលធ្វើឡើងដោយអ្នកប្រើប្រាស់នៃ subsite ជាក់លាក់មួយ មិនអាចចូលមើលបាននៅលើ subsite ផ្សេងទៀតឡើយ។

- Subsite នីមួយៗមានផ្ទាំងគ្រប់គ្រង (admin panel) ផ្ទាល់ខ្លួន ហើយអាចបើកដំណើរការ ឬបិទភ្នាក់ងារបន្ថែម (plugins) ឬប្រធានបទ (themes) បាន លុះត្រាតែវាត្រូវបានធ្វើឱ្យសកម្មនៅក្នុងបណ្តាញដោយ super admin។

- តារាងមូលដ្ឋានទិន្នន័យ (database tables) ភាគច្រើនត្រូវបានបង្កើតឡើងសម្រាប់ subsite នីមួយៗ ដែលមានន័យថា posts, comments, pages, settings និងអ្វីផ្សេងទៀតត្រូវបានកំណត់វិសាលភាពសម្រាប់ subsite នីមួយៗ។

## ការគ្រប់គ្រងអ្នកប្រើប្រាស់នៅលើ WordPress Multisite {#user-management-on-wordpress-multisite}

ប្រធានបទដ៏រសើបមួយនៅក្នុង WordPress multisite គឺការគ្រប់គ្រងអ្នកប្រើប្រាស់ (user management)។ តារាងអ្នកប្រើប្រាស់របស់ WordPress គឺជាតារាងមួយក្នុងចំណោមតារាងតិចតួចដែលត្រូវបានចែករំលែករវាង subsites ទាំងអស់។

ការរៀបចំនេះអាចបង្កបញ្ហាខ្លះអាស្រ័យលើអ្វីដែលអ្នកគ្រោងនឹងបង្កើតជាមួយបណ្តាញរបស់អ្នក។ ឧទាហរណ៍ខាងក្រោមនេះជួយបង្ហាញពីបញ្ហាដ៏បន្ទាន់បំផុតមួយ។

សូមស្រមៃមើលស្ថានភាពដូចខាងក្រោម៖

អ្នកបានបង្កើតបណ្តាញ WordPress multisite ហើយចាប់ផ្តើមផ្តល់សេវា subsites ក្នុងតម្លៃប្រចាំខែដល់មនុស្សដែលចង់មានហាងលក់ទំនិញអនឡាញ។

អ្នកទទួលបានអតិថិជនចំណូលដំបូងគឺ John។ អ្នកបានបង្កើតគេហទំព័រមួយសម្រាប់ John នៅលើបណ្តាញរបស់អ្នក ដំឡើង plugin ដែលចាំបាច់ទាំងអស់ រួចបង្កើត user មួយសម្រាប់ John ដើម្បីឱ្យគាត់អាចគ្រប់គ្រងហាងរបស់គាត់បាន។

បន្ទាប់មក អតិថិជនទីពីរគឺ Alice បានចូលមកដល់។ អ្នកធ្វើដូចគ្នាសម្រាប់នាង ហើយឥឡូវនេះនាងក៏មានហាងនៅលើបណ្តាញរបស់អ្នកដែរ។

John និង Alice គឺជាអតិថិជនរបស់អ្នកទាំងពីរ ប៉ុន្តែពួកគេមិនស្គាល់គ្នាទេ។ សំខាន់ជាងនេះទៅទៀត ប្រសិនបើម្នាក់ក្នុងចំណោមពួកគេចូលមើលគេហទំព័ររបស់ម្នាក់ទៀតទេ គ្មានវិធីដើម្បីដឹងថាហាងនេះត្រូវបានបង្ហោះនៅលើបណ្តាញគេហទំព័រដូចគ្នាឡើយ។

ថ្ងៃមួយ John ចង់ទិញស្បែកជើងថ្មី ហើយគាត់បានរកឃើញស្បែកជើងដ៏ល្អឥតខ្ចោះនៅក្នុងហាងរបស់ Alice។ នៅពេលដែលគាត់ព្យាយាមបញ្ចប់ការទិញ គាត់ទទួលបានសារកំហុសថា "email already in use" (អ៊ីមែលត្រូវបានប្រើរួចហើយ) ដែលវាគួរឱ្យភ្ញាក់ផ្អើលណាស់ ព្រោះ John ប្រាកដ ១០០% ថាវានេះជាលើកទីមួយដែលគាត់ធ្លាប់ចូលមើលគេហទំព័ររបស់ Alice។

អ្វីដែលបានកើតឡើងនៅទីនេះគឺ user របស់ John ត្រូវបានចែករំលែកឆ្លងកាត់បណ្តាញទាំងមូល។ ដូច្នេះ នៅពេលគាត់ព្យាយាមបង្កើតគណនីដើម្បីបង់ប្រាក់នៅលើគេហទំព័ររបស់ Alice WordPress នឹងរកឃើញថាមាន user ដែលប្រើ email Address ដូចគ្នាស្រាប់ ហើយបង្ហាញសារកំហុស។

_ចំណាំ៖ យើងដឹងថាវានិរន្តរភាពខ្លាំងប៉ុណ្ណាអាស្រ័យលើករណីប្រើប្រាស់របស់អ្នក ដូច្នេះ Ultimate Multisite មានជម្រើសមួយដែលបypass ការត្រួតពិនិត្យធម្មតាក្នុងការរកមើល user ដែលមានរួចហើយ ដោយអនុញ្ញាតឱ្យបង្កើតគណនីច្រើនដោយប្រើ email Address ដូចគ្នា។ គណនីនីមួយៗត្រូវបានភ្ជាប់ទៅនឹង subsites របស់វា ដូច្នេះហានិភ័យនៃការប៉ះទង្គិចគ្នានៃឈ្មោះ (collision) ត្រូវបានរក្សាឱ្យនៅកម្រិតអប្បបរមា។ ក្នុងឧទាហរណ៍ខាងលើ John នឹងមិនទទួលបានសារកំហុសទេ ហើយគាត់អាចទិញស្បែកជើងទាំងនោះបានដោយគ្មានបញ្ហា។ ជម្រើសនេះហៅថា Enable Multiple Accounts (បើកគណនីច្រើន) ហើយអាចបើកដំណើរការបាននៅ Ultimate Multisite → Settings → Login & Registration។_

ទោះបីជាតារាងអ្នកប្រើប្រាស់ត្រូវបានចែករំលែកក៏ដោយ អ្នកគ្រប់គ្រង subsites ឬ super admin អាចបន្ថែម និងដកអ្នកប្រើប្រាស់ចេញពី subsites បាន ហើយពួកគេក៏អាចមានតួនាទីអ្នកប្រើប្រាស់ខុសៗគ្នាលើ subsites ផ្សេងៗគ្នាបានដែរ។

## ការពិចារណាផ្នែកដំណើរការ (Performance considerations) {#performance-considerations}

WordPress multisite មានអំណាចខ្លាំងណាស់ទាក់ទងនឹងចំនួនគេហទំព័រដែលវាអាចទ្រទ្រង់បាន។ អ្នកអាចសាកល្បងចំណុចនេះដោយមើលថា [WordPress.com](https://WordPress.com), Edublogs, និង Campuspress គឺជាសេវាកម្មផ្អែកលើ multisite ហើយអ្នកបង្ហោះនីមួយៗមានគេហទំព័ររាប់ពាន់។

ក្នុងទ្រឹស្តី គ្មានចំនួនកំណត់នៃគេហទំព័រដែលអ្នកអាចបង្ហោះនៅលើការដំឡើង WordPress multisite តែមួយនោះទេ ប៉ុន្តែនៅក្នុងការអនុវត្តជាក់ស្តែង ចំនួនគេហទំព័រដែលអ្នកអាចដំណើរការបានដោយពេញចិត្តអាចប្រែប្រួលយ៉ាងខ្លាំងទៅតាមកត្តាផ្សេងៗគ្នាជាច្រើន៖ តើគេហទំព័រមានភាពផ្លាស់ប្តូរ (dynamic) ខ្លាំងប៉ុណ្ណា, តើ plugin ណាខ្លះដែលមានសម្រាប់ subsites ជាដើម។

ជាគោលការណ៍ អ្នកកាន់តែមានបណ្តាញដែលសាមញ្ញទៅ កាន់តែល្អ។ ការផ្តល់អាទិភាពដល់គេហទំព័រដែលខ្លឹមសារមិនផ្លាស់ប្តូរច្រើន (ដែលធ្វើឱ្យពួកវាជាបេក្ខជនដ៏ល្អសម្រាប់យុទ្ធសាស្ត្រ caching ដ៏ខ្លាំង) និងការរក្សា stack plugin ឱ្យស្រាលបំផុត (ចំនួន active plugins តិចតួចប៉ុណ្ណា កាន់តែល្អ) អាចបង្កើនចំនួន subsites ដែលអ្នកអាចបង្ហោះបានយ៉ាងច្រើន។

ចំណុចល្អបំផុតគឺ ដោយសារវាទាំងអស់សុទ្ធតែជា WordPress អ្នកប្រើប្រាស់ឧបករណ៍ដូចគ្នាដែលអ្នកធ្លាប់ស្គាល់ និងចូលចិត្តសម្រាប់ការកែលម្អដំណើរការ ក៏នឹងដំណើរការសម្រាប់បណ្តាញ multisite ផងដែរ។

ឧបសគ្គចម្បងសម្រាប់ multisite គឺ database ប៉ុន្តែប្រសិនបើអ្វីៗផ្សេងទៀតត្រូវបានរៀបចំត្រឹមត្រូវ វាអាចចំណាយពេលដល់គេហទំព័រមួយចំនួនពីរពាន់ zanim អ្នកត្រូវការព្រួយបារម្ភអំពីវា។ សូម្បីតែនៅពេលនោះ ក៏មានដំណោះស្រាយដែលអាចបន្ថែមទៅជាលំដាប់បាន (ដូចជា solutions database sharding ជាឧទាហរណ៍)។
