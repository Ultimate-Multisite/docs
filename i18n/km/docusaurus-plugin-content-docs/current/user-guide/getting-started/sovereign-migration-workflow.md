# ដំណើរការផ្លាស់ប្តូរទៅជា Sovereign {#sovereign-migration-workflow}

Ultimate Multisite: Multi-Tenancy 1.2.0 បានបន្ថែមច្រកត្រួតពិនិត្យការផ្លាស់ប្តូរ (migration verification gates) ដើម្បីធ្វើការផ្លាស់ប្តូរ Subsite ស្តង់ដារទៅក្នុងហេដ្ឋារចនាសម្ព័ន្ធ tenant sovereign។

## មុនពេលចាប់ផ្តើម {#before-you-start}

សូមបញ្ជាក់ថាអ្នកមាន៖

- ការสำรองទិន្នន័យ (backup) បច្ចុប្បន្ននៃគេហទំព័រដើម (source site)។
- Addon Multi-Tenancy កំពុងដំណើរការ។
- មូលដ្ឋានទិន្នន័យ tenant គោលដៅ (destination tenant database), ប្រព័ន្ធឯកសារ (filesystem root), និងដែន (domain) រួចរាល់។
- ការភ្ជាប់ host នៃមូលដ្ឋានទិន្នន័យដែលត្រូវនឹងបរិយាកាសគោលដៅ។
- ការចូលប្រើដើម្បីដំណើរការ WP-CLI commands សម្រាប់បណ្តាញ (network)។

## ដំណើរការដែលបានណែនាំ {#recommended-workflow}

1. រៀបចំមូលដ្ឋានទិន្នន័យ tenant និងប្រព័ន្ធឯកសារគោលដៅ។
2. ចុះឈ្មោះ ឬធ្វើបច្ចុប្បន្នលើការកំណត់ការបំបែក tenant (tenant isolation settings)។
3. ដំណើរការការផ្លាស់ប្តូរ tenant (tenant migration)។
4. បង្កើត (provision) ឬផ្ទៀងផ្ទាត់អ្នកប្រើប្រាស់ tenant។
5. បណ្តេញ (drain) ប្រតិបត្តិការ migrration ដែលមិនទាន់បានបញ្ចប់ (async migration jobs)។
6. ដំណើរការការត្រួតពិនិត្យការផ្លាស់ប្តូរ sovereign (sovereign migration verification)។
7. ចូលទៅកាន់ tenant ដោយប្រើ SSO។
8. ប្តូរ DNS ឬ routing នៅពេលដែលការត្រួតពិនិត្យបានឆ្លងកាត់ជោគជ័យប៉ុណ្ណោះ។

## ច្រកត្រួតពិនិត្យ (Verification gates) {#verification-gates}

ដំណើរការត្រួតពិនិត្យនេះនឹងពិនិត្យការផ្លាស់ប្តូរពីមុំជាច្រើន៖

- ស្គ្រៀន (schema) tenant មានវត្តមាន និងអាចត្រូវបានសរសេរដោយអ្នកសរសេរមូលដ្ឋានទិន្នន័យ tenant បាន។
- Host មូលដ្ឋានទិន្នន័យដែលបានកំណត់មានការអនុញ្ញាត (grants) ដែលត្រឹមត្រូវ។
- អ្នកប្រើប្រាស់ tenant មានវត្តមាន និងត្រូវនឹងចំនួនអ្នកប្រើប្រាស់ដែលរំពឹងទុកសម្រាប់ការដំឡើង sovereign ។
- កម្មវិធី push មិនទាន់បានបញ្ចប់ (async push queue) អាចបណ្តេញចេញដោយជោគជ័យ។
- បណ្តាញទិន្នន័យចាស់ៗនៅខាងក្រៅ (legacy network-side data paths) មិនត្រូវបានទាមទារទៀតទេសម្រាប់ tenant sovereign ។

សូមចាត់ទុកការបរាជ័យនៃការត្រួតពិនិត្យថាជាឧបសគ្គមុនពេលដាក់ឱ្យដំណើរការ។ ប្រសិនបើមានបញ្ហាដែលបានរាយការណ៍អំពីមូលដ្ឋានទិន្នន័យ អ្នកប្រើប្រាស់ ឬកម្មវិធី push សូមជួសជុលវា រួចដំណើរការការត្រួតពិនិត្យម្តងទៀត មុននឹងបង្ហាញ tenant ទៅកាន់អតិថិជន។

## ការចូលប្រើលើកដំបូងក្នុងផលិតកម្ម (First production visit) {#first-production-visit}

បន្ទាប់ពីការត្រួតពិនិត្យបានឆ្លងកាត់ សូមប្រើ **Visit (SSO)** ពីអេក្រង់គ្រប់គ្រងគេហទំព័រ (site management screen) ដើម្បីចូលប្រើជាអ្នកគ្រប់គ្រងលើកដំបូង។ ការធ្វើបែបនេះបញ្ជាក់អំពីការកំណត់เส้นทาง tenant (tenant routing), ការគ្រប់គ្រង token SSO, ការកំណត់ប្រភពដើម (origin pinning), និងការផ្តល់សិទ្ធិនៃអ្នកប្រើប្រាស់នៅខាង tenant ក្នុងជំហានដែលបានគ្រប់គ្រងតែមួយ។
