# Multi-Tenancy

Distribuisci i sottositi tra server e database per prestazioni e sicurezza

## Punti salienti della versione 1.2.0 {#version-120-release-highlights}

Multi-Tenancy 1.2.0 aggiunge la base per tenant sovrani per i flussi di lavoro di isolamento in produzione:

- Autologin tenant stateless con visite SSO di breve durata e protette dalla ripetizione.
- Isolamento del database e del filesystem per sottosito per tenant sovrani.
- Funzionalità Hostinger multi-tenancy per il provisioning di tenant consapevole dell'host.
- Gate di verifica della migrazione sovrana per controlli sui dati legacy ed elaborazione push asincrona.
- Associazione configurabile dell'host database sulla stessa macchina per installazioni Bedrock, FrankenPHP e localhost.
- Miglioramenti al bootstrap del tenant, al writer del database, al bootstrap dello schema e al provisioning degli utenti.
- Teardown sovrano più sicuro che pulisce le credenziali del tenant.

Consulta le guide utente per [flusso di lavoro della migrazione sovrana](/user-guide/getting-started/sovereign-migration-workflow), [isolamento multi-tenancy](/user-guide/configuration/multi-tenancy-isolation) e [operazioni sui tenant sovrani](/user-guide/administration/sovereign-tenant-operations).

## Requisiti {#requirements}

- WordPress 5.3 o superiore
- PHP 7.4 o superiore
- Plugin Multisite Ultimate (attivo)

## Installazione {#installation}

1. Carica i file dell'addon nella directory `/wp-content/plugins/`
2. Attiva il plugin tramite il menu 'Plugin' in WordPress
3. Configura le impostazioni dell'addon nell'amministrazione di rete

## Configurazione {#configuration}

L'addon supporta varie opzioni di configurazione accessibili tramite l'interfaccia di amministrazione di rete.

### Configurazione del tenant sovrano {#sovereign-tenant-configuration}

Per i tenant sovrani, configura il database del tenant, la radice del filesystem, l'associazione dell'host database e la funzionalità dell'host prima di instradare il traffico verso il tenant. Esegui la verifica della migrazione dopo ogni modifica che influisce sulle credenziali del database, sull'associazione dell'host, sul routing o sul provisioning degli utenti.

Usa `localhost` solo quando il runtime del tenant e i grant del database prevedono accesso tramite socket locale o sulla stessa macchina. Usa `127.0.0.1` o un hostname di servizio quando quello è l'host di grant configurato dal tuo provider.

### Funzionalità Hostinger {#hostinger-capability}

La funzionalità Hostinger multi-tenancy funziona con il token di integrazione Hostinger condiviso. Conferma che il dominio sia presente in hPanel, che il DNS punti all'Account Hostinger e che le impostazioni del database del tenant corrispondano a hPanel prima della verifica.
