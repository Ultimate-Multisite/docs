# ملٹی ٹیننسی انضمام {#multi-tenancy-integration}

Ultimate Multisite: ملٹی ٹیننسی 1.2.0 خود مختار ٹیننٹس، مائیگریشن توثیق، اور ٹیننٹ لائف سائیکل خودکاری کے لیے کئی انضمامی ٹچ پوائنٹس تبدیل کرتا ہے۔

## ٹیننٹ بوٹسٹریپ فلو {#tenant-bootstrap-flow}

وہ انضمامات جو ٹیننٹس بناتے یا تبدیل کرتے ہیں، انہیں اس ترتیب پر عمل کرنا چاہیے:

1. ٹیننٹ رجسٹری ریکارڈ اور آئسولیشن ماڈل حل کریں۔
2. ٹیننٹ ڈیٹابیس رائٹر بنائیں یا اس کی توثیق کریں۔
3. ٹیننٹ اسکیما بوٹسٹریپ کریں۔
4. ٹیننٹ صارفین فراہم کریں۔
5. ٹیننٹ روٹنگ اور فائل سسٹم راستے رجسٹر کریں۔
6. ٹیننٹ کو ظاہر کرنے سے پہلے مائیگریشن توثیق چلائیں۔

یہ فرض نہ کریں کہ خود مختار ٹیننٹ نیٹ ورک ڈیٹابیس کنکشن دوبارہ استعمال کر سکتا ہے۔ ایڈ آن کی فراہم کردہ ٹیننٹ رجسٹری اور رائٹر تجریدات استعمال کریں۔

## SSO اور REST ہکس {#sso-and-rest-hooks}

اسٹیٹ لیس ٹیننٹ آٹولاگ اِن مختصر مدت کے ٹوکنز استعمال کرتا ہے جن میں purpose claim، JTI ری پلے تحفظ، ایک اختتامی حد، اور اصل کی پننگ شامل ہوتی ہے۔ وہ انضمامات جو لاگ اِن بٹن یا ریموٹ مینجمنٹ لنکس شامل کرتے ہیں، انہیں ٹیننٹ لاگ اِن URLs براہِ راست بنانے کے بجائے معاونت شدہ SSO فلو کے ذریعے ٹیننٹ وزٹس جنریٹ کرنے چاہئیں۔

نیٹ ورک سائیڈ API آڈٹ ایونٹس اور روزانہ خلاصے خود مختار ٹیننٹ گیٹ ویز کے لیے دستیاب ہیں۔ بیرونی سسٹمز کی ڈیبگنگ کرتے وقت جو ٹیننٹ لائف سائیکل endpoints کو کال کرتے ہیں، یہ لاگز استعمال کریں۔

## خود مختار کسٹمر ایکشن URLs {#sovereign-customer-action-urls}

Ultimate Multisite v2.13.0 خود مختار ٹیننٹ کسٹمر ایکشنز کو Account، checkout، billing، invoice، سائٹ، ٹیمپلیٹ سوئچنگ، اور ڈومین میپنگ فلوز کے لیے واپس مین سائٹ پر روٹ کرتا ہے۔ وہ انضمامات جو ٹیننٹ سائیڈ مینجمنٹ لنکس رینڈر کرتے ہیں، انہیں ان ایکشنز کو مین سائٹ کسٹمر پینل کی طرف اشارہ کرنا چاہیے اور جب صارف کو ایکشن مکمل کرنے کے بعد ٹیننٹ پر واپس جانے کے قابل ہونا چاہیے تو ایک توثیق شدہ واپسی ہدف شامل کرنا چاہیے۔

کراس ڈومین مینجمنٹ لنکس کے لیے کور SSO ریپر استعمال کریں:

```php
$url = wu_with_sso($main_site_customer_url);
```

جنریٹ شدہ URL `wu_sso_url` کے ذریعے فلٹر کے قابل رہتا ہے، جو SSO URL، موجودہ صارف، ہدف سائٹ ID، اور ری ڈائریکٹ سیاق و سباق وصول کرتا ہے۔ ایڈ آنز اس فلٹر کو پرووائیڈر مخصوص سیاق و سباق شامل کرنے یا Ultimate Multisite کی ٹوکن توثیق برقرار رکھتے ہوئے بروکر URL تبدیل کرنے کے لیے استعمال کر سکتے ہیں۔

خود مختار ٹیننٹ کے اندر membership، invoice، billing-address، template، یا domain-management حالت کی نقل نہ بنائیں۔ ٹیننٹ ڈیش بورڈ کو لانچر اور مین سائٹ کسٹمر پینل کو managed actions کے لیے system of record سمجھیں۔

## مائیگریشن توثیق {#migration-verification}

مائیگریشن یا لائف سائیکل انضمام ٹیننٹ ڈیٹا تبدیل کرنے کے بعد، توثیقی گیٹس چلائیں:

- `wp tenant verify-no-legacy --site=<site-id>` تصدیق کرتا ہے کہ ٹیننٹ اب legacy نیٹ ورک سائیڈ راستوں پر منحصر نہیں ہے۔
- `wp tenant verify-sovereign-push --site=<site-id>` تصدیق کرتا ہے کہ خود مختار push jobs پراسیس اور drain کر سکتے ہیں۔

انضمامات کو ناکام توثیق کو deployment blocker سمجھنا چاہیے اور ناکامی حل ہونے تک ٹیننٹ کو live نشان زد کرنے سے گریز کرنا چاہیے۔

## ٹیننٹ حذف کرنا {#tenant-deletion}

حذف کرنے کے فلوز کو ایڈ آن teardown راستہ کال کرنا چاہیے تاکہ ٹیننٹ ڈیٹابیس credentials صاف ہو جائیں۔ بیرونی انضمامات teardown کامیاب ہونے کے بعد پرووائیڈر resources ہٹا سکتے ہیں، لیکن جب تک verification یا async push jobs چل رہے ہوں، host databases یا folders حذف نہیں کرنے چاہئیں۔

## فرسودہ ڈیٹابیس راؤٹر {#deprecated-database-router}

legacy `Database_Router` کو deprecation stub سے بدل دیا گیا ہے۔ نئے انضمامات کو پرانی راؤٹر class پر انحصار کرنے کے بجائے موجودہ سائٹ راؤٹر اور ٹیننٹ رجسٹری APIs کے ذریعے ٹیننٹس حل کرنے چاہئیں۔
